# 5.2 推理优化：让每一毫秒都有用

## 不只关心步数

上一节讨论的蒸馏是在算法层面减少"需要多少步"。这一节讨论的是工程层面——在给定的步数下，**每一步跑多快、吃多少显存**。蒸馏让你从 30 步到 4 步（可能损失质量），推理优化让你同样 30 步，但每一步从 100ms 到 30ms（不损失质量）。两者正交且互补。

Stable Diffusion 的推理瓶颈集中在两个模块上：U-Net 中的注意力计算和 VAE 解码器的上采样。前者占了总推理时间的约 60% 到 70%，后者约占 20% 到 25%。优化这两个模块是推理工程的主要战场。

## FlashAttention 和 xFormers：让注意力不再 O(n²)

标准的多头注意力计算中，中间产物——$QK^T$ 得到的注意力分数矩阵——尺寸为 $N \times N$，其中 $N$ 是序列长度（特征图展平后的空间位置数）。在 U-Net 的 64×64 分辨率层，$N = 4096$，这个矩阵虽然大（4096×4096 = 16.7M 个标量），但还算可以承受。问题在于这个矩阵的完整形式必须被存储在 HBM（GPU 显存）中才能进行后续的 softmax 和乘以 V 的操作——虽然计算量不是特别大，但 HBM 的读写带宽是瓶颈。在更高分辨率的推理（如 1024×1024）时，$N$ 增大到 16384，注意力分数矩阵达到约 268M 个元素、约 1GB（fp16），读写这 1GB 的数据就成为了一个严重的瓶颈。

FlashAttention 的核心思想是"**算力换带宽**"。它不是一次性把整个 $QK^T$ 矩阵算出来再存到 HBM 再读回来做 softmax，而是把矩阵分块（tiling）：每次只加载 Q 和 K 的一个小块到 SRAM（GPU 的片上缓存，带宽远比 HBM 高），在小块上完成局部的矩阵乘法和 softmax，然后直接在 SRAM 上乘 V 得到输出的一小块，写回 HBM。整个过程中，完整的 $N \times N$ 注意力分数矩阵从未被完整地存储在 HBM 中——它只以小块的形式在 SRAM 中短暂地存在，存在时间极短。FlashAttention 在数学上和标准注意力完全等价（它通过在线 softmax 算法保证了分块计算和完整计算在数值上精确等价），而不是像剪枝或量化那样做近似。

对于 SD 1.5 在 512×512 上的推理，FlashAttention 带来的速度提升约在 20%–30%；在 1024×1024 上（SDXL 的常用分辨率），提升可以超过 40%，因为注意力矩阵的存储瓶颈越往后越大。

xFormers 是 Meta 开发的一个通用注意力优化库，其 `memory_efficient_attention` 实现和 FlashAttention 思想类似——通过分块计算降低显存占用。在 Diffusers 中启用 xFormers 只需一行代码 `pipe.enable_xformers_memory_efficient_attention()`，它会自动将所有注意力层替换为 xFormers 的优化版本。xFormers 的另一个优势是它对各种 GPU 架构都有良好的兼容性——FlashAttention 对 Ampere 以上的架构最优（利用了 Tensor Cores 的特定指令），xFormers 则在 Turing 和更老的架构上也有不错的加速效果。

## Token Merging：少算一些 token

Token Merging（ToMe）来自一个直觉性的观察：在 U-Net 的中间层，很多相邻空间位置的特征向量高度相似——它们的注意力模式几乎相同，去噪的方向也几乎一致——合并它们不会丢失有意义的视觉信息。ToMe 在 U-Net 的每一层自注意力前，将最相似的特征向量对合并为一个（通过简单的特征平分或加权平均），减少后续注意力计算的序列长度。合并是自适应的——在粗糙层（低分辨率）合并更多（因为这里的 token 本来就更冗余），在精细层（高分辨率）合并更少（细节需要更高的空间分辨率）。

ToMe 在 SD 推理中的应用几乎不需要任何训练——它是一个纯推理时的后处理操作，直接插入在 U-Net 的前向过程中。它可以和 FlashAttention 叠加使用：ToMe 减少序列长度，FlashAttention 加速剩余序列的注意力计算。两者叠加可以让同一张 512×512 图像的生成时间相比未优化版本降低约 40%–50%。

## VAE Tiling：大图的 VAE 瓶颈

VAE 解码器是推理的另一个瓶颈，而且它对分辨率的敏感度比 U-Net 更高。VAE 解码器的输入是一个 $h \times w \times 4$ 的潜变量，经过一系列卷积和上采样后输出 $H \times W \times 3$ 的 RGB 图像。在 1024×1024 的分辨率下，VAE 解码器中间层的激活张量可能达到 $1024 \times 1024 \times 256$——约 2.68 亿个浮点数，单层就需要约 500MB 显存（fp16）。如果 U-Net 还在显存中，两者同时存在的峰值显存可以轻松超过 24GB。

VAE Tiling 通过将潜变量在空间上切成多个重叠的瓦片（tiles），逐一解码每个瓦片，然后将瓦片拼回完整图像。重叠区域用于消除瓦片边缘的不连续性——相邻瓦片之间有一定像素的重叠带，在这些带中做线性淡入淡出混合。解码完一片瓦片后，它的中间激活可以释放，只保留解码后的像素结果——解码下一片时显存被复用。这让一张原本需要 32GB 以上显存的 4096×4096 图像可以在 8GB 的卡上完成生成。

在 Diffusers 中，`pipe.enable_vae_tiling()` 可以开启 VAE Tiling，`pipe.enable_vae_slicing()` 是另一个相关功能，它把 batch 中的多张图的 VAE 解码切片化处理——每次只解码 batch 中的一张，减少 batch 维度上的峰值显存。

## FP16 和半精度推理

Stable Diffusion 的 U-Net 和 VAE 在 FP32 下训练，但推理时改用 FP16（半精度）几乎不会产生可感知的画质损失，却能带来近 2 倍的吞吐量提升和一半的显存占用。这背后的原理是：扩散模型的去噪轨迹在潜空间中相对平滑，网络在每一步的输入-输出 mapping 对精度不太敏感——噪声预测的误差在 FP16 精度下远小于潜空间中不同的采样噪声之间的差异。

在 Diffusers 中，`pipe = StableDiffusionPipeline.from_pretrained(model_id, torch_dtype=torch.float16)` 指定加载到 FP16，`pipe.to("cuda")` 将模型移到 GPU。FP16 推理时需要注意一些数值稳定的细节：VAE 解码器在某些像素上的浮点值可能超出 FP16 的表示范围（-65504 到 65504），需要偶尔做 clamp 或 scale 处理；注意力层中的 softmax 计算在 FP16 下可能因极小的分母产生 NaN 梯度，FlashAttention 和 xFormers 内部处理了这些细节。

BF16（bfloat16）是另一个选择。和 FP16 相比，BF16 的指数范围更大（和 FP32 一样有 8 位指数），但尾数精度更低（只有 7 位 vs FP16 的 10 位）。更大的指数范围意味着不会出现 overflow，对 VAE 解码器更友好。但 BF16 需要 Ampere 以上架构的 GPU（A100、RTX 3090、4090 等）才有硬件加速支持——在 Turing 或更老的 GPU 上 BF16 会退化为软模拟且比 FP16 慢。

## TensorRT 和模型编译

TensorRT 和 ONNX Runtime 走的是一种更底层的加速路线——它们将 PyTorch 模型的计算图编译为高度优化的 GPU 内核，消除了 Python 解释器的开销和 PyTorch 框架的动态图构建开销。

编译的过程通常包括：算子融合（将连续的卷积 / batch norm / 激活函数合并为一个 GPU kernel，减少 kernel launch 开销和 HBM 读写次数）；内存规划（预先为整个计算图的所有中间张量分配合并的显存块，消除显存碎片和运行时分配开销）；精度校准（在 FP16 和 INT8 之间选择每层的最优精度，用少量校准数据测出每层的精度对生成的敏感度）。对于 SD 1.5，TensorRT 编译后的推理速度通常比未优化的 PyTorch FP16 快 30% 到 70%，具体取决于 GPU 和 batch size。

TensorRT 的主要代价是部署复杂度。模型需要先导出为 ONNX，再通过 TensorRT parser 编译为 `.trt` engine——这个过程对环境（CUDA 版本、TensorRT 版本、GPU 架构）敏感，一个在一台机器上编译好的 engine 通常不能在另一台有不同 CUDA 版本或不同 GPU 的机器上使用。此外，TensorRT 对动态形状（变分辨率、变 batch size）的支持比 PyTorch 复杂得多，需要预先为每个可能的输入形状编译单独的 engine 或使用较慢的动态形状模式。这使得它在需要灵活性的场景下不如原生 PyTorch 方便，但在部署固定的推理管线时是标准优化选择。

## 显存管理：让大模型活在小的卡上

对于显存有限的 GPU，有几个额外的显存优化技术可以叠加使用。

**模型卸载**（Model Offloading）将 U-Net 的三个组件在 GPU 和 CPU 内存之间按需迁移。VAE 只在编码（训练）和解码（最后一步）时被用到，CLIP 文本编码器只在第一步计算文本嵌入时被用到，U-Net 在每一步去噪中都会被用到。模型卸载策略是：CLIP 编码文本后将文本嵌入留在 GPU，CLIP 本体移回 CPU；VAE 只在需要编码或解码时临时加载到 GPU；U-Net 常驻 GPU（因为它每一步都跑）。`pipe.enable_model_cpu_offload()` 在 Diffusers 中实现了这个策略，可以让 SDXL（约 7GB 的完整模型）在 8GB 以下的 GPU 上运行。

**注意力切片**（Attention Slicing）将注意力计算中的 batch 维度切成多个子 batch——每次只处理总序列长度的一小部分，多个步骤串行完成——减少注意力计算的单次显存峰值。这比 FlashAttention 更极端——FlashAttention 降低的是注意力分数矩阵的显存占用，注意力切片降低的是 Q、K、V 和输出在整个 attention 计算管线中的所有中间张量的累积显存占用。代价是速度（串行化加长 runtime），但在显存极度紧张的场景下是救命稻草。`pipe.enable_attention_slicing()` 是它的开关。

## 优化组合建议

这些技术不是互斥的——它们在不同层次上发挥作用，可以叠加：

| 优化组合 | 适用场景 | 单图 512×512 生成时间（参考） |
|----------|---------|------------------------------|
| 基准（FP16 + xFormers） | 所有场景的默认起点 | ~1.5s (20 steps) |
| + ToMe | 显存紧张或追求极限速度 | ~1.0s |
| + TensorRT | 生产部署、固定管线 | ~0.7s |
| + LCM/蒸馏模型 | 实时交互 | ~0.1–0.3s (4 steps) |
| 仅显存优化（offload + slicing） | < 8GB 显卡 | ~5–10s（更慢但能跑） |

速度数字粗略参考 A10/RTX 3090 级别 GPU，SD 1.5, 512×512, 20 steps DPM-Solver。不同硬件、分辨率、软件版本下数值会有波动。
