# 2.4 CLIP 与条件控制

## 语言与像素之间的桥梁

U-Net 能做去噪，但它本身并不理解自然语言。你需要一种方式，把"a cat sitting on a sofa in a sunlit room"这段文本翻译成 U-Net 能消化的信号——一个高维向量序列，编码了文本中的语义和句法信息，并且和图像内容在同一个语义空间中"对齐"。这就是 CLIP 做的事情。

CLIP（Contrastive Language-Image Pre-training）是 OpenAI 在 2021 年发布的一个双编码器模型。它的核心思想是用**对比学习**训练一个图像编码器和一个文本编码器，使得配对的图像-文本对在嵌入空间中靠近，而不配对的则被推远。训练数据是 4 亿个来自互联网的图像-文本对——这个规模是 CLIP 强大泛化能力的基础。

## 对比学习的智慧

CLIP 的训练不需要手工标注的类别标签，只需要图像和它们的自然语言描述。对于一个包含 $N$ 个图像-文本对的 batch，CLIP 训练两个编码器：一个将图像映射为嵌入向量，一个将文本映射为同样维度的嵌入向量。然后计算所有 $N \times N$ 对配对之间的余弦相似度，形成一个相似度矩阵。

训练目标是让对角线（正确配对）上的相似度尽可能高，其余位置尽可能低——这是一个对称的交叉熵损失：

$$\mathcal{L}_{\text{CLIP}} = -\frac{1}{2} \sum_{i=1}^{N} \left[ \log \frac{\exp(s_{ii}/\tau)}{\sum_{j} \exp(s_{ij}/\tau)} + \log \frac{\exp(s_{ii}/\tau)}{\sum_{j} \exp(s_{ji}/\tau)} \right]$$

其中 $s_{ij}$ 是第 $i$ 张图像和第 $j$ 段文本的余弦相似度，$\tau$ 是可学习的温度参数。第一项是"给定图像，选对文本"，第二项是"给定文本，选对图像"——两者同时优化。

这个训练目标的精妙之处在于，它不要求模型理解每一个像素和每一个单词之间的精确对应关系，它只需要模型学会"哪段文字描述和哪张图是同一个东西"。但正是这种粗粒度的对比目标，奇迹般地催生了细粒度的语义对齐能力——因为你没法仅靠整体色调或大致物体类别就在 4 亿对中精准匹配，模型被迫学会理解物体属性、空间关系、甚至风格和情感。

## CLIP 文本编码器：只看文本就够了

Stable Diffusion 只使用了 CLIP 的文本编码器，图像编码器被丢弃——VAE 已经提供了图像端的表示，而 CLIP 图像编码器的输出维度（通常 768 或 1024 维）远不如 VAE 潜变量（64×64×4 = 16,384 维）保留的信息丰富。所以 CLIP 在 Stable Diffusion 中专注于它最擅长的事：把文字变成富含语义的向量。

CLIP 的文本编码器是一个 Transformer 编码器，类似于 GPT 的文本处理部分但加上了因果掩码的变体。SD 1.x 使用 CLIP ViT-L/14 的文本编码器（12 层 Transformer，隐藏维度 768），SD 2.x 使用 OpenCLIP ViT-H 的文本编码器（更大，隐藏维度 1024）。

一段文本的旅程如下：首先通过 tokenizer 分解为 token——"a cat sitting on a sofa" 变成类似 `[<|startoftext|>, a, cat, sitting, on, a, sofa, <|endoftext|>]` 的序列，填充或截断到 77 个 token。每个 token 映射为一个嵌入向量，加上位置编码后送入 12 层 Transformer。最终输出是一个 $L \times d$ 的矩阵，其中 $L=77$，$d=768$——每个 token 位置都有一个上下文相关的嵌入向量。

这个 $77 \times 768$ 的矩阵就是注入 U-Net 的文本条件。在 Diffusers 中，它通常被称为 `encoder_hidden_states`。

## 77 个 token 够用吗

77 个 token 是 CLIP 文本编码器的硬限制——这是训练时就固定下来的上下文窗口大小。对于绝大多数提示词，77 足够了："a cat sitting on a sofa in a sunlit room" 只有 11 个 token。但对于长提示词、复杂场景描述或多主体交互，77 可能不够。超出 77 个 token 的输入会被截断，后面部分直接丢弃。

社区有一些 hack 来解决这个问题，例如把长提示词切分成多段，每段独立编码后再在 token 维度上拼接，但这样做会丢失段间的上下文依赖关系。SDXL 为了解决这个问题，同时用了两个文本编码器（CLIP ViT-L 和 OpenCLIP ViT-G），两者互补，增强了文本理解的整体容量。但截断问题在根本上并没有消失——它仍然是当前文本到图像生成的一个结构性瓶颈。

## 交叉注意力：文本如何"指挥"U-Net

在 2.3 节简略提了交叉注意力的概念，这里展开其细节。交叉注意力是 Stable Diffusion 中文本信号注入 U-Net 的唯一途径。如果把它删掉，U-Net 就只能在没有任何语言引导的情况下做无条件去噪——产出的图像和提示词没有任何关系。

交叉注意力的 Query 来自 U-Net 中的空间特征图（展平为 $(h \cdot w) \times c_{\text{latent}}$），Key 和 Value 来自 CLIP 文本嵌入（$77 \times 768$，经过了线性投影到相同的注意力维度）。和自注意力不同，交叉注意力中的 K 和 V 在整个去噪过程中是固定的——文本嵌入只计算一次，然后被 U-Net 各个层级的交叉注意力模块反复查询。Query 在每一步去噪中都会更新，因为它来自不断变化的潜变量 $z_t$。

在实现层面，交叉注意力紧跟在自注意力之后——先让图像空间位置之间互相交流（自注意力），再让每个位置去"查阅"文本语义（交叉注意力）：

$$h_{\text{spatial}} \leftarrow h_{\text{spatial}} + \text{SelfAttn}(h_{\text{spatial}})$$

$$h_{\text{spatial}} \leftarrow h_{\text{spatial}} + \text{CrossAttn}(h_{\text{spatial}}, \tau_\theta(y))$$

交叉注意力的位置遍布 U-Net 的各个分辨率层级——在 64×64、32×32、16×16、8×8 上都有。这让文本同时从全局构图和局部细节两个尺度上影响生成。8×8 层的交叉注意力主要影响"这件东西应该放在画面的大致哪个位置"，64×64 层的交叉注意力则影响"这块区域的纹理是毛皮还是木板"。

## 提示词工程的正名

理解了 CLIP 和交叉注意力的工作机制后，"提示词工程"就不再是玄学了。它本质上是关于如何用有限的语言表达，向一个在 4 亿对图文上训练过的文本编码器传递最精确的语义信号。

CLIP 不是大语言模型，它对复杂语法结构的理解有限。它见过大量"a photo of X"、"X in the style of Y"这类模板化的描述（训练数据中的自然语言标题往往就是这种形式），所以简单直接的句法往往比复杂修辞更有效。形容词的位置、标点的使用、单词的顺序都会微妙地影响嵌入向量的方向，进而通过交叉注意力影响图像输出。

这也解释了为什么不同模型（SD 1.5 vs SDXL vs FLUX）对同一个提示词的响应不同——它们使用了不同的文本编码器，即使相同的文本，编码出的嵌入向量也不同，交叉注意力的投影矩阵也不同，最终生成的图像取向自然不同。
