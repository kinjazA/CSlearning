# MySQL 索引学习笔记

---

## 一、索引是什么？为什么需要它？

索引是数据库中一种**用空间换时间**的数据结构，本质类似书籍的目录——不用逐页翻阅，直接定位到目标位置

**核心价值：** 减少磁盘 I/O 次数，将全表扫描变为精确定位

**代价：** 额外的存储空间 + 写操作变慢（每次 INSERT/UPDATE/DELETE 都需要同步维护索引结构）

**建索引的基本判断：**

| 适合建索引 | 不适合建索引 |
|---|---|
| 高频出现在 WHERE / ORDER BY / JOIN 中的字段 | 数据量极小的表（全表扫描更快） |
| 区分度高的字段（如用户ID、手机号） | 区分度极低的字段（如性别、状态位） |
| 读多写少的业务场景 | 写操作远多于读的场景 |

---

## 二、索引分类全景图

### 按功能划分

| 类型 | 特点 | 典型场景 |
|---|---|---|
| **主键索引** | 唯一、非空、每表仅一个 | 用户ID、订单ID |
| **唯一索引** | 值唯一，允许一个 NULL | 手机号、邮箱 |
| **普通索引** | 无约束，允许重复和 NULL | 普通查询过滤字段 |
| **联合索引** | 多列组合，遵循最左前缀 | 多条件组合查询 |
| **全文索引** | 分词匹配，用于文本搜索 | 文章内容搜索（较少使用） |

### 按存储结构划分（InnoDB 核心概念）

- **聚簇索引（Clustered Index）：** 主键索引。叶子节点直接存储**完整行数据**，数据按主键顺序物理存储。**每张表有且仅有一个**
- **二级索引（Secondary Index）：** 非主键索引。叶子节点存储的是**索引列的值 + 主键值**，不含完整行数据

> 理解这两者的区别是掌握回表、覆盖索引等后续概念的基础

## 三、B+ 树——MySQL 索引的底层引擎

### 为什么是 B+ 树？

数据库索引的核心矛盾：数据在磁盘上，而磁盘 I/O 极慢。好的索引结构必须**尽可能少地访问磁盘**

各数据结构的对比：

| 结构 | 关键缺陷 | 磁盘友好度 |
|---|---|---|
| 二叉搜索树/红黑树 | 树高 \($O(\log_2 n)$\)，百万数据约 20 层，即 20 次磁盘 I/O | ✗ |
| 哈希表 | 等值查询 \($O(1)$\)，但**完全不支持范围查询和排序** | 部分 |
| B 树 | 多路降低树高，但数据分散在所有节点，范围查询需要中序遍历 | 较好 |
| **B+ 树** | 多路 + 数据只在叶子节点 + 叶子节点形成有序链表 | **最优** |

### B+ 树核心特征

```
          [30 | 60]                    ← 非叶子节点：只存键值，用于导航
         /    |    \
    [10|20] [40|50] [70|80]            ← 非叶子节点
     / | \   / | \   / | \
   [数据] [数据] [数据] ...            ← 叶子节点：存储实际数据
    ←→     ←→     ←→                   ← 叶子节点之间通过双向链表相连
```

1. **非叶子节点只存索引键**，不存数据 → 单个节点能容纳更多键 → 树更矮（通常 3～4 层即可覆盖千万级数据）
2. **所有数据都在叶子节点** → 每次查询路径长度一致，性能稳定
3. **叶子节点形成有序双向链表** → 范围查询和排序只需沿链表顺序扫描，极其高效

### 一个直觉估算

InnoDB 默认页大小 16KB，假设主键为 bigint（8字节）+ 指针（6字节），单个非叶子节点可存约 \($16KB / 14B \approx 1170$\) 个指针。三层 B+ 树理论上可索引 \($1170 \times 1170 \times 16 \approx 2000万$\) 行数据，仅需 **3 次磁盘 I/O**。

## 四、InnoDB 索引核心机制

### 4.1 聚簇索引 vs 二级索引的查询路径

```css
查询：SELECT * FROM user WHERE name = '张三'
假设 name 上有二级索引

步骤 1：在 name 的 B+ 树中找到 '张三' → 获取对应主键值 id=42
步骤 2：拿 id=42 去聚簇索引（主键 B+ 树）中再查一次 → 获取完整行数据
       └── 这第二步就是「回表」
```

### 4.2 回表（Back to Table）

**定义：** 通过二级索引找到主键后，再到聚簇索引中查找完整行数据的过程

**性能影响：** 每一次回表都是一次额外的 B+ 树查找（额外的磁盘 I/O）。当查询命中大量行时，回表代价极高

### 4.3 覆盖索引（Covering Index）

**定义：** 查询所需的所有列都包含在索引中，无需回表

```sql
-- 假设有联合索引 idx(name, age)

SELECT name, age FROM user WHERE name = '张三';
-- ✅ 覆盖索引：name 和 age 都在索引中，直接返回，不回表

SELECT name, age, email FROM user WHERE name = '张三';
-- ❌ 需要回表：email 不在索引中
```

在 EXPLAIN 中，覆盖索引的标志是 Extra 列出现 **`Using index`**

**核心思路：** 尽量让高频查询走覆盖索引，是优化的重要手段

## 五、联合索引与最左前缀原则

### 5.1 最左前缀原则

联合索引 `(a, b, c)` 在 B+ 树中的排序方式：**先按 a 排序，a 相同再按 b 排序，b 相同再按 c 排序。**

这意味着只有从最左列开始连续匹配，索引才能被有效利用：

| 查询条件 | 是否命中索引 | 说明 |
|---|---|---|
| `WHERE a = 1 AND b = 2 AND c = 3` | ✅ 完全命中 | |
| `WHERE a = 1 AND b = 2` | ✅ 命中 a, b | |
| `WHERE a = 1` | ✅ 命中 a | |
| `WHERE b = 2 AND c = 3` | ❌ 未命中 | 缺少最左列 a |
| `WHERE a = 1 AND c = 3` | ⚠️ 仅命中 a | b 缺失导致 c 无法利用索引 |
| `WHERE a > 1 AND b = 2` | ⚠️ 仅命中 a | a 用了范围查询后，b 无法继续利用 |

### 5.2 列顺序设计原则

1. **区分度高的列放前面**（区分度 = 不同值数量 / 总行数）
2. **等值查询的列放在范围查询的列前面**（范围条件会"截断"后续列的索引利用）
3. **兼顾排序需求**：如果 WHERE 用 a 过滤，ORDER BY 用 b 排序，`(a, b)` 索引可同时优化两者

## 六、索引失效的常见陷阱

以下情况会导致索引无法被使用，查询退化为全表扫描：

### 1. 对索引列使用函数或表达式

```sql
-- ❌ 失效
WHERE YEAR(create_time) = 2025
WHERE id + 1 = 10

-- ✅ 改写
WHERE create_time >= '2025-01-01' AND create_time < '2026-01-01'
WHERE id = 9
```

### 2. 隐式类型转换

```sql
-- phone 字段是 VARCHAR 类型
WHERE phone = 13800001111    -- ❌ 数字与字符串比较，触发隐式转换，索引失效
WHERE phone = '13800001111'  -- ✅ 类型匹配
```

### 3. LIKE 以通配符开头

```sql
WHERE name LIKE '张%'   -- ✅ 前缀匹配，可用索引
WHERE name LIKE '%张%'  -- ❌ 全模糊匹配，索引失效
```

### 4. OR 条件中有非索引列

```sql
-- a 有索引，b 无索引
WHERE a = 1 OR b = 2   -- ❌ 整体无法走索引（除非 a、b 都有索引，可能走 index_merge）
```

### 5. 联合索引不满足最左前缀

见上一节的表格

### 6. 范围查询截断后续列

联合索引 `(a, b, c)` 中，`WHERE a = 1 AND b > 10 AND c = 3`，c 列无法被索引利用

## 七、EXPLAIN 执行计划速查

使用 `EXPLAIN SELECT ...` 查看 SQL 的执行计划，重点关注以下字段：

### 核心字段

| 字段 | 含义 | 关注点 |
|---|---|---|
| **type** | 访问类型（性能从好到差排列） | 目标：至少 `range` 以上 |
| **key** | 实际使用的索引 | NULL 表示未走索引 |
| **rows** | 预估扫描行数 | 越小越好 |
| **Extra** | 附加信息 | 关注是否有回表、文件排序等 |

### type 性能级别（从优到差）

| 级别 | 含义 | 示例 |
|---|---|---|
| `const` | 主键/唯一索引等值查询，最多一行 | `WHERE id = 1` |
| `eq_ref` | JOIN 中被驱动表用主键/唯一索引匹配 | 一对一关联 |
| `ref` | 非唯一索引等值查询 | `WHERE name = '张三'` |
| `range` | 索引范围扫描 | `WHERE id > 100` |
| `index` | 全索引扫描（遍历整棵索引树） | 覆盖索引但无过滤条件 |
| `ALL` | **全表扫描，最差** | 无索引可用 |

### Extra 关键信息

| 值 | 含义 | 好坏 |
|---|---|---|
| `Using index` | 覆盖索引，无需回表 | ✅ 好 |
| `Using where` | 在存储引擎返回数据后再过滤 | 中性 |
| `Using filesort` | 额外的排序操作，未利用索引排序 | ⚠️ 需优化 |
| `Using temporary` | 使用了临时表（常见于 GROUP BY） | ⚠️ 需优化 |

## 八、索引设计原则

### 实用规则清单

1. **高区分度 + 高频查询的字段优先建索引**
2. **联合索引优于多个单列索引**：一个 `(a, b, c)` 联合索引可以覆盖 a、(a,b)、(a,b,c) 三种查询
3. **控制数量**：单表索引建议不超过 5～6 个，过多索引严重拖慢写入
4. **优先考虑覆盖索引**：让高频查询的 SELECT 字段都在索引中
5. **等值列在前，范围列在后**

### 主键设计原则

| 原则 | 原因 |
|---|---|
| **尽量短** | 主键存在于每个二级索引的叶子节点中，主键越长，二级索引越大 |
| **尽量有序（推荐自增）** | 无序主键（如 UUID）导致频繁的页分裂，写入性能差 |
| **尽量稳定** | 主键变更会导致聚簇索引数据物理迁移+所有二级索引更新 |

## 九、索引与锁的关系

这是容易被忽略但在生产环境中至关重要的知识点：

- **InnoDB 的行锁是加在索引上的**，不是加在数据行上的
- 如果 UPDATE/DELETE 的 WHERE 条件没有命中索引，InnoDB **无法精确定位行**，会退化为对所有扫描到的记录加锁，甚至锁全表
- **间隙锁（Gap Lock）** 和 **临键锁（Next-Key Lock）** 也基于索引实现，用于防止幻读

**实际影响：** 在高并发写入场景下，缺少合适的索引不仅会让查询变慢，还会因锁范围扩大导致大量锁等待甚至死锁

## 十、InnoDB vs MyISAM 索引差异

| 特性 | InnoDB | MyISAM |
|---|---|---|
| 聚簇索引 | ✅ 主键索引即数据 | ❌ 索引和数据分离存储 |
| 二级索引叶子节点 | 存储主键值（回表走聚簇索引） | 存储数据的物理地址（直接定位） |
| 事务支持 | ✅ | ❌ |
| 行级锁 | ✅ | ❌（仅表锁） |

> 现代业务几乎都使用 InnoDB，因为它支持事务、行锁和崩溃恢复

## 十一、高频面试题速答

**Q：为什么 MySQL 用 B+ 树而不是红黑树/B 树/哈希？**
> 红黑树层级太深，磁盘 I/O 多；B 树数据散布在所有节点，范围查询需要中序遍历；哈希不支持范围查询和排序。B+ 树多路平衡降低树高、数据只在叶子节点且通过链表相连，最适合磁盘存储和范围查询

**Q：什么是回表？怎么避免？**
> 通过二级索引查到主键后，再去聚簇索引取完整数据的过程。通过设计覆盖索引避免

**Q：联合索引为什么要遵循最左前缀？**
> 因为 B+ 树按联合索引的列从左到右依次排序，跳过左边的列就无法利用有序性定位

**Q：主键为什么建议自增？**
> 自增主键保证新数据总是追加到 B+ 树末端，避免页分裂，写入更高效；且 bigint 类型仅占 8 字节，二级索引存储开销小

**Q：索引是不是越多越好？**
> 不是。每个索引都是一棵独立的 B+ 树，会占用存储空间，并且每次写操作都要同步维护所有索引，过多索引会严重影响写入性能

## 十二、实战优化思路

遇到慢查询时的排查步骤：

```css
1. 用 EXPLAIN 查看执行计划
   ↓
2. 检查 type 是否为 ALL（全表扫描）
   ↓
3. 检查 key 是否为 NULL（未使用索引）
   ↓
4. 检查 Extra 是否有 Using filesort / Using temporary
   ↓
5. 分析原因：
   - 索引不存在 → 创建合适索引
   - 索引存在但失效 → 排查失效原因（函数、类型转换、最左前缀等）
   - 需要回表 → 考虑覆盖索引
   - 深分页问题 → 改用延迟关联或基于游标的分页
   ↓
6. 优化后再次 EXPLAIN 确认效果
```

### 深分页优化示例

```sql
-- ❌ 慢：OFFSET 越大越慢（需要扫描并丢弃前 100000 行）
SELECT * FROM orders ORDER BY id LIMIT 100000, 20;

-- ✅ 延迟关联：先在索引上定位主键，再回表取少量数据
SELECT * FROM orders
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 20) AS t
ON orders.id = t.id;

-- ✅ 游标分页：记住上一页最后的 id
SELECT * FROM orders WHERE id > 上一页最大id ORDER BY id LIMIT 20;
```
