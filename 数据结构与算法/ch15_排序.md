## 1. 引言：为什么排序如此重要？

排序是计算机科学中**被研究最深**的问题之一。为什么？

1. **基础设施地位**：二分查找、去重、分组、Top-K、数据库索引、中位数……几乎所有高级操作都要求"先排序"。
2. **优化收益巨大**：把一个 \(O(n^2)\) 的排序升级到 \(O(n \log n)\)，在 \(n = 10^6\) 时大约能把 1 分钟压缩到 0.02 秒。
3. **思想集大成**：**分治、递归、堆、基数、空间换时间……** 几乎所有算法思想都能在排序家族里找到。

一句话：

> **学懂排序，就学懂了一半的算法思想。**

## 2. 排序算法的三大核心属性

在学具体算法之前，先掌握三个**评判维度**，后面每个算法都会对照这三项。

### 2.1 稳定性（Stability）

> **相等元素排序后相对顺序是否保持不变。**

```
原数组（按"分数"排序，保留姓名）:
  [("Alice", 90), ("Bob", 85), ("Carol", 90), ("Dave", 85)]

稳定排序结果（Alice 在 Carol 前，Bob 在 Dave 前）:
  [("Bob", 85), ("Dave", 85), ("Alice", 90), ("Carol", 90)]

不稳定排序可能是:
  [("Dave", 85), ("Bob", 85), ("Carol", 90), ("Alice", 90)]   ← 相对顺序乱了
```

**为什么在乎稳定性？** 工程中常做"**二次排序**"：先按部门排，再按工资排——如果第二次排序不稳定，同部门内同工资的员工顺序就会乱掉。

### 2.2 原地性（In-place）

> **是否只使用 \(O(1)\) 或 \(O(\log n)\) 的额外空间**（递归栈可以忽略）。

- 原地：选择、冒泡、插入、希尔、堆、快速（基本算原地，递归栈 \(O(\log n)\)）
- 非原地：归并（需要 \(O(n)\) 辅助数组）、计数、桶、基数

### 2.3 自适应性（Adaptive）

> **输入"接近有序"时能否更快。**

- 自适应：冒泡（加优化后）、插入、Tim Sort
- 非自适应：选择、归并、堆（不管数据长什么样都是固定复杂度）

### 2.4 对照表（先有个全局印象，后面逐个讲）

| 算法 | 平均时间 | 最坏时间 | 空间 | 稳定 | 原地 | 自适应 |
|------|---------|---------|------|------|------|--------|
| 选择排序 | \(O(n^2)\) | \(O(n^2)\) | \(O(1)\) | ❌ | ✅ | ❌ |
| 冒泡排序 | \(O(n^2)\) | \(O(n^2)\) | \(O(1)\) | ✅ | ✅ | ✅（优化版） |

| 快速排序 | \(O(n\log n)\) | \(O(n^2)\) | \(O(\log n)\) | ❌ | ✅ | ❌ |

## 3. 选择排序（Selection Sort）

### 3.1 核心思想

> **每轮从未排序区中选出最小值，放到已排序区的末尾。**

```
初始:  [4, 2, 5, 1, 3]
              ↓ 找最小 = 1，和位置 0 交换
第1轮: [1 | 2, 5, 4, 3]   ("|" 左边已排序)
              ↓ 在右侧找最小 = 2，已在位，不动
第2轮: [1, 2 | 5, 4, 3]
              ↓ 最小 = 3，和位置 2 交换
第3轮: [1, 2, 3 | 4, 5]
              ↓ 最小 = 4，不动
第4轮: [1, 2, 3, 4 | 5]
完成:  [1, 2, 3, 4, 5]
```

### 3.2 代码实现

```python
from typing import List


def selection_sort(nums: List[int]) -> None:
    """选择排序（原地、不稳定）。

    Args:
        nums: 待排序的整数列表，就地修改。

    Complexity:
        Time:  O(n^2) —— 无论输入如何，比较次数都是 n(n-1)/2，不自适应。
        Space: O(1)   —— 只需常数个临时变量。
    """
    n: int = len(nums)
    # 外层循环：i 表示"已排序区的右边界下一个位置"
    # 只需要排到 n-2，因为最后一个元素自动就位
    for i in range(n - 1):
        min_idx: int = i                            # 假设未排序区第一个元素就是最小值
        # 内层循环：在 [i+1, n) 区间找真正的最小值下标
        for j in range(i + 1, n):
            if nums[j] < nums[min_idx]:             # 发现更小的，更新最小下标
                min_idx = j
        # 一轮扫描完后，把真正最小值交换到位置 i
        # 即使 min_idx == i 也无妨，自己和自己交换没副作用（但略微浪费，可加 if 优化）
        if min_idx != i:                            # 小优化：避免无意义的交换
            nums[i], nums[min_idx] = nums[min_idx], nums[i]
```

### 3.3 为什么选择排序是**不稳定**的？

**反例**：`[5a, 5b, 2]`（5a 和 5b 值相等，用下标区分）

```
初始:  [5a, 5b, 2]
第1轮: 扫描发现最小值是 2（下标 2），与位置 0 的 5a 交换
       [2, 5b, 5a]    ← 5a 和 5b 的相对顺序已经反了！
```

**关键原因**：选择排序的**交换是"长距离跨越"**，中间被跳过的相等元素位置相对关系就可能乱。

### 3.4 Pitfalls

1. `range(n - 1)` 不是 `range(n)`：外层最后一次不需要执行，末尾元素已是最大。
2. 想让它稳定可以改用"**用插入而不是交换**"，但那样就不是原本意义上的选择排序了。
3. 比较次数固定，**不要指望它对"基本有序"的数据变快**。

## 4. 冒泡排序（Bubble Sort）

### 4.1 核心思想

> **相邻元素两两比较，逆序就交换，每轮把当前最大值"冒"到右端。**

```
[5, 1, 4, 2, 3]    一轮流程:
 5,1 -> 1,5        [1, 5, 4, 2, 3]
    5,4 -> 4,5     [1, 4, 5, 2, 3]
       5,2 -> 2,5  [1, 4, 2, 5, 3]
          5,3 -> 3,5  [1, 4, 2, 3, 5]  ← 5 冒到末尾
```

### 4.2 代码实现（带**已排序提前退出**优化）

```python
def bubble_sort(nums: List[int]) -> None:
    """冒泡排序（原地、稳定、自适应）。

    加了"某轮未发生交换就提前退出"的经典优化：
    对已经有序的数据，时间复杂度从 O(n^2) 降到 O(n)。

    Args:
        nums: 待排序列表，就地修改。

    Complexity:
        Time:  最好 O(n)（已排序），平均/最坏 O(n^2)。
        Space: O(1)
    """
    n: int = len(nums)
    # 外层：每一轮把一个"当前最大值"冒到最右端
    # 因此第 i 轮结束后，右侧 i 个元素已就位，内层只需扫 [0, n-1-i)
    for i in range(n - 1):
        swapped: bool = False                       # 标记：本轮是否发生过交换
        # 内层：相邻两两比较，边界是 n-1-i（留一个空隙做 j+1 访问）
        for j in range(n - 1 - i):
            if nums[j] > nums[j + 1]:               # 严格大于才交换，保证稳定性（相等不动）
                nums[j], nums[j + 1] = nums[j + 1], nums[j]
                swapped = True                      # 记录"这一轮有过交换"
        # 若整整一轮都没交换，说明数组已经有序，直接跳出
        if not swapped:
            break
```

### 4.3 为什么冒泡排序是**稳定**的？

**关键点**：只在 `nums[j] > nums[j + 1]` 时才交换——**相等时不动**。所以相等元素的相对顺序永远不会被破坏。

> 💡 **记忆口诀**：稳定排序看"**是否会让相等元素互相越过**"。冒泡相等不换，不越过，稳定。选择排序长距离跳跃跨越，不稳定。

### 4.4 Pitfalls

1. 交换条件写成 `>=` 会导致不稳定（且增加无效交换）。
2. 内层循环边界 `n - 1 - i`，少减 1 会越界（`nums[j+1]` 越界）。
3. 不加 `swapped` 优化，基本有序的输入也会全程跑 \(O(n^2)\)，**在实际中几乎等同于选择排序的慢**。

## 5. 插入排序 & 希尔排序

### 5.1 插入排序核心思想

> **像整理扑克牌**：从左到右扫，把每张新牌插到"左侧已排好的牌堆"的正确位置。

```
[3, 1, 4, 1, 5, 9, 2]
 ↑ 已排序区: [3]，未排序区: [1,4,1,5,9,2]

取 1，从右向左比，3 比 1 大 → 3 后移 → 插在 3 前：
[1, 3, 4, 1, 5, 9, 2]

取 4，与 3 比，不用动：
[1, 3, 4, 1, 5, 9, 2]

取 1，从右向左：4→3→插在 3 前：
[1, 1, 3, 4, 5, 9, 2]
...
```

### 5.2 插入排序代码

```python
def insertion_sort(nums: List[int]) -> None:
    """插入排序（原地、稳定、自适应）。

    特别适合：
    - 小规模数据（n < 50）
    - 基本有序的数据（最好 O(n)）

    事实上，Java/Python 的内置排序在小块数据上都会退化为插入排序，
    因为它的常数因子极小。

    Args:
        nums: 待排序列表，就地修改。

    Complexity:
        Time:  最好 O(n)（已排序），平均/最坏 O(n^2)。
        Space: O(1)
    """
    n: int = len(nums)
    # i 从 1 开始：把 nums[0] 视为"已排序区"的初始单元素
    for i in range(1, n):
        base: int = nums[i]                         # 本轮要插入的"新牌"；先暂存，避免覆盖
        j: int = i - 1                              # 从已排序区最右端开始向左扫
        # 只要已排序区元素比 base 大，就把它向右挪一格
        # 注意是 > 而不是 >=，这样相等元素不会越过 base，保持稳定性
        while j >= 0 and nums[j] > base:
            nums[j + 1] = nums[j]                   # 后移一格（不是交换，是覆盖）
            j -= 1                                  # 继续比前一个
        # 循环退出时，j 指向第一个 <= base 的位置（或 -1）
        # 所以 base 的正确位置是 j + 1
        nums[j + 1] = base                          # 把暂存的 base 放到正确位置
```

### 5.3 为什么插入排序对**小数据/基本有序数据**极快？

- **小数据**：常数因子非常小，没有递归开销、没有复杂判断，CPU 缓存友好。
- **基本有序**：内层 `while` 几乎立刻退出，总比较次数接近 \(n\)，逼近 \(O(n)\)。

**这也是工业界真实排序器的秘密**：`Tim Sort` 对长度 < 32 的小区间一律用插入排序。

### 5.4 希尔排序（Shell Sort）—— 插入排序的进化版

#### 5.4.1 为什么需要希尔？

插入排序慢的根源：**每次只能把一个元素向前挪一步**。如果一个元素要从末尾挪到开头，得动 \(n\) 次。

**希尔的核心想法**：

> **先按一个较大的"步长" gap 分组排序**（每组内部用插入排序），这样大幅减少每个元素的移动距离；然后**逐步缩小 gap**，直到 gap = 1（此时就是标准插入排序，但数据已经"大致有序"了）。

```
nums = [8, 3, 1, 5, 9, 2, 7, 4, 6]
gap = 4:
  分组: (8,9,6)  (3,2)  (1,7)  (5,4)   ← 每组组内做插入排序
  结果: [6, 2, 1, 4, 8, 3, 7, 5, 9]

gap = 2:
  分组: (6,1,8,7,9)  (2,4,3,5)         ← 再组内插入排序
  结果: [1, 2, 6, 3, 7, 4, 8, 5, 9]

gap = 1:
  普通插入排序，但此时已几乎有序，运行极快
  结果: [1, 2, 3, 4, 5, 6, 7, 8, 9]
```

#### 5.4.2 希尔排序代码

```python
def shell_sort(nums: List[int]) -> None:
    """希尔排序（原地、不稳定、部分自适应）。

    Args:
        nums: 待排序列表，就地修改。

    Complexity:
        Time:  取决于 gap 序列，常见版本 O(n^1.3)~O(n^1.5)，最坏 O(n^2)。
        Space: O(1)
    """
    n: int = len(nums)
    gap: int = n // 2                               # 初始 gap 常取 n/2（Shell 最初版本）
    # 逐步缩小 gap，直到为 0
    while gap > 0:
        # 对每个"偏移起点"做间隔为 gap 的插入排序
        # 等价于：对 gap 个子序列分别做插入排序，只不过我们把它们交错在一起处理
        for i in range(gap, n):
            base: int = nums[i]
            j: int = i - gap                        # 在同一子序列里，前一个元素下标是 i - gap
            # 在同一子序列内做插入排序的"向前比较-后移"逻辑
            while j >= 0 and nums[j] > base:
                nums[j + gap] = nums[j]             # 同样是后移（距离为 gap）
                j -= gap
            nums[j + gap] = base
        gap //= 2                                   # 缩小 gap（最经典的对半缩减）
```

#### 5.4.3 为什么希尔是**不稳定**的？

相等元素可能被分到不同的"gap 子序列"里，子序列内排序时它们的相对顺序会被打乱。

### 5.5 Pitfalls

1. 插入排序 `while` 循环条件：`j >= 0 and nums[j] > base`。少了 `j >= 0` 会 IndexError。
2. 希尔排序的 gap 序列选择会显著影响性能，工程中常用 **Knuth 序列**：`1, 4, 13, 40, 121...`（`gap = 3*gap + 1`），最坏复杂度接近 \(O(n^{1.5})\)。
3. 插入排序是 **数据接近有序** 时的首选；但数据完全随机时，插入排序比快速排序慢得多。

## 6. 快速排序（Quick Sort）

快排是**平均性能最好的通用比较排序**，工业界实际应用最广。

### 6.1 核心思想：分治 + 分区

```
1. 选一个"基准值 pivot"。
2. 把数组划分为两部分：左边都 ≤ pivot，右边都 ≥ pivot（pivot 本身放在中间）。
3. 递归地对左、右两部分排序。
```

```
[3, 7, 1, 8, 5, 2, 4]  pivot = 3
↓ 分区
[1, 2, 3, 8, 5, 7, 4]
        ↑ pivot 已就位
递归左 [1,2]，递归右 [8,5,7,4]
...
```

### 6.2 基础版：Lomuto 分区法

```python
def quick_sort(nums: List[int], lo: int = 0, hi: int = -1) -> None:
    """快速排序（原地、不稳定、非自适应）。

    这里使用 Lomuto 分区法：简单易懂，代码短。

    Args:
        nums: 待排序列表。
        lo:   待排序子区间左端（含）。
        hi:   待排序子区间右端（含）；默认 -1 表示 len(nums)-1。

    Complexity:
        Time:  平均 O(n log n)，最坏 O(n^2)（退化场景：已排序数组 + 取第一个作 pivot）。
        Space: 递归栈 O(log n) 平均，O(n) 最坏。
    """
    if hi == -1:                                    # 首次调用的便利处理
        hi = len(nums) - 1
    if lo >= hi:                                    # 区间长度 <= 1，无需排序
        return

    # ------ 分区：选最后一个元素作 pivot ------
    pivot: int = nums[hi]                           # 取区间末尾元素作 pivot
    i: int = lo - 1                                 # i 指向"小于 pivot 区"的右边界
    # j 扫描 [lo, hi-1]，把所有 <= pivot 的元素搬到左边
    for j in range(lo, hi):
        if nums[j] <= pivot:                        # 发现一个该放左边的元素
            i += 1                                  # 小于区右边界右移一格
            nums[i], nums[j] = nums[j], nums[i]     # 把它交换到小于区末尾
    # 循环结束后，nums[lo..i] 都 <= pivot，nums[i+1..hi-1] 都 > pivot
    # 最后把 pivot 放到 i+1 位置（它和左右两边的分界线）
    nums[i + 1], nums[hi] = nums[hi], nums[i + 1]
    p: int = i + 1                                  # p 是 pivot 最终位置

    # ------ 递归排序两个子区间 ------
    # 注意：不包含 p 本身（它已就位）
    quick_sort(nums, lo, p - 1)
    quick_sort(nums, p + 1, hi)
```

### 6.3 最坏情况与"随机化"救星

**最坏情况**：每次 pivot 都是当前区间的最小（或最大）值——分区极不平衡，递归深度退化为 \(n\)，复杂度 \(O(n^2)\)。

典型触发：**对已排序数组用"取末尾"作 pivot**。

**解法：随机化选 pivot**，或"**三数取中**"（取首、中、尾三个元素的中位数）。

```python
import random

def quick_sort_randomized(nums: List[int], lo: int = 0, hi: int = -1) -> None:
    """随机化快速排序。核心差异：分区前把 pivot 与随机位置交换。

    Args:
        同 quick_sort。
    """
    if hi == -1:
        hi = len(nums) - 1
    if lo >= hi:
        return
    # 在 [lo, hi] 中随机选一个下标，和末尾交换，实现"随机 pivot"
    # 这样对任意输入，期望分区都是平衡的，避免 O(n^2) 退化
    rand_idx: int = random.randint(lo, hi)
    nums[rand_idx], nums[hi] = nums[hi], nums[rand_idx]
    # 下面和 Lomuto 分区一样
    pivot: int = nums[hi]
    i: int = lo - 1
    for j in range(lo, hi):
        if nums[j] <= pivot:
            i += 1
            nums[i], nums[j] = nums[j], nums[i]
    nums[i + 1], nums[hi] = nums[hi], nums[i + 1]
    p: int = i + 1
    quick_sort_randomized(nums, lo, p - 1)
    quick_sort_randomized(nums, p + 1, hi)
```

### 6.4 三路快排（应对大量重复元素）

**痛点**：当数组有**大量重复元素**时，普通快排会把"等于 pivot 的元素"也递归处理，浪费大量时间。

**三路快排**把数组分成三段：`< pivot` / `== pivot` / `> pivot`，中间那段直接跳过。

```python
def quick_sort_3way(nums: List[int], lo: int = 0, hi: int = -1) -> None:
    """三路快速排序：适合大量重复元素的场景。

    把区间划分为三部分：
        [lo, lt)   : < pivot
        [lt, gt]   : == pivot
        (gt, hi]   : > pivot

    Args:
        nums, lo, hi: 与 quick_sort 相同。
    """
    if hi == -1:
        hi = len(nums) - 1
    if lo >= hi:
        return

    pivot: int = nums[lo]                           # 取首元素作 pivot（可换随机）
    lt: int = lo                                    # < 区的右边界（开区间）
    gt: int = hi                                    # > 区的左边界（开区间）
    i: int = lo + 1                                 # 扫描指针
    # 三指针扫描
    while i <= gt:
        if nums[i] < pivot:
            # 放入 < 区
            nums[lt], nums[i] = nums[i], nums[lt]
            lt += 1                                 # < 区扩张
            i += 1                                  # i 继续前进（交换来的元素已处理）
        elif nums[i] > pivot:
            # 放入 > 区
            nums[i], nums[gt] = nums[gt], nums[i]
            gt -= 1                                 # > 区扩张
            # 注意 i 不前进！因为交换过来的元素尚未扫描
        else:
            # 等于 pivot，留在中间区，i 继续
            i += 1
    # 中间 [lt, gt] 已全是 pivot，无需递归
    quick_sort_3way(nums, lo, lt - 1)
    quick_sort_3way(nums, gt + 1, hi)
```

### 6.5 为什么快排**不稳定**？

分区过程中需要"跨越"交换，相等元素的相对位置会被打乱。例如分区时把一个 `5a` 和 `3` 交换、再把 `5b` 留在原处，`5a` 和 `5b` 的顺序就乱了。

### 6.6 Pitfalls

1. **取第一个/最后一个作 pivot + 有序输入 = \(O(n^2)\)**。务必随机化。
2. 递归基线写错：`if lo >= hi: return`，少了会无限递归或越界。
3. Lomuto 分区的 `<=` 若写成 `<`，相等元素会被分到错的一边，导致分区不平衡。
4. 三路快排里 `i` 在和 `gt` 交换后**不能前进**——这是个极易错的细节。

## 7. 归并排序（Merge Sort）

### 7.1 核心思想：分治

```
1. 分：把数组从中间劈成两半。
2. 治：递归地把两半分别排序。
3. 合：将两个有序半段合并成一个有序整体。
```

```
[5, 2, 8, 1, 9, 3, 7, 4]
       ↓ 分
[5, 2, 8, 1]    [9, 3, 7, 4]
    ↓               ↓
[5,2] [8,1]     [9,3] [7,4]
  ↓     ↓         ↓     ↓
[2,5] [1,8]     [3,9] [4,7]   ← 合并
       ↓               ↓
[1, 2, 5, 8]    [3, 4, 7, 9]  ← 继续合并
           ↓
[1, 2, 3, 4, 5, 7, 8, 9]
```

### 7.2 自顶向下实现（递归版）

```python
def merge_sort(nums: List[int], lo: int = 0, hi: int = -1) -> None:
    """归并排序（非原地、稳定、非自适应）。

    自顶向下递归实现。

    Args:
        nums: 待排序列表，就地修改。
        lo:   子区间左端（含）。
        hi:   子区间右端（含）。

    Complexity:
        Time:  O(n log n)（最好 / 平均 / 最坏 全是这个）
        Space: O(n) 辅助数组 + O(log n) 递归栈
    """
    if hi == -1:
        hi = len(nums) - 1
    if lo >= hi:                                    # 单个元素或空区间，直接返回
        return

    mid: int = (lo + hi) // 2                       # 中点，分成 [lo, mid] 和 [mid+1, hi]
    merge_sort(nums, lo, mid)                       # 递归排左半
    merge_sort(nums, mid + 1, hi)                   # 递归排右半
    _merge(nums, lo, mid, hi)                       # 合并两段有序


def _merge(nums: List[int], lo: int, mid: int, hi: int) -> None:
    """将 nums[lo..mid] 和 nums[mid+1..hi] 两段有序区间合并。

    使用辅助数组 tmp（这就是归并非原地的根源）。
    """
    # 关键：一次性复制到辅助数组，之后原数组用于写入
    tmp: List[int] = nums[lo:hi + 1]                # 切片复制整段
    # 两个指针分别指向 tmp 中的两段
    i: int = 0                                      # 左段起点（在 tmp 中是 0）
    j: int = mid - lo + 1                           # 右段起点（在 tmp 中的偏移）
    left_end: int = mid - lo                        # 左段结束（含）
    right_end: int = hi - lo                        # 右段结束（含）

    # 指针 k 指向原数组写入位置
    for k in range(lo, hi + 1):
        if i > left_end:                            # 左段已耗尽，剩下全取右段
            nums[k] = tmp[j]
            j += 1
        elif j > right_end:                         # 右段已耗尽，剩下全取左段
            nums[k] = tmp[i]
            i += 1
        elif tmp[i] <= tmp[j]:                      # <= 保证稳定性：相等时取左段（左段在原数组更靠前）
            nums[k] = tmp[i]
            i += 1
        else:
            nums[k] = tmp[j]
            j += 1
```

### 7.3 为什么归并排序是**稳定**的？

合并时，只要 `tmp[i] <= tmp[j]` 就**优先取左段**。当左右两段各有一个相等元素时，左段的先被放入——而左段元素在原数组中本来就排在前面，所以相对顺序不变。

> ⚠️ 如果你把合并条件写成 `<`（严格小于），就不稳定了。**稳定性往往就藏在这个等号里**。

### 7.4 自底向上实现（迭代版，省去递归栈）

```python
def merge_sort_bottom_up(nums: List[int]) -> None:
    """自底向上归并排序（迭代版）。

    先合并长度为 1 的相邻对子，再长度 2、4、8...逐步翻倍。
    避免递归栈开销，更适合链表排序。

    Args:
        nums: 待排序列表，就地修改。

    Complexity:
        Time:  O(n log n), Space: O(n)
    """
    n: int = len(nums)
    size: int = 1                                   # 每轮要合并的子区间长度
    while size < n:                                 # 直到子区间覆盖整个数组
        lo: int = 0
        # 把数组切成长度 2*size 的窗口，一个个合并
        while lo < n - size:                        # 右段必须存在才有合并意义
            mid: int = lo + size - 1
            # 右段可能不足 size，用 min 截断
            hi: int = min(lo + 2 * size - 1, n - 1)
            _merge(nums, lo, mid, hi)
            lo += 2 * size                          # 跳到下一个窗口
        size *= 2                                   # 子区间翻倍
```

### 7.5 Pitfalls

1. `_merge` 里 `tmp[i] < tmp[j]` 会让归并**不稳定**。必须是 `<=`。
2. 辅助数组开在外面（一次分配）还是每次 `_merge` 内部分配？**工程推荐外面分配一次**，避免反复申请释放内存。
3. 递归深度 \(\log n\)，Python 默认限制够用；但 Java/C++ 在超大数组上可能栈溢出。

## 8. 堆排序（Heap Sort）

### 8.1 前置：堆是什么？

**堆**是一种**完全二叉树**，满足：

- **大顶堆**：父节点 ≥ 两个子节点
- **小顶堆**：父节点 ≤ 两个子节点

用数组表示堆时，下标关系（0-indexed）：

```
父节点 i 的左子: 2*i + 1
父节点 i 的右子: 2*i + 2
子节点 j 的父:  (j - 1) // 2
```

### 8.2 核心思想：堆排序 = 反复取堆顶

```
升序排序 → 建"大顶堆"（不是小顶堆！）
1. 把数组建成大顶堆（整个数组都是堆区）
2. 把堆顶 nums[0]（最大值）和末尾交换 → 最大值就位
3. 堆区长度 -1，对新的堆顶做"下沉"修复堆性质
4. 重复步骤 2~3，直到堆区长度为 1
```

> 🤔 **为什么升序排序要用大顶堆？**
> 因为每轮我们要把"当前最大值"固定到末尾，所以每次取"最大值"就对了——这正是大顶堆堆顶的值。

### 8.3 核心操作：下沉（sift down）

```
         5                        9
       /   \                    /   \
      9     7    →   下沉 5 →  5     7
     / \               或     / \
    3   1                    3   1    ← 继续下沉
    
    其实是:
             5                9
           /   \            /   \
          9     7   →      5     7
         /                /
        3                3   ← 5 和更大的子(9)交换
```

### 8.4 堆排序完整代码

```python
def heap_sort(nums: List[int]) -> None:
    """堆排序（原地、不稳定、非自适应）。

    Args:
        nums: 待排序列表，就地修改。

    Complexity:
        Time:  O(n log n) —— 最好/平均/最坏 全是。
        Space: O(1)
    """
    n: int = len(nums)

    # ---------- 阶段 1：建大顶堆 ----------
    # 从最后一个"非叶子节点"开始倒着下沉
    # 最后一个非叶子节点的下标 = (n-1-1)//2 = n//2 - 1
    for i in range(n // 2 - 1, -1, -1):
        _sift_down(nums, n, i)

    # ---------- 阶段 2：反复取堆顶 ----------
    # 每轮:
    #   1) 把堆顶（最大）和末尾交换 → 最大值就位
    #   2) 堆区长度减 1（末尾不再算入堆）
    #   3) 新堆顶做下沉，恢复堆性质
    for heap_end in range(n - 1, 0, -1):
        nums[0], nums[heap_end] = nums[heap_end], nums[0]
        _sift_down(nums, heap_end, 0)               # 注意：新堆长度就是 heap_end


def _sift_down(nums: List[int], heap_size: int, i: int) -> None:
    """对位置 i 执行"下沉"操作，使其符合大顶堆性质。

    Args:
        nums:      数组（堆的数组表示）。
        heap_size: 当前堆的有效长度（只关心 [0, heap_size)）。
        i:         要下沉的节点下标。
    """
    while True:
        left: int = 2 * i + 1                       # 左子下标
        right: int = 2 * i + 2                      # 右子下标
        largest: int = i                            # 假设当前节点就是三者里最大

        # 比较左子，只在左子存在时比较
        if left < heap_size and nums[left] > nums[largest]:
            largest = left
        # 比较右子
        if right < heap_size and nums[right] > nums[largest]:
            largest = right

        # 如果当前节点就是三者最大，下沉结束
        if largest == i:
            break
        # 否则和更大的子交换，继续下沉
        nums[i], nums[largest] = nums[largest], nums[i]
        i = largest                                 # 从交换后的新位置继续检查
```

### 8.5 为什么建堆阶段是 \(O(n)\) 而不是 \(O(n\log n)\)？

直觉上 `n/2` 次下沉每次 \(O(\log n)\) 应该是 \(O(n\log n)\)，但**实际是 \(O(n)\)**。

原因：越接近叶子的节点越多但下沉深度越浅，数学求和得到 \(O(n)\)。不过第二阶段仍是 \(O(n \log n)\)，所以总复杂度是 \(O(n\log n)\)。

### 8.6 为什么堆排序**不稳定**？

建堆和下沉过程中，相等元素会被大量跨越交换，相对位置无法保留。

### 8.7 Pitfalls

1. **升序排序用大顶堆，不是小顶堆**——这是最常见的混淆点。
2. 下沉循环中 `left < heap_size` 的边界检查不能漏，否则 `nums[left]` 越界。
3. 建堆阶段的循环从 `n // 2 - 1` 开始，倒序到 0。不是从 `n - 1` 开始（叶子节点不需要下沉）。
4. 第二阶段调 `_sift_down(nums, heap_end, 0)` 时第二个参数是 **`heap_end`（新堆长度）而不是 `n`**。

## 9. 桶排序（Bucket Sort）

### 9.1 核心思想

> **按值域划分若干"桶"，把元素分发到对应桶，桶内排序，再依次串起来。**

### 9.2 工作流程

```
输入 (浮点数，范围 [0, 1)): [0.42, 0.13, 0.88, 0.56, 0.17, 0.91, 0.30]
假设分 5 个桶 (每个桶范围 0.2)：

桶0 [0.0, 0.2):  [0.13, 0.17]
桶1 [0.2, 0.4):  [0.30]
桶2 [0.4, 0.6):  [0.42, 0.56]
桶3 [0.6, 0.8):  []
桶4 [0.8, 1.0):  [0.88, 0.91]

每个桶内部排序后，依次拼接:
[0.13, 0.17, 0.30, 0.42, 0.56, 0.88, 0.91]
```

### 9.3 代码实现

```python
from typing import List


def bucket_sort(nums: List[float]) -> List[float]:
    """桶排序（非原地、稳定、非比较）。

    假设输入是 [0, 1) 范围内的浮点数；对其他范围要做归一化。

    Args:
        nums: 输入的浮点数列表，范围 [0, 1)。

    Returns:
        排好序的新列表。

    Complexity:
        Time:  平均 O(n + k)（k 是桶数），最坏 O(n^2)（所有元素挤在一个桶）。
        Space: O(n + k)
    """
    n: int = len(nums)
    if n == 0:
        return []

    # 桶数一般取 n，让每个桶平均约 1 个元素（这样期望时间最好）
    k: int = n
    buckets: List[List[float]] = [[] for _ in range(k)]

    # ---------- 分发：把每个元素扔到对应的桶 ----------
    for x in nums:
        # idx = floor(x * k)，保证 x ∈ [0,1) → idx ∈ [0, k)
        # 若 x == 1.0 会溢出，这里用 min 兜底
        idx: int = min(int(x * k), k - 1)
        buckets[idx].append(x)

    # ---------- 每个桶内排序 ----------
    # 桶内用任何稳定排序都行；Python 内置 sort 即稳定且高效
    for bucket in buckets:
        bucket.sort()

    # ---------- 合并：按桶顺序串起来 ----------
    result: List[float] = []
    for bucket in buckets:
        result.extend(bucket)                       # extend 比逐个 append 快
    return result
```

### 9.4 什么时候用桶排序？

**理想条件**：

- 数据**均匀分布**在已知区间
- 数据量较大（小数据用插入就行）
- 需要稳定排序

**非理想场景**（不要用桶排序）：

- 数据集中分布（比如 99% 的值在某个小区间）→ 大部分桶空，极少数桶巨大，退化为 \(O(n^2)\)
- 无法估计值域 → 桶数难选

### 9.5 Pitfalls

1. 桶数过少 → 退化为在一个大桶里排序，和原来一样慢。
2. 桶数过多 → 空间浪费，且合并阶段变慢。
3. 输入 `x == 1.0` 时 `int(x * k) == k`，越界。务必 `min(idx, k-1)`。

## 10. 计数排序（Counting Sort）

### 10.1 核心思想

> **不做任何比较**，而是统计每个值出现的次数，然后根据次数还原有序序列。

**适用条件**：元素都是"**小范围的非负整数**"（或能映射到它）。

### 10.2 工作流程（不稳定版 —— 简化理解）

```
输入: [4, 2, 2, 8, 3, 3, 1]
1. 找最大值 max = 8 → 开长度为 max+1=9 的计数数组 cnt
2. 统计: cnt = [0, 1, 2, 2, 1, 0, 0, 0, 1]
   (下标 = 值, 元素 = 出现次数)
3. 按 cnt 逐值铺出: 1 次 "1"、2 次 "2"、2 次 "3"、1 次 "4"、1 次 "8"
   → [1, 2, 2, 3, 3, 4, 8]
```

```python
def counting_sort_simple(nums: List[int]) -> List[int]:
    """计数排序（简化版，不处理 key=record 型数据，因此不稳定语义）。

    Args:
        nums: 非负整数列表。

    Returns:
        排好序的新列表。

    Complexity:
        Time:  O(n + k)（k 是值域大小）。
        Space: O(n + k)
    """
    if not nums:
        return []
    max_val: int = max(nums)                        # 找最大值决定计数数组长度
    cnt: List[int] = [0] * (max_val + 1)            # cnt[v] = 值 v 的出现次数
    for x in nums:
        cnt[x] += 1                                 # 计数
    # 展开 cnt 生成结果
    result: List[int] = []
    for v, c in enumerate(cnt):
        result.extend([v] * c)                      # 有 c 个 v，就加 c 个
    return result
```

### 10.3 稳定版（工业级实现）

简化版在"整数本身排序"时稳定性无意义（相等的元素本就无法区分）。但真正的应用是给**带 key 的记录**排序，这时就需要**稳定版**。

```python
def counting_sort_stable(nums: List[int]) -> List[int]:
    """稳定版计数排序：使用"前缀和"确定每个值在结果数组的位置。

    这种写法才能推广到"按 key 给记录排序"的真实应用场景（见基数排序）。

    Args:
        nums: 非负整数列表。

    Returns:
        排好序的新列表。
    """
    if not nums:
        return []
    max_val: int = max(nums)
    cnt: List[int] = [0] * (max_val + 1)
    for x in nums:
        cnt[x] += 1
    # ---------- 关键步骤：前缀和 ----------
    # 做完前缀和后，cnt[v] 表示"值 ≤ v 的元素总数"
    # 也就是"值 v 的最后一个应放的位置 + 1"
    for i in range(1, max_val + 1):
        cnt[i] += cnt[i - 1]
    # ---------- 倒序填入 result（倒序是稳定性的关键）----------
    result: List[int] = [0] * len(nums)
    # 为什么倒序？从后往前扫 nums，对相等元素，后出现的会被放到更后面的位置
    # 这样就保留了原数组中的相对顺序
    for x in reversed(nums):
        cnt[x] -= 1                                 # 该值在结果数组的"下一个"位置
        result[cnt[x]] = x                          # 把 x 放到该位置
    return result
```

### 10.4 前缀和 + 倒序填入：稳定性的秘密

假设有两个相等元素 `5a`（原数组较前）、`5b`（原数组较后）：

- 倒序扫时，**先遇到 `5b`**，它占据"值 5 的最后一个位置"。
- 然后 `cnt[5] -= 1`，下次遇到 `5a` 时占据"值 5 的倒数第二个位置"。
- 结果：`5a` 在 `5b` 前面，和原数组一致 → **稳定** ✅

### 10.5 Pitfalls

1. **值域太大会爆内存**。若 `nums = [0, 1, 10^9]`，要开 \(10^9+1\) 长的计数数组——不现实。
2. 处理**负数**时需要偏移：`cnt[x - min_val] += 1`。
3. **倒序填入**是稳定性的关键。写成正序就不稳定了（在纯整数排序场景下看不出，但作为基数排序的"子过程"就会错）。

## 11. 基数排序（Radix Sort）

### 11.1 核心思想

> **按位数分别排序**：从低位到高位（LSD）或高位到低位（MSD），每一位都用"稳定"的计数排序，最终得到整体有序。

这是**计数排序**的推广：当值域巨大但位数少时（例如 32 位整数），按位排序远优于直接对整个值域计数。

### 11.2 LSD（Least Significant Digit）流程

```
输入: [170, 45, 75, 90, 2, 802, 24, 66]

按个位排（稳定!）: [170, 90, 2, 802, 24, 45, 75, 66]
按十位排（稳定!）: [2, 802, 24, 45, 66, 170, 75, 90]
按百位排（稳定!）: [2, 24, 45, 66, 75, 90, 170, 802]
完成 ✓
```

**核心要求**：每一轮的子排序**必须稳定**！否则前一轮排的序会被打乱。

### 11.3 代码实现

```python
def radix_sort(nums: List[int]) -> List[int]:
    """基数排序（LSD 从低位到高位）。仅支持非负整数。

    Args:
        nums: 非负整数列表。

    Returns:
        排好序的新列表。

    Complexity:
        Time:  O(d * (n + k))，d 是最大位数，k 是基数（一般 10）。
        Space: O(n + k)
    """
    if not nums:
        return []
    max_val: int = max(nums)
    exp: int = 1                                    # exp = 当前处理的位的权值（1、10、100、...）
    # 只要最高位还有意义就继续
    while max_val // exp > 0:
        nums = _counting_sort_by_digit(nums, exp)
        exp *= 10
    return nums


def _counting_sort_by_digit(nums: List[int], exp: int) -> List[int]:
    """按"当前位"做一轮稳定计数排序。

    Args:
        nums: 输入。
        exp:  该位的权值，1=个位, 10=十位, 100=百位……

    Returns:
        仅按该位排序的新列表。
    """
    n: int = len(nums)
    result: List[int] = [0] * n
    cnt: List[int] = [0] * 10                       # 十进制，0-9 共 10 个"桶"

    # ---------- 按当前位计数 ----------
    for x in nums:
        digit: int = (x // exp) % 10                # 提取当前位的数字，如 x=345, exp=10 → digit=4
        cnt[digit] += 1

    # ---------- 前缀和（确定每个数字的输出位置）----------
    for i in range(1, 10):
        cnt[i] += cnt[i - 1]

    # ---------- 倒序填入，保持稳定性 ----------
    for x in reversed(nums):
        digit = (x // exp) % 10
        cnt[digit] -= 1
        result[cnt[digit]] = x

    return result
```

### 11.4 LSD vs MSD

| 维度 | LSD（低位优先） | MSD（高位优先） |
| :--- | :--- | :--- |
| 思路 | 一次性排完所有数据，按位递增 | 分桶递归，像快排那样拆分 |
| 实现 | 简单、迭代 | 复杂、通常递归 |
| 常见场景 | 定长整数、定长字符串 | 变长字符串、字典序比较 |
| 稳定性 | 天然稳定 | 需要小心实现 |

### 11.5 Pitfalls

1. 子排序**必须稳定**，否则前几轮的结果会被后面打乱。
2. 负数：先把全数组加上一个偏移量变为非负，或分正负处理。
3. 浮点数：需要按 IEEE 754 位模式处理，直接套基数排序不行。
4. `exp` 溢出：在其他语言中 `exp *= 10` 可能溢出 int；Python 无此问题。

## 12. Tim Sort：Python 的内置排序

> **Python 的 `list.sort()` 和 `sorted()` 底层用的不是我们学过的任何一种纯算法，而是 Tim Sort。**

### 12.1 为什么要了解 Tim Sort？

因为它是**工业级排序的真实样貌**：

1. 在真实数据上**速度最优**
2. **稳定**
3. 在"已排序"、"逆序"、"部分有序"等常见场景下都非常快
4. Python、Java 7+、Android、V8 都采用它

### 12.2 核心思想（简化版）

Tim Sort = **归并排序 + 插入排序 + 识别已有顺序的 run**

```
1. 把数组扫一遍，识别出已有的"有序段"（run），太短的用插入排序扩展到 minrun（通常 32）。
2. 维护一个 run 栈，按规则合并（避免合并太不平衡的两段）。
3. 最终合并成整个有序数组。
```

**三个关键优化**：

- **识别自然 run**：如果输入已经包含大量有序段，几乎直接 merge 出来。
- **小段插入排序**：长度 < 32 的段用插入排序，利用其低常数因子。
- **合并时二分 + gallop**：在合并两个 run 时用二分法跳跃式定位，减少比较次数。

### 12.3 你只需要知道的使用细节

```python
from typing import List

# ---------- 1. list.sort() 是就地排序，返回 None ----------
a: List[int] = [3, 1, 4, 1, 5, 9]
a.sort()                                            # 就地，a 被修改
print(a)                                            # [1, 1, 3, 4, 5, 9]

# ---------- 2. sorted(iterable) 返回新列表，不修改原数据 ----------
b = (3, 1, 4, 1, 5, 9)
c = sorted(b)                                       # c = [1, 1, 3, 4, 5, 9]
print(b)                                            # 原元组不变

# ---------- 3. key 参数：按自定义键排序（稳定）----------
words = ["banana", "Apple", "cherry"]
words.sort(key=str.lower)                           # 忽略大小写排序
print(words)                                        # ['Apple', 'banana', 'cherry']

# ---------- 4. reverse 参数：降序 ----------
nums = [3, 1, 4, 1, 5]
nums.sort(reverse=True)                             # 降序
print(nums)                                         # [5, 4, 3, 1, 1]

# ---------- 5. 多关键字排序：利用稳定性 ----------
# 比如先按姓名升序，再按年龄降序
people = [("Alice", 30), ("Bob", 25), ("Alice", 25), ("Bob", 30)]
# 技巧：稳定排序下，先按次要 key 排，再按主要 key 排，结果就是"主要升次要降"
people.sort(key=lambda x: x[1], reverse=True)       # 第 1 步：按年龄降序
people.sort(key=lambda x: x[0])                     # 第 2 步：按姓名升序（稳定！）
print(people)
# [('Alice', 30), ('Alice', 25), ('Bob', 30), ('Bob', 25)]
```

### 12.4 实战建议

1. **99% 的场景直接用 `sorted()` / `list.sort()`**。手写排序只在面试或特殊需求（如原地 + 稳定 + 自定义分区）时才做。
2. 性能敏感：用 `key` 而不是 `cmp_to_key`，前者每个元素只调用一次 key 函数。
3. 大数据排序涉及对象时，考虑把排序键提取到数组，排完后再映射回去。

## 13. 外部排序概览（数据超过内存）

### 13.1 场景

> 你要给一个 1TB 的日志文件按时间戳排序，但内存只有 16GB。

### 13.2 经典解法：多路归并

```
1. 分块：把大文件分成能装进内存的小块（如 8GB 一块），每块用内存排序后写回磁盘。
2. 归并：对所有有序小文件做 k 路归并（用小顶堆当"当前最小值"选择器），边读边写。
```

### 13.3 关键技术点

- **最小堆**（`heapq.merge`）做 k 路归并
- **缓冲 I/O**：每个文件维护一个内存缓冲区
- **磁盘顺序读写**远快于随机读写，所以归并顺序写入

Python 示例：

```python
import heapq
from typing import Iterator, List

def external_merge(files: List[str], output: str) -> None:
    """对多个已有序文件做 k 路归并，输出到 output。

    Args:
        files:  输入文件路径列表，每个文件内部已按升序排列（每行一个整数）。
        output: 输出文件路径。
    """
    # 打开所有文件，并用生成器表达式把每个文件变成"产出 int 的迭代器"
    handles = [open(f, 'r') for f in files]
    try:
        iters: List[Iterator[int]] = [
            (int(line) for line in fh) for fh in handles
        ]
        # heapq.merge 是懒惰的 k 路归并，内存只需 O(k)
        with open(output, 'w') as out:
            for x in heapq.merge(*iters):
                out.write(f"{x}\n")
    finally:
        for fh in handles:
            fh.close()
```

> 💡 真实的数据库（MySQL 的 filesort、Hadoop 的 shuffle 阶段）都在用这个思路。

## 14. 排序算法全景对比

### 14.1 按时间复杂度分类

```
O(n^2) 家族（简单但慢）:
  选择、冒泡、插入

O(n^1.3)~O(n^1.5):
  希尔

O(n log n) 家族（比较排序的理论下界）:
  快速、归并、堆、Tim Sort

O(n + k) / O(n * d) 家族（非比较排序，突破下界）:
  计数、桶、基数
```

### 14.2 如何选？实战决策树

```
元素可比较吗？
├── 否 → 自定义 key 或 cmp_to_key
└── 是
    ├── 数据量 < 50？
    │   └── 插入排序（或直接用 Tim Sort 也行）
    │
    ├── 是整数且值域小（例如 [0, 10^5]）？
    │   └── 计数排序
    │
    ├── 是定长整数/字符串？
    │   └── 基数排序
    │
    ├── 数据均匀分布在已知区间？
    │   └── 桶排序
    │
    ├── 需要稳定 + 内存足够？
    │   └── 归并排序（或 Tim Sort）
    │
    ├── 需要原地 + 最坏 O(n log n)？
    │   └── 堆排序
    │
    ├── 一般场景，追求实际最快？
    │   └── 快速排序（随机化）或 Tim Sort
    │
    └── 数据超过内存？
        └── 外部归并排序
```

### 14.3 面试高频考点

| 问题 | 答案要点 |
| :--- | :--- |
| 最好的 O(n log n) 通用排序？ | 快排（平均）、归并（稳定）、Tim Sort（实际）|
| 空间复杂度 O(1) 且最坏 O(n log n)？ | **堆排序**（这是它的独门优势）|
| 稳定的 O(n log n)？ | **归并排序**、Tim Sort |
| 基本有序数据最快？ | 插入排序、Tim Sort |
| 重复元素多时最快？ | 三路快速排序 |
| Python 内置用什么？ | Tim Sort |

## 16. 总结与复习要点

### 16.1 必背三大属性（每个算法都要背）

对每个算法问自己三个问题：

1. **时间复杂度** —— 最好 / 平均 / 最坏
2. **是否稳定**
3. **是否原地**

### 16.2 必背清单 ✅

**基础三剑客（O(n²)）**

- [ ] **选择排序**：每轮选最小放前，**不稳定**，原地
- [ ] **冒泡排序**：相邻交换，加 swapped 优化后最好 O(n)，**稳定**，原地
- [ ] **插入排序**：像整理扑克牌，**稳定**，原地，**小数据最快**

**进阶三剑客（O(n log n)）**

- [ ] **快速排序**：分治 + 分区，平均最快，**不稳定**，基本原地
  - [ ] Lomuto 分区、Hoare 分区、三路快排
  - [ ] 随机化防退化
- [ ] **归并排序**：分治 + 合并，**稳定**，需要 O(n) 辅助空间
  - [ ] 合并条件 `<=` 保证稳定
  - [ ] 自顶向下 vs 自底向上
- [ ] **堆排序**：大顶堆 + 反复取顶，**不稳定**，**O(1) 空间**
  - [ ] 建堆 O(n)、下沉 O(log n)
  - [ ] 升序用大顶堆（不是小顶堆！）

**非比较三剑客（O(n+k)）**

- [ ] **计数排序**：值域小的整数，**稳定**（需前缀和+倒序）
- [ ] **桶排序**：均匀分布数据，**稳定**
- [ ] **基数排序**：按位排序，每轮必须稳定

**工业实战**

- [ ] **Tim Sort** = Python 内置，归并+插入+run 识别，**稳定**
- [ ] **希尔排序**：插入排序进化版，gap 逐步缩小，**不稳定**
- [ ] **外部排序**：k 路归并 + 小顶堆

### 16.3 易错速查

| 算法 | 常见错误 | 防御 |
| :--- | :--- | :--- |
| 选择排序 | 外层循环写成 `range(n)` | 应是 `range(n-1)` |
| 冒泡排序 | 忘加 `swapped` 优化 | 始终写上，对基本有序数据救命 |
| 冒泡/插入 | 比较用 `>=` 导致不稳定 | 严格 `>`，相等不换 |
| 插入排序 | `while j >= 0 and nums[j] > base` 少前半 | 必须先判 `j >= 0` 再访问 |
| 快速排序 | 有序输入 + 取首元素 = O(n²) | 必须随机化 pivot |
| 快速排序 | 三路快排里 i 误前进 | 和 gt 交换后 i 不前进 |
| 归并排序 | 合并写 `<` 导致不稳定 | 务必 `<=` |
| 堆排序 | 升序用小顶堆 | 升序必须用**大顶堆** |
| 堆排序 | `_sift_down` 漏边界 | `left < heap_size` 不能少 |
| 计数排序 | 值域巨大爆内存 | 改用基数排序 |
| 计数排序 | 正序填入导致不稳定 | 必须倒序填入 |
| 基数排序 | 子排序不稳定 | 必须用稳定的计数排序 |

### 16.4 复习建议动作

1. **手绘一张"10 种排序过程"大挂图**，每种用 `[5, 2, 4, 6, 1, 3]` 跑一遍，每轮画出数组状态。
2. **实现挑战**：不看书独立实现 10 种排序，每种都能跑通 `assert sorted(a) == a_sorted_by_your_impl`。
3. **稳定性实验**：用带 tag 的数据 `[(5, 'a'), (5, 'b'), (3, 'c')]` 跑每个算法，验证稳定性。
4. **性能对比**：用随机数组、已排序数组、逆序数组、大量重复数组分别测试，体会"自适应"和"最坏情况"。
5. **源码阅读**：读一遍 CPython 的 `listobject.c` 中 Tim Sort 的实现（英文注释很棒）。
