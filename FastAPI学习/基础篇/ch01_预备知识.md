## 1. Python 类型注解（Type Hints）

### 1.1 为什么要学这个

FastAPI 的**核心设计**就建立在类型注解上。它通过你写的类型注解来：

- 自动校验请求数据是否合法
- 自动生成 API 文档
- 提供编辑器智能提示

### 1.2 基本类型注解

类型注解就是在变量或函数参数后面加个冒号，告诉别人（和工具）"这个东西应该是什么类型"

```python
# ==================== 变量类型注解 ====================

name: str = "张三"          # 字符串
age: int = 25               # 整数
price: float = 99.5         # 浮点数
is_active: bool = True      # 布尔值

# 注意：类型注解不会强制报错，Python 解释器不会拦你
# 它只是一个"标记"，配合工具使用（比如 mypy、FastAPI）
age: int = "二十五"         # 不会报错，但工具会警告你
```

### 1.3 函数参数与返回值注解

```python
# 参数注解：冒号后面写类型
# 返回值注解：箭头后面写类型
def greet(name: str, times: int) -> str:
    """
    参数:
        name: 用户名，字符串类型
        times: 重复次数，整数类型
    返回:
        拼接后的问候语，字符串类型
    """
    return f"你好，{name}！" * times


# 调用
result: str = greet("小明", 3)
print(result)  # 你好，小明！你好，小明！你好，小明！
```

### 1.4 容器类型注解

当变量是列表、字典这类容器时，还可以标注里面装的是什么类型

```python
from typing import List, Dict, Tuple, Set

# ==================== 列表 ====================
# "一个列表，里面每个元素都是字符串"
names: List[str] = ["张三", "李四", "王五"]

# "一个列表，里面每个元素都是整数"
scores: List[int] = [88, 92, 76]


# ==================== 字典 ====================
# "一个字典，键是字符串，值是浮点数"
prices: Dict[str, float] = {"苹果": 5.5, "香蕉": 3.2}

# "一个字典，键是字符串，值是整数"
student_ages: Dict[str, int] = {"小明": 20, "小红": 22}


# ==================== 元组 ====================
# "一个元组，包含一个字符串和一个整数"
record: Tuple[str, int] = ("张三", 95)


# ==================== 集合 ====================
# "一个集合，里面都是字符串"
tags: Set[str] = {"Python", "FastAPI", "Docker"}
```

> **Python 3.9+ 的简写**：如果你的 Python 版本 >= 3.9，可以直接用小写 `list[str]`、`dict[str, int]`，不需要从 `typing` 导入。FastAPI 两种写法都支持

```python
# Python 3.9+ 简写形式
names: list[str] = ["张三", "李四"]
prices: dict[str, float] = {"苹果": 5.5}
```

### 1.5 Optional —— 可以为 None 的类型

实际开发中，很多字段是"可填可不填"的。`Optional` 表示这个值可以是某个类型，也可以是 `None`。

```python
from typing import Optional

# Optional[str] 等价于 str | None
# 意思是：这个值要么是字符串，要么是 None
def find_user(user_id: int) -> Optional[str]:
    """根据 ID 查找用户名，找不到就返回 None"""
    users = {1: "张三", 2: "李四"}
    return users.get(user_id)  # 找不到时 .get() 返回 None


result = find_user(1)   # "张三"
result = find_user(999) # None
```

```python
# ========== 在 FastAPI 中的典型用法 ==========
# 用户注册时，昵称是选填的
from typing import Optional

class UserCreate:
    username: str                       # 必填
    email: str                          # 必填
    nickname: Optional[str] = None      # 选填，默认为 None
```

> **Python 3.10+ 简写**：可以用 `str | None` 替代 `Optional[str]`，效果一样

### 1.6 Union —— 多种类型都可以

```python
from typing import Union

# 这个参数既可以是 int，也可以是 str
def process_id(item_id: Union[int, str]) -> str:
    """
    接受整数或字符串形式的 ID
    """
    return f"处理 ID: {item_id}"


process_id(42)       # 合法
process_id("abc")    # 合法
```

> **Python 3.10+ 简写**：`Union[int, str]` 可以写成 `int | str`

### 1.7 Pydantic BaseModel（重要！FastAPI 核心）

这个严格说不算 Python 自带的，但它是 FastAPI 的核心依赖，必须提前了解。Pydantic 的作用：**定义数据的形状，并自动校验**

```python
from pydantic import BaseModel
from typing import Optional

# ==================== 定义一个数据模型 ====================
class BidRequest(BaseModel):
    """投标请求的数据格式"""
    project_name: str                       # 必填，字符串
    control_price: float                    # 必填，浮点数
    num_competitors: int                    # 必填，整数
    notes: Optional[str] = None             # 选填，默认 None


# ==================== 使用：传入正确的数据 ====================
req = BidRequest(
    project_name="南昌路面改造",
    control_price=1200.5,
    num_competitors=8
)
print(req.project_name)    # "南昌路面改造"
print(req.notes)           # None（没传就用默认值）


# ==================== 自动类型转换 ====================
# 传入字符串 "10"，Pydantic 会自动转成 int 类型 10
req2 = BidRequest(
    project_name="测试项目",
    control_price="999.9",   # 字符串 -> 自动转成 float
    num_competitors="10"     # 字符串 -> 自动转成 int
)
print(type(req2.control_price))  # <class 'float'>


# ==================== 自动校验：传入非法数据会报错 ====================
# 下面这行会抛出 ValidationError，因为 "abc" 没法转成 int
# req3 = BidRequest(
#     project_name="错误示例",
#     control_price=100.0,
#     num_competitors="abc"   # 报错！
# )
```

**关键点**：在 FastAPI 里，你的每个接口"接收什么数据"和"返回什么数据"，都是用 Pydantic 模型定义的。FastAPI 拿到你的模型定义后，会自动做校验、自动生成文档、自动转 JSON

### 1.8 嵌套模型

真实业务中，数据结构经常是嵌套的。如果不用嵌套模型，没法告诉 FastAPI "history 里面每个元素应该长什么样"。你只能写 `history: list`，那 FastAPI 就不知道列表里该有哪些字段、每个字段什么类型，也没法帮你校验

```python
from pydantic import BaseModel
from typing import List, Optional


class HistoricalBid(BaseModel):
    """单条历史投标记录"""
    project_name: str
    bid_price: float
    won: bool                  # 是否中标


class AnalysisRequest(BaseModel):
    """分析请求：包含当前项目信息 + 历史数据"""
    current_project: str
    control_price: float
    competitor_count: int
    # 嵌套：一个列表，里面每个元素都是 HistoricalBid 模型
    history: List[HistoricalBid]
    strategy: Optional[str] = "default"


# 使用时，JSON 数据可以直接被解析成嵌套结构
data = {
    "current_project": "路面改造",
    "control_price": 1500.0,
    "competitor_count": 6,
    "history": [
        {"project_name": "项目A", "bid_price": 980.0, "won": True},
        {"project_name": "项目B", "bid_price": 1100.0, "won": False},
    ]
}

req = AnalysisRequest(**data)
print(req.history[0].project_name)  # "项目A"
print(req.history[0].won)           # True
```

## 2. HTTP协议基础

### 2.1 HTTP 是什么

HTTP（HyperText Transfer Protocol，超文本传输协议）是浏览器和服务器之间通信的规则



你在浏览器输入一个网址按回车，浏览器就发了一个 HTTP **请求**给服务器，服务器处理后返回一个 HTTP **响应**。你写的 FastAPI 程序就是那个服务器。整个过程就是：

```css
客户端（浏览器/App/代码）                      服务器（你的 FastAPI）
        |                                           |
        |  -------- HTTP 请求（Request）-------->   |
        |                                           |  处理逻辑...
        |  <------- HTTP 响应（Response）--------   |
        |                                           |
```

### 2.2 HTTP 请求的组成

一个请求包含四个部分，理解这四个部分非常重要：

```css
POST /api/predict?verbose=true HTTP/1.1     ← 请求行（方法 + 路径 + 协议版本）
Host: localhost:8000                         ← 请求头（Headers）
Content-Type: application/json               ← 请求头（告诉服务器：我发的是 JSON）
Authorization: Bearer abc123                 ← 请求头（认证信息）
                                             ← 空行（分隔头和体）
{"control_price": 1200.5, "competitors": 8}  ← 请求体（Body，实际数据）
```

拆解说明：

| 组成部分     | 内容                     | 说明                                           |
| ------------ | ------------------------ | ---------------------------------------------- |
| **请求方法** | `POST`                   | 告诉服务器你要干什么（取数据/提交数据/删除等） |
| **路径**     | `/api/predict`           | 你要访问哪个接口                               |
| **查询参数** | `?verbose=true`          | 附加在 URL 后面的键值对参数                    |
| **请求头**   | `Content-Type: ...`      | 附加信息（数据格式、认证、语言等）             |
| **请求体**   | `{"control_price": ...}` | 实际发送的数据，通常是 JSON                    |

### 2.3 HTTP 响应的组成

```css
HTTP/1.1 200 OK                              ← 状态行（协议版本 + 状态码 + 描述）
Content-Type: application/json               ← 响应头
                                             ← 空行
{"recommended_price": 1150.0, "prob": 0.68}  ← 响应体（返回的数据）
```

### 2.4 HTTP 请求方法

这是你每天都会用到的东西。HTTP 定义了几种"动作"，告诉服务器你想干什么：

```css
GET     → 获取数据（"我要看东西"）
POST    → 提交数据（"我给你一些数据，帮我处理一下"）
PUT     → 整体更新（"用我给的数据替换掉旧的"）
PATCH   → 部分更新（"只改其中几个字段"）
DELETE  → 删除数据（"把这条记录删掉"）
```

**具体区别和使用场景**：

```python
# ==================== GET：获取数据 ====================
# 特点：不带请求体，参数放在 URL 里
# 场景：查询、搜索、获取列表

# 请求示例：
# GET /api/projects?page=1&limit=10
# 意思：获取项目列表，第1页，每页10条


# ==================== POST：提交数据 ====================
# 特点：数据放在请求体里，通常是 JSON
# 场景：创建新资源、提交表单、触发计算

# 请求示例：
# POST /api/predict
# Body: {"control_price": 1200, "competitors": 8}
# 意思：提交这些参数，让服务器算一个推荐报价


# ==================== PUT：整体更新 ====================
# 特点：用新数据完整替换旧数据
# 场景：修改用户信息、更新配置

# 请求示例：
# PUT /api/projects/42
# Body: {"name": "新名称", "price": 1500, "status": "active"}
# 意思：把 42 号项目的信息整体替换成这个


# ==================== DELETE：删除 ====================
# 特点：通常不带请求体，用路径指定要删的东西
# 场景：删除记录

# 请求示例：
# DELETE /api/projects/42
# 意思：删除 42 号项目
```

**GET 和 POST 的核心区别**：

```css
                GET                              POST
参数位置     URL 里（?key=value）              请求体里（Body）
数据量       有长度限制（URL 不能太长）          没有明显限制
可见性       参数直接暴露在 URL 里              参数在 Body 里，相对隐蔽
缓存         可以被浏览器缓存                   不会被缓存
用途         获取数据，不修改服务器状态          提交数据，可能修改服务器状态
幂等性       是（请求多次结果一样）             否（每次提交可能创建新数据）
```

### 2.5 HTTP 状态码

服务器返回的三位数字，告诉客户端"结果怎么样了"。

```css
2xx 成功
├── 200 OK                → 请求成功，返回了数据
├── 201 Created           → 创建成功（比如新建了一条记录）
└── 204 No Content        → 成功，但没有返回内容（比如删除操作）

3xx 重定向
└── 301/302               → 资源搬家了，去新地址找
                            （基本用不到，了解即可）

4xx 客户端错误（请求方的问题）
├── 400 Bad Request       → 请求格式不对（比如 JSON 格式错了）
├── 401 Unauthorized      → 没有认证（没登录）
├── 403 Forbidden         → 认证了但没权限（登录了但你不是管理员）
├── 404 Not Found         → 你请求的东西不存在
└── 422 Unprocessable     → 数据格式对了但内容不合法
                            （FastAPI 校验失败时常返回这个）

5xx 服务端错误（你的代码的问题）
├── 500 Internal Error    → 服务器内部错误（你代码崩了）
└── 503 Service Unavail.  → 服务暂时不可用（服务器过载）
```

**在 FastAPI 中手动返回状态码**：

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.get("/projects/{project_id}")
def get_project(project_id: int):
    projects = {1: "路面改造", 2: "桥梁建设"}

    if project_id not in projects:
        # 主动抛出 404 错误
        raise HTTPException(
            status_code=404,
            detail=f"项目 {project_id} 不存在"
        )

    return {"id": project_id, "name": projects[project_id]}
```

### 2.6 请求头（Headers）

请求头是附加在请求里的元信息，键值对格式。常见的有：

```
Content-Type: application/json     → 告诉服务器"我发的数据是 JSON 格式"
Accept: application/json           → 告诉服务器"我希望你返回 JSON"
Authorization: Bearer <token>      → 认证信息（登录令牌）
User-Agent: Mozilla/5.0 ...        → 客户端标识（什么浏览器/程序在请求）
```

你不需要记住所有请求头。在 FastAPI 里，大部分请求头是自动处理的，你只需要在需要认证的时候关注 `Authorization`

## 3. 请求与响应模型

当你用浏览器或程序向服务器发请求时：

- **请求头（Headers）**：说明“我是谁、我要干嘛”
- **请求体（Body）**：真正“带过去的数据”

👉 就像寄快递：

- 请求头 = 快递单（寄件人、收件人、类型）
- 请求体 = 包裹里面的东西

### 3.1 完整的请求-响应流程

把前面学的串起来，一个完整的 API 调用过程是这样的：

```
Step 1: 客户端构造请求
┌─────────────────────────────────────────────┐
│ POST /api/predict HTTP/1.1                  │ ← 方法和路径
│ Content-Type: application/json              │ ← 请求头
│                                             │
│ {                                           │
│   "control_price": 1200.5,                  │ ← 请求体
│   "num_competitors": 8                      │   （JSON 格式）
│ }                                           │
└─────────────────────────────────────────────┘
                    |
                    ↓
Step 2: FastAPI 接收并处理
┌─────────────────────────────────────────────┐
│ 1. 路由匹配 → 找到 /api/predict 对应的函数  │
│ 2. 数据校验 → 用 Pydantic 模型校验 JSON     │
│ 3. 执行逻辑 → 跑你的模型，算出结果          │
│ 4. 构造响应 → 把结果包成 JSON 返回          │
└─────────────────────────────────────────────┘
                    |
                    ↓
Step 3: 服务器返回响应
┌─────────────────────────────────────────────┐
│ HTTP/1.1 200 OK                             │ ← 状态码
│ Content-Type: application/json              │ ← 响应头
│                                             │
│ {                                           │
│   "recommended_price": 1150.0,              │ ← 响应体
│   "win_probability": 0.68,                  │   （JSON 格式）
│   "strategy": "稳健型"                      │
│ }                                           │
└─────────────────────────────────────────────┘
```

### 3.2 在 FastAPI 中的代码实现

把上面的流程翻译成真实代码：

```python
from fastapi import FastAPI
from pydantic import BaseModel
from typing import Optional

app = FastAPI()


# ========== 定义请求数据格式 ==========
class PredictRequest(BaseModel):
    """客户端发过来的数据长这样"""
    control_price: float
    num_competitors: int
    strategy: Optional[str] = "default"


# ========== 定义响应数据格式 ==========
class PredictResponse(BaseModel):
    """服务器返回的数据长这样"""
    recommended_price: float
    win_probability: float
    strategy: str


# ========== 定义接口 ==========
@app.post("/api/predict", response_model=PredictResponse)
def predict_bid(request: PredictRequest):
    """
    接收投标参数，返回推荐报价

    - FastAPI 看到参数类型是 PredictRequest，
      就知道要从请求体里取 JSON 数据并校验
    - response_model 告诉 FastAPI 返回值的格式，
      用于生成文档和校验响应
    """
    # 这里调用你的模型逻辑（简化示意）
    recommended = request.control_price * 0.95
    probability = 0.68

    return PredictResponse(
        recommended_price=recommended,
        win_probability=probability,
        strategy=request.strategy
    )
```

## 4. URL、路由与路径

### 4.1 URL 的结构

```
https://api.example.com:8000/api/v1/projects/42?verbose=true&lang=zh
└─┬──┘ └──────┬───────┘└┬─┘└──────┬──────────┘ └──────────┬────────┘
 协议       主机名     端口      路径                   查询参数
```

各部分说明：

```
协议（scheme）    → https:// 或 http://，加密与否
主机名（host）    → 服务器地址，本地开发时通常是 localhost 或 127.0.0.1
端口（port）      → 程序监听的端口号，FastAPI 默认 8000
路径（path）      → 指向哪个接口，这就是"路由"
查询参数（query） → ?key=value&key2=value2，附加的筛选/配置参数
```

### 4.2 路由（Routing）

路由就是"URL 路径和处理函数的映射关系"——用户访问某个路径，执行对应的函数

```python
from fastapi import FastAPI

app = FastAPI()


# 路由 1：访问 GET /  → 执行 root() 函数
@app.get("/")
def root():
    return {"message": "欢迎使用投标决策系统"}


# 路由 2：访问 GET /health  → 执行 health_check() 函数
@app.get("/health")
def health_check():
    return {"status": "ok"}


# 路由 3：访问 POST /api/predict  → 执行 predict() 函数
@app.post("/api/predict")
def predict():
    return {"result": "计算中..."}
```

### 4.3 路径参数（Path Parameters）

路径参数是 URL 路径中的动态部分，用花括号 `{}` 标记

```python
# /projects/42 中的 42 就是路径参数
@app.get("/projects/{project_id}")
def get_project(project_id: int):
    """
    {project_id} 对应函数参数 project_id
    类型注解为 int，FastAPI 会自动把 URL 中的字符串转成整数
    如果传的不是数字（比如 /projects/abc），自动返回 422 错误
    """
    return {"project_id": project_id, "name": f"项目{project_id}"}


# 多个路径参数
@app.get("/users/{user_id}/projects/{project_id}")
def get_user_project(user_id: int, project_id: int):
    return {
        "user_id": user_id,
        "project_id": project_id
    }

# 访问 /users/5/projects/42
# → user_id=5, project_id=42
```

### 4.4 查询参数（Query Parameters）

查询参数是 URL 中 `?` 后面的键值对。在函数中，**不在路径里出现的参数就是查询参数**。

```python
@app.get("/projects")
def list_projects(
    page: int = 1,              # 查询参数，默认值 1
    limit: int = 10,            # 查询参数，默认值 10
    keyword: str | None = None  # 查询参数，可选
):
    """
    请求示例:
      GET /projects?page=2&limit=20&keyword=路面
      → page=2, limit=20, keyword="路面"

      GET /projects
      → page=1, limit=10, keyword=None（全用默认值）
    """
    return {
        "page": page,
        "limit": limit,
        "keyword": keyword
    }
```

### 4.5 路径参数 vs 查询参数 vs 请求体

```python
from pydantic import BaseModel


class BidData(BaseModel):
    control_price: float
    competitors: int


@app.post("/projects/{project_id}/bid")
def submit_bid(
    project_id: int,             # 路径参数（URL 路径里有 {project_id}）
    verbose: bool = False,       # 查询参数（不在路径里，且有默认值）
    bid: BidData = ...,          # 请求体（类型是 Pydantic 模型）
):
    """
    完整请求示例:
      POST /projects/42/bid?verbose=true
      Body: {"control_price": 1200.5, "competitors": 8}

    FastAPI 的判断规则:
      1. 参数名出现在路径的 {} 里 → 路径参数
      2. 参数类型是 Pydantic 模型  → 请求体
      3. 其他                      → 查询参数
    """
    result = {"project_id": project_id, "bid_price": bid.control_price}
    if verbose:
        result["detail"] = "详细模式"
    return result
```

### 5.2 JSON 类型对照表

```
JSON 类型          Python 类型          示例
─────────────────────────────────────────────────

number (整数)     int                  42
number (小数)     float                3.14
boolean           bool                 true/false → True/False
null              None                 null → None
```

### 5.3 Python 中处理 JSON

```python
import json

# ==================== dict → JSON 字符串 ====================
data = {
    "project": "路面改造",
    "price": 1200.5,
    "competitors": ["公司A", "公司B"]
}

json_str = json.dumps(data, ensure_ascii=False, indent=2)
# ensure_ascii=False → 中文不会被转成 \uXXXX

print(parsed["project"])  # "路面改造"
print(type(parsed))       # <class 'dict'>
```

## 6. Python 装饰器

### 6.1 为什么要学

FastAPI 中定义接口全靠装饰器：

```python
@app.get("/")       # ← 这个 @ 开头的东西就是装饰器
def homepage():
    return {"msg": "hello"}
```

如果你不理解装饰器，你会不明白这个 `@app.get("/")` 到底干了什么。

### 6.2 装饰器的本质

装饰器就是**一个函数，它接收一个函数作为参数，返回一个新的函数**。它在不修改原函数代码的前提下，给函数"附加"了额外功能。

```python
import time

# ==================== 一个简单的计时装饰器 ====================
def timer(func):
    """装饰器：测量函数执行时间"""
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)           # 执行原函数
        end = time.time()
        print(f"{func.__name__} 耗时 {end - start:.4f} 秒")
        return result
    return wrapper


# 使用装饰器
@timer
def slow_function():
    """一个很慢的函数"""
    time.sleep(1)
    return "done"


slow_function()
# 输出: slow_function 耗时 1.0012 秒
```

**`@timer` 本质上等价于**：

```python
def slow_function():
    time.sleep(1)
    return "done"

slow_function = timer(slow_function)  # 手动装饰
```

### 6.3 带参数的装饰器

FastAPI 的装饰器 `@app.get("/path")` 是带参数的，理解一下它的结构：

```python
# @app.get("/") 拆解：
# 1. app.get("/") 被调用，返回一个装饰器函数
# 2. 这个装饰器函数接收 homepage 函数
# 3. 装饰器把 homepage 注册到 "/" 路径上

# 等价于：
decorator = app.get("/")       # 先调用 app.get("/"），拿到装饰器
homepage = decorator(homepage)  # 再用装饰器装饰函数
```

不需要自己写带参数的装饰器，只需要理解 FastAPI 的装饰器在"注册路由"——告诉框架"这个路径对应这个函数"

## 7. async/await 异步基础

### 7.1 为什么需要异步

想象一个场景：你的 API 要查数据库，数据库需要 0.5 秒才能返回。

- **同步模式**：在这 0.5 秒里，服务器干等着，什么都不干。如果同时来了 100 个请求，就要排队。
- **异步模式**：在等数据库的 0.5 秒里，服务器去处理其他请求。数据库返回了再回来继续处理。

异步不会让单个请求变快，但能让服务器**同时处理更多请求**。

### 7.2 同步 vs 异步代码对比

```python
import asyncio
import time

# ==================== 同步版本 ====================
def sync_query_db():
    """模拟一次数据库查询（同步，阻塞 1 秒）"""
    time.sleep(1)
    return "数据"

def sync_main():
    """同步：顺序执行两次查询"""
    r1 = sync_query_db()  # 等 1 秒
    r2 = sync_query_db()  # 再等 1 秒
    # 总耗时: 2 秒


# ==================== 异步版本 ====================
async def async_query_db():
    """模拟一次数据库查询（异步，不阻塞）"""
    await asyncio.sleep(1)   # await = "让出控制权，等好了再回来"
    return "数据"

async def async_main():
    """异步：并发执行两次查询"""
    r1, r2 = await asyncio.gather(
        async_query_db(),    # 两个查询同时发出
        async_query_db()
    )
    # 总耗时: 1 秒（并发执行）
```

### 7.3 在 FastAPI 中使用

FastAPI 同时支持同步和异步写法，你可以根据需要选择：

```python
from fastapi import FastAPI

app = FastAPI()


# ========== 同步写法（完全可以用，FastAPI 会自动处理） ==========
@app.get("/sync")
def sync_endpoint():
    """普通函数，FastAPI 会放到线程池里执行，不会阻塞其他请求"""
    return {"mode": "sync"}


# ========== 异步写法（推荐，但不是必须） ==========
@app.get("/async")
async def async_endpoint():
    """异步函数，适合有 IO 等待的场景（数据库、网络请求等）"""
    # 注意：await 只能在 async def 函数里用
    # await some_async_database_query()
    return {"mode": "async"}
```

**初学建议**：先用同步写法，完全不影响学习和使用。等你对 FastAPI 熟悉了，需要优化性能的时候再换成异步。

### 7.4 关键概念速记

```
async def    → 定义异步函数（协程）
await        → "等这个操作完成，期间去干别的"
              只能在 async def 函数里使用
asyncio      → Python 内置的异步库

记住一句话：
  async def 定义的函数不会自动变快，
  它只是告诉 Python "这个函数里有等待的地方，等的时候可以去干别的"。
```

## 8. REST API 设计风格

### 8.1 REST 是什么

REST（Representational State Transfer）不是一个技术或框架，而是一种**设计风格/约定**。它规定了 API 应该怎么设计才"规范"。

核心思想：**用 URL 表示资源，用 HTTP 方法表示操作**。

### 8.2 RESTful URL 设计

```
推荐（RESTful）                    不推荐
─────────────────────────────────────────────────────
GET    /projects                   GET    /getProjectList
GET    /projects/42                GET    /getProjectById?id=42
POST   /projects                   POST   /createProject
PUT    /projects/42                POST   /updateProject
DELETE /projects/42                POST   /deleteProject?id=42
```

设计原则：

```
1. URL 用名词（/projects），不用动词（/getProjects）
   → 动作由 HTTP 方法表达（GET/POST/PUT/DELETE）

2. 用复数形式（/projects 而不是 /project）

3. 层级关系用嵌套路径表示
   → /users/5/projects      （5号用户的所有项目）
   → /users/5/projects/42   （5号用户的42号项目）

4. 过滤、排序、分页用查询参数
   → /projects?status=active&sort=date&page=2
```
