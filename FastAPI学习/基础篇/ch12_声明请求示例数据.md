前面学了定义接口、校验数据、嵌套模型。但当你打开 `/docs` 页面，别人看到你的接口时，他可能会问："这个接口到底应该传什么格式的 JSON？"

光看字段名和类型有时不够直观，**一个具体的示例比十行文字说明都管用。**这一节讲的就是：**怎么给你的接口加上示例数据，让文档里直接展示"照着这个格式传就对了"。**

---

## 1. 为什么需要示例数据

### 1.1 没有示例时

打开 `/docs`，用户看到的是：

```css
name: string (required)
description: string or null
price: number (required)
tax: number or null
```

他知道每个字段的类型，但不知道具体传什么值。`price` 传 42 还是 0.01？`name` 传中文还是英文

### 1.2 有示例时

文档里直接显示：

```json
{
    "name": "南昌路面改造项目",
    "description": "市政道路修缮工程",
    "price": 1200.5,
    "tax": 72.03
}
```

用户一看就懂，复制过去改改就能用。示例数据是最好的文档

---

## 2. 三种添加示例的方式

FastAPI 提供了三种方式，适用于不同场景。先给一个总览，再逐个讲解：

```css
方式                                    用在哪                  效果
─────────────────────────────────────────────────────────────────────
1. model_config + json_schema_extra   模型类内部        给整个模型加一组完整示例
2. Field(examples=[...])              模型字段上        给单个字段加示例值
3. Body(examples=[...])               函数参数上        给请求体参数加示例
   Body(openapi_examples={...})       函数参数上        给请求体加多个带描述的示例
```

想象你有一个 `Item` 模型，被两个接口使用：

```python
class Item(BaseModel):
    name: str
    price: float


# 接口 A：创建商品
@app.post("/items/")
def create_item(item: Item): ...

# 接口 B：更新商品
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item): ...
```

现在你想加示例数据，问题来了：**示例加在哪里？**

```css
选项 1：加在 Item 模型上
  → 不管哪个接口用了 Item，都显示同一个示例
  → 相当于给"设计图纸"贴了一张样品照片

选项 2：加在 Item 的某个字段上（比如 price）
  → 只给 price 字段一个参考值，不管整体
  → 相当于给设计图纸上的某个尺寸标注了"比如 35.4"

选项 3：加在接口参数上
  → 接口 A 可以用"创建场景"的示例，接口 B 可以用"更新场景"的示例
  → 同一个模型，不同接口展示不同示例
  → 相当于给每个使用场景单独拍了样品照片
```

---

## 3. 方式一：model_config + json_schema_extra

给整个模型一个"标准示例"。不管这个模型被哪个接口使用，文档里都显示这个示例

### 3.1 完整代码

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()


class Item(BaseModel):
    """商品模型——示例直接写在模型里"""
    name: str
    price: float
    description: str | None = None

    # ↓↓↓ 关键：在模型内部通过 model_config 添加示例 ↓↓↓
    model_config = {
        "json_schema_extra": {
            "examples": [
                {
                    "name": "苹果",
                    "price": 5.5,
                    "description": "新鲜红富士",
                }
            ]
        }
    }


# 接口 A：创建商品 → 文档里显示上面的苹果示例
@app.post("/items/")
def create_item(item: Item):
    return item


# 接口 B：更新商品 → 文档里也显示上面的苹果示例（因为示例绑在模型上）
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item):
    return {"item_id": item_id, "item": item}
```

### 3.2 效果

打开 `/docs`，不管是"创建商品"还是"更新商品"，请求体输入框里预填的都是：

```json
{
    "name": "苹果",
    "price": 5.5,
    "description": "新鲜红富士"
}
```

### 3.3 语法拆解

```python
model_config = {                        # Pydantic 的模型配置（你在第七节见过它）
    "json_schema_extra": {              # "往 JSON Schema 里加点额外的东西"
        "examples": [                   # 示例列表
            {                           # 一个完整的示例对象
                "name": "苹果",         # 每个字段都给一个具体的值
                "price": 5.5,
                "description": "新鲜红富士",
            }
        ]
    }
}
```

`model_config` 你已经用过了——第七节用 `{"extra": "forbid"}` 禁止多传参数。现在多加了一个 `"json_schema_extra"` 键，往生成的文档 Schema 里塞示例数据

### 3.4 适用场景

- 模型比较简单，一个示例就能说清楚
- 这个模型在所有接口里的示例都一样
- 你想让示例和模型定义放在一起，好维护

### 3.5 局限

==**所有用到这个模型的接口，文档里显示的都是同一个示例。** 如果"创建商品"想展示一个完整示例，"更新商品"想展示一个只改价格的示例——做不到，因为示例绑在模型上，不区分接口==

---

## 4. 方式二：Field(examples=[...])

不给整个模型一个完整示例，而是给**每个字段单独**加一个参考值。文档里每个字段旁边会显示"比如：xxx"

### 4.1 语法

```python
from pydantic import BaseModel, Field


class Item(BaseModel):
    name: str = Field(examples=["Foo"])
    description: str | None = Field(default=None, examples=["A very nice Item"])
    price: float = Field(examples=[35.4])
    tax: float | None = Field(default=None, examples=[3.2])
```

### 4.2 和方式一的区别

```
方式一（model_config）：给整个模型一个完整的示例 JSON
  → 示例是一个完整的对象 {"name": "Foo", "price": 35.4, ...}

方式二（Field examples）：给每个字段单独加示例值
  → 每个字段自己有自己的示例值
```

**方式二更细粒度**——每个字段独立设置。适合你想让文档里每个字段旁边都显示一个参考值的场景

### 4.3 注意 examples 是列表

```python
# examples 参数接收一个列表，即使只有一个示例也要用 [...]
name: str = Field(examples=["Foo"])          # ✅ 列表
name: str = Field(examples="Foo")            # ❌ 不是列表，会出问题
```

---

## 5. 方式三：Body(examples=[...])

**同一个模型在不同接口里展示不同的示例。** 示例不绑定在模型上，而是绑定在具体的接口参数上

### 5.1 语法

```python
from typing import Annotated
from fastapi import Body

@app.put("/items/{item_id}")
def update_item(
    item_id: int,
    item: Annotated[
        Item,
        Body(
            examples=[
                {
                    "name": "Foo",
                    "description": "A very nice Item",
                    "price": 35.4,
                    "tax": 3.2,
                }
            ],
        ),
    ],
):
    return {"item_id": item_id, "item": item}
```

### 5.2 和方式一的区别

```css
方式一（model_config）：示例绑定在模型上
  → 这个模型不管用在哪个接口，示例都一样

方式三（Body examples）：示例绑定在接口参数上
  → 同一个模型在不同接口里可以有不同的示例
```

**方式三更灵活**——同一个 `Item` 模型，在"创建"接口和"更新"接口里可以展示不同的示例

看这个场景：

```python
class Item(BaseModel):
    name: str
    price: float
    description: str | None = None


# 创建商品：应该传所有字段
@app.post("/items/")
def create_item(item: Item):
    # 文档里最好展示：{"name": "苹果", "price": 5.5, "description": "新鲜红富士"}
    ...

# 更新商品：可能只改价格
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item):
    # 文档里最好展示：{"name": "苹果", "price": 6.0, "description": null}
    ...
```

"创建"和"更新"是不同的使用场景，应该有不同的示例。如果用方式一把示例绑在模型上，两个接口只能显示同一个示例。**方式三就是为了解决这个问题——把示例绑在接口参数上

### 5.3 多个示例

```python
item: Annotated[
    Item,
    Body(
        examples=[
            {
                "name": "普通示例",
                "price": 35.4,
                "tax": 3.2,
            },
            {
                "name": "只有必填字段",
                "price": 100.0,
            },
            {
                "name": "会报错的示例",
                "price": "这不是数字",     # 故意写错，展示错误情况
            },
        ],
    ),
],
```

可以放多个示例，展示正常情况、最简情况、错误情况等

## 6. openapi_examples——带描述的多个示例（推荐用于复杂接口）

### 6.1 问题

`examples=[...]` 只能放示例数据本身，没法加标题和说明。如果你有三个示例，用户在文档里看到的就是三坨 JSON，不知道每个代表什么场景。

### 6.2 用 openapi_examples 给每个示例加描述

```python
item: Annotated[
    Item,
    Body(
        openapi_examples={
            "normal": {
                "summary": "正常示例",
                "description": "一个包含所有字段的标准请求",
                "value": {
                    "name": "Foo",
                    "description": "A very nice Item",
                    "price": 35.4,
                    "tax": 3.2,
                },
            },
            "minimal": {
                "summary": "最简示例",
                "description": "只传必填字段，其他用默认值",
                "value": {
                    "name": "Bar",
                    "price": 100.0,
                },
            },
            "invalid": {
                "summary": "错误示例",
                "description": "price 传了字符串，会返回 422 错误",
                "value": {
                    "name": "Baz",
                    "price": "thirty five point four",
                },
            },
        },
    ),
],
```

### 6.3 结构拆解

```python
openapi_examples={                    # 字典，不是列表
    "normal": {                       # 键名：示例的标识符（英文，不显示给用户）
        "summary": "正常示例",         # 简短标题（文档下拉菜单里显示这个）
        "description": "...",         # 详细描述（支持 Markdown）
        "value": {                    # 实际的示例数据
            "name": "Foo",
            "price": 35.4,
        },
    },
    "minimal": {                      # 第二个示例
        ...
    },
}
```

**注意和 `examples` 的区别**：

```
examples=[...]            → 列表，每个元素直接是示例数据
openapi_examples={...}    → 字典，每个值包含 summary + description + value
```

### 6.4 在文档 UI 里的效果

打开 `/docs`，在请求体区域会出现一个下拉菜单，列出所有示例的 `summary`。选择不同的示例，请求体输入框里的 JSON 会切换成对应的 `value`。用户可以直接选一个示例，改改就能发送请求。

### 6.5 openapi_examples 的每个字段

```python
{
    "summary": "简短标题",           # 显示在下拉菜单里
    "description": "详细描述",       # 选中后显示在下方，支持 Markdown
    "value": {...},                 # 实际的示例数据
    "externalValue": "https://...", # 外部 URL（替代 value，很少用）
}
```

`summary` 和 `value` 是最常用的。`description` 和 `externalValue` 可选。

## 7. 三种方式怎么选

```
场景                                          推荐方式
─────────────────────────────────────────────────────────────────────
模型比较简单，一个示例就够了                    model_config + json_schema_extra
想让文档里每个字段旁边都有参考值                Field(examples=[...])
同一个模型在不同接口需要不同示例                Body(examples=[...])
复杂接口，需要展示多种场景并附带说明            Body(openapi_examples={...})
```

实际项目中，**最常用的是 `model_config` 和 `openapi_examples`**。前者简单省事，后者功能最强。

它们可以组合使用——模型上用 `model_config` 设一个默认示例，具体接口上用 `openapi_examples` 覆盖或补充



用一个场景把三种方式放在一起，一目了然：

```python
from typing import Annotated
from fastapi import Body, FastAPI
from pydantic import BaseModel, Field

app = FastAPI()


# ==================== 方式一：示例加在模型上 ====================
# 特点：示例跟着模型走，所有用到这个模型的接口都显示同一个示例

class ItemV1(BaseModel):
    name: str
    price: float

    model_config = {
        "json_schema_extra": {
            "examples": [
                {"name": "苹果", "price": 5.5}
            ]
        }
    }


@app.post("/v1/items/")
def create_item_v1(item: ItemV1):
    """文档里显示的示例：{"name": "苹果", "price": 5.5}"""
    return item


@app.put("/v1/items/{item_id}")
def update_item_v1(item_id: int, item: ItemV1):
    """文档里显示的示例：也是 {"name": "苹果", "price": 5.5}（和上面一样）"""
    return {"item_id": item_id, "item": item}


# ==================== 方式二：示例加在字段上 ====================
# 特点：每个字段独立加参考值，不构成完整的示例对象

class ItemV2(BaseModel):
    name: str = Field(
        examples=["苹果", "香蕉"]    # name 的参考值
    )
    price: float = Field(
        gt=0,
        examples=[5.5, 3.0]          # price 的参考值
    )


@app.post("/v2/items/")
def create_item_v2(item: ItemV2):
    """文档里：name 字段旁边显示 "苹果"，price 字段旁边显示 5.5"""
    return item


# ==================== 方式三：示例加在接口参数上 ====================
# 特点：同一个模型在不同接口可以有不同的示例

class ItemV3(BaseModel):
    name: str
    price: float


@app.post("/v3/items/")
def create_item_v3(
    item: Annotated[
        ItemV3,
        Body(
            openapi_examples={
                "fruit": {
                    "summary": "水果示例",
                    "value": {"name": "苹果", "price": 5.5},
                },
            }
        ),
    ]
):
    """文档里显示：水果示例 {"name": "苹果", "price": 5.5}"""
    return item


@app.put("/v3/items/{item_id}")
def update_item_v3(
    item_id: int,
    item: Annotated[
        ItemV3,
        Body(
            openapi_examples={
                "price_update": {
                    "summary": "只改价格",
                    "value": {"name": "苹果", "price": 6.0},  # 不同的示例！
                },
            }
        ),
    ],
):
    """文档里显示：只改价格 {"name": "苹果", "price": 6.0}"""
    return {"item_id": item_id, "item": item}
```

### 总结表

```css
方式                示例绑在谁身上     不同接口能用不同示例吗    写在哪里
──────────────────────────────────────────────────────────────────────────────
model_config        模型              ❌ 不能，所有接口一样     模型类内部
Field(examples)     单个字段          ❌ 不能，字段值固定       模型字段定义处
Body(examples)      接口参数          ✅ 能，每个接口独立       函数参数的 Annotated 里
Body(openapi_       接口参数          ✅ 能，还能加标题描述     函数参数的 Annotated 里
  examples)
```

## 8. Query、Path 也可以加示例

`examples` 和 `openapi_examples` 不仅能用在 `Body()` 上，`Query()`、`Path()`、`Header()`、`Cookie()` 也都支持：

```python
from fastapi import Query, Path

@app.get("/items/{item_id}")
def read_item(
    item_id: Annotated[int, Path(examples=[42, 100, 7])],
    q: Annotated[str | None, Query(examples=["搜索关键词", "路面改造"])] = None,
):
    ...
```

不过查询参数和路径参数通常比较简单，加不加示例影响不大。**示例最有价值的地方是复杂的请求体**

## 9. 对应到你的投标项目

```python
from typing import Annotated
from fastapi import FastAPI, Path, Body
from pydantic import BaseModel, Field, HttpUrl

app = FastAPI(title="智能投标决策系统")


# ==================== 方式一：model_config 给模型加默认示例 ====================

class CompetitorInfo(BaseModel):
    company_name: str = Field(min_length=1, max_length=100)
    estimated_price: float | None = Field(default=None, gt=0)
    strength: str = Field(default="unknown")

    model_config = {
        "json_schema_extra": {
            "examples": [
                {
                    "company_name": "XX建设集团",
                    "estimated_price": 1150.0,
                    "strength": "strong",
                }
            ]
        }
    }


# ==================== 方式二：Field(examples) 给每个字段加示例 ====================

class BidPredictRequest(BaseModel):
    control_price: float = Field(
        gt=0,
        description="控制价，单位：万元",
        examples=[1200.5, 850.0, 2300.0]
    )
    num_competitors: int = Field(
        ge=2, le=100,
        description="预估竞争对手数量",
        examples=[8, 15, 5]
    )
    strategy: str = Field(
        default="moderate",
        description="投标策略",
        examples=["aggressive", "moderate", "conservative"]
    )
    competitors: list[CompetitorInfo] = Field(
        default=[],
        description="已知的竞争对手信息"
    )


# ==================== 方式三：openapi_examples 在接口上展示多个带描述的场景 ====================

@app.post("/api/projects/{project_id}/predict")
def predict_bid(
    project_id: Annotated[int, Path(ge=1, examples=[42, 101])],
    request: Annotated[
        BidPredictRequest,
        Body(
            openapi_examples={
                "aggressive": {
                    "summary": "激进策略",
                    "description": "竞争对手多、控制价高的大项目，采用激进低价策略",
                    "value": {
                        "control_price": 2500.0,
                        "num_competitors": 15,
                        "strategy": "aggressive",
                        "competitors": [
                            {
                                "company_name": "XX建设集团",
                                "estimated_price": 2400.0,
                                "strength": "strong",
                            },
                            {
                                "company_name": "YY工程公司",
                                "estimated_price": 2350.0,
                                "strength": "medium",
                            },
                        ],
                    },
                },
                "moderate": {
                    "summary": "稳健策略",
                    "description": "中等规模项目，竞争适中，采用稳健报价",
                    "value": {
                        "control_price": 1200.5,
                        "num_competitors": 8,
                        "strategy": "moderate",
                        "competitors": [],
                    },
                },
                "minimal": {
                    "summary": "最简请求",
                    "description": "只传必填字段，其余用默认值",
                    "value": {
                        "control_price": 800.0,
                        "num_competitors": 3,
                    },
                },
            },
        ),
    ],
):
    """
    根据投标参数预测最优报价。

    在文档页面的请求体区域，可以从下拉菜单中选择不同的示例场景，
    直接点击 "Try it out" 测试。
    """
    recommended = request.control_price * 0.95
    return {
        "project_id": project_id,
        "recommended_price": recommended,
        "strategy": request.strategy,
        "competitor_count": len(request.competitors),
    }
```

打开 `/docs` 页面，你会看到：

- 请求体区域有一个下拉菜单，列出"激进策略"、"稳健策略"、"最简请求"三个选项
- 选择不同选项，输入框里的 JSON 会自动切换
- 每个字段旁边有 `examples` 提供的参考值
- 用户可以直接点 "Try it out"，选一个示例，改改参数就能测试

## 10. 本节核心知识点速查

```css
方式                                    位置              格式              特点
──────────────────────────────────────────────────────────────────────────────────
model_config + json_schema_extra       模型类内部        examples: [...]   示例跟着模型走
Field(examples=[...])                  模型字段上        列表              给单个字段加参考值
Body(examples=[...])                   接口参数上        列表              给整个请求体加示例
Body(openapi_examples={...})           接口参数上        字典              多个带标题描述的示例
```

**本节最核心的一句话**：示例数据是最好的文档。简单场景用 `model_config` 或 `Field(examples)`，复杂接口用 `Body(openapi_examples={...})` 给每个场景加标题和描述，文档下拉菜单直接切换，用户选了就能测
