前面学了用 POST 创建数据，这一节学**修改已有的数据**(就是客户端传数据到后台要修改已有的数据，所以叫请求体更新)。修改数据有两种策略：

- **PUT（全量替换）**：客户端发来完整的新数据，直接覆盖旧数据
- **PATCH（部分更新）**：客户端只发来要改的字段，其他字段保持不变



PUT 简单但有陷阱（默认值会覆盖已有数据），PATCH 安全但流程多几步。这一节的核心就是讲清楚这两种策略的区别，以及 PATCH 的标准实现流程

---

## 1. 先看数据：三条已有记录

整节课都围绕这份数据展开，先看清楚每条记录有什么、缺什么：

```python
from pydantic import BaseModel


class Item(BaseModel):
    name: str | None = None
    description: str | None = None
    price: float | None = None
    tax: float = 10.5          # 注意：默认值是 10.5
    tags: list[str] = []       # 注意：默认值是空列表


# 数据库里已经有这三条数据
items = {
    "foo": {"name": "Foo", "price": 50.2},
    "bar": {"name": "Bar", "description": "The bartenders", "price": 62, "tax": 20.2},
    "baz": {"name": "Baz", "description": None, "price": 50.2, "tax": 10.5, "tags": []},
}
```

三条记录各自的特点——后面的例子全靠这些数据来理解：

```
foo：只存了 name 和 price
     没有 description（数据库里根本没这个字段）
     没有 tax（数据库里根本没这个字段，但模型默认值是 10.5）
     没有 tags（数据库里根本没这个字段）

bar：存了 name、description、price、tax
     tax = 20.2（被显式设置过，不是默认值 10.5）
     没有 tags

baz：存了所有字段
     description = None（显式设为 None）
     tax = 10.5（显式设为 10.5，恰好和默认值一样）
     tags = []（显式设为空列表，恰好和默认值一样）
```

**注意 bar 的 tax**：它是 `20.2`，不是默认值 `10.5`。这个值是我们下面演示 PUT 陷阱的关键

## 2. PUT 全量替换——简单但有严重陷阱

### 2.1 代码

```python
from fastapi import FastAPI
from fastapi.encoders import jsonable_encoder

app = FastAPI()


@app.put("/items/{item_id}", response_model=Item)
def update_item(item_id: str, item: Item):
    update_item_encoded = jsonable_encoder(item)
    items[item_id] = update_item_encoded       # 直接替换整条记录
    return update_item_encoded
```

逻辑非常简单：客户端发来什么，就用什么替换旧数据。不看旧数据，不做合并，直接覆盖

### 2.2 PUT 的陷阱：一步步走一遍你就明白了

**场景**：bar 原来的数据是这样的：

```python
# bar 的原始数据
{"name": "Bar", "description": "The bartenders", "price": 62, "tax": 20.2}
#                                                              ^^^^^^^^^^^
#                                                              tax 是 20.2
```

现在客户端想用 PUT 更新 bar，**只想改 name 和 price**，发来这个请求体：

```json
{
    "name": "Barz",
    "price": 3,
    "description": null
}
```

客户端没有传 `tax` 和 `tags`——因为客户端觉得"我不传就是不改"。但事情不是这样发展的

**第一步：FastAPI 收到请求体，用 Item 模型解析**

```python
# FastAPI 拿客户端发来的 JSON 去创建 Item 对象
item = Item(name="Barz", price=3, description=None)

# 但是！Item 模型里 tax 有默认值 10.5，tags 有默认值 []
# 客户端没传这两个字段，Pydantic 自动用默认值填充：
# item.name = "Barz"          ← 客户端传的
# item.description = None     ← 客户端传的
# item.price = 3.0            ← 客户端传的
# item.tax = 10.5             ← 默认值！不是客户端传的！
# item.tags = []              ← 默认值！不是客户端传的！
```

**第二步：jsonable_encoder 把模型转成字典**

```python
update_item_encoded = jsonable_encoder(item)
# {
#     "name": "Barz",
#     "description": null,
#     "price": 3.0,
#     "tax": 10.5,        ← 默认值 10.5，不是原来的 20.2
#     "tags": []           ← 默认值
# }
```

**第三步：直接覆盖旧数据**

```python
items["bar"] = update_item_encoded
# bar 的数据现在变成了：
# {
#     "name": "Barz",
#     "description": null,
#     "price": 3.0,
#     "tax": 10.5,        ← 原来是 20.2，现在变成 10.5 了！数据丢失！
#     "tags": []
# }
```

**灾难发生了**：bar 的 `tax` 原来是 `20.2`，现在被默认值 `10.5` 覆盖了。客户端根本没想改 `tax`，但 `tax` 还是被改了

### 2.3 用图来理解这个过程

```css
客户端发来的 JSON          Pydantic 解析后（填充了默认值）     数据库里的旧数据
─────────────────         ──────────────────────────         ──────────────────
name: "Barz"              name: "Barz"                       name: "Bar"
description: null         description: null                  description: "The bartenders"
price: 3                  price: 3.0                         price: 62
（没传 tax）               tax: 10.5  ← 默认值自动填充        tax: 20.2  ← 原始值
（没传 tags）              tags: []   ← 默认值自动填充        tags: （没有）


PUT 直接用中间列替换右边列 → tax 从 20.2 变成 10.5，数据丢失
```

### 2.4 PUT 的正确用法

使用 PUT 时，**客户端有责任发来所有字段的完整数据**：

```json
{
    "name": "Barz",
    "description": null,
    "price": 3,
    "tax": 20.2,
    "tags": []
}
```

这样 `tax` 就不会丢了。PUT 的设计初衷就是"给你一个完整的新版本，你替换掉旧的"，缺了字段就用默认值——这是**符合规范的行为**，但在实际项目中非常容易导致 bug

## 3. PATCH 部分更新——只改传了的字段

### 3.1 PATCH 要解决的核心问题

客户端发来 `{"name": "Barz", "price": 3}`，`tax` 字段不在 JSON 里。

我们需要区分两种情况：

```
情况 A：客户端没传 tax → 意思是"不改 tax"，保持原值 20.2
情况 B：客户端显式传了 tax: 10.5 → 意思是"把 tax 改成 10.5"
```

但 Pydantic 解析完之后，两种情况看起来一模一样——`item.tax` 都是 `10.5`

怎么区分？**靠 Pydantic 内部的跟踪机制**。Pydantic 会记录哪些字段是客户端显式传的，哪些是自动填的默认值。`model_dump(exclude_unset=True)` 就能把"自动填的"排除掉

### 3.2 深入理解 `exclude_unset=True`

```python
# 客户端发来：{"name": "Barz", "price": 3}
# Pydantic 解析后：
item = Item(name="Barz", price=3)
# 内部状态：
#   item.name = "Barz"         → 客户端显式传的（set）
#   item.description = None    → 默认值，没传（unset）
#   item.price = 3.0           → 客户端显式传的（set）
#   item.tax = 10.5            → 默认值，没传（unset）
#   item.tags = []             → 默认值，没传（unset）
```

用不同方式 dump 看看区别：

```python
# 不加 exclude_unset → 所有字段都出来，分不清哪些是客户端传的
item.model_dump()
# {
#     "name": "Barz",          ← 客户端传的
#     "description": None,     ← 默认值（分不清！）
#     "price": 3.0,            ← 客户端传的
#     "tax": 10.5,             ← 默认值（分不清！）
#     "tags": []               ← 默认值（分不清！）
# }


# 加上 exclude_unset=True → 只保留客户端真正传了的字段
item.model_dump(exclude_unset=True)
# {
#     "name": "Barz",          ← 客户端传的
#     "price": 3.0             ← 客户端传的
# }
# description、tax、tags 全部被排除了——因为客户端没有显式传这些字段
```

**再看一个关键的对比场景**：如果客户端**显式传了** `tax: 10.5`（虽然值和默认值一样）：

```python
# 客户端发来：{"name": "Barz", "price": 3, "tax": 10.5}
item = Item(name="Barz", price=3, tax=10.5)

item.model_dump(exclude_unset=True)
# {
#     "name": "Barz",
#     "price": 3.0,
#     "tax": 10.5             ← 虽然和默认值一样，但客户端显式传了，所以保留！
# }
```

**核心理解**：`exclude_unset` 判断的不是"值是否等于默认值"，而是"客户端是否显式传了这个字段"。即使传的值恰好等于默认值，只要传了就算 set

### 3.3 PATCH 的五步标准流程

```python
@app.patch("/items/{item_id}", response_model=Item)
def update_item(item_id: str, item: Item):
    # 第一步：取出旧数据
    stored_item_data = items[item_id]

    # 第二步：把旧数据转成 Pydantic 模型
    stored_item_model = Item(**stored_item_data)

    # 第三步：从客户端数据里只提取"真正传了的字段"
    update_data = item.model_dump(exclude_unset=True)

    # 第四步：复制旧模型，用新数据覆盖对应字段
    updated_item = stored_item_model.model_copy(update=update_data)

    # 第五步：转成 JSON 兼容格式，存回数据库
    items[item_id] = jsonable_encoder(updated_item)
    return updated_item
```

### 3.4 用 bar 的真实数据逐步图解

**起始状态**：

```css
数据库中 bar 的数据：
  name = "Bar"
  description = "The bartenders"
  price = 62
  tax = 20.2               ← 这个值很重要，我们不希望它被改掉

客户端发来的 PATCH 请求体（只想改 name 和 price）：
  {"name": "Barz", "price": 3}
```

---

**第一步：取出旧数据**

```python
stored_item_data = items["bar"]
# 结果：{"name": "Bar", "description": "The bartenders", "price": 62, "tax": 20.2}
```

目的：我们需要旧数据，才能在上面做"修补"

---

**第二步：旧数据转成 Pydantic 模型**

```python
stored_item_model = Item(**stored_item_data)
# 等价于：Item(name="Bar", description="The bartenders", price=62, tax=20.2)
#
# 完整状态：
#   name = "Bar"
#   description = "The bartenders"
#   price = 62.0
#   tax = 20.2                    ← 从数据库来的，不是默认值
#   tags = []                     ← 数据库里没存 tags，用的模型默认值
```

目的：后面要用 `model_copy()` 方法来做合并，这个方法只有 Pydantic 模型才有，字典没有

---

**第三步：提取客户端真正传了的字段（灵魂步骤）**

```python
# 客户端发来的 item（Pydantic 自动解析后的完整模型）：
# Item(name="Barz", description=None, price=3.0, tax=10.5, tags=[])
#                    ~~~~~~~~ 默认值   ~~~~ 默认值  ~~ 默认值

update_data = item.model_dump(exclude_unset=True)
# 结果：{"name": "Barz", "price": 3.0}
#
# 只有 name 和 price！
# description、tax、tags 都被排除了——它们是默认值，不是客户端传的
```

目的：**过滤掉默认值**。如果不过滤，`tax: 10.5` 这个默认值就会在下一步把数据库里的 `20.2` 覆盖掉，和 PUT 的陷阱一模一样

---

**第四步：复制旧模型，合并新数据**

```python
updated_item = stored_item_model.model_copy(update=update_data)
```

`model_copy(update=...)` 的内部逻辑：

```css
旧模型 stored_item_model 的字段：
  name = "Bar"
  description = "The bartenders"
  price = 62.0
  tax = 20.2
  tags = []

update_data 的内容：
  {"name": "Barz", "price": 3.0}

合并规则：
  - update_data 里有的字段 → 用新值覆盖
  - update_data 里没有的字段 → 保持旧值不变

合并结果 updated_item：
  name = "Barz"                   ← 被覆盖（"Bar" → "Barz"）
  description = "The bartenders"  ← update_data 里没有，保持旧值
  price = 3.0                     ← 被覆盖（62.0 → 3.0）
  tax = 20.2                      ← update_data 里没有，保持旧值！！！
  tags = []                       ← update_data 里没有，保持旧值
```

**tax 保住了！** 因为第三步 `exclude_unset=True` 把 tax 从 update_data 里排除了，所以第四步合并时 tax 不在覆盖名单里，保持了原来的 `20.2`

---

**第五步：转成 JSON 兼容格式，存回数据库**

```python
items["bar"] = jsonable_encoder(updated_item)
# {
#     "name": "Barz",
#     "description": "The bartenders",
#     "price": 3.0,
#     "tax": 20.2,              ← 安全！保持了原值
#     "tags": []
# }
```

---

### 3.5 PUT vs PATCH 最终对比

同样的请求体 `{"name": "Barz", "price": 3}` 更新 bar（原 `tax=20.2`）：

```css
字段             PUT 的结果              PATCH 的结果
───────────────────────────────────────────────────────────
name            "Barz" ← 更新了        "Barz" ← 更新了
description     null ← 被默认值覆盖     "The bartenders" ← 保持原值
price           3.0 ← 更新了           3.0 ← 更新了
tax             10.5 ← 被默认值覆盖！   20.2 ← 保持原值！
tags            [] ← 被默认值覆盖       [] ← 保持原值
```

PUT 覆盖了 3 个不该覆盖的字段，PATCH 只改了客户端传的 2 个字段

## 4. 深入理解 `model_copy`

### 4.1 基本行为

```python
original = Item(name="Bar", description="The bartenders", price=62, tax=20.2)

# 不传 update → 纯粹复制，得到一个新对象
copy1 = original.model_copy()
# Item(name="Bar", description="The bartenders", price=62.0, tax=20.2, tags=[])
# copy1 和 original 的值一样，但是两个不同的对象（修改 copy1 不影响 original）

# 传 update → 复制 + 覆盖指定字段
copy2 = original.model_copy(update={"name": "Barz", "price": 3})
# Item(name="Barz", description="The bartenders", price=3.0, tax=20.2, tags=[])
#      ^^^^^^^^^^^                                  ^^^^^^^^^
#      被 update 覆盖                                被 update 覆盖
```

### 4.2 model_copy 不会修改原对象

```python
original = Item(name="Bar", price=62, tax=20.2)
updated = original.model_copy(update={"name": "Barz"})

print(original.name)   # "Bar"   ← 原对象没变
print(updated.name)    # "Barz"  ← 新对象才是修改后的
```

这是一个**不可变**的操作——返回新对象，不改旧对象。在并发场景下这很重要

### 4.3 Pydantic v1 vs v2 的方法名区别

```python
# Pydantic v1（旧版）
updated = original.copy(update={"name": "Barz"})

# Pydantic v2（新版，FastAPI 现在默认用的）
updated = original.model_copy(update={"name": "Barz"})
```

方法名从 `copy` 改成了 `model_copy`，功能一样。网上老教程用 `copy()`，新项目用 `model_copy()`

## 5. 为什么 PATCH 模型的字段都必须是可选的

### 5.1 问题

假设你的创建模型里 name 和 price 是必填的：

```python
class ItemCreate(BaseModel):
    name: str                          # 必填
    price: float                       # 必填
    description: str | None = None
    tax: float = 10.5
    tags: list[str] = []
```

如果 PATCH 接口也用这个模型，客户端只传 `{"name": "Barz"}` 会怎样？

```
客户端发来：{"name": "Barz"}
Pydantic 解析：price 是必填的但没传 → 422 验证错误！
```

客户端只想改个名字，却因为没传 price 被拒绝了——这就不是"部分更新"了

### 5.2 解决方案：创建和更新用不同的模型

```python
# 创建模型：name 和 price 必填
class ItemCreate(BaseModel):
    name: str                          # 必填！创建时不能省
    price: float                       # 必填！创建时不能省
    description: str | None = None
    tax: float = 10.5
    tags: list[str] = []


# 更新模型：所有字段都是 类型 | None = None
class ItemUpdate(BaseModel):
    name: str | None = None            # 可选
    price: float | None = None         # 可选
    description: str | None = None     # 可选
    tax: float | None = None           # 可选
    tags: list[str] | None = None      # 可选
```

### 5.3 为什么更新模型里 tax 的默认值要从 10.5 改成 None

创建模型里是 `tax: float = 10.5`，但更新模型里改成了 `tax: float | None = None`。为什么不继续用 `10.5`？

```python
# 如果更新模型还用 tax: float = 10.5
class ItemUpdateBad(BaseModel):
    name: str | None = None
    tax: float = 10.5              # 还是 10.5

# 功能上：exclude_unset 确实能正确排除没传的 tax ✓
# 但问题出在另一个场景：

# 如果客户端想把 tax 设为 null（清空税费）：
# 请求体：{"name": "Barz", "tax": null}
# Pydantic 报错：tax 的类型是 float，不接受 null ✗
```

把默认值改成 `None` 后，三种客户端意图都能正确表达：

```css
客户端不传 tax      → tax = None（默认值）→ exclude_unset 排除 → 不更新
客户端传 tax: 20.5  → tax = 20.5 → exclude_unset 保留 → 更新成 20.5
客户端传 tax: null  → tax = None → exclude_unset 保留 → 更新成 None（清空）
```

### 5.4 两个模型配合使用

```python
@app.post("/items/", status_code=201)
def create_item(item: ItemCreate):
    # name 和 price 必填，不传就 422
    ...

@app.patch("/items/{item_id}")
def update_item(item_id: str, item: ItemUpdate):
    # 所有字段可选，只传要改的
    update_data = item.model_dump(exclude_unset=True)
    ...
```

## 6. PATCH 的另一种实现：直接操作字典

官方文档用 `model_copy` 方式，但直接操作字典也很常见：

```python
@app.patch("/items/{item_id}", response_model=Item)
def update_item(item_id: str, item: ItemUpdate):
    # 取旧数据
    stored_data = items[item_id]

    # 提取客户端传了的字段
    update_data = item.model_dump(exclude_unset=True)

    # 直接在字典上更新
    for key, value in update_data.items():
        stored_data[key] = jsonable_encoder(value)

    # 存回去
    items[item_id] = stored_data
    return stored_data
```

两种方式对比：

```css
model_copy 方式（官方文档）：
  旧数据 → 转模型 → model_copy(update=...) → jsonable_encoder → 存回去
  优点：经过 Pydantic 再次验证，类型更安全
  缺点：步骤多

字典方式（更直接）：
  旧数据 → 遍历 update_data → 逐个覆盖键值 → 存回去
  优点：简单直观，步骤少
  缺点：不经过 Pydantic 再次验证（合并后的数据没被校验过）
```

两种都对，核心都是 `exclude_unset=True`。

## 7. 实际项目中怎么选 PUT / PATCH

```css
只用 PUT：
  适合简单项目，或前端总是发完整数据
  前端是自己团队开发，可以约定"PUT 必须传所有字段"

只用 PATCH：
  大部分场景更安全
  前端经常只想改一两个字段

两个都用（最规范）：
  PUT  → 完整替换（客户端必须传所有字段）
  PATCH → 部分更新（客户端只传要改的字段）

用 PUT 做部分更新：
  很多团队这么干——路由用 PUT，但内部用 exclude_unset 逻辑
  FastAPI 不强制你用哪个 HTTP 方法，怎么方便怎么来
```

## 8. 本节核心知识点速查

```css
概念                            做法
─────────────────────────────────────────────────────────────────────
PUT 全量替换                    @app.put(...)，直接用新数据覆盖
PATCH 部分更新                  @app.patch(...)，五步流程
只取客户端传了的字段             item.model_dump(exclude_unset=True)
复制模型并覆盖部分字段           old.model_copy(update={"field": value})
转 JSON 兼容格式                jsonable_encoder(model)
创建模型（必填字段）             name: str
更新模型（全可选字段）           name: str | None = None
```

**本节最核心的三句话**：

1. **PUT 的陷阱**：客户端没传的字段会被 Pydantic 的默认值填充，然后覆盖掉数据库里的原值。所以 PUT 要求客户端必须发完整数据——这是 PUT "替换"语义的本质

2. **PATCH 的灵魂是 `model_dump(exclude_unset=True)`**：它利用 Pydantic 内部的字段跟踪机制，区分"客户端显式传了的字段"和"模型自动填的默认值"，只保留前者。没有它，PATCH 就退化成了 PUT

3. **创建和更新要用不同的模型**：创建模型的关键字段是必填的（`name: str`），更新模型的所有字段都是 `类型 | None = None`。这样创建时不能缺字段，更新时可以只传要改的字段
