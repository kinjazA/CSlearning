上一节学了"输入模型和输出模型要分开"，用继承消除重复代码。这一节把这个思路往前推一步：**一个实体（比如"用户"）在整个生命周期中，可能需要 3 个甚至更多模型**



核心场景：

- 用户**注册时**传明文密码 → `UserIn`
- 用户信息**返回给前端**时不能有密码 → `UserOut`
- 用户**存进数据库**时要存哈希密码 → `UserInDB`

这一节讲：怎么优雅地管理这些相关但不同的模型，以及模型之间怎么转换数据

---

## 1. 问题：一个用户，三种形态

### 1.1 三种形态分别长什么样

```css
用户注册（UserIn）        返回给前端（UserOut）      存进数据库（UserInDB）
─────────────────       ─────────────────        ─────────────────
username: str           username: str             username: str
password: str  ← 明文    email: EmailStr           hashed_password: str  ← 哈希后的
email: EmailStr         full_name: str | None     email: EmailStr
full_name: str | None                             full_name: str | None
```

三种形态的区别只在于**密码字段**：

```css
UserIn    → 有 password（明文，用户输入的）
UserOut   → 没有任何密码字段（返回给前端，绝不能暴露）
UserInDB  → 有 hashed_password（哈希后的，存数据库用的）
```

### 1.2 笨办法：写三个完全独立的模型

```python
class UserIn(BaseModel):
    username: str           # 重复 ×3
    password: str
    email: EmailStr         # 重复 ×3
    full_name: str | None = None   # 重复 ×3


class UserOut(BaseModel):
    username: str           # 重复 ×3
    email: EmailStr         # 重复 ×3
    full_name: str | None = None   # 重复 ×3


class UserInDB(BaseModel):
    username: str           # 重复 ×3
    hashed_password: str
    email: EmailStr         # 重复 ×3
    full_name: str | None = None   # 重复 ×3
```

三个字段写了三遍。以后加一个 `avatar` 字段，要改三个地方，忘改一个就是 bug

## 2. 正确做法：用继承消除重复

### 2.1 提取公共基类

```python
from pydantic import BaseModel, EmailStr


# 基类：所有用户模型共有的字段
class UserBase(BaseModel):
    username: str
    email: EmailStr
    full_name: str | None = None


# 注册输入：基类 + 明文密码
class UserIn(UserBase):
    password: str


# 返回输出：和基类完全一样（没有任何密码）
class UserOut(UserBase):
    pass    # 不加任何字段，直接继承基类的所有字段


# 数据库存储：基类 + 哈希密码
class UserInDB(UserBase):
    hashed_password: str
```

### 2.2 继承关系图

```css
            UserBase（公共字段）
            ├── username: str
            ├── email: EmailStr
            └── full_name: str | None
                    │
         ┌──────────┼──────────┐
         ↓          ↓          ↓
      UserIn     UserOut    UserInDB
      + password   （无新字段）  + hashed_password
      （明文）                  （哈希后）
```

**公共字段只写一次**，每个子类只声明自己独有的字段。以后要加 `avatar`，只改 `UserBase` 一个地方，三个子类自动继承

### 2.3 `UserOut` 的 `pass` 是什么意思

```python
class UserOut(UserBase):
    pass
```

`pass` 是 Python 的"什么都不做"语句。`UserOut` 继承了 `UserBase` 的所有字段，自己不加任何新字段——它就是 `UserBase` 本身，但有一个独立的类名，语义更清晰。你可能会问：那为什么不直接用 `UserBase` 当输出模型？可以，但单独建一个 `UserOut` 有两个好处：

```css
1. 语义清晰：看到 UserOut 就知道是"给前端的输出"，看到 UserBase 不知道是干什么用的
2. 方便扩展：以后 UserOut 可能要加 created_at 等字段，有独立的类就不影响 UserBase
```

## 3. 模型之间的数据转换——`model_dump()` 和字典解包

这一节的另一个核心知识点：**怎么把一个模型的数据传给另一个模型**

### 3.1 `model_dump()`：把pydantic数据模型变成字典

```python
user_in = UserIn(
    username="alice",
    password="secret123",
    email="alice@example.com",
    full_name="Alice Wang"
)

# model_dump() 返回一个普通的 Python 字典
user_dict = user_in.model_dump()
print(user_dict)
# {
#     "username": "alice",
#     "password": "secret123",
#     "email": "alice@example.com",
#     "full_name": "Alice Wang"
# }
```

**`model_dump()` 是 Pydantic v2 的方法名**，v1 里叫 `dict()`。功能一样：把模型对象转成一个普通的 Python 字典

### 3.2 `**` 字典解包：把字典变回关键字参数

Python 的 `**` 操作符可以把字典"展开"成关键字参数：

```python
user_dict = {
    "username": "alice",
    "password": "secret123",
    "email": "alice@example.com",
    "full_name": "Alice Wang"
}

# 这两种写法完全等价：

# 写法一：手动传每个参数
UserInDB(
    username="alice",
    password="secret123",
    email="alice@example.com",
    full_name="Alice Wang",
    hashed_password="hashed_secret123"
)

# 写法二：用 ** 把字典展开，再加额外参数
UserInDB(**user_dict, hashed_password="hashed_secret123")
```

==`**user_dict` 相当于把字典里的每个键值对都拿出来，当作关键字参数传进去==

### 3.3 连起来用：从一个模型创建另一个模型

```python
def fake_password_hasher(raw_password: str) -> str:
    """模拟密码哈希（实际项目中用 bcrypt 等库）"""
    return "hashed_" + raw_password


def fake_save_user(user_in: UserIn) -> UserInDB:
    """模拟保存用户到数据库"""

    # 第一步：对明文密码做哈希
    hashed_password = fake_password_hasher(user_in.password)

    # 第二步：把 UserIn 的数据传给 UserInDB，同时加上 hashed_password
    user_in_db = UserInDB(
        **user_in.model_dump(),           # 展开 UserIn 的所有字段
        hashed_password=hashed_password   # 加上哈希密码
    )
    # 等价于：
    # UserInDB(
    #     username="alice",
    #     password="secret123",       ← UserInDB 没有这个字段，会被忽略
    #     email="alice@example.com",
    #     full_name="Alice Wang",
    #     hashed_password="hashed_secret123"
    # )

    return user_in_db
```

### 3.4 等一下，`password` 字段怎么处理的？

你可能注意到了：`user_in.model_dump()` 里有 `password` 字段，但 `UserInDB` 没有定义 `password` 字段。这不会报错吗？**默认不会报错。** Pydantic 默认会忽略模型里没定义的字段（除非你设了 `extra = "forbid"`）。所以 `password` 会被自动丢弃，只有 `UserInDB` 定义了的字段会被保留

完整的数据流：

```css
UserIn 的 model_dump() 输出：
{
    "username": "alice",
    "password": "secret123",        ← UserInDB 没定义，被丢弃
    "email": "alice@example.com",
    "full_name": "Alice Wang"
}

加上额外参数 hashed_password="hashed_secret123"

UserInDB 最终拿到：
{
    "username": "alice",            ← 来自 UserIn
    "email": "alice@example.com",   ← 来自 UserIn
    "full_name": "Alice Wang",      ← 来自 UserIn
    "hashed_password": "hashed_..."  ← 额外添加的
}
```

## 4. 完整流程：注册 → 存储 → 返回

把所有东西串起来：

```python
from fastapi import FastAPI
from pydantic import BaseModel, EmailStr

app = FastAPI()


# ==================== 模型定义 ====================

class UserBase(BaseModel):
    username: str
    email: EmailStr
    full_name: str | None = None

class UserIn(UserBase):
    password: str

class UserOut(UserBase):
    pass

class UserInDB(UserBase):
    hashed_password: str


# ==================== 工具函数 ====================

def fake_password_hasher(raw_password: str) -> str:
    return "hashed_" + raw_password

def fake_save_user(user_in: UserIn) -> UserInDB:
    hashed_password = fake_password_hasher(user_in.password)
    user_in_db = UserInDB(**user_in.model_dump(), hashed_password=hashed_password)
    print("User saved!")
    return user_in_db


# ==================== 接口 ====================

@app.post("/user/", response_model=UserOut)
def create_user(user_in: UserIn):
    user_saved = fake_save_user(user_in)
    return user_saved
```

完整的数据流：

```css
第一步：客户端发来注册数据
POST /user/
{
    "username": "alice",
    "password": "secret123",
    "email": "alice@example.com"
}

第二步：FastAPI 自动解析为 UserIn 对象
→ user_in = UserIn(username="alice", password="secret123", email="alice@example.com")

第三步：fake_save_user 处理
→ 密码哈希："secret123" → "hashed_secret123"
→ 创建 UserInDB：拿 UserIn 的数据 + hashed_password
→ UserInDB(username="alice", email="alice@example.com", hashed_password="hashed_secret123")

第四步：函数返回 UserInDB 对象（包含 hashed_password）

第五步：FastAPI 用 response_model=UserOut 做过滤
→ UserOut 只有 username、email、full_name
→ hashed_password 被过滤掉

第六步：客户端收到干净的响应
{
    "username": "alice",
    "email": "alice@example.com",
    "full_name": null
}
```

**全程没有密码泄露**——明文密码在哈希后就不再使用，哈希密码在返回时被 `response_model` 过滤掉

## 5. Union 类型：一个接口返回多种模型

### 5.1 场景

有时候同一个接口可能返回不同类型的数据。比如一个"商品"接口，可能返回汽车或飞机，它们有不同的字段：

```python
from typing import Union
from pydantic import BaseModel


class BaseItem(BaseModel):
    description: str
    type: str


class CarItem(BaseItem):
    type: str = "car"
    # 汽车没有额外字段


class PlaneItem(BaseItem):
    type: str = "plane"
    size: int              # 飞机有 size 字段，汽车没有
```

### 5.2 用 `Union` 声明"可能是 A 也可能是 B"

```python
items = {
    "item1": {"description": "经典老爷车", "type": "car"},
    "item2": {"description": "波音 737", "type": "plane", "size": 5},
}


@app.get("/items/{item_id}", response_model=Union[PlaneItem, CarItem])
def read_item(item_id: str):
    return items[item_id]
```

**`Union[PlaneItem, CarItem]`** 告诉 FastAPI：这个接口可能返回 `PlaneItem`，也可能返回 `CarItem`

### 5.3 顺序很重要：具体的类型放前面

```python
# ✅ 正确：更具体的 PlaneItem 放前面
response_model=Union[PlaneItem, CarItem]

# ❌ 不好：CarItem 放前面可能导致 PlaneItem 的数据被当成 CarItem 处理
response_model=Union[CarItem, PlaneItem]
```

为什么？因为 Pydantic 会按顺序尝试匹配。`PlaneItem` 比 `CarItem` 多一个 `size` 字段，更"具体"。如果把 `CarItem` 放前面，一条飞机数据可能会被 `CarItem` 先匹配上（因为 `CarItem` 的字段 `PlaneItem` 都有），`size` 字段就丢了

==**规则：字段多的（更具体的）放前面，字段少的（更宽泛的）放后面**==

### 5.4 `Union` 在装饰器参数里不能用 `|`

```python
# ✅ 在 response_model 参数里，用 Union[]
response_model=Union[PlaneItem, CarItem]

# ❌ 在 response_model 参数里，不能用 |
response_model=PlaneItem | CarItem    # 报错！Python 会把 | 当成位运算

# ✅ 在类型注解里，可以用 |（Python 3.10+）
def read_item(item_id: str) -> PlaneItem | CarItem:
    ...
```

**区别在于**：`->` 后面是类型注解，Python 知道 `|` 表示"或"；但 `response_model=` 后面是普通的赋值表达式，Python 会把 `|` 当成位运算符，报错

## 6. 返回列表和字典

### 6.1 返回模型列表

```python
class Item(BaseModel):
    name: str
    description: str


@app.get("/items/", response_model=list[Item])
def read_items():
    return [
        {"name": "Foo", "description": "第一个"},
        {"name": "Bar", "description": "第二个"},
    ]
```

`response_model=list[Item]` 表示返回的是一个 `Item` 列表。FastAPI 会验证列表里的每一项都符合 `Item` 模型

### 6.2 返回任意字典

有时候你事先不知道有哪些键——比如关键词和权重的映射：

```python
@app.get("/keyword-weights/", response_model=dict[str, float])
def read_keyword_weights():
    return {"foo": 2.3, "bar": 3.4}
```

`response_model=dict[str, float]` 表示返回的是一个字典，键是字符串，值是浮点数。不需要定义 Pydantic 模型——因为你不知道具体有哪些键

### 6.3 什么时候用 `dict[str, ...]` 而不是 Pydantic 模型

```css
用 Pydantic 模型：
  - 你明确知道有哪些字段（name、price、description 等）
  - 需要每个字段的类型验证
  - 需要在 /docs 里展示字段说明

用 dict[str, ...]：
  - 键是动态的，你事先不知道有哪些键
  - 比如：关键词权重、标签计数、动态配置项
```

## 7. `model_dump()` 的进阶用法

`model_dump()` 不只是简单地转字典，它还有很多实用的参数：

```python
user_in = UserIn(
    username="alice",
    password="secret123",
    email="alice@example.com",
    full_name=None
)

# 基本用法：转成字典
user_in.model_dump()
# {"username": "alice", "password": "secret123", "email": "alice@example.com", "full_name": None}

# 排除某些字段
user_in.model_dump(exclude={"password"})
# {"username": "alice", "email": "alice@example.com", "full_name": None}

# 只包含某些字段
user_in.model_dump(include={"username", "email"})
# {"username": "alice", "email": "alice@example.com"}

# 排除值为 None 的字段
user_in.model_dump(exclude_none=True)
# {"username": "alice", "password": "secret123", "email": "alice@example.com"}

# 排除没有被显式赋值的字段
user_in.model_dump(exclude_unset=True)
# 取决于创建时显式传了哪些字段
```

**结合模型转换**：

```python
def fake_save_user(user_in: UserIn) -> UserInDB:
    hashed_password = fake_password_hasher(user_in.password)

    # 用 exclude 排除明文密码，再加上哈希密码
    user_in_db = UserInDB(
        **user_in.model_dump(exclude={"password"}),
        hashed_password=hashed_password
    )
    return user_in_db
```

这种写法更明确——显式排除了 `password`，而不是依赖 Pydantic 默认忽略多余字段的行为

## 8. 对应到你的投标项目

### 8.1 投标系统的完整模型体系

```python
from datetime import datetime
from decimal import Decimal
from uuid import UUID, uuid4

from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI(title="智能投标决策系统")


# ==================== 用户模型体系 ====================

class UserBase(BaseModel):
    """用户公共字段"""
    username: str = Field(min_length=3, max_length=50)
    email: str
    full_name: str | None = None
    role: str = Field(default="viewer", description="admin / editor / viewer")


class UserCreate(UserBase):
    """用户注册——输入"""
    password: str = Field(min_length=8)


class UserPublic(UserBase):
    """用户信息——对外输出"""
    id: UUID
    created_at: datetime


class UserInDB(UserBase):
    """用户信息——数据库存储"""
    id: UUID
    hashed_password: str
    created_at: datetime


# ==================== 项目模型体系 ====================

class ProjectBase(BaseModel):
    """项目公共字段"""
    project_name: str = Field(min_length=1, max_length=200)
    location: str
    control_price: Decimal = Field(gt=0, description="控制价，万元")
    project_type: str = Field(description="市政 / 公路 / 房建")
    tags: set[str] = set()


class ProjectCreate(ProjectBase):
    """创建项目——输入（含内部敏感信息）"""
    cost_estimate: Decimal | None = Field(default=None, description="成本估算")
    internal_note: str | None = Field(default=None, description="内部备注")


class ProjectPublic(ProjectBase):
    """项目信息——对外输出（不含敏感信息）"""
    id: UUID
    created_at: datetime
    bid_count: int = Field(default=0, description="已收到的投标数量")


class ProjectInDB(ProjectBase):
    """项目信息——数据库存储（含所有信息）"""
    id: UUID
    cost_estimate: Decimal | None = None
    internal_note: str | None = None
    created_at: datetime


# ==================== 投标记录模型体系 ====================

class BidBase(BaseModel):
    """投标记录公共字段"""
    project_id: UUID
    bid_price: Decimal = Field(gt=0, description="投标报价，万元")
    strategy: str = Field(default="moderate", description="投标策略")


class BidCreate(BidBase):
    """创建投标——输入"""
    confidence_score: float | None = Field(
        default=None, ge=0, le=100,
        description="AI 预测的中标概率"
    )


class BidPublic(BidBase):
    """投标记录——对外输出"""
    id: UUID
    created_at: datetime
    won: bool | None = None
    rank: int | None = None


class BidInDB(BidBase):
    """投标记录——数据库存储"""
    id: UUID
    confidence_score: float | None = None
    created_at: datetime
    won: bool | None = None
    rank: int | None = None
    profit_margin: Decimal | None = Field(
        default=None, description="利润率（绝密）"
    )


# ==================== 接口：用户注册 ====================

# 模拟数据库
fake_users_db: dict[str, UserInDB] = {}


def fake_hash_password(raw: str) -> str:
    return "hashed_" + raw


@app.post("/api/users/", response_model=UserPublic)
def create_user(user_in: UserCreate):
    """
    数据流：
    UserCreate（含 password） → 哈希密码 → UserInDB（含 hashed_password）
    → 返回时按 UserPublic 过滤（不含任何密码）
    """
    now = datetime.now()
    user_id = uuid4()

    user_in_db = UserInDB(
        **user_in.model_dump(exclude={"password"}),
        id=user_id,
        hashed_password=fake_hash_password(user_in.password),
        created_at=now,
    )

    fake_users_db[user_in.username] = user_in_db
    return user_in_db    # 返回 UserInDB，FastAPI 按 UserPublic 过滤


# ==================== 接口：创建项目 ====================

fake_projects_db: dict[UUID, ProjectInDB] = {}


@app.post("/api/projects/", response_model=ProjectPublic)
def create_project(project_in: ProjectCreate):
    """
    数据流：
    ProjectCreate（含 cost_estimate、internal_note）
    → ProjectInDB（保存所有信息）
    → 返回时按 ProjectPublic 过滤（不含敏感信息）
    """
    now = datetime.now()
    project_id = uuid4()

    project_in_db = ProjectInDB(
        **project_in.model_dump(),
        id=project_id,
        created_at=now,
    )

    fake_projects_db[project_id] = project_in_db
    return project_in_db


# ==================== 接口：Union 类型——不同投标策略返回不同结构 ====================

from typing import Union


class AggressiveBidResult(BaseModel):
    """激进策略结果"""
    strategy: str = "aggressive"
    recommended_price: Decimal
    risk_level: str = "high"
    potential_profit: Decimal


class ConservativeBidResult(BaseModel):
    """保守策略结果"""
    strategy: str = "conservative"
    recommended_price: Decimal
    risk_level: str = "low"
    safety_margin: Decimal


@app.get(
    "/api/projects/{project_id}/recommend",
    response_model=Union[AggressiveBidResult, ConservativeBidResult],
)
def get_bid_recommendation(project_id: UUID, strategy: str = "conservative"):
    """
    根据策略返回不同结构的推荐结果。
    激进策略返回 potential_profit，保守策略返回 safety_margin。
    """
    if strategy == "aggressive":
        return AggressiveBidResult(
            recommended_price=Decimal("1050.00"),
            potential_profit=Decimal("150.00"),
        )
    else:
        return ConservativeBidResult(
            recommended_price=Decimal("1150.00"),
            safety_margin=Decimal("50.00"),
        )


# ==================== 接口：返回字典——关键词权重 ====================

@app.get("/api/keyword-weights/", response_model=dict[str, float])
def get_keyword_weights():
    """
    返回投标文件关键词的权重，键是动态的。
    """
    return {
        "技术方案": 3.5,
        "工期承诺": 2.8,
        "业绩经验": 4.2,
        "报价合理性": 5.0,
        "安全措施": 2.1,
    }
```

### 8.2 模型体系总览

```css
用户 User：
    UserBase（公共字段）
    ├── UserCreate   + password        ← 注册输入
    ├── UserPublic   + id, created_at  ← 对外输出
    └── UserInDB     + id, hashed_password, created_at  ← 数据库

项目 Project：
    ProjectBase（公共字段）
    ├── ProjectCreate  + cost_estimate, internal_note   ← 创建输入
    ├── ProjectPublic  + id, created_at, bid_count      ← 对外输出
    └── ProjectInDB    + id, cost_estimate, internal_note, created_at  ← 数据库

投标 Bid：
    BidBase（公共字段）
    ├── BidCreate    + confidence_score     ← 创建输入
    ├── BidPublic    + id, created_at, won, rank  ← 对外输出
    └── BidInDB      + 所有字段 + profit_margin   ← 数据库
```

**每个实体都是同一个模式**：Base → Create / Public / InDB。这是 FastAPI 项目中最常见的模型组织方式

## 9. 本节核心知识点速查

```css
概念                              做法
──────────────────────────────────────────────────────────────────
提取公共字段                      定义 Base 模型，其他模型继承它
输入模型                          Base + 用户提交的字段（password 等）
输出模型                          Base + 服务器生成的字段（id、created_at）
数据库模型                        Base + 所有需要持久化的字段（hashed_password）
模型转字典                        user.model_dump()
字典转模型                        UserInDB(**user_dict)
模型 A → 模型 B                   ModelB(**model_a.model_dump(), extra_field=value)
排除字段再转换                     model_a.model_dump(exclude={"password"})
返回多种类型                      response_model=Union[TypeA, TypeB]（具体的放前面）
返回列表                          response_model=list[Item]
返回任意字典                      response_model=dict[str, float]
```

**本节最核心的三句话**：

1. 一个实体在不同场景下需要不同的模型：输入模型（含密码）、输出模型（不含密码）、数据库模型（含哈希密码）。用继承提取公共字段，消除重复

2. `model_dump()` + `**` 字典解包是模型间数据传递的标准方式：`ModelB(**model_a.model_dump(), extra=value)` 可以把一个模型的数据传给另一个模型，同时添加额外字段

3. `Union[TypeA, TypeB]` 让一个接口可以返回多种类型的数据，字段多的（更具体的）类型要放前面
