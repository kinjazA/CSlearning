前面所有例子都是"正常情况"——数据合法、资源存在、用户有权限。但现实中，**出错才是常态**：ID 不存在、密码不对、没有权限、数据格式错误……



这一节讲的就是：**怎么在出错时返回规范的错误响应，而不是让服务器直接崩掉。**核心就一个东西：**`raise HTTPException`**——不是 return，是 raise。抛出异常后，函数立刻终止，FastAPI 自动把错误信息包装成 JSON 返回给客户端

---

## 1. 最基本的用法：`raise HTTPException`

### 1.1 语法

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

items = {"foo": "The Foo Wrestlers"}


@app.get("/items/{item_id}")
def read_item(item_id: str):
    if item_id not in items:
        raise HTTPException(status_code=404, detail="Item not found")
    return {"item": items[item_id]}
```

### 1.2 两种结果

请求存在的 item（`/items/foo`）→ 正常返回 200：

```json
{"item": "The Foo Wrestlers"}
```

请求不存在的 item（`/items/bar`）→ 返回 404 错误：

```json
{"detail": "Item not found"}
```

### 1.3 为什么是 `raise` 不是 `return`

```python
# ❌ 用 return 的问题
@app.get("/items/{item_id}")
def read_item(item_id: str):
    if item_id not in items:
        return {"detail": "Item not found"}   # 状态码还是 200！客户端以为成功了
    return {"item": items[item_id]}

# ✅ 用 raise 的好处
@app.get("/items/{item_id}")
def read_item(item_id: str):
    if item_id not in items:
        raise HTTPException(status_code=404, detail="Item not found")
        # 1. 状态码是 404，客户端明确知道出错了
        # 2. 函数立刻终止，后面的代码不会执行
        # 3. 即使在嵌套的工具函数里 raise，也能直接终止整个请求
    return {"item": items[item_id]}
```

`raise` 的三个好处：

```css
1. 状态码正确——客户端通过状态码（404/401/403）就能判断错误类型
2. 立刻终止——不需要写 else 或 return，后面的代码保证不会执行
3. 穿透调用栈——在任何嵌套函数里 raise，都能直接终止请求
```

### 1.4 第三个好处展开：穿透调用栈

```python
def get_item_from_db(item_id: str) -> dict:
    """工具函数：从数据库查数据"""
    item = db.get(item_id)
    if not item:
        raise HTTPException(status_code=404, detail=f"Item {item_id} not found")
        # 异常直接穿透到路径操作函数，整个请求终止
    return item


@app.get("/items/{item_id}")
def read_item(item_id: str):
    item = get_item_from_db(item_id)   # 如果 item 不存在，这行直接抛异常
    return {"item": item}              # 这行不会执行
```

不需要在路径操作函数里写 `if not item` 的判断——工具函数里已经处理了。这让代码更干净，错误处理逻辑可以集中管理

## 2. `HTTPException` 的三个参数

```python
raise HTTPException(
    status_code=404,                    # 必填：HTTP 状态码
    detail="Item not found",            # 必填：错误详情
    headers={"X-Error": "custom"},      # 可选：自定义响应头
)
```

### 2.1 `status_code`——告诉客户端错误类型

```python
# 资源不存在
raise HTTPException(status_code=404, detail="...")

# 没有登录
raise HTTPException(status_code=401, detail="...")

# 登录了但没权限
raise HTTPException(status_code=403, detail="...")

# 通用的客户端错误
raise HTTPException(status_code=400, detail="...")

# 数据冲突（比如用户名已存在）
raise HTTPException(status_code=409, detail="...")
```

### 2.2 `detail`——告诉客户端具体怎么了

`detail` 可以是任何能转成 JSON 的值，不只是字符串：

```python
# 字符串
raise HTTPException(status_code=404, detail="项目不存在")

# 字典——可以携带更多信息
raise HTTPException(status_code=400, detail={
    "error": "invalid_bid_price",
    "message": "投标报价不能高于控制价",
    "bid_price": 1500,
    "control_price": 1200,
})

# 列表——可以报告多个错误
raise HTTPException(status_code=400, detail=[
    {"field": "bid_price", "error": "不能为负数"},
    {"field": "project_name", "error": "不能为空"},
])
```

客户端收到的 JSON：

```json
{
    "detail": {
        "error": "invalid_bid_price",
        "message": "投标报价不能高于控制价",
        "bid_price": 1500,
        "control_price": 1200
    }
}
```

### 2.3 `headers`——添加自定义响应头

```python
raise HTTPException(
    status_code=401,
    detail="认证令牌已过期",
    headers={"WWW-Authenticate": "Bearer"},
    # WWW-Authenticate 告诉客户端需要用什么方式认证
)
```

大部分情况用不到 `headers`，知道有这个参数就行。主要在 OAuth2 认证场景会用

## 3. `HTTPException` vs FastAPI 的自动验证错误

FastAPI 里有两种错误：

```css
你手动抛的错误（HTTPException）：
  - 业务逻辑层面的错误
  - 资源不存在（404）、没有权限（403）、数据冲突（409）等
  - 你自己写 raise HTTPException(...)

FastAPI 自动抛的错误（RequestValidationError）：
  - 数据格式/类型层面的错误
  - 类型不对（传了字符串但需要整数）、约束不满足（min_length=3 但传了 2 个字符）
  - 你不需要写任何代码，FastAPI 自动返回 422
```

```python
@app.get("/items/{item_id}")
def read_item(item_id: int):          # 要求 int
    if item_id == 0:
        raise HTTPException(          # 你手动抛的：业务逻辑错误
            status_code=400,
            detail="ID 不能为 0"
        )
    return {"item_id": item_id}

# GET /items/abc → 422（FastAPI 自动的：abc 不是 int）
# GET /items/0   → 400（你手动抛的：ID 不能为 0）
# GET /items/42  → 200（正常）
```

## 4. 自定义异常处理程序

### 4.1 为什么需要

默认情况下，`HTTPException` 返回的是 JSON 格式的错误。但你可能想：

```css
- 统一错误格式（加上错误码、时间戳等）
- 记录错误日志
- 把 JSON 改成纯文本
- 发送错误通知
```

### 4.2 自定义异常 + 全局处理器

```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse


# 第一步：定义自定义异常
class BidException(Exception):
    def __init__(self, error_code: str, message: str, status_code: int = 400):
        self.error_code = error_code
        self.message = message
        self.status_code = status_code


app = FastAPI()


# 第二步：注册全局处理器
@app.exception_handler(BidException)
async def bid_exception_handler(request: Request, exc: BidException):
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "error_code": exc.error_code,
            "message": exc.message,
            "path": str(request.url),
        },
    )


# 第三步：在任何地方 raise 这个异常
@app.post("/api/bids/")
def create_bid(bid_price: float, control_price: float):
    if bid_price > control_price:
        raise BidException(
            error_code="BID_PRICE_TOO_HIGH",
            message=f"投标报价 {bid_price} 不能高于控制价 {control_price}",
        )
    return {"bid_price": bid_price}
```

客户端收到：

```json
{
    "error_code": "BID_PRICE_TOO_HIGH",
    "message": "投标报价 1500.0 不能高于控制价 1200.0",
    "path": "http://localhost:8000/api/bids/"
}
```

**好处**：整个项目的所有业务错误都走统一的格式（有 `error_code`、`message`、`path`），前端只需要一套错误处理逻辑

### 4.3 处理流程

```css
你的代码 raise BidException(...)
    ↓
FastAPI 捕获到这个异常
    ↓
查找：有没有注册过 BidException 的处理器？
    ↓
找到了 → 调用 bid_exception_handler(request, exc)
    ↓
处理器返回 JSONResponse → 发给客户端
```

## 5. 覆盖默认的异常处理程序

### 5.1 覆盖验证错误处理器

FastAPI 默认的 422 验证错误返回的 JSON 格式比较复杂。你可以改成更简洁的格式：

```python
from fastapi import FastAPI
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from fastapi.encoders import jsonable_encoder

app = FastAPI()


@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request, exc: RequestValidationError):
    """把默认的 422 错误格式改成更简洁的版本"""
    errors = []
    for error in exc.errors():
        errors.append({
            "field": " → ".join(str(loc) for loc in error["loc"]),
            "message": error["msg"],
        })
    return JSONResponse(
        status_code=422,
        content={
            "error_code": "VALIDATION_ERROR",
            "message": "请求数据验证失败",
            "errors": errors,
        },
    )
```

之前的默认格式：

```json
{
    "detail": [
        {
            "type": "int_parsing",
            "loc": ["path", "item_id"],
            "msg": "Input should be a valid integer...",
            "input": "abc",
            "url": "https://errors.pydantic.dev/..."
        }
    ]
}
```

改成你的格式：

```json
{
    "error_code": "VALIDATION_ERROR",
    "message": "请求数据验证失败",
    "errors": [
        {
            "field": "path → item_id",
            "message": "Input should be a valid integer..."
        }
    ]
}
```

### 5.2 覆盖 HTTPException 处理器

```python
from starlette.exceptions import HTTPException as StarletteHTTPException


@app.exception_handler(StarletteHTTPException)
async def http_exception_handler(request, exc):
    """统一所有 HTTPException 的格式"""
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "error_code": f"HTTP_{exc.status_code}",
            "message": exc.detail,
        },
    )
```

**注意**：覆盖 `HTTPException` 处理器时，要导入的是 **Starlette** 的 `HTTPException`，不是 FastAPI 的。因为 FastAPI 的继承自 Starlette 的，注册 Starlette 的才能捕获所有 HTTP 异常（包括 FastAPI 内部抛的和 Starlette 抛的）

```python
# ✅ 注册 Starlette 的 HTTPException（捕获范围更广）
from starlette.exceptions import HTTPException as StarletteHTTPException

@app.exception_handler(StarletteHTTPException)
async def handler(request, exc):
    ...

# 在代码中 raise 的还是 FastAPI 的 HTTPException（能传 JSON 作为 detail）
from fastapi import HTTPException

raise HTTPException(status_code=404, detail={"key": "value"})
```

### 5.3 重用默认处理器 + 加点自己的逻辑

最常见的需求：不改错误格式，只是想**在出错时打印日志**：

```python
from fastapi import FastAPI, HTTPException
from fastapi.exception_handlers import (
    http_exception_handler,
    request_validation_exception_handler,
)
from fastapi.exceptions import RequestValidationError
from starlette.exceptions import HTTPException as StarletteHTTPException

app = FastAPI()


@app.exception_handler(StarletteHTTPException)
async def custom_http_handler(request, exc):
    # 加一行日志
    print(f"[HTTP Error] {exc.status_code}: {exc.detail}")
    # 然后用 FastAPI 默认的处理器返回响应
    return await http_exception_handler(request, exc)


@app.exception_handler(RequestValidationError)
async def custom_validation_handler(request, exc):
    # 加一行日志
    print(f"[Validation Error] {exc.errors()}")
    # 然后用默认的处理器
    return await request_validation_exception_handler(request, exc)
```

**这是最实用的模式**——不改变客户端收到的错误格式（保持兼容），只是在服务端加上日志记录

## 6. 错误处理的最佳实践

### 6.1 错误格式统一

整个项目的所有错误响应应该有统一的 JSON 结构：

```json
{
    "error_code": "RESOURCE_NOT_FOUND",
    "message": "项目 xxx 不存在",
    "details": {}
}
```

**不要**一个接口返回 `{"detail": "..."}` 另一个返回 `{"error": "..."}` 又一个返回 `{"message": "..."}`——前端崩溃

### 6.2 错误码使用常量

```python
from fastapi import status

# ✅ 用常量
raise HTTPException(
    status_code=status.HTTP_404_NOT_FOUND,
    detail="项目不存在",
)

# ❌ 用魔法数字
raise HTTPException(status_code=404, detail="项目不存在")
```

### 6.3 错误信息要有用

```python
# ❌ 没用的错误信息
raise HTTPException(status_code=404, detail="Not found")

# ✅ 有用的错误信息——告诉客户端具体是什么不存在
raise HTTPException(
    status_code=404,
    detail=f"项目 {project_id} 不存在"
)

# ✅ 更好的——带上错误码，方便前端做不同处理
raise HTTPException(
    status_code=400,
    detail={
        "error_code": "BID_DEADLINE_PASSED",
        "message": f"项目 {project_name} 的投标截止时间已过",
        "deadline": str(deadline),
    }
)
```

## 7. 对应到你的投标项目

```python
from datetime import datetime
from decimal import Decimal
from uuid import UUID, uuid4
from typing import Annotated

from fastapi import FastAPI, HTTPException, Request, status
from fastapi.exception_handlers import http_exception_handler
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from starlette.exceptions import HTTPException as StarletteHTTPException
from pydantic import BaseModel, Field

app = FastAPI(title="智能投标决策系统")


# ==================== 统一错误格式 ====================

class ErrorResponse(BaseModel):
    """统一的错误响应格式"""
    error_code: str
    message: str
    details: dict | None = None
    timestamp: datetime


# ==================== 自定义业务异常 ====================

class BidSystemException(Exception):
    """投标系统业务异常基类"""
    def __init__(
        self,
        error_code: str,
        message: str,
        status_code: int = 400,
        details: dict | None = None,
    ):
        self.error_code = error_code
        self.message = message
        self.status_code = status_code
        self.details = details


# ==================== 全局异常处理器 ====================

@app.exception_handler(BidSystemException)
async def bid_system_exception_handler(request: Request, exc: BidSystemException):
    """处理所有业务异常"""
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "error_code": exc.error_code,
            "message": exc.message,
            "details": exc.details,
            "timestamp": datetime.now().isoformat(),
        },
    )


@app.exception_handler(StarletteHTTPException)
async def custom_http_exception_handler(request: Request, exc):
    """统一 HTTPException 的格式"""
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "error_code": f"HTTP_{exc.status_code}",
            "message": exc.detail if isinstance(exc.detail, str) else str(exc.detail),
            "details": exc.detail if isinstance(exc.detail, dict) else None,
            "timestamp": datetime.now().isoformat(),
        },
    )


@app.exception_handler(RequestValidationError)
async def custom_validation_handler(request: Request, exc: RequestValidationError):
    """统一验证错误的格式"""
    errors = []
    for error in exc.errors():
        errors.append({
            "field": " → ".join(str(loc) for loc in error["loc"]),
            "message": error["msg"],
        })
    return JSONResponse(
        status_code=422,
        content={
            "error_code": "VALIDATION_ERROR",
            "message": "请求数据验证失败",
            "details": {"errors": errors},
            "timestamp": datetime.now().isoformat(),
        },
    )


# ==================== 模型 ====================

class ProjectBase(BaseModel):
    project_name: str = Field(min_length=1, max_length=200)
    control_price: Decimal = Field(gt=0)
    bid_deadline: datetime


class BidCreate(BaseModel):
    bid_price: Decimal = Field(gt=0, description="投标报价，万元")
    strategy: str = "moderate"


# ==================== 模拟数据库 ====================

fake_projects = {
    "proj-001": {
        "id": "proj-001",
        "project_name": "南昌路面改造工程",
        "control_price": Decimal("1200.00"),
        "bid_deadline": datetime(2025, 12, 31, 17, 0, 0),
        "status": "open",
    },
    "proj-002": {
        "id": "proj-002",
        "project_name": "九江桥梁加固工程",
        "control_price": Decimal("850.00"),
        "bid_deadline": datetime(2024, 6, 1, 17, 0, 0),  # 已过期
        "status": "closed",
    },
}


# ==================== 工具函数 ====================

def get_project_or_404(project_id: str) -> dict:
    """获取项目，不存在则抛 404"""
    project = fake_projects.get(project_id)
    if not project:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail=f"项目 {project_id} 不存在",
        )
    return project


# ==================== 接口 ====================

@app.get("/api/projects/{project_id}")
def get_project(project_id: str):
    """
    查询项目

    可能的错误：
    - 404：项目不存在
    """
    project = get_project_or_404(project_id)
    return project


@app.post(
    "/api/projects/{project_id}/bids",
    status_code=status.HTTP_201_CREATED,
)
def submit_bid(project_id: str, bid: BidCreate):
    """
    提交投标

    可能的错误：
    - 404：项目不存在
    - BID_DEADLINE_PASSED：投标截止时间已过
    - BID_PRICE_EXCEEDS_CONTROL：报价超过控制价
    - PROJECT_CLOSED：项目已关闭
    - 422：数据验证失败（FastAPI 自动处理）
    """
    # 1. 项目是否存在
    project = get_project_or_404(project_id)

    # 2. 项目是否已关闭
    if project["status"] == "closed":
        raise BidSystemException(
            error_code="PROJECT_CLOSED",
            message=f"项目「{project['project_name']}」已关闭，不再接受投标",
            status_code=403,
        )

    # 3. 是否过了截止时间
    if datetime.now() > project["bid_deadline"]:
        raise BidSystemException(
            error_code="BID_DEADLINE_PASSED",
            message=f"项目「{project['project_name']}」的投标截止时间已过",
            status_code=400,
            details={
                "deadline": project["bid_deadline"].isoformat(),
                "now": datetime.now().isoformat(),
            },
        )

    # 4. 报价是否超过控制价
    if bid.bid_price > project["control_price"]:
        raise BidSystemException(
            error_code="BID_PRICE_EXCEEDS_CONTROL",
            message="投标报价不能高于控制价",
            status_code=400,
            details={
                "bid_price": float(bid.bid_price),
                "control_price": float(project["control_price"]),
            },
        )

    # 5. 一切正常，创建投标
    bid_id = str(uuid4())
    return {
        "bid_id": bid_id,
        "project_id": project_id,
        "bid_price": bid.bid_price,
        "strategy": bid.strategy,
        "submitted_at": datetime.now().isoformat(),
    }
```

## 8. 错误处理的层次总结

```css
层次              谁负责              示例
──────────────────────────────────────────────────────────────
数据格式/类型错误   FastAPI 自动处理    传了 "abc" 但需要 int → 422
业务逻辑错误       你手动 raise        资源不存在 → 404，没权限 → 403
意外错误           Python 自动         代码 bug → 500（不需要你处理）
```

```css
工具                    作用                      何时用
──────────────────────────────────────────────────────────────
HTTPException           抛出标准 HTTP 错误          最常用，适合 90% 场景
自定义异常 + handler     抛出业务异常，统一格式      项目规模较大时
覆盖默认 handler         改变 FastAPI 默认错误格式   需要统一全局错误格式时
重用默认 handler         在默认基础上加日志等逻辑    只想加日志，不改格式时
```

## 9. 本节核心知识点速查

```css
需求                          做法
──────────────────────────────────────────────────────────────────
资源不存在                    raise HTTPException(status_code=404, detail="...")
没有权限                      raise HTTPException(status_code=403, detail="...")
未认证                        raise HTTPException(status_code=401, detail="...")
数据冲突                      raise HTTPException(status_code=409, detail="...")
通用业务错误                  raise HTTPException(status_code=400, detail="...")
detail 传字典                 detail={"error_code": "...", "message": "..."}
自定义异常                    class MyException(Exception) + @app.exception_handler
覆盖 422 格式                 @app.exception_handler(RequestValidationError)
覆盖 HTTPException 格式       @app.exception_handler(StarletteHTTPException)
加日志不改格式                 自定义 handler 里调用默认 handler
```

**本节最核心的三句话**：

1. 用 `raise HTTPException` 而不是 `return` 来返回错误——它会设置正确的状态码、立刻终止函数执行、能穿透嵌套调用栈

2. `detail` 参数可以传任何能转 JSON 的值（字符串、字典、列表），不只是字符串。传字典可以携带错误码和额外信息，方便前端处理

3. 项目规模大了之后，用自定义异常 + 全局处理器统一错误格式：定义一个业务异常类（`BidSystemException`），注册全局处理器（`@app.exception_handler`），所有接口的错误响应自动保持一致的 JSON 结构
