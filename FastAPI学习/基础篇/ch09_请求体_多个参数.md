## 1. 请求体也可以是可选的

### 1.1 之前的写法：请求体必填

```python
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item):
    # item 是 Pydantic 模型，没有默认值 → 必填
    ...
```

用户必须在请求体里发 JSON，不发就报错

### 1.2 加上 `= None` 变成可选

```python
@app.put("/items/{item_id}")
def update_item(
    item_id: Annotated[int, Path(ge=0, le=1000)],
    q: str | None = None,
    item: Item | None = None,        # 请求体也可以是可选的
):
    results = {"item_id": item_id}
    if q:
        results.update({"q": q})
    if item:
        results.update({"item": item})
    return results
```

**关键变化**：`item: Item | None = None`

```python
item: Item | None = None
#      ↑      ↑      ↑
#   正常类型  可为None  默认None
```

这样用户可以不发请求体，`item` 就是 `None`。也可以发，FastAPI 照常校验。**和查询参数一样的规则**：有默认值（包括 `None`）就是可选的，没默认值就是必填的。这条规则对路径参数、查询参数、请求体**全部适用**

## 2. 多个请求体模型

### 2.1 问题：一个接口需要接收两种不同的数据

假设要更新一个项目，同时要传"项目信息"和"操作人信息"。它们是两类不同的数据，放在同一个模型里不合理

### 2.2 直接声明两个 Pydantic 模型参数

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()


class Item(BaseModel):
    name: str
    description: str | None = None
    price: float
    tax: float | None = None


class User(BaseModel):
    username: str
    full_name: str | None = None


@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item, user: User):
    return {"item_id": item_id, "item": item, "user": user}
```

FastAPI 看到两个参数都是 Pydantic 模型，就知道它们都从请求体取

### 2.3 JSON 结构的关键变化

**这是本节最重要的知识点。** 只有一个请求体模型和有多个请求体模型时，FastAPI 期望的 JSON 结构是不同的

**只有一个模型时**——JSON 就是模型本身，平铺的：

```python
# 只有一个请求体参数
def update_item(item_id: int, item: Item):
    ...
```

```json
{
    "name": "Foo",
    "price": 42.0,
    "tax": 3.2
}
```

JSON 的顶层直接就是 `Item` 的字段

**有两个模型时**——JSON 变成了嵌套结构，每个模型用参数名作为键：

```python
# 两个请求体参数
def update_item(item_id: int, item: Item, user: User):
    ...
```

```json
{
    "item": {
        "name": "Foo",
        "price": 42.0,
        "tax": 3.2
    },
    "user": {
        "username": "dave",
        "full_name": "Dave Grohl"
    }
}
```

外层多了 `"item"` 和 `"user"` 两个键，对应函数参数的名字。**FastAPI 自动用参数名把多个模型"包"了一层**

### 2.4 为什么要这样

如果两个模型的字段平铺在同一层，万一它们有同名字段（比如都有 `name`），FastAPI 就分不清这个 `name` 属于谁了。所以多个模型必须用键名隔开

### 2.5 图示对比

```css
一个模型时的 JSON 结构：

{
    "name": "...",        ← 直接就是 Item 的字段
    "price": 42.0,
    "tax": 3.2
}


两个模型时的 JSON 结构：

{
    "item": {             ← 用参数名 item 包了一层
        "name": "...",
        "price": 42.0
    },
    "user": {             ← 用参数名 user 包了一层
        "username": "..."
    }
}
```

## 3. Body()——把简单值也放进请求体

### 3.1 问题：除了两个模型，还想传一个简单的数字

```python
def update_item(item_id: int, item: Item, user: User, importance: int):
```

`importance` 是 `int` 类型（简单类型），按之前学的规则，FastAPI 会把它当成**查询参数**。但你想让它成为请求体的一部分

### 3.2 用 Body() 强制标记为请求体

```python
from fastapi import Body

@app.put("/items/{item_id}")
def update_item(
    item_id: int,
    item: Item,
    user: User,
    importance: Annotated[int, Body()],     # 强制标记为请求体
):
    return {
        "item_id": item_id,
        "item": item,
        "user": user,
        "importance": importance
    }
```

==`Annotated[int, Body()]` 告诉 FastAPI："虽然 `importance` 是简单类型，但别当查询参数，去请求体里找"==

### 3.3 此时 JSON 长什么样

```json
{
    "item": {
        "name": "Foo",
        "price": 42.0,
        "tax": 3.2
    },
    "user": {
        "username": "dave",
        "full_name": "Dave Grohl"
    },
    "importance": 5
}
```

`importance` 和两个模型并列，成为 JSON 顶层的第三个键

### 3.4 Body() 也支持约束

`Body()` 和 `Query()`、`Path()` 是一家人，支持同样的约束参数：

```python
importance: Annotated[int, Body(gt=0, le=10, title="重要程度")]
```

### 3.5 完整的参数判断规则

加上 `Body()` 之后，判断规则的完整版是这样的：

```css
FastAPI 看到一个函数参数，判断它从哪里取数据：

1. 参数名在路径 {} 里                → 路径参数
2. 参数类型是 Pydantic 模型          → 请求体
3. 参数被 Annotated[..., Body()] 标记 → 请求体（即使是简单类型）
4. 参数被 Annotated[..., Query()] 标记 → 查询参数（即使是复杂类型）
5. 以上都不是，且是简单类型            → 查询参数
```

**`Body()` 和 `Query()` 就是用来"推翻默认判断"的**。默认情况下简单类型是查询参数，但你可以用 `Body()` 强制把它塞进请求体。反过来，默认情况下 Pydantic 模型是请求体，但你可以用 `Query()` 强制从查询参数取

## 4. 多个请求体 + 查询参数混合

### 4.1 所有类型的参数同时存在

```python
@app.put("/items/{item_id}")
def update_item(
    *,     # py语法，* 之后的所有参数，在调用时必须以关键字参数（keyword argument）的形式传入
    item_id: int,                              # 路径参数
    item: Item,                                # 请求体（Pydantic 模型）
    user: User,                                # 请求体（Pydantic 模型）
    importance: Annotated[int, Body(gt=0)],    # 请求体（Body 强制标记）
    q: str | None = None,                      # 查询参数（简单类型）
):
    results = {
        "item_id": item_id,
        "item": item,
        "user": user,
        "importance": importance
    }
    if q:
        results.update({"q": q})
    return results
```

**完整的请求**：

```
PUT /items/42?q=搜索关键词
Content-Type: application/json

{
    "item": {
        "name": "Foo",
        "price": 42.0,
        "tax": 3.2
    },
    "user": {
        "username": "dave"
    },
    "importance": 5
}
```

**数据来源**：

```css
item_id = 42              ← 从 URL 路径取
q = "搜索关键词"           ← 从 URL 查询参数取
item = Item(...)           ← 从请求体 JSON 的 "item" 键取
user = User(...)           ← 从请求体 JSON 的 "user" 键取
importance = 5             ← 从请求体 JSON 的 "importance" 键取
```

### 4.2 简单类型不加 Body() 会怎样

```python
# 不加 Body()
def update_item(item_id: int, item: Item, user: User, importance: int):
#                                                       ↑
#                                          简单类型，被当成查询参数

# 请求变成这样：
# PUT /items/42?importance=5
# Body: {"item": {...}, "user": {...}}

# importance 从 URL 查询参数取，不在请求体里
```

加了 `Body()`：

```python
# 加了 Body()
def update_item(item_id: int, item: Item, user: User, importance: Annotated[int, Body()]):
#                                                                    ↑
#                                                           强制放进请求体

# 请求变成这样：
# PUT /items/42
# Body: {"item": {...}, "user": {...}, "importance": 5}

# importance 在请求体里
```

**区别就是：`importance` 这个值放在 URL 里还是放在 JSON 里。** 用 `Body()` 就放 JSON 里

## 5. embed=True——强制给单个模型加一层包装

### 5.1 问题

只有一个请求体模型时，JSON 是平铺的：

```python
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item):
    ...
```

```json
{
    "name": "Foo",
    "price": 42.0,
    "tax": 3.2
}
```

但有时候你希望 JSON 结构和多个模型时一致——也就是外面包一层键名：

```json
{
    "item": {
        "name": "Foo",
        "price": 42.0,
        "tax": 3.2
    }
}
```

### 5.2 用 Body(embed=True)

```python
@app.put("/items/{item_id}")
def update_item(
    item_id: int,
    item: Annotated[Item, Body(embed=True)],
):
    return {"item_id": item_id, "item": item}
```

`embed=True` 告诉 FastAPI："即使只有一个模型，也要用参数名包一层。"

### 5.3 对比

```python
# 不加 embed（默认行为）
item: Item

#  JSON：
{
    "name": "Foo",
    "price": 42.0
}


# 加了 embed=True
item: Annotated[Item, Body(embed=True)]

#  JSON：
{
    "item": {
        "name": "Foo",
        "price": 42.0
    }
}
```

### 5.4 什么时候用 embed

说实话，大部分时候不用。但有两个场景需要：

**场景一：前端约定好了 JSON 结构**。前端说"我发的数据外面一定有一层 key"，那你就加 `embed=True`

**场景二：为了和多模型接口保持一致**。你的系统里有些接口收两个模型（自动有嵌套），有些只收一个模型。如果你希望所有接口的 JSON 结构风格统一，就给单模型的也加上 `embed=True`

## 6. 总结：不同情况下 JSON 的结构变化

这是本节的核心，用一张表总结清楚：

```css
函数参数情况                                 期望的 JSON 结构
──────────────────────────────────────────────────────────────────

一个模型                                    直接是模型字段（平铺）
  item: Item                                {"name": "...", "price": 42}

一个模型 + embed=True                       用参数名包一层
  item: Annotated[Item, Body(embed=True)]   {"item": {"name": "...", "price": 42}}

两个模型                                    每个模型用参数名包一层（自动）
  item: Item, user: User                    {"item": {...}, "user": {...}}

两个模型 + 一个 Body 值                     三个键并列
  item: Item, user: User,                   {"item": {...}, "user": {...},
  importance: Annotated[int, Body()]          "importance": 5}
```

**记住这个规律**：只要请求体里有"多个东西"（多个模型、或者模型 + Body 值），FastAPI 就自动给每个东西加一层键名包装。只有一个模型时默认不包装，除非你用 `embed=True` 手动要求

## 7. 对应到你的投标项目

```python
from typing import Annotated
from fastapi import FastAPI, Path, Body
from pydantic import BaseModel, Field

app = FastAPI(title="智能投标决策系统")


class BidInfo(BaseModel):
    """投标信息"""
    control_price: float = Field(gt=0)
    num_competitors: int = Field(ge=1, le=100)
    strategy: str = "moderate"


class ProjectInfo(BaseModel):
    """项目信息"""
    project_name: str = Field(min_length=1, max_length=200)
    location: str | None = None
    category: str = "市政"


class OperatorInfo(BaseModel):
    """操作人信息"""
    username: str
    department: str | None = None


# ==================== 场景一：单个请求体模型（最常见） ====================
@app.post("/api/predict")
def predict_bid(bid: BidInfo):
    """
    JSON 结构（平铺）：
    {
        "control_price": 1200,
        "num_competitors": 8,
        "strategy": "aggressive"
    }
    """
    return {"recommended_price": bid.control_price * 0.95}


# ==================== 场景二：两个请求体模型 ====================
@app.post("/api/projects/{project_id}/bid")
def submit_bid(
    project_id: Annotated[int, Path(ge=1)], # 因为写在数据类模型里，就一定要用Annotated来追加限制条件
    project: ProjectInfo,
    bid: BidInfo,
):
    """
    JSON 结构（自动嵌套）：
    {
        "project": {
            "project_name": "南昌路面改造",
            "location": "南昌市",
            "category": "市政"
        },
        "bid": {
            "control_price": 1200,
            "num_competitors": 8,
            "strategy": "moderate"
        }
    }
    """
    return {
        "project_id": project_id,
        "project": project,
        "bid": bid,
    }


# ==================== 场景三：两个模型 + Body 值 + 查询参数 ====================
@app.put("/api/bids/{bid_id}")
def update_bid(
    bid_id: Annotated[int, Path(ge=1)],            # 路径参数
    bid: BidInfo,                                    # 请求体模型
    operator: OperatorInfo,                          # 请求体模型
    priority: Annotated[int, Body(ge=1, le=5)],     # 请求体简单值
    recalculate: bool = False,                       # 查询参数
):
    """
    PUT /api/bids/42?recalculate=true

    JSON 结构：
    {
        "bid": {
            "control_price": 1300,
            "num_competitors": 10,
            "strategy": "aggressive"
        },
        "operator": {
            "username": "zhangsan",
            "department": "投标部"
        },
        "priority": 3
    }

    数据来源：
    bid_id = 42              ← URL 路径
    recalculate = True       ← URL 查询参数
    bid = BidInfo(...)       ← 请求体 "bid" 键
    operator = OperatorInfo  ← 请求体 "operator" 键
    priority = 3             ← 请求体 "priority" 键
    """
    result = {
        "bid_id": bid_id,
        "bid": bid,
        "operator": operator,
        "priority": priority,
    }
    if recalculate:
        result["new_recommendation"] = bid.control_price * 0.95
    return result
```

## 8. 本节核心知识点速查

```css
概念                              写法                              效果
──────────────────────────────────────────────────────────────────────────
可选请求体          item: Item | None = None             不发请求体也行
多个请求体模型      item: Item, user: User               JSON 自动嵌套
简单值进请求体      Annotated[int, Body()]               强制从请求体取
单模型加包装        Annotated[Item, Body(embed=True)]    JSON 外面多一层键名
```

**参数判断规则完整版**：

```css
1. 名字在路径 {} 里                         → 路径参数
2. Pydantic 模型                            → 请求体
3. Annotated[..., Body()] 标记              → 请求体
4. Annotated[..., Query()] 标记             → 查询参数
5. 简单类型（int/str/bool/float），以上都不是 → 查询参数
```

**本节最核心的一句话**：一个模型时 JSON 平铺，多个模型时 JSON 自动嵌套（用参数名做键）。简单类型想放进请求体就用 `Body()` 标记。`Body()`、`Query()`、`Path()` 是同一家族，支持一样的约束参数
