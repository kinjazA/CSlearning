## 1. 为什么需要请求体

### 1.1 查询参数装不下复杂数据

回想一下查询参数的样子：

```css
GET /predict?control_price=1200&competitors=8&strategy=moderate
```

简单的几个参数还行。但如果你要传一个包含 10 条历史投标记录的列表呢？

```css
GET /predict?control_price=1200&competitors=8&history[0][name]=项目A&history[0][price]=980&history[0][won]=true&history[1][name]=项目B&history[1][price]=1100&history[1][won]=false&...
```

这根本不现实。URL 有长度限制，而且这样写也没法阅读和调试。**请求体就是来解决这个问题的**：把复杂数据打包成 JSON，放在 HTTP 请求的"身体"里发送

```css
POST /predict
Content-Type: application/json

{
    "control_price": 1200.5,
    "num_competitors": 8,
    "history": [
        {"project_name": "项目A", "bid_price": 980, "won": true},
        {"project_name": "项目B", "bid_price": 1100, "won": false}
    ]
}
```

整洁、清晰、没有长度限制，嵌套多深都可以

### 1.2 请求体 vs 响应体

这两个词经常一起出现，先分清楚：

```css
请求体（Request Body）：客户端 → 发给 → 服务器 的数据
  "用户提交的东西"

响应体（Response Body）：服务器 → 返回给 → 客户端 的数据
  "你返回的结果"
```

每个接口几乎都有响应体（总要返回点东西给用户）。但不是每个接口都需要请求体——==GET 请求通常就不需要，用户只是想看数据，不需要提交什么==

### 1.3 哪些 HTTP 方法用请求体

```css
POST    → 最常用请求体（创建数据、提交计算）
PUT     → 常用（整体更新数据）
PATCH   → 常用（部分更新数据）
DELETE  → 偶尔用（删除时附带条件）

GET     → 一般不用请求体
          技术上可以，但不推荐，很多工具和中间件不支持
```

## 2. 用 Pydantic 模型定义请求体

### 2.1 完整的代码示例

```python
from fastapi import FastAPI
from pydantic import BaseModel


# ==================== 第一步：定义数据模型 ====================
class Item(BaseModel):
    name: str                          # 必填，字符串
    description: str | None = None     # 可选，默认 None
    price: float                       # 必填，浮点数
    tax: float | None = None           # 可选，默认 None


app = FastAPI()


# ==================== 第二步：在接口中使用 ====================
@app.post("/items/")
def create_item(item: Item):
    return item
```

### 2.2 第一步拆解：定义数据模型

```python
class Item(BaseModel):
    name: str
    description: str | None = None
    price: float
    tax: float | None = None
```

定义了==客户端应该发什么样的 JSON 数据，放在函数的参数里，而响应体该符合什么数据，是在装饰器的参数里，这里是指在装饰器里用 response_model参数==：

```json
{
    "name": "Foo",              ← 必填（没有默认值）
    "description": "一段描述",   ← 可选（默认 None，不传也行）
    "price": 45.2,              ← 必填（没有默认值）
    "tax": 3.5                  ← 可选（默认 None，不传也行）
}
```

**必填 vs 可选的判断规则和查询参数一模一样**：有默认值就是可选的，没默认值就是必填的。所以下面这个 JSON 也是合法的（只传了必填字段）：

```json
{
    "name": "Foo",
    "price": 45.2
}
```

`description` 和 `tax` 会自动被填成 `None`

### 2.3 第二步拆解：在接口中使用

```python
@app.post("/items/")
def create_item(item: Item):
    return item
```

**关键就一行**：`item: Item`

```python
def create_item(item: Item):
#                 ↑     ↑
#              参数名  类型是 Pydantic 模型
```

FastAPI 看到参数类型是 Pydantic 模型（`Item`），就知道：**这个参数的数据要从请求体里取。**然后 FastAPI 自动做以下事情：

```css
1. 读取请求体中的 JSON 数据
2. 用 Item 模型的定义去校验每个字段
   - name 是不是字符串？✓
   - price 是不是数字？✓
   - 必填字段都传了吗？✓
3. 校验通过 → 创建一个 Item 对象，赋给 item 参数
4. 校验失败 → 自动返回 422 错误，告诉用户哪里不对
```

你在函数里拿到的 `item` 已经是一个校验过的、类型安全的 Python 对象。直接用 `item.name`、`item.price` 访问属性就行

### 2.4 FastAPI 怎么知道去请求体里找数据？

这个判断规则在第三节已经提过，现在加上请求体的完整版：

```python
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item, q: str | None = None):
#                  ↑             ↑            ↑
#               路径参数       请求体        查询参数
```

**FastAPI 的判断逻辑**：

```css
1. 参数名出现在路径的 {} 里     → 路径参数   （item_id）
2. 参数类型是 Pydantic 模型     → 请求体     （item）
3. 参数类型是简单类型（int/str/bool/float 等） → 查询参数 （q）
```

**就这三条规则，没有例外。** FastAPI 根据这个自动判断每个参数从哪里取数据，不需要做任何标记

## 3. 在函数里怎么用请求体数据

### 3.1 直接访问属性

```python
@app.post("/items/")
def create_item(item: Item):
    # item 是一个 Item 对象，用点号访问属性
    print(item.name)          # "Foo"
    print(item.price)         # 45.2
    print(item.description)   # None（如果用户没传）
    print(item.tax)           # 3.5（如果用户传了）
    return item
```

### 3.2 用 `.model_dump()` 转成字典

有时候需要把 Pydantic 对象转成普通字典来操作：

```python
@app.post("/items/")
def create_item(item: Item):
    # .model_dump() 把 Pydantic 对象转成 Python 字典
    item_dict = item.model_dump()
    print(item_dict)
    # {"name": "Foo", "description": None, "price": 45.2, "tax": 3.5}

    # 转成字典后就能用字典的方法了
    print(type(item_dict))  # <class 'dict'>

    return item_dict
```

> **注意**：老版本 Pydantic（v1）用的是 `.dict()`，新版本（v2）改成了 `.model_dump()`。FastAPI 现在用的是 Pydantic v2，所以用 `.model_dump()`

### 3.3 基于请求体数据做计算

```python
@app.post("/items/")
def create_item(item: Item):
    item_dict = item.model_dump()

    # 如果用户传了 tax，就计算含税价格
    if item.tax is not None:
        price_with_tax = item.price + item.tax
        item_dict["price_with_tax"] = price_with_tax
        # 注意：这里是往字典里加了一个新字段
        # 原始的 item 对象没有 price_with_tax 这个属性

    return item_dict
```

**请求**：

```json
{
    "name": "Foo",
    "price": 45.2,
    "tax": 3.5
}
```

**响应**（多了一个计算出来的字段）：

```json
{
    "name": "Foo",
    "description": null,
    "price": 45.2,
    "tax": 3.5,
    "price_with_tax": 48.7
}
```

### 3.4 为什么用 `if item.tax is not None` 而不是 `if item.tax`

和第三节讲查询参数时的道理一样：

```python
item.tax = 0.0    # 合法值，表示"免税"
item.tax = None   # 表示"用户没传这个字段"

if item.tax:           # 0.0 是 falsy → 被误判为"没传" → 错误！
if item.tax is not None:  # 只有 None 才算"没传" → 正确！
```

**凡是数值类型的可选字段，都用 `is not None` 来判断。** 这是个好习惯，能避免很多隐蔽的 bug

## 4. 请求体 + 路径参数

### 4.1 两者混合使用

```python
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item):
    """
    item_id: 从 URL 路径里取（路径参数）
    item:    从请求体里取（Pydantic 模型）
    """
    return {"item_id": item_id, **item.model_dump()}
```

**请求**：

```
PUT /items/42
Content-Type: application/json

{
    "name": "更新后的名称",
    "price": 99.9
}
```

**响应**：

```json
{
    "item_id": 42,
    "name": "更新后的名称",
    "description": null,
    "price": 99.9,
    "tax": null
}
```

### 4.2 `**item.model_dump()` 是什么语法

这是 Python 的**字典解包**语法，==后面的model_dump是把这个数据对象转成字典，然后\**再把字典拆开成键值对==

```python
# item.model_dump() 返回一个字典：
# {"name": "Foo", "description": None, "price": 99.9, "tax": None}

# ** 把字典"拆开"成键值对：
# name="Foo", description=None, price=99.9, tax=None

# 放在另一个字典里，就是合并：
result = {"item_id": 42, **item.model_dump()}
# 等价于：
result = {
    "item_id": 42,
    "name": "Foo",
    "description": None,
    "price": 99.9,
    "tax": None
}
```

这个写法比手动一个个字段拼要简洁很多

## 5. 请求体 + 路径参数 + 查询参数——三者同时用

### 5.1 完整示例

```python
@app.put("/items/{item_id}")
def update_item(item_id: int, item: Item, q: str | None = None):
    """
    三种参数同时存在，FastAPI 自动区分：
      item_id → 路径参数（在路径的 {} 里）
      item    → 请求体  （类型是 Pydantic 模型）
      q       → 查询参数（简单类型，不在路径里）
    """
    result = {"item_id": item_id, **item.model_dump()}
    if q:
        result["q"] = q
    return result
```

**完整的请求长这样**：

```css
PUT /items/42?q=搜索关键词
Content-Type: application/json

{
    "name": "更新后的名称",
    "price": 99.9,
    "tax": 5.0
}
```

**FastAPI 从三个不同的地方取数据**：

```css
URL 路径:    /items/42          → item_id = 42
URL 查询:    ?q=搜索关键词       → q = "搜索关键词"
请求体:      {"name": ...}      → item = Item(name="更新后的名称", ...)
```

### 5.2 完整的判断规则

```css
参数判断优先级：

1. 参数名在路径 {} 里        → 路径参数    例：item_id: int
2. 参数类型是 Pydantic 模型  → 请求体      例：item: Item
3. 参数类型是简单类型         → 查询参数    例：q: str | None = None

FastAPI 按这个顺序自动判断
```

这三条规则贯穿整个 FastAPI 开发

## 6. 请求体的自动校验

### 6.1 少了必填字段

```json
// 请求
{
    "name": "Foo"
}
// 少了必填字段 price
```

```json
// 响应：422 错误
{
    "detail": [
        {
            "type": "missing",
            "loc": ["body", "price"],
            "msg": "Field required",
            "input": {"name": "Foo"}
        }
    ]
}
```

注意 `"loc": ["body", "price"]`——告诉你错误在请求体的 `price` 字段上。

### 6.2 类型不对

```json
// 请求
{
    "name": "Foo",
    "price": "不是数字"
}
```

```json
// 响应：422 错误
{
    "detail": [
        {
            "type": "float_parsing",
            "loc": ["body", "price"],
            "msg": "Input should be a valid number",
            "input": "不是数字"
        }
    ]
}
```

### 6.3 多传了字段

```json
// 请求
{
    "name": "Foo",
    "price": 45.2,
    "color": "红色"
}
// 多传了 color 字段，模型里没定义
```

**默认情况下不会报错**。Pydantic 会忽略模型里没定义的字段。`item` 对象里不会有 `color` 属性，相当于这个字段白传了

### 6.4 和路径参数/查询参数的错误信息对比

```css
路径参数错误 → "loc": ["path", "item_id"]
查询参数错误 → "loc": ["query", "needy"]
请求体错误   → "loc": ["body", "price"]
```

`loc` 字段告诉你错误的数据来自哪里（path / query / body）以及具体是哪个字段。调试的时候非常方便

## 8. 常见疑问

### 8.1 GET 请求能不能带请求体？

技术上 FastAPI 支持，但**强烈不推荐**。原因是：

- HTTP 规范没有定义 GET 请求带请求体的行为
- 很多浏览器、代理服务器、CDN 会忽略或丢弃 GET 的请求体
- Swagger UI 文档页面不会显示 GET 请求的请求体

所以：需要接收复杂数据的接口，用 POST

### 8.2 请求体和返回值能用同一个模型吗？

可以，但不推荐。因为接收的数据和返回的数据往往不完全一样：

```python
# ❌ 不推荐：输入和输出用同一个模型
# 问题：用户提交时不需要传 id 和 created_at，但返回时需要有

# ✅ 推荐：分开定义
class BidCreate(BaseModel):
    """用户提交时的数据（没有 id）"""
    control_price: float
    num_competitors: int

class BidResponse(BaseModel):
    """返回给用户的数据（有 id 和计算结果）"""
    id: int
    control_price: float
    num_competitors: int
    recommended_price: float
    created_at: str
```

一般命名习惯是：`XxxCreate`（创建用）、`XxxUpdate`（更新用）、`XxxResponse`（返回用）。

### 8.3 `return item` 直接返回 Pydantic 对象可以吗？

可以。FastAPI 会自动把 Pydantic 对象转成 JSON：

```python
@app.post("/items/")
def create_item(item: Item):
    return item  # 直接返回 Pydantic 对象，自动转 JSON
```

返回字典也行：

```python
@app.post("/items/")
def create_item(item: Item):
    return item.model_dump()  # 返回字典，也会自动转 JSON
```

两种写法效果一样。直接返回 Pydantic 对象更简洁。

### 8.4 怎么测试带请求体的接口？

**方法一：用 Swagger UI（最方便）**

打开 `/docs`，找到你的 POST 接口，点 "Try it out"，编辑 JSON，点 "Execute"。

**方法二：用 curl**

```bash
curl -X POST http://127.0.0.1:8000/api/predict \
  -H "Content-Type: application/json" \
  -d '{
    "control_price": 1200.5,
    "num_competitors": 8,
    "strategy": "aggressive"
  }'
```

`-X POST` 指定方法，`-H` 设置请求头，`-d` 后面跟请求体。

**方法三：用 Python requests**

```python
import requests

response = requests.post(
    "http://127.0.0.1:8000/api/predict",
    json={
        "control_price": 1200.5,
        "num_competitors": 8,
        "strategy": "aggressive"
    }
)
print(response.json())
```

`json=` 参数会自动设置 `Content-Type: application/json` 并把字典转成 JSON。

## 9. 本节核心知识点速查

```css
概念                   写法                              从哪里取数据
──────────────────────────────────────────────────────────────────
路径参数             item_id: int（名字在路径 {} 里）    URL 路径
查询参数             q: str = None（简单类型）           URL ?后面
请求体               item: Item（Pydantic 模型）        HTTP 请求的 Body

判断规则（FastAPI 自动执行）：
  1. 名字在路径 {} 里            → 路径参数
  2. 类型是 Pydantic 模型        → 请求体
  3. 类型是简单类型（int/str等）  → 查询参数
```

**本节最核心的一句话**：函数参数的类型如果是 Pydantic 模型，FastAPI 就去请求体里找 JSON 数据。不需要手动解析 JSON、不需要手动校验字段、不需要手动转换类型，全部自动
