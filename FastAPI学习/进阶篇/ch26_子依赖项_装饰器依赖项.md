之前学依赖注入时，我们学了最基本的用法：一个函数依赖另一个函数，但真实项目里，依赖关系往往是**多层嵌套**的：

```
 路径操作函数
→ 依赖项 A
   → 子依赖项 B
        → 子子依赖项 C
```

比如：获取当前用户 → 需要先解析 Token → 需要先从请求头提取 Token。FastAPI 会自动帮你按正确的顺序解析整棵依赖树

另外还有一种场景：依赖项只是做验证（比如检查 Token 是否有效），你不需要它的返回值。这时候把它放在**装饰器的 dependencies 参数**里更干净

---

## 1. 子依赖项——依赖项也可以有自己的依赖项

### 1.1 什么是子依赖项

```css
普通依赖项：
  路径操作函数 → 依赖项 A

子依赖项：
  路径操作函数 → 依赖项 A → 依赖项 B（A 的子依赖项）

更深的嵌套：
  路径操作函数 → A → B → C → D → ……

FastAPI 会自动从最底层开始解析，逐层往上传递结果。
```

你不需要手动管理调用顺序。FastAPI 看到依赖树后，会自动按正确的顺序执行，把每一层的返回值传给上一层

### 1.2 示例逐行拆解

```python
from typing import Annotated
from fastapi import Cookie, Depends, FastAPI

app = FastAPI()


# ===== 第一层：最底层的依赖项 =====
def query_extractor(q: str | None = None):
    """从查询参数里提取 q"""
    return q


# ===== 第二层：依赖第一层的依赖项 =====
def query_or_cookie_extractor(
    q: Annotated[str, Depends(query_extractor)],       # ← 依赖 query_extractor
    last_query: Annotated[str | None, Cookie()] = None, # ← 从 Cookie 里读
):
    """优先用查询参数 q，没有就用 Cookie 里保存的上次查询"""
    if not q:
        return last_query
    return q


# ===== 路径操作函数：只声明第二层，不需要知道第一层的存在 =====
@app.get("/items/")
async def read_query(
    query_or_default: Annotated[str, Depends(query_or_cookie_extractor)],
):
    return {"q_or_cookie": query_or_default}
```

### 1.3 执行流程

```css
请求到达 GET /items/?q=hello

第一步：FastAPI 分析依赖树
  read_query
    → 依赖 query_or_cookie_extractor
        → 依赖 query_extractor

第二步：从最底层开始执行
  ① 执行 query_extractor(q="hello")
     → 返回 "hello"

  ② 执行 query_or_cookie_extractor(q="hello", last_query=None)
     → q 有值，返回 "hello"

  ③ 执行 read_query(query_or_default="hello")
     → 返回 {"q_or_cookie": "hello"}
```

再看一个没有查询参数但有 Cookie 的情况：

```css
请求到达 GET /items/
请求头里有 Cookie: last_query=previous_search

第一步：FastAPI 分析依赖树（同上）

第二步：从最底层开始执行
  ① 执行 query_extractor(q=None)
     → 返回 None

  ② 执行 query_or_cookie_extractor(q=None, last_query="previous_search")
     → q 没有值，返回 "previous_search"（Cookie 里的值）

  ③ 执行 read_query(query_or_default="previous_search")
     → 返回 {"q_or_cookie": "previous_search"}
```

### 1.4 关键理解

```css
1. 路径操作函数只需要声明它直接依赖的依赖项
   → read_query 只写了 Depends(query_or_cookie_extractor)
   → 它不需要知道 query_extractor 的存在
   → FastAPI 自动发现并解析整棵依赖树

2. 子依赖项的参数来源和普通依赖项完全一样
   → query_extractor 的 q 来自查询参数（因为声明了 q: str | None = None）
   → query_or_cookie_extractor 的 last_query 来自 Cookie（因为用了 Cookie()）
   → 唯一的区别是 q 不是直接来自请求，而是来自另一个依赖项的返回值

3. 依赖项可以无限嵌套
   → A 依赖 B，B 依赖 C，C 依赖 D……
   → FastAPI 都能自动解析
```

### 1.5 用图展示依赖树

```
             read_query
                 │
                 │  Depends(query_or_cookie_extractor)
                 ▼
    query_or_cookie_extractor
         │              │
         │ Depends()    │ Cookie()
         ▼              ▼
   query_extractor    last_query (来自 Cookie)
         │
         │ 查询参数
         ▼
    q (来自 URL ?q=xxx)
```

FastAPI 自动从叶子节点（最底层）开始执行，逐层往上，直到路径操作函数

## 2. 实际场景——认证的依赖链

上面的例子太简单了，看个更真实的场景：

```python
from typing import Annotated
from fastapi import Depends, Header, HTTPException

# ===== 第一层：从请求头提取 Token =====
def get_token(authorization: Annotated[str, Header()]):
    """从 Authorization 头提取 Token"""
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="无效的 Authorization 头")
    token = authorization.removeprefix("Bearer ")
    return token


# ===== 第二层：解析 Token，获取用户信息 =====
def get_current_user(token: Annotated[str, Depends(get_token)]):
    """用 Token 查找用户"""
    # 实际项目中这里会解码 JWT Token
    if token == "valid-token-for-alice":
        return {"username": "alice", "role": "admin"}
    raise HTTPException(status_code=401, detail="Token 无效或已过期")


# ===== 第三层：检查用户权限 =====
def require_admin(user: Annotated[dict, Depends(get_current_user)]):
    """检查用户是否是管理员"""
    if user["role"] != "admin":
        raise HTTPException(status_code=403, detail="需要管理员权限")
    return user


# ===== 路径操作函数：只声明最外层依赖 =====
@app.get("/admin/stats")
async def admin_stats(user: Annotated[dict, Depends(require_admin)]):
    return {"stats": "...", "accessed_by": user["username"]}
```

依赖链条：

```
admin_stats
    → require_admin（检查是否管理员）
        → get_current_user（解析 Token 获取用户）
            → get_token（从请求头提取 Token）
                → Authorization 请求头

执行顺序（从底到上）：
  ① 提取 Authorization 头 → "Bearer valid-token-for-alice"
  ② get_token → "valid-token-for-alice"
  ③ get_current_user → {"username": "alice", "role": "admin"}
  ④ require_admin → {"username": "alice", "role": "admin"}（检查通过）
  ⑤ admin_stats 拿到 user，返回响应
```

任何一层验证失败都会抛出 HTTPException，请求被拦截，后面的不执行。

这就是子依赖项的真正威力——**每一层只关心自己的事**：

```
get_token       → 只负责从请求头提取 Token
get_current_user → 只负责用 Token 查用户
require_admin   → 只负责检查权限
admin_stats     → 只负责业务逻辑
```

职责清晰，每一层都可以独立复用。比如有的接口不需要管理员权限，只需要登录就行，直接依赖 `get_current_user` 就好：

```python
@app.get("/profile")
async def my_profile(user: Annotated[dict, Depends(get_current_user)]):
    return {"user": user}
```

## 3. 依赖项缓存——同一请求内只执行一次

### 3.1 问题场景

```python
def get_db_session():
    """创建数据库会话"""
    print("创建数据库会话！")  # 观察调用次数
    with Session(engine) as session:
        yield session

def get_current_user(session: Annotated[Session, Depends(get_db_session)]):
    """依赖数据库会话来查用户"""
    ...

def get_project(
    project_id: int,
    session: Annotated[Session, Depends(get_db_session)],   # 也依赖数据库会话
    user: Annotated[dict, Depends(get_current_user)],        # 间接依赖数据库会话
):
    ...
```

`get_db_session` 被依赖了两次——`get_project` 直接依赖它，`get_current_user` 也依赖它。一个请求里会创建两个数据库会话吗？

### 3.2 答案：不会，默认缓存

```
FastAPI 默认行为：

同一个请求中，同一个依赖项函数只执行一次。
第一次执行的返回值被缓存，后续的"依赖方"直接拿缓存值。

所以上面的例子：
  get_db_session 只执行一次
  get_project 和 get_current_user 拿到的是同一个 session 对象
  "创建数据库会话！" 只打印一次
```

这是完全正确的行为——我们**确实想要**在一个请求里共享同一个数据库会话。

### 3.3 可视化缓存机制

```
请求到达 → FastAPI 分析依赖树：

get_project
├── Depends(get_db_session)      ← 第一次调用，执行并缓存结果
├── Depends(get_current_user)
│   └── Depends(get_db_session)  ← 第二次需要，直接用缓存值
└── project_id（路径参数）

实际执行顺序：
  ① get_db_session()  → session 对象（缓存）
  ② get_current_user(session=缓存的session)  → user 对象
  ③ get_project(session=缓存的session, user=user, project_id=1)
```

### 3.4 禁用缓存：use_cache=False

极少数情况下，你希望同一个依赖项在一个请求里执行多次（比如每次需要一个全新的随机值）：

```python
import random

def get_random_value():
    return random.randint(1, 100)


@app.get("/test")
async def test(
    value1: Annotated[int, Depends(get_random_value)],
    value2: Annotated[int, Depends(get_random_value, use_cache=False)],
):
    # 默认缓存：value1 和没标 use_cache=False 的依赖共享同一个值
    # use_cache=False：value2 会重新调用 get_random_value，得到不同的值
    return {"value1": value1, "value2": value2}
```

**99% 的情况下你不需要 `use_cache=False`**。默认的缓存行为是你想要的。

## 4. 装饰器依赖项——只执行不要返回值

### 4.1 问题：有些依赖项只做验证，返回值没人用

```python
async def verify_token(x_token: Annotated[str, Header()]):
    if x_token != "fake-super-secret-token":
        raise HTTPException(status_code=400, detail="X-Token header invalid")
    # 注意：这个函数没有返回值（返回 None）
    # 它只是验证，验证通过就什么都不做，验证失败就抛异常


async def verify_key(x_key: Annotated[str, Header()]):
    if x_key != "fake-super-secret-key":
        raise HTTPException(status_code=400, detail="X-Key header invalid")
    return x_key  # 虽然返回了值，但我们根本不需要这个值
```

如果用参数级依赖的方式：

```python
@app.get("/items/")
async def read_items(
    token_result: Annotated[None, Depends(verify_token)],   # ← 接收了值但根本不用
    key_result: Annotated[str, Depends(verify_key)],         # ← 接收了值但根本不用
):
    return [{"item": "Foo"}, {"item": "Bar"}]
```

问题：

```
1. token_result 和 key_result 在函数体里根本没用到
2. 编辑器会提示"未使用的参数"（黄色警告）
3. 新手看到这些参数会疑惑："这些参数有什么用？"
4. 函数签名被不必要的参数污染了
```

### 4.2 解决：把依赖项放在装饰器的 dependencies 参数里

```python
@app.get("/items/", dependencies=[Depends(verify_token), Depends(verify_key)])
async def read_items():
    return [{"item": "Foo"}, {"item": "Bar"}]
```

效果完全一样——`verify_token` 和 `verify_key` 都会被执行，验证失败会抛异常。但路径操作函数的签名干干净净，没有多余的参数。

### 4.3 两种方式的对比

```python
# 方式一：参数级依赖（需要返回值时用）
@app.get("/users/me")
async def get_me(user: Annotated[dict, Depends(get_current_user)]):
    return user    # ← 需要用到 user 的值，所以用参数级依赖

# 方式二：装饰器依赖（只做验证，不需要返回值时用）
@app.get("/items/", dependencies=[Depends(verify_token)])
async def read_items():
    return [...]   # ← 不需要 verify_token 的返回值，只需要它执行验证
```

判断用哪种的标准非常简单：

```
你需要依赖项的返回值吗？
  → 需要 → 放在函数参数里（参数级依赖）
  → 不需要 → 放在装饰器的 dependencies 里（装饰器依赖）
```

### 4.4 装饰器依赖的完整能力

装饰器依赖和参数级依赖的能力完全一样，除了"返回值不传给函数"这一点：

```
能力                            参数级依赖    装饰器依赖
──────────────────────────────────────────────────────────
从请求中提取数据（Header等）      ✓            ✓
抛出 HTTPException               ✓            ✓
有自己的子依赖项                  ✓            ✓
返回值传给路径操作函数            ✓            ✗（丢弃）
可以有 yield（清理逻辑）          ✓            ✓
```

### 4.5 装饰器依赖也可以有子依赖项

```python
async def get_db_connection():
    """获取数据库连接"""
    ...

async def verify_admin(
    conn: Annotated[Connection, Depends(get_db_connection)],
    x_token: Annotated[str, Header()],
):
    """用数据库查 Token 对应的用户是否是管理员"""
    user = await conn.fetch_user_by_token(x_token)
    if not user or user.role != "admin":
        raise HTTPException(status_code=403, detail="需要管理员权限")


# verify_admin 有子依赖项 get_db_connection
# 放在装饰器依赖里完全没问题
@app.delete("/items/{item_id}", dependencies=[Depends(verify_admin)])
async def delete_item(item_id: int):
    return {"ok": True}
```

## 5. 三个层级的依赖项——完整视图

在第二十八节我们学了 `APIRouter` 和 `FastAPI` 级别的依赖项。现在把所有层级放在一起看：

### 5.1 三个层级

```python
# 层级一：全局依赖项（加在 FastAPI 上）
app = FastAPI(dependencies=[Depends(verify_api_key)])
# → 所有接口都执行

# 层级二：Router 级依赖项（加在 APIRouter 上或 include_router 时）
router = APIRouter(dependencies=[Depends(verify_token)])
# → 这个 Router 下的所有接口都执行

# 层级三：路径操作级依赖项（加在装饰器上）
@router.get("/admin", dependencies=[Depends(require_admin)])
async def admin_panel():
    ...
# → 只有这个接口执行
```

### 5.2 执行顺序

```
一个请求到达时，依赖项的执行顺序：

① 全局依赖项（FastAPI 级）
    → verify_api_key

② Router 级依赖项（APIRouter 级）
    → verify_token

③ 路径操作级依赖项（装饰器 dependencies 参数）
    → require_admin

④ 参数级依赖项（函数参数里的 Depends）
    → get_current_user、get_db_session 等

任何一层失败 → 抛异常 → 后面的不执行
```

### 5.3 用一张图看全景

```
FastAPI(dependencies=[...])          ← 全局：所有接口都要经过
    │
    ├── include_router(
    │       router,
    │       dependencies=[...]       ← Router 级：该模块所有接口
    │   )
    │   │
    │   ├── @router.get(
    │   │       "/items/",
    │   │       dependencies=[...]   ← 路径操作级：只有这个接口
    │   │   )
    │   │   async def read_items(
    │   │       session: SessionDep,  ← 参数级：需要返回值的依赖
    │   │       user: UserDep,        ← 参数级：需要返回值的依赖
    │   │   ):
    │   │       ...
```

## 6. 对应到你的投标项目

### 6.1 依赖链：认证 → 权限

```python
# app/dependencies.py

from typing import Annotated
from fastapi import Depends, Header, HTTPException
from sqlmodel import Session

from .database import get_session

SessionDep = Annotated[Session, Depends(get_session)]


# ===== 第一层：提取 Token =====
async def get_token(authorization: Annotated[str, Header()]):
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="请提供有效的 Bearer Token")
    return authorization.removeprefix("Bearer ")


# ===== 第二层：解析 Token → 获取用户（依赖第一层） =====
async def get_current_user(
    token: Annotated[str, Depends(get_token)],
    session: SessionDep,
):
    # 实际项目中这里用 JWT 解码
    user = session.exec(select(User).where(User.token == token)).first()
    if not user:
        raise HTTPException(status_code=401, detail="Token 无效")
    return user

UserDep = Annotated[dict, Depends(get_current_user)]


# ===== 第三层：检查管理员权限（依赖第二层） =====
async def require_admin(user: UserDep):
    if user.get("role") != "admin":
        raise HTTPException(status_code=403, detail="需要管理员权限")
    return user
```

### 6.2 在 Router 里使用

```python
# app/routers/projects.py

from fastapi import APIRouter, Depends
from ..dependencies import SessionDep, UserDep

router = APIRouter(
    prefix="/projects",
    tags=["项目管理"],
)


# 需要登录 + 需要返回值 → 参数级依赖
@router.get("/")
async def list_projects(session: SessionDep, user: UserDep):
    # user 用于记录"谁查询了项目列表"
    return session.exec(select(Project)).all()


# 只需要验证登录，不需要返回值 → 装饰器依赖也行，参数级也行
# 但这里我们需要 user 来记录"谁创建了项目"，所以用参数级
@router.post("/")
async def create_project(project: ProjectCreate, session: SessionDep, user: UserDep):
    db_project = Project.model_validate(project)
    db_project.created_by = user["username"]
    session.add(db_project)
    session.commit()
    session.refresh(db_project)
    return db_project
```

### 6.3 管理后台用装饰器依赖

```python
# app/main.py

from .dependencies import require_admin

# 管理后台整个模块都需要管理员权限
# 用 include_router 的 dependencies 参数
# require_admin 内部依赖 get_current_user → 依赖 get_token
# FastAPI 自动解析整条链
app.include_router(
    admin.router,
    prefix="/admin",
    tags=["管理后台"],
    dependencies=[Depends(require_admin)],  # ← 装饰器依赖：不需要返回值
)
```

### 6.4 完整的依赖树

```
请求到达 POST /admin/clear-cache

① 全局依赖项（如果有的话）
② include_router 的 dependencies
   → require_admin
       → get_current_user（子依赖项）
           → get_token（子子依赖项）
               → 从 Authorization 头提取 Token
           → get_session（数据库会话，缓存共享）
               → 创建 Session
③ clear_cache 路径操作函数执行

任何一层失败：
  没有 Authorization 头 → 422 (FastAPI 自动校验)
  Token 格式不对 → 401 (get_token 抛出)
  Token 查不到用户 → 401 (get_current_user 抛出)
  用户不是管理员 → 403 (require_admin 抛出)
  全部通过 → 执行 clear_cache，返回结果
```

## 7. 本节核心知识点速查

```
概念                                  做法
──────────────────────────────────────────────────────────────────────────
创建子依赖项                          在依赖项的参数里再用 Depends()
依赖链执行顺序                        从最底层开始，逐层往上
依赖缓存                              同一请求内，同一依赖项只执行一次（默认）
禁用缓存                              Depends(func, use_cache=False)
装饰器依赖（不要返回值）               @app.get("/", dependencies=[Depends(func)])
参数级依赖（要返回值）                 def handler(result: Annotated[T, Depends(func)])
全局依赖                              FastAPI(dependencies=[Depends(func)])
Router 级依赖                         APIRouter(dependencies=[Depends(func)])
                                      或 include_router(..., dependencies=[Depends(func)])
路径操作级依赖                         @router.get("/", dependencies=[Depends(func)])
```

**本节最核心的三句话**：

1. **依赖项可以无限嵌套，FastAPI 自动解析整棵依赖树**。你只需在每个函数里声明"我依赖谁"，FastAPI 从叶子节点开始执行，逐层往上传递返回值，直到路径操作函数。这让你可以把认证、权限、数据库会话等逻辑拆成独立的层，各管各的。

2. **同一请求内，同一依赖项默认只执行一次**。FastAPI 缓存第一次执行的返回值，后续的"依赖方"直接拿缓存。这保证了一个请求里用的是同一个数据库会话、同一个用户对象，既正确又高效。

3. **不需要返回值的依赖项放在 `dependencies` 参数里**。验证类的依赖项（检查 Token、检查权限、检查频率限制）只做"通过/不通过"的判断，不给路径操作函数传数据——放在装饰器的 `dependencies` 里，函数签名更干净，代码意图更清晰。
