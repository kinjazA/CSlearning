从这一节开始，进入了 FastAPI 的**进阶篇**。基础篇教"怎么接收数据、怎么返回数据"，进阶篇教"怎么写出不重复、可维护的代码"



**依赖注入（Dependency Injection）** 是 FastAPI 最强大的特性之一。名字听起来很唬人，但本质非常简单：**把多个接口共用的逻辑提取成一个函数，让 FastAPI 自动调用它、自动把结果传给你**



在前面学过 `Query()`、`Path()`、`Body()`、`Cookie()`、`Header()`、`Form()`、`File()`，现在再加一个 `Depends()`，它不是从请求里取数据，而是**运行一个函数，把函数的返回值注入给你**

---

## 1. 先看问题：为什么需要依赖注入

### 1.1 没有依赖注入时的代码

假设有两个接口，都需要分页参数（`q`、`skip`、`limit`）：

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/items/")
def read_items(q: str | None = None, skip: int = 0, limit: int = 100):
    # 用 q、skip、limit 查询商品...
    return {"q": q, "skip": skip, "limit": limit}


@app.get("/users/")
def read_users(q: str | None = None, skip: int = 0, limit: int = 100):
    # 用 q、skip、limit 查询用户...
    return {"q": q, "skip": skip, "limit": limit}
```

问题出在哪？

```css
两个接口的参数完全一样：q: str | None = None, skip: int = 0, limit: int = 100
                        ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                        复制粘贴了一遍

如果以后要加一个参数（比如 sort_by），两个接口都要改
如果有 20 个接口都用这组参数呢？改 20 个地方，漏改一个就是 bug
```

### 1.2 有依赖注入之后

```python
from typing import Annotated
from fastapi import Depends, FastAPI

app = FastAPI()


# 第一步：把共用的逻辑提取成一个函数
async def common_parameters(q: str | None = None, skip: int = 0, limit: int = 100):
    return {"q": q, "skip": skip, "limit": limit}


# 第二步：用 Depends() 让 FastAPI 自动调用这个函数
@app.get("/items/")
async def read_items(commons: Annotated[dict, Depends(common_parameters)]):
    return commons


@app.get("/users/")
async def read_users(commons: Annotated[dict, Depends(common_parameters)]):
    return commons
```

`q`、`skip`、`limit` 只写了一次（在 `common_parameters` 函数里）。20 个接口都用这组参数？每个接口只需要写一个 `commons: Annotated[dict, Depends(common_parameters)]`。要加参数？只改 `common_parameters` 一个地方

## 2. `Depends()` 到底做了什么

### 2.1 一步步拆解执行过程

当客户端发请求 `GET /items/?q=hello&skip=10&limit=5` 时：

```css
第一步：FastAPI 收到请求，匹配到 read_items 函数

第二步：FastAPI 发现 read_items 的参数里有 Depends(common_parameters)
       → "哦，这个参数需要先调用 common_parameters 函数才能得到"

第三步：FastAPI 看 common_parameters 函数的参数签名：
       q: str | None = None, skip: int = 0, limit: int = 100
       → "这些参数和路径操作函数的参数一样，我从请求里取"
       → 从 URL 查询参数里取出 q="hello", skip=10, limit=5

第四步：FastAPI 调用 common_parameters(q="hello", skip=10, limit=5)
       → 返回 {"q": "hello", "skip": 10, "limit": 5}

第五步：FastAPI 把返回值赋给 read_items 的 commons 参数
       → commons = {"q": "hello", "skip": 10, "limit": 5}

第六步：FastAPI 调用 read_items(commons={"q": "hello", "skip": 10, "limit": 5})
       → 执行你的业务逻辑
```

用一张图表示：

```css
客户端请求：GET /items/?q=hello&skip=10&limit=5
                  │
                  ▼
       FastAPI 发现 Depends(common_parameters)
                  │
                  ▼
       自动调用 common_parameters(q="hello", skip=10, limit=5)
                  │
                  ▼
       返回值：{"q": "hello", "skip": 10, "limit": 5}
                  │
                  ▼
       把返回值注入到 read_items 的 commons 参数
                  │
                  ▼
       执行 read_items(commons={"q": "hello", ...})
```

### 2.2 "注入"这个词的含义

"依赖注入"里的"注入"就是指：**你声明"我需要某个东西"，框架帮你准备好并传给你**

你不需要手动调用 `common_parameters()`，FastAPI 帮你调了，还帮你传了正确的参数，最后把结果塞进你的函数参数里。你只要说"我依赖 `common_parameters`"，其他的 FastAPI 全包了

## 3. 依赖项函数的本质——就是一个普通函数

### 3.1 它和路径操作函数长得一模一样

```python
# 这是一个路径操作函数（有装饰器）
@app.get("/items/")
async def read_items(q: str | None = None, skip: int = 0, limit: int = 100):
    return {"q": q, "skip": skip, "limit": limit}


# 这是一个依赖项函数（没有装饰器，其他完全一样）
async def common_parameters(q: str | None = None, skip: int = 0, limit: int = 100):
    return {"q": q, "skip": skip, "limit": limit}
```

唯一的区别：**依赖项函数没有 `@app.get()` 之类的装饰器**。其他方面——参数声明、类型注解、默认值、返回值和路径操作函数完全一样

### 3.2 依赖项函数的参数也会被 FastAPI 解析

依赖项函数里的参数遵循和路径操作函数一样的规则：

```python
async def common_parameters(
    q: str | None = None,     # ← FastAPI 从查询参数取（简单类型默认是查询参数）
    skip: int = 0,            # ← FastAPI 从查询参数取
    limit: int = 100,         # ← FastAPI 从查询参数取
):
    return {"q": q, "skip": skip, "limit": limit}
```

FastAPI 从请求里取数据的规则你已经学得很熟了：简单类型默认查询参数，Pydantic 模型默认请求体，显式标记 `Cookie()`/`Header()` 等。**这些规则在依赖项函数里完全一样**

### 3.3 依赖项函数可以返回任何东西

```python
# 返回字典
async def common_parameters(...):
    return {"q": q, "skip": skip, "limit": limit}

# 返回 Pydantic 模型
async def get_current_user(...):
    return User(username="alice", role="admin")

# 返回数据库连接
def get_db():
    db = SessionLocal()
    return db

# 返回布尔值
def is_admin(user: User):
    return user.role == "admin"

# 甚至返回 None（只是执行某些逻辑，不需要返回值）
def verify_token(token: str):
    if token != "valid":
        raise HTTPException(status_code=401)
    # 没有 return → 返回 None
```

## 4. `Depends()` 的语法详解

### 4.1 基本语法

```python
from typing import Annotated
from fastapi import Depends


# 写法：Annotated[返回值类型, Depends(依赖项函数)]
commons: Annotated[dict, Depends(common_parameters)]
```

拆解每一部分：

```css
commons                    → 参数名（你随便起）
Annotated[                 → 类型注解语法
    dict,                  → 依赖项函数的返回值类型（让编辑器知道 commons 是什么类型）
    Depends(               → 告诉 FastAPI "这个参数要通过依赖注入获得"
        common_parameters  → 依赖项函数（注意！不加括号，不调用它）
    )
]
```

### 4.2 关键点：传函数本身，不是调用函数

```python
# ✅ 正确：传函数本身（不加括号）
Depends(common_parameters)      # 把函数交给 FastAPI，让 FastAPI 去调用

# ❌ 错误：调用了函数（加了括号）
Depends(common_parameters())    # 你自己调用了函数，把返回值传给了 Depends
```

你只需要告诉 FastAPI"我需要这个函数的结果"，具体什么时候调用、传什么参数，由 FastAPI 决定

### 4.3 和 Query / Path / Body 的对比

```python
# Query：从 URL 查询参数取值
q: Annotated[str | None, Query()] = None

# Path：从 URL 路径取值
item_id: Annotated[int, Path()]

# Body：从 JSON 请求体取值
item: Annotated[Item, Body()]

# Cookie：从 Cookie 取值
session: Annotated[str, Cookie()]

# Header：从请求头取值
token: Annotated[str, Header()]

# Depends：从依赖项函数的返回值取值
commons: Annotated[dict, Depends(common_parameters)]
```

全都是 `Annotated[类型, 标记()]` 的统一格式。`Depends()` 只是又多了一种"数据来源"——来源不是请求里的某个位置，而是另一个函数的返回值

## 5. 消除重复：类型别名

### 5.1 问题

如果 20 个接口都依赖 `common_parameters`，每个都要写：

```python
commons: Annotated[dict, Depends(common_parameters)]
```

虽然比复制粘贴参数好多了，但这行代码本身也在重复

### 5.2 用类型别名进一步精简

```python
# 定义类型别名（在文件顶部定义一次）
CommonsDep = Annotated[dict, Depends(common_parameters)]


# 所有接口都用这个别名
@app.get("/items/")
async def read_items(commons: CommonsDep):
    return commons


@app.get("/users/")
async def read_users(commons: CommonsDep):
    return commons


@app.get("/orders/")
async def read_orders(commons: CommonsDep):
    return commons
```

`CommonsDep` 就是 Python 的类型别名，把一个复杂的类型表达式赋给一个变量名。这不是 FastAPI 的特殊功能，是标准 Python 语法

**好处**：

```css
1. 代码更短：commons: CommonsDep 比完整写法短多了
2. 改一处改全部：如果依赖项函数换了，只改 CommonsDep 的定义
3. 编辑器支持不变：自动补全、类型检查、内联错误全部正常工作
```

## 6. `async def` 还是 `def`

### 6.1 两种都行，可以混着用

```python
# 依赖项是 async def，路径操作是 def → ✅ 没问题
async def get_user():
    ...

@app.get("/items/")
def read_items(user: Annotated[User, Depends(get_user)]):
    ...


# 依赖项是 def，路径操作是 async def → ✅ 也没问题
def get_db():
    ...

@app.get("/items/")
async def read_items(db: Annotated[Session, Depends(get_db)]):
    ...
```

FastAPI 内部会正确处理：`async def` 的用 `await` 调用，普通 `def` 的放到线程池里调用

### 6.2 怎么选

```css
用 async def：
  依赖项内部有 await 操作（异步数据库查询、异步 HTTP 请求等）

用普通 def：
  依赖项内部没有 await 操作（简单计算、同步数据库、参数组装等）
  不确定的时候用 def 也行——FastAPI 会自动放到线程池里跑，不会阻塞
```

## 7. 依赖注入的 OpenAPI 集成

### 7.1 依赖项的参数会出现在 /docs 里

```python
async def common_parameters(q: str | None = None, skip: int = 0, limit: int = 100):
    return {"q": q, "skip": skip, "limit": limit}


@app.get("/items/")
async def read_items(commons: Annotated[dict, Depends(common_parameters)]):
    return commons
```

打开 `/docs` 页面，你会看到 `/items/` 接口有三个查询参数：`q`、`skip`、`limit`

**这些参数来自依赖项函数**，不是来自路径操作函数本身。但 FastAPI 会把它们合并展示在一起，客户端开发者看到的就是"这个接口需要 q、skip、limit 三个参数"——完全透明，不知道后面有依赖注入

### 7.2 验证也是自动的

```python
async def common_parameters(q: str | None = None, skip: int = 0, limit: int = 100):
    ...
```

如果客户端传了 `skip=abc`（不是整数），FastAPI 自动返回 422 验证错误——和参数直接写在路径操作函数里的效果一模一样

## 8. 依赖注入的本质：用一句话理解

> **依赖注入就是：你声明"我需要某个函数的返回值"，FastAPI 自动调用那个函数、传入正确的参数、拿到返回值、传给你**

你和 FastAPI 之间的"契约"是：

```css
你说：commons: Annotated[dict, Depends(common_parameters)]
意思是：
  1. 在执行我的函数之前，先执行 common_parameters
  2. common_parameters 需要的参数（q、skip、limit），你从请求里取
  3. 把 common_parameters 的返回值赋给 commons
  4. 然后再执行我的函数
```

FastAPI 负责第 1-3 步，你只负责第 4 步里的业务逻辑

## 9. 依赖注入能用来做什么（预告）

这一节只讲了最基本的用法（共享参数），但依赖注入的威力远不止如此：

```css
本节学的：
  共享参数逻辑 → 多个接口复用同一组参数

后面会学的：
  数据库连接 → 每个请求获取一个数据库会话，请求结束自动关闭
  用户认证 → 每个接口自动检查登录状态，没登录就 401
  权限控制 → 根据用户角色决定能不能访问某个接口
  子依赖项 → 依赖项 A 依赖于依赖项 B，B 又依赖于 C，形成链条
  依赖项层级 → 公共接口不需要登录，私有接口需要登录，管理接口需要管理员权限
```

## 10. 对应到你的投标项目

```python
from typing import Annotated

from fastapi import Depends, FastAPI, HTTPException, Query, status
from pydantic import BaseModel, Field

app = FastAPI(title="智能投标决策系统")


# ==================== 依赖项：通用分页参数 ====================

async def pagination_params(
    page: Annotated[int, Query(ge=1, description="页码")] = 1,
    page_size: Annotated[int, Query(ge=1, le=100, description="每页条数")] = 20,
):
    """
    通用分页参数——所有列表接口都用。

    返回 skip 和 limit，方便数据库查询。
    """
    skip = (page - 1) * page_size
    return {"skip": skip, "limit": page_size, "page": page, "page_size": page_size}


# 类型别名：在多个接口中复用
PaginationDep = Annotated[dict, Depends(pagination_params)]


# ==================== 依赖项：通用筛选参数 ====================

async def project_filters(
    keyword: Annotated[str | None, Query(description="搜索关键词")] = None,
    project_type: Annotated[str | None, Query(description="项目类型")] = None,
    location: Annotated[str | None, Query(description="项目所在地")] = None,
    min_price: Annotated[float | None, Query(ge=0, description="最低控制价")] = None,
    max_price: Annotated[float | None, Query(description="最高控制价")] = None,
):
    """
    项目筛选参数——项目列表、投标列表等多个接口共用。

    把五个筛选条件封装在一起，不用每个接口都写一遍。
    """
    filters = {}
    if keyword:
        filters["keyword"] = keyword
    if project_type:
        filters["project_type"] = project_type
    if location:
        filters["location"] = location
    if min_price is not None:
        filters["min_price"] = min_price
    if max_price is not None:
        filters["max_price"] = max_price
    return filters


FiltersDep = Annotated[dict, Depends(project_filters)]


# ==================== 依赖项：模拟用户认证 ====================

class User(BaseModel):
    username: str
    role: str = "viewer"


async def get_current_user(
    token: Annotated[str | None, Query(description="认证令牌")] = None,
):
    """
    模拟认证——后面学安全章节时会用 Header 里的 Bearer Token。
    这里简化为查询参数，只为演示依赖注入。
    """
    if not token:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="请提供认证令牌",
        )
    # 模拟：根据 token 查用户
    fake_users = {
        "admin-token": User(username="admin", role="admin"),
        "editor-token": User(username="editor", role="editor"),
        "viewer-token": User(username="viewer", role="viewer"),
    }
    user = fake_users.get(token)
    if not user:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="无效的认证令牌",
        )
    return user


CurrentUser = Annotated[User, Depends(get_current_user)]


# ==================== 接口：使用依赖项 ====================

@app.get("/api/projects/", tags=["项目管理"], summary="查询项目列表")
async def list_projects(
    pagination: PaginationDep,    # 分页参数（从依赖项注入）
    filters: FiltersDep,          # 筛选参数（从依赖项注入）
    user: CurrentUser,            # 当前用户（从依赖项注入）
):
    """
    查询项目列表——同时使用三个依赖项。

    请求示例：
    GET /api/projects/?page=2&page_size=10&keyword=路面&project_type=市政&token=admin-token

    FastAPI 的处理顺序：
    1. 调用 pagination_params(page=2, page_size=10) → 得到分页信息
    2. 调用 project_filters(keyword="路面", project_type="市政", ...) → 得到筛选条件
    3. 调用 get_current_user(token="admin-token") → 得到当前用户
    4. 把三个返回值分别注入到 pagination、filters、user 参数
    5. 执行 list_projects 的业务逻辑
    """
    return {
        "user": user.username,
        "role": user.role,
        "pagination": pagination,
        "filters": filters,
        "results": ["...项目列表..."],
    }


@app.get("/api/bids/", tags=["投标管理"], summary="查询投标列表")
async def list_bids(
    pagination: PaginationDep,    # 同一个分页依赖项，复用！
    user: CurrentUser,            # 同一个认证依赖项，复用！
    project_id: Annotated[str | None, Query(description="按项目筛选")] = None,
):
    """
    查询投标列表——复用了分页和认证两个依赖项。

    分页参数和认证逻辑和项目列表接口完全一样，
    但不需要重复写任何代码。
    """
    return {
        "user": user.username,
        "pagination": pagination,
        "project_id": project_id,
        "results": ["...投标列表..."],
    }


@app.get("/api/statistics/", tags=["数据分析"], summary="查看统计数据")
async def get_statistics(
    user: CurrentUser,            # 只需要认证，不需要分页
):
    """
    查看统计数据——只用了认证依赖项。

    每个接口可以选择性地使用需要的依赖项，不是全有或全无。
    """
    return {
        "user": user.username,
        "total_projects": 42,
        "total_bids": 156,
        "win_rate": 0.68,
    }
```

**三个依赖项写一次，在多个接口中反复使用**：

```
pagination_params → list_projects ✓   list_bids ✓   get_statistics ✗
project_filters   → list_projects ✓   list_bids ✗   get_statistics ✗
get_current_user  → list_projects ✓   list_bids ✓   get_statistics ✓
```

如果以后要给分页加一个 `sort_by` 参数，只改 `pagination_params` 一个函数，所有用到它的接口自动生效。

## 11. 本节核心知识点速查

```css
概念                          做法
──────────────────────────────────────────────────────────────────────
定义依赖项                    写一个普通函数（和路径操作函数长得一样，只是没有装饰器）
使用依赖项                    commons: Annotated[返回类型, Depends(函数名)]
注意不要加括号                Depends(func) ✓    Depends(func()) ✗
类型别名复用                  CommonsDep = Annotated[dict, Depends(func)]
async 和 def 都行             可以混着用，FastAPI 自动处理
/docs 自动集成                依赖项的参数会出现在接口文档里
验证自动生效                  依赖项参数的类型检查和约束和普通参数一样
```

**本节最核心的三句话**：

1. 依赖注入的本质就是：**你声明"我需要某个函数的返回值"（`Depends(func)`），FastAPI 自动调用那个函数、从请求里取参数、把返回值注入给你**。你不需要手动调用，不需要注册，不需要配置

2. 依赖项函数就是一个**普通的 Python 函数**，和路径操作函数的写法一模一样（参数声明、类型注解、返回值），唯一的区别是没有 `@app.get()` 装饰器。FastAPI 解析它参数的方式也完全一样

3. 依赖注入解决的核心问题是**代码复用**：多个接口共用的逻辑（分页、筛选、认证、数据库连接）提取成依赖项函数，写一次到处用。要修改只改一个地方，所有使用的接口自动生效
