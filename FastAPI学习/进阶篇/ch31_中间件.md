前面学的所有东西——路径参数、查询参数、请求体、依赖注入——都是在**单个接口**层面工作的。中间件不一样，它在**所有接口之外**工作，像一个门卫一样守在应用的入口和出口



**每一个请求进来，先过中间件；每一个响应出去，再过中间件。**无论你有 5 个接口还是 500 个接口，==中间件都会拦截所有的请求和响应==



中间件的典型用途：计算请求耗时、记录日志、身份验证、CORS 跨域、请求限流、异常捕获。这些逻辑如果写在每个接口里会重复无数遍，放在中间件里只写一次

---

## 1. 中间件的本质：请求和响应的"收费站"

### 1.1 没有中间件时

```css
客户端请求 → FastAPI 路由匹配 → 路径操作函数 → 返回响应
```

### 1.2 有中间件时

```css
客户端请求
    │
    ▼
  中间件（请求进来时执行一些代码）
    │
    ▼
  FastAPI 路由匹配 → 路径操作函数
    │
    ▼
  中间件（响应出去前执行一些代码）
    │
    ▼
客户端收到响应
```

中间件把整个应用"包"了一层。请求进来时经过它，响应出去时再经过它。所以中间件可以做两件事：

```css
1. 在请求到达路径操作函数之前：修改请求、记录日志、检查权限、拒绝请求...
2. 在响应返回给客户端之前：修改响应、添加头信息、记录耗时、统一处理异常...
```

### 1.3 和依赖注入的区别

```css
依赖注入（Depends）：
  作用范围：你选择哪些接口用，就在哪些接口生效
  粒度：接口级别——可以只给某几个接口加认证
  方式：在路径操作函数的参数里声明
  适合：认证、权限、分页、数据库连接等"部分接口需要"的逻辑

中间件（Middleware）：
  作用范围：所有接口，无一例外
  粒度：应用级别——全局拦截
  方式：在 app 上注册
  适合：日志、耗时统计、CORS、全局异常处理等"每个请求都要做"的逻辑
```

简单来说：**依赖注入是"按需使用"，中间件是"全局生效"**

## 2. 第一个中间件：计算请求耗时

### 2.1 完整代码

```python
import time

from fastapi import FastAPI, Request

app = FastAPI()


@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start_time = time.perf_counter()
    response = await call_next(request)
    process_time = time.perf_counter() - start_time
    response.headers["X-Process-Time"] = str(process_time)
    return response
```

就这么短。但里面每一行都很重要，下面逐行拆解

### 2.2 逐行拆解

**第一行：装饰器**

```python
@app.middleware("http")
```

告诉 FastAPI："这个函数是一个 HTTP 中间件，请在每个请求进出时调用它。"`"http"` 是目前唯一支持的中间件类型

**第二行：函数签名**

```python
async def add_process_time_header(request: Request, call_next):
```

中间件函数**必须**接收两个参数：

```css
request    → 当前的请求对象（包含 URL、方法、头信息、请求体等）
call_next  → 一个函数，调用它就是"把请求交给下一层处理"
             下一层可能是另一个中间件，也可能是路径操作函数
```

**第三行：记录开始时间**

```python
start_time = time.perf_counter()
```

在请求被处理**之前**记录时间。用 `time.perf_counter()` 而不是 `time.time()`，因为前者精度更高，专门用来测量时间间隔

**第四行：把请求交给下一层**

```python
response = await call_next(request)
```

这是中间件最核心的一行。`call_next(request)` 做了这些事：

```css
1. 把 request 传给下一层（可能是另一个中间件，也可能是路由匹配 + 路径操作函数）
2. 等待下一层处理完毕
3. 拿到下一层返回的 response
4. 把 response 返回给你
```

**`await call_next(request)` 之前的代码 = 请求进来时执行**，**`await call_next(request)` 之后的代码 = 响应出去前执行**

**第五行：计算耗时**

```python
process_time = time.perf_counter() - start_time
```

在响应返回**之后**计算时间差。这个 `process_time` 就是整个请求的处理耗时

**第六行：把耗时加到响应头里**

```python
response.headers["X-Process-Time"] = str(process_time)
```

给响应添加一个自定义头 `X-Process-Time`，值是耗时的秒数。`X-` 前缀表示这是自定义头（不是 HTTP 标准头）

**第七行：返回响应**

```python
return response
```

中间件**必须返回 response**。如果你不返回，客户端就收不到任何响应

### 2.3 用图来理解 call_next 的分界作用

```css
@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    ┌─────────────────────────────────────────┐
    │  start_time = time.perf_counter()       │  ← 请求进来时执行（Before）
    └─────────────────────────────────────────┘
                        │
                        ▼
    ┌─────────────────────────────────────────┐
    │  response = await call_next(request)    │  ← 分界线：交给下一层处理
    └─────────────────────────────────────────┘
                        │
                        ▼
    ┌─────────────────────────────────────────┐
    │  process_time = ...                     │  ← 响应出去前执行（After）
    │  response.headers["X-Process-Time"] = ..│
    └─────────────────────────────────────────┘
                        │
                        ▼
    return response                              ← 把响应交还给客户端
```

## 3. call_next 到底发生了什么

### 3.1 没有 call_next 会怎样

如果你不调用 `call_next`，请求就永远不会到达你的路径操作函数：

```python
@app.middleware("http")
async def block_everything(request: Request, call_next):
    # 没有调用 call_next！
    # 请求被拦在这里了，路径操作函数根本不会执行
    from starlette.responses import JSONResponse
    return JSONResponse({"detail": "所有请求都被中间件拦截了"}, status_code=403)
```

这就是中间件实现"全局拦截"的原理——你可以在调用 `call_next` 之前检查条件，不满足就直接返回错误响应，满足了才放行

### 3.2 call_next 的完整流程

```css
你的中间件调用 call_next(request)
        │
        ▼
如果还有其他中间件 → 进入下一个中间件
        │
        ▼
没有更多中间件了 → 进入 FastAPI 路由匹配
        │
        ▼
匹配到路径操作函数 → 执行依赖项 → 执行路径操作函数
        │
        ▼
路径操作函数返回 response
        │
        ▼
response 沿着来时的路反向传回
        │
        ▼
你的中间件收到 response → await call_next(request) 返回了
```

## 4. 中间件的完整生命周期

### 4.1 一个请求从头到尾经历了什么

假设有一个中间件和一个依赖项：

```python
@app.middleware("http")
async def my_middleware(request: Request, call_next):
    print("1. 中间件：请求进来了")
    response = await call_next(request)
    print("5. 中间件：响应要出去了")
    return response


async def my_dependency():
    print("2. 依赖项：开始执行")
    yield "some_resource"
    print("4. 依赖项：yield 后面的清理代码")


@app.get("/items/")
async def read_items(dep: Annotated[str, Depends(my_dependency)]):
    print("3. 路径操作函数：正在执行")
    return {"result": "ok"}
```

执行顺序（按打印顺序）：

```
1. 中间件：请求进来了         ← call_next 之前
2. 依赖项：开始执行           ← 路径操作函数的依赖
3. 路径操作函数：正在执行      ← 业务逻辑
4. 依赖项：yield 后面的清理代码 ← 依赖项的退出代码
5. 中间件：响应要出去了        ← call_next 之后
```

关键点：**带 yield 的依赖项的清理代码（yield 后面的部分）在中间件的 `call_next` 返回之后执行，但在中间件处理响应之前**。实际上从中间件的角度来看，`call_next` 返回时，依赖项的清理已经完成了

### 4.2 如果有后台任务（BackgroundTask）

后台任务在**所有中间件之后**运行。也就是说，中间件返回响应之后、客户端已经收到响应了，后台任务才开始执行

```
中间件（Before） → 依赖项 → 路径操作函数 → 依赖项清理 → 中间件（After）→ 客户端收到响应 → 后台任务
```

## 5. 多个中间件的执行顺序——洋葱模型

### 5.1 注册顺序 vs 执行顺序

```python
@app.middleware("http")
async def middleware_a(request: Request, call_next):
    print("A 进")
    response = await call_next(request)
    print("A 出")
    return response


@app.middleware("http")
async def middleware_b(request: Request, call_next):
    print("B 进")
    response = await call_next(request)
    print("B 出")
    return response
```

你可能以为先注册的 A 先执行。但实际输出是：

```
B 进      ← 后注册的先执行（最外层）
A 进
（路径操作函数执行）
A 出
B 出      ← 后注册的最后执行
```

### 5.2 为什么是反的——堆栈模型

每个中间件都"包装"了它里面的东西。后注册的中间件包在最外面：

```
注册顺序：先注册 A，再注册 B

最终的包装结构：
┌─── B（最外层）──────────────────────────┐
│  B 进                                   │
│  ┌─── A ──────────────────────────┐     │
│  │  A 进                          │     │
│  │  ┌─── 路径操作函数 ─────┐      │     │
│  │  │  处理请求            │      │     │
│  │  └─────────────────────┘      │     │
│  │  A 出                          │     │
│  └────────────────────────────────┘     │
│  B 出                                   │
└─────────────────────────────────────────┘

请求方向（从外到内）：B 进 → A 进 → 路径操作函数
响应方向（从内到外）：路径操作函数 → A 出 → B 出
```

这就像**剥洋葱**：请求从外到内穿过每一层，响应从内到外再穿回来

### 5.3 用 add_middleware 注册效果一样

```python
# 用装饰器注册
@app.middleware("http")
async def middleware_a(...): ...

@app.middleware("http")
async def middleware_b(...): ...

# 等价于用 add_middleware 注册
app.add_middleware(MiddlewareA)
app.add_middleware(MiddlewareB)
```

两种方式的规则一样：**后注册的中间件包在最外层，请求先经过它**。

### 5.4 记忆口诀

```css
注册顺序：A → B → C
请求顺序：C → B → A → 路径操作函数
响应顺序：路径操作函数 → A → B → C

最后注册的最先收到请求、最后发出响应。
就像叠盘子：最后放上去的盘子在最上面（最先被拿到）。
```

## 6. 中间件能做什么——实用场景

### 6.1 请求日志

```python
import logging

logger = logging.getLogger("api")


@app.middleware("http")
async def log_requests(request: Request, call_next):
    logger.info(f"收到请求: {request.method} {request.url}")
    response = await call_next(request)
    logger.info(f"返回响应: {response.status_code}")
    return response
```

### 6.2 全局异常捕获

```python
from starlette.responses import JSONResponse


@app.middleware("http")
async def catch_exceptions(request: Request, call_next):
    try:
        response = await call_next(request)
        return response
    except Exception as e:
        # 所有未捕获的异常都会到这里
        logger.error(f"未捕获的异常: {e}")
        return JSONResponse(
            {"detail": "服务器内部错误"},
            status_code=500,
        )
```

### 6.3 简单的请求限流

```python
from collections import defaultdict
from starlette.responses import JSONResponse

request_counts: dict[str, int] = defaultdict(int)


@app.middleware("http")
async def rate_limit(request: Request, call_next):
    client_ip = request.client.host
    request_counts[client_ip] += 1

    if request_counts[client_ip] > 100:  # 简化版：每个 IP 最多 100 次
        return JSONResponse(
            {"detail": "请求过于频繁"},
            status_code=429,
        )

    response = await call_next(request)
    return response
```

一旦某个 IP 的 `request_counts[client_ip]` 被累加到 **101**（或以上），以后**无论隔多久**（哪怕几天、几周），只要服务器还在运行，这个 IP 每次发请求进来，代码都会立刻检查 `if request_counts[client_ip] > 100`，然后**直接返回 429 错误**，根本不会执行后面的业务逻辑。因为 `request_counts` 是**全局的内存变量**（程序启动时创建，一直存在于服务器的 RAM 里），它**不会自动过期或重置**。只有当重启 FastAPI 服务（例如 `uvicorn` 进程被 kill 再启动），整个字典才会清空，所有 IP 的计数重新从 0 开始

### 6.4 在响应中添加安全头

```python
@app.middleware("http")
async def add_security_headers(request: Request, call_next):
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["X-XSS-Protection"] = "1; mode=block"
    return response
```

## 7. 中间件的注意事项

### 7.1 中间件必须是 async def

```python
# ✅ 正确：async def
@app.middleware("http")
async def my_middleware(request: Request, call_next):
    response = await call_next(request)
    return response


# ❌ 错误：普通 def
@app.middleware("http")
def my_middleware(request: Request, call_next):  # 不行！
    response = call_next(request)  # 没有 await，拿不到真正的 response
    return response
```

因为 `call_next` 是一个异步函数，必须用 `await` 调用

### 7.2 中间件必须返回 response

```python
# ✅ 正确：返回 response
@app.middleware("http")
async def my_middleware(request: Request, call_next):
    response = await call_next(request)
    return response         # 必须 return


# ❌ 错误：忘记返回
@app.middleware("http")
async def my_middleware(request: Request, call_next):
    response = await call_next(request)
    # 忘记 return response → 客户端收不到任何响应
```

### 7.3 不要在中间件里读取请求体

```python
# ⚠️ 不推荐：在中间件里读请求体
@app.middleware("http")
async def bad_middleware(request: Request, call_next):
    body = await request.body()    # 读了请求体
    # 问题：请求体是一个流，读完一次就没了
    # 路径操作函数再读就读不到了！
    response = await call_next(request)
    return response
```

请求体（body）是一个流（stream），只能读一次。如果中间件读了，路径操作函数就读不到了。如果确实需要在中间件里读请求体，需要做额外的处理（把读到的内容重新放回去），但这属于高级用法，通常应该避免

### 7.4 性能考虑

中间件作用于**每一个请求**。如果中间件里有耗时操作（比如查数据库、调外部 API），会拖慢所有接口的响应速度。中间件里的逻辑应该尽量轻量

## 8. 对应到你的投标项目

```python
import logging
import time

from fastapi import FastAPI, Request
from starlette.responses import JSONResponse

# ==================== 配置 ====================

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("bidding-system")

app = FastAPI(title="智能投标决策系统")


# ==================== 中间件 1：请求耗时 + 日志 ====================

@app.middleware("http")
async def request_logging(request: Request, call_next):
    """
    记录每个请求的方法、路径、耗时、状态码。

    注册顺序：第一个注册 → 最内层
    执行顺序：请求时最后执行，响应时最先执行
    """
    start = time.perf_counter()

    # 记录请求信息
    logger.info(f"→ {request.method} {request.url.path}")

    response = await call_next(request)

    # 计算耗时并添加到响应头
    duration = time.perf_counter() - start
    response.headers["X-Process-Time"] = f"{duration:.4f}"

    # 记录响应信息
    logger.info(f"← {request.method} {request.url.path} [{response.status_code}] {duration:.4f}s")

    return response


# ==================== 中间件 2：全局异常捕获 ====================

@app.middleware("http")
async def global_exception_handler(request: Request, call_next):
    """
    捕获所有未处理的异常，避免返回 500 裸错误。

    注册顺序：第二个注册 → 中间层
    """
    try:
        response = await call_next(request)
        return response
    except Exception as e:
        logger.error(f"未捕获异常: {request.method} {request.url.path} → {type(e).__name__}: {e}")
        return JSONResponse(
            status_code=500,
            content={"detail": "服务器内部错误，请联系管理员"},
        )


# ==================== 中间件 3：安全响应头 ====================

@app.middleware("http")
async def security_headers(request: Request, call_next):
    """
    给所有响应加上安全相关的头信息。

    注册顺序：第三个注册 → 最外层
    请求时最先执行，响应时最后执行。
    """
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    return response


# ==================== 三个中间件的执行顺序 ====================
#
# 注册顺序：request_logging → global_exception_handler → security_headers
#
# 请求进来时（从外到内）：
#   security_headers 进 → global_exception_handler 进 → request_logging 进
#   → 路径操作函数
#
# 响应出去时（从内到外）：
#   路径操作函数 → request_logging 出 → global_exception_handler 出 → security_headers 出
#
# 所以最终的响应会包含：
#   - X-Process-Time 头（request_logging 加的）
#   - X-Content-Type-Options 头（security_headers 加的）
#   - X-Frame-Options 头（security_headers 加的）


# ==================== 路径操作函数 ====================

@app.get("/api/projects/", tags=["项目管理"])
async def list_projects():
    """
    这个接口被三层中间件包裹。
    每次请求都会：记日志、捕获异常、添加安全头。
    接口本身不需要写任何相关代码。
    """
    return {
        "projects": [
            {"id": "proj-001", "name": "南昌路面改造工程", "price": 1200},
            {"id": "proj-002", "name": "赣州公路扩建项目", "price": 3500},
        ]
    }


@app.get("/api/health", tags=["系统"])
async def health_check():
    """
    健康检查接口——也会经过所有中间件。
    中间件是全局的，无法让某个接口跳过。
    """
    return {"status": "ok"}
```

**一个请求的完整日志输出**：

```
INFO → GET /api/projects/
INFO ← GET /api/projects/ [200] 0.0012s
```

**响应头里会多出这些**：

```
X-Process-Time: 0.0012
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
```

三个中间件各司其职，路径操作函数里一行相关代码都不用写

## 9. 中间件 vs 依赖注入——什么时候用哪个

```css
场景                              用中间件              用依赖注入
──────────────────────────────────────────────────────────────────
请求耗时统计                      ✓ 每个请求都要        ✗
全局日志记录                      ✓ 每个请求都要        ✗
CORS 跨域设置                     ✓ 全局配置            ✗
安全响应头                        ✓ 每个响应都要        ✗
用户认证                          ✗                    ✓ 部分接口需要
权限检查                          ✗                    ✓ 不同接口不同权限
分页参数                          ✗                    ✓ 列表接口才需要
数据库连接                        ✗                    ✓ 需要连接的接口才用
```

判断标准很简单：**如果是所有请求都要做的事 → 中间件。如果是部分接口需要的事 → 依赖注入**

## 10. 本节核心知识点速查

```css
概念                                做法
───────────────────────────────────────────────────────────────────────────
注册中间件                          @app.middleware("http") 装饰器
中间件函数签名                      async def func(request: Request, call_next)
把请求交给下一层                     response = await call_next(request)
call_next 之前的代码                 请求进来时执行
call_next 之后的代码                 响应出去前执行
修改响应头                          response.headers["X-Custom"] = "value"
拦截请求不放行                      不调用 call_next，直接返回自己的 response
多个中间件的执行顺序                 后注册的先收到请求（洋葱模型 / 堆栈模型）
必须是 async def                    因为 call_next 是异步函数
必须返回 response                   否则客户端收不到响应
```

**本节最核心的三句话**：

1. **中间件是全局拦截器**：每一个请求进来都会经过它，每一个响应出去也会经过它。和依赖注入不同，中间件没有"选择性使用"的概念——注册了就对所有接口生效

2. **`call_next(request)` 是分界线**：它之前的代码在请求进来时执行，它之后的代码在响应出去前执行。不调用 `call_next` 就能拦截请求、直接返回自定义响应

3. **多个中间件按洋葱模型执行**：后注册的中间件包在最外层，请求先经过它，响应最后经过它。请求从外到内，响应从内到外，像剥洋葱一样
