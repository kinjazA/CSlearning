上一节学了中间件的基本概念，这一节讲中间件最常见的实际应用：**CORS**。如果你正在开发前后端分离的项目（前端 React/Vue 跑在 `localhost:3000`，后端 FastAPI 跑在 `localhost:8000`），你一定会遇到这个报错：

 ```
 Access to fetch at 'http://localhost:8000/api/...' from origin 'http://localhost:3000'
 has been blocked by CORS policy
 ```
这不是你代码写错了，是**浏览器的安全策略**在阻止你。CORS 中间件就是告诉浏览器："放行，这个前端是我认识的"

---

## 1. 先搞懂"源（Origin）"是什么

### 1.1 源 = 协议 + 域名 + 端口

```css
https://example.com:443
─────   ───────────  ───
协议     域名          端口

一个"源"就是这三个部分的组合。
三个部分任何一个不同，就是不同的源。
```

### 1.2 哪些是不同的源

```css
http://localhost        ← 协议 http，端口 80（默认）
https://localhost       ← 协议 https，端口 443（默认）
http://localhost:8080   ← 协议 http，端口 8080

这三个虽然都是 localhost，但它们是三个不同的源！
因为协议或端口不一样。
```

再看几个例子：

```css
源 A                          源 B                          同源？
──────────────────────────────────────────────────────────────────────
http://localhost:3000         http://localhost:8000         ✗ 端口不同
http://example.com            https://example.com           ✗ 协议不同
https://api.example.com       https://www.example.com       ✗ 域名不同
https://example.com:443       https://example.com           ✓ 同源（443是https默认端口）
```

### 1.3 为什么"源"这么重要

浏览器有一个安全策略叫**同源策略（Same-Origin Policy）**：

> **浏览器里的 JavaScript 默认只能向同源的服务器发请求。跨源请求会被浏览器拦截**

注意：是**浏览器**拦截的，不是服务器拦截的。用 Postman、curl、Python 的 requests 库直接调接口完全没问题——因为它们不是浏览器，没有同源策略

## 2. 跨域请求被拦截的完整过程

### 2.1 场景

```css
前端：http://localhost:3000（React 开发服务器）
后端：http://localhost:8000（FastAPI）

前端的 JavaScript 要调用后端的接口：
fetch("http://localhost:8000/api/projects/")
```

### 2.2 浏览器做了什么

```css
第一步：浏览器发现这是跨域请求
  前端的源：http://localhost:3000
  请求的目标：http://localhost:8000
  → 端口不同，不是同源！

第二步：浏览器先发一个"预检请求"（Preflight Request）
  这是一个 OPTIONS 请求（不是你代码里写的 GET/POST）
  浏览器自动发的，你控制不了

  OPTIONS http://localhost:8000/api/projects/
  Origin: http://localhost:3000
  Access-Control-Request-Method: GET

  意思是："后端，我是 http://localhost:3000 的前端，
          我想发一个 GET 请求给你，你允许吗？"

第三步：后端响应预检请求
  情况 A：后端没有配置 CORS → 响应里没有 Access-Control-Allow-Origin 头
    → 浏览器："后端没说允许，拦截！"
    → 你看到 CORS 报错

  情况 B：后端配置了 CORS，允许 http://localhost:3000
    → 响应里有：Access-Control-Allow-Origin: http://localhost:3000
    → 浏览器："后端说允许了，放行！"

第四步（只有情况 B 才到这一步）：浏览器发送真正的请求
  GET http://localhost:8000/api/projects/

第五步：后端返回真正的响应，浏览器把数据交给前端的 JavaScript
```

### 2.3 用图表示

```css
前端 (localhost:3000)                浏览器                    后端 (localhost:8000)
        │                              │                              │
        │  fetch("/api/projects/")     │                              │
        │─────────────────────────────>│                              │
        │                              │                              │
        │                              │  ① OPTIONS /api/projects/    │
        │                              │  Origin: localhost:3000      │
        │                              │─────────────────────────────>│
        │                              │                              │
        │                              │  ② 响应（有/没有 CORS 头）    │
        │                              │<─────────────────────────────│
        │                              │                              │
        │                    没有 CORS 头？                            │
        │                    拦截！报错！                               │
        │<── CORS Error ──────────────│                              │
        │                              │                              │
        │                    有 CORS 头？                              │
        │                              │  ③ GET /api/projects/        │
        │                              │─────────────────────────────>│
        │                              │                              │
        │                              │  ④ 响应数据                   │
        │                              │<─────────────────────────────│
        │                              │                              │
        │<── 数据 ────────────────────│                              │
```

### 2.4 关键理解

```css
1. CORS 是浏览器的行为，不是服务器的行为
   → 服务器正常收到了请求，也正常返回了响应
   → 是浏览器看到响应里没有 CORS 头，选择不把数据交给 JavaScript

2. 预检请求（OPTIONS）是浏览器自动发的
   → 你的代码里没有写 OPTIONS 请求
   → 浏览器在发真正的请求之前，先问一下后端"你允许吗"

3. 不是所有跨域请求都会触发预检
   → 简单请求（GET/POST + 简单头）可能直接发，不预检
   → 但浏览器仍然会检查响应里的 CORS 头
```

## 3. FastAPI 里配置 CORS——一共就四步

### 3.1 完整代码

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

# 第一步：列出允许的源
origins = [
    "http://localhost:3000",      # React 开发服务器
    "http://localhost:5173",      # Vite 开发服务器
    "https://www.myapp.com",      # 生产环境的前端域名
]

# 第二步：添加 CORS 中间件
app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,          # 允许哪些源
    allow_credentials=True,         # 允许携带 Cookie
    allow_methods=["*"],            # 允许所有 HTTP 方法
    allow_headers=["*"],            # 允许所有请求头
)


# 第三步：正常写你的接口（不需要做任何 CORS 相关的事）
@app.get("/api/projects/")
async def list_projects():
    return {"projects": [...]}
```

就这么简单。加了这个中间件之后，来自 `origins` 列表里的前端就可以正常调用后端接口了

### 3.2 每个参数的含义

```python
app.add_middleware(
    CORSMiddleware,

    # 允许哪些源发跨域请求
    # 只有列表里的源才能访问你的接口
    allow_origins=["http://localhost:3000", "https://www.myapp.com"],

    # 是否允许携带凭证（Cookie、Authorization 头等）
    # 如果你的前端需要发送 Cookie 或 Bearer Token，设为 True
    allow_credentials=True,

    # 允许哪些 HTTP 方法
    # ["*"] 表示允许所有方法（GET、POST、PUT、DELETE、PATCH、OPTIONS 等）
    # 默认只允许 ["GET"]
    allow_methods=["*"],

    # 允许哪些请求头
    # ["*"] 表示允许所有头（Content-Type、Authorization、X-Custom-Header 等）
    # 默认是 []（只允许简单头）
    allow_headers=["*"],
)
```

### 3.3 还有几个不太常用但重要的参数

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=[...],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],

    # 允许前端 JavaScript 读取哪些响应头
    # 默认情况下浏览器只让前端读少数几个"安全"响应头
    # 如果你在中间件里加了自定义响应头（比如 X-Process-Time），
    # 要在这里声明，前端才能读到
    expose_headers=["X-Process-Time", "X-Request-Id"],

    # 预检请求的缓存时间（秒）
    # 浏览器在这个时间内不会重复发送 OPTIONS 预检请求
    # 默认 600 秒（10 分钟）
    max_age=600,

    # 用正则表达式匹配允许的源
    # 比如允许所有 *.example.com 的子域名
    allow_origin_regex=r"https://.*\.example\.com",
)
```

## 4. 通配符 `"*"` 的陷阱

### 4.1 看似最简单的配置

```python
# "允许所有源"——看起来最省事
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],        # 所有源都行
    allow_methods=["*"],        # 所有方法都行
    allow_headers=["*"],        # 所有头都行
)
```

### 4.2 但它有一个严重限制

```python
# ❌ 这个组合是不允许的！
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],        # 通配符
    allow_credentials=True,     # 允许凭证
)
```

**当 `allow_credentials=True` 时，`allow_origins`、`allow_methods`、`allow_headers` 都不能用 `["*"]`**。

为什么？这是浏览器的安全规则，不是 FastAPI 的限制：

```css
allow_origins=["*"] + allow_credentials=True 的含义是：
"任何网站都可以携带用户的 Cookie 来访问我的接口"

这意味着：
  恶意网站 evil.com 可以利用用户浏览器里存的 Cookie
  冒充用户向你的接口发请求——这就是 CSRF 攻击

所以浏览器规定：要携带凭证，就必须明确指定允许的源，不能用通配符。
```

### 4.3 正确的做法

```python
# ✅ 开发环境：明确列出源，允许凭证
app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000", "http://localhost:5173"],
    allow_credentials=True,
    allow_methods=["*"],        # 这里用 * 可以，只要 origins 不是 *
    allow_headers=["*"],        # 这里用 * 可以，只要 origins 不是 *
)

# ✅ 公开 API（不需要 Cookie）：可以用通配符
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=False,    # 不允许凭证时可以用 *
    allow_methods=["GET"],      # 只读 API 只开放 GET
    allow_headers=["*"],
)
```

### 4.4 allow_credentials 为 True 时的严格规则

这条规则非常容易踩坑，单独拿出来强调：

```css
当 allow_credentials=True 时：
  allow_origins    不能是 ["*"]  → 必须明确列出每个源
  allow_methods    不能是 ["*"]  → 实际上 Starlette 会自动展开为具体方法列表，所以这里写 ["*"] 通常没问题
  allow_headers    不能是 ["*"]  → 同上，Starlette 会处理

  但最安全的做法是：三个都明确写出来，不用通配符。
```

## 5. 预检请求（Preflight）和简单请求——深入理解

### 5.1 什么请求会触发预检

不是所有跨域请求都会先发 OPTIONS 预检。浏览器把跨域请求分成两类：

**简单请求（Simple Request）**——不触发预检，直接发：

```css
同时满足以下所有条件：
  1. 方法是 GET、HEAD 或 POST
  2. 没有自定义请求头（只有 Accept、Accept-Language、Content-Language、Content-Type）
  3. Content-Type 只能是：
     - application/x-www-form-urlencoded
     - multipart/form-data
     - text/plain
```

**非简单请求（需要预检）**——浏览器先发 OPTIONS：

```css
以下任一条件成立就触发预检：
  1. 方法是 PUT、DELETE、PATCH 等
  2. 有自定义请求头（比如 Authorization、X-Custom-Header）
  3. Content-Type 是 application/json（最常见的触发原因！）
```

### 5.2 为什么 JSON API 几乎总是触发预检

```css
前端发 POST 请求，Content-Type: application/json
→ application/json 不在简单请求的允许列表里
→ 触发预检
→ 浏览器先发 OPTIONS，等后端确认才发真正的 POST
```

所以如果你的后端是 JSON API（绝大多数 FastAPI 项目都是），**几乎每个非 GET 请求都会触发预检**

### 5.3 预检请求长什么样

```css
OPTIONS /api/projects/ HTTP/1.1
Host: localhost:8000
Origin: http://localhost:3000
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization
```

后端的 CORS 中间件收到这个 OPTIONS 请求后，检查配置，返回：

```css
HTTP/1.1 200 OK
Access-Control-Allow-Origin: http://localhost:3000
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 600
```

浏览器看到这些头，确认"后端允许了"，才发送真正的请求

### 5.4 max_age 的作用

```python
app.add_middleware(
    CORSMiddleware,
    ...
    max_age=600,    # 预检结果缓存 600 秒
)
```

浏览器在 600 秒内不会对同一个接口重复发送 OPTIONS 预检请求。这能减少一半的网络请求量（每个非简单请求本来要发两次：OPTIONS + 真正请求）

## 6. expose_headers——为什么前端读不到自定义响应头

### 6.1 问题

你在上一节的中间件里加了 `X-Process-Time` 响应头：

```python
response.headers["X-Process-Time"] = str(process_time)
```

但前端 JavaScript 读取时：

```javascript
const response = await fetch("http://localhost:8000/api/projects/");
console.log(response.headers.get("X-Process-Time"));
// null ← 读不到！
```

### 6.2 原因

浏览器默认只让前端 JavaScript 读取以下"安全"响应头：

```css
Cache-Control
Content-Language
Content-Length
Content-Type
Expires
Last-Modified
Pragma
```

**其他所有头（包括你自定义的 X-Process-Time）都被浏览器隐藏了。** 响应头确实存在（用浏览器开发者工具的 Network 面板能看到），但 JavaScript 代码读不到

### 6.3 解决：用 expose_headers

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=[...],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
    expose_headers=["X-Process-Time", "X-Request-Id"],    # ← 加这行
)
```

加了之后，前端就能读到这些头了：

```javascript
console.log(response.headers.get("X-Process-Time"));
// "0.0034" ← 读到了！
```

## 8. 常见报错和排查

### 8.1 报错："No 'Access-Control-Allow-Origin' header is present"

```
原因：后端没有配置 CORS 中间件，或者前端的源不在 allow_origins 列表里。

排查：
  1. 确认你加了 CORSMiddleware
  2. 确认 allow_origins 里有前端的完整源（协议 + 域名 + 端口）
  3. 注意：http://localhost:3000 和 http://127.0.0.1:3000 是不同的源！
```

### 8.2 报错："The value of 'Access-Control-Allow-Origin' header must not be '*' when credentials mode is 'include'"

```
原因：allow_origins=["*"] 和 allow_credentials=True 同时使用了。

解决：把 allow_origins 改成具体的源列表。
```

### 8.3 报错："Method PUT is not allowed by Access-Control-Allow-Methods"

```
原因：allow_methods 没有包含你要用的 HTTP 方法。

解决：把 allow_methods 改成 ["*"] 或明确加上需要的方法。
     默认值是 ["GET"]，只允许 GET 请求。
```

### 8.4 报错一切看起来正确但就是不生效

```
排查清单：
  1. 确认 add_middleware 在所有路由定义之前执行（通常放在 app 创建之后的最前面）
  2. 确认前端源的端口号是对的（开发服务器可能换端口了）
  3. 确认 http 和 https 没搞混
  4. 用浏览器开发者工具的 Network 面板看 OPTIONS 请求的响应头
  5. 用 curl 直接测试，确认后端确实返回了 CORS 头：
     curl -X OPTIONS http://localhost:8000/api/projects/ \
       -H "Origin: http://localhost:3000" \
       -H "Access-Control-Request-Method: GET" -v
```

## 9. 对应到你的投标项目

```python
import os
import time

from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI(title="智能投标决策系统")




# ==================== 其他中间件（在 CORS 之前注册） ====================

@app.middleware("http")
async def add_process_time(request: Request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    response.headers["X-Process-Time"] = f"{time.perf_counter() - start:.4f}"
    return response


# ==================== CORS 配置 ====================

# 从环境变量读取允许的源，方便在不同环境切换
raw_origins = os.getenv(
    "CORS_ORIGINS",
    "http://localhost:3000,http://localhost:5173"  # 开发环境默认值
)
origins = [origin.strip() for origin in raw_origins.split(",")]

app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    allow_credentials=True,           # 允许前端发送 Cookie 和 Authorization 头
    allow_methods=["*"],              # 允许所有 HTTP 方法
    allow_headers=["*"],              # 允许所有请求头
    expose_headers=["X-Process-Time"],# 让前端能读到自定义响应头
    max_age=600,                      # 预检请求缓存 10 分钟
)


# ==================== 路径操作函数 ====================

@app.get("/api/projects/", tags=["项目管理"])
async def list_projects():
    """
    前端调用这个接口时：
    1. 浏览器发现跨域 → 先发 OPTIONS 预检请求
    2. CORS 中间件收到 OPTIONS → 检查 Origin 是否在允许列表 → 返回 CORS 头
    3. 浏览器确认允许 → 发送真正的 GET 请求
    4. CORS 中间件在真正的响应里也加上 CORS 头
    5. 浏览器把数据交给前端 JavaScript
    """
    return {
        "projects": [
            {"id": "proj-001", "name": "南昌路面改造工程", "price": 1200},
            {"id": "proj-002", "name": "赣州公路扩建项目", "price": 3500},
        ]
    }


@app.post("/api/projects/", tags=["项目管理"])
async def create_project():
    """
    POST 请求 + Content-Type: application/json → 一定触发预检。
    因为 application/json 不是"简单请求"的 Content-Type。
    """
    return {"id": "proj-003", "name": "新项目"}
```

**前端代码（React/Vue 都一样）**：

```javascript
// 不需要做任何特殊处理——CORS 是后端的事
const response = await fetch("http://localhost:8000/api/projects/", {
    method: "GET",
    credentials: "include",    // 如果需要发送 Cookie
});
const data = await response.json();

// 读取自定义响应头（因为后端配了 expose_headers）
const processTime = response.headers.get("X-Process-Time");
console.log(`请求耗时: ${processTime}s`);
```

## 10. CORS 配置的注意事项

### 10.1 中间件注册顺序

```python
# CORS 中间件通常放在最后面（用 add_middleware 注册）
# 因为 add_middleware 后注册的在最外层，最先处理请求
# CORS 需要最先拦截 OPTIONS 预检请求

app.add_middleware(OtherMiddleware, ...)    # 先注册 → 内层
app.add_middleware(CORSMiddleware, ...)     # 最后注册 → 最外层 → 最先处理

#  CORSMiddleware 没法用装饰器，只能用 add_middleware
```

### 10.2 CORS 不能替代身份验证

```
CORS 保护的是：阻止未授权的前端域名调用你的接口
CORS 不能防止：Postman、curl、Python 脚本等非浏览器工具调用你的接口

所以 CORS 只是"第一道门"，不能替代真正的身份验证（Token、JWT 等）。
两者都要配置。
```

## 11. 本节核心知识点速查

```css
概念                              做法
──────────────────────────────────────────────────────────────────────────
配置 CORS                        app.add_middleware(CORSMiddleware, ...)
允许特定源                        allow_origins=["http://localhost:3000"]
允许所有源                        allow_origins=["*"]（不能和 credentials 同时用）
允许携带凭证                      allow_credentials=True
允许所有方法                      allow_methods=["*"]（默认只允许 GET）
允许所有请求头                    allow_headers=["*"]（默认是空列表）
让前端读取自定义响应头             expose_headers=["X-Process-Time"]
预检请求缓存                      max_age=600（秒）
正则匹配源                        allow_origin_regex=r"https://.*\.example\.com"
```

**本节最核心的三句话**：

1. **CORS 是浏览器的安全策略，不是服务器的限制**。浏览器禁止前端 JavaScript 向不同源的服务器发请求（或读取响应）。CORS 中间件通过在响应里加特定头信息，告诉浏览器"这个前端是我信任的，放行"

2. **配置 CORS 最关键的一步是 `allow_origins`**：把前端的完整源（协议 + 域名 + 端口）加进列表里。如果需要携带 Cookie 或 Authorization 头，设 `allow_credentials=True`，但此时不能用通配符 `["*"]` 作为源

3. **JSON API 几乎总是触发预检请求**：因为 `Content-Type: application/json` 不是"简单请求"允许的类型，浏览器会先发 OPTIONS 预检。CORS 中间件自动处理这些 OPTIONS 请求，你不需要手动写 OPTIONS 路由
