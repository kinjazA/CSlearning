# 01 asyncio：异步编程

## 01.1 为什么需要异步

传统同步代码在遇到 I/O 操作（网络请求、文件读写、数据库查询）时，线程会**阻塞等待**，什么都不做。异步编程的核心思想是：**在等待 I/O 的时候，把控制权交还给事件循环，让它去执行别的任务**

```css
同步模型（串行等待）：
  请求A发出 ──等待──── 响应A ── 请求B发出 ──等待──── 响应B
  总耗时 = A耗时 + B耗时

异步模型（并发等待）：
  请求A发出 ── 请求B发出 ──等待──── 响应A
                                    └── 响应B
  总耗时 ≈ max(A耗时, B耗时)
```

关键区分：**异步 ≠ 多线程**。asyncio 是单线程模型，通过协程切换实现并发，没有线程安全问题，也没有 GIL 的困扰。它适合 **I/O 密集型** 任务，不适合 CPU 密集型计算

## 01.2 事件循环（Event Loop）

事件循环是 asyncio 的引擎，本质上是一个**无限循环**，不断做三件事：

1. 检查有没有可以执行的协程（比如某个 await 的 I/O 完成了）
2. 执行就绪的协程，直到它遇到下一个 await 或执行完毕
3. 回到步骤 1

```python
import asyncio

# 获取当前事件循环（在协程内部）
loop = asyncio.get_running_loop()

# 启动事件循环并运行协程（程序入口，通常只调用一次）
asyncio.run(main())
```

**`asyncio.run()` 做了什么：**

- 创建一个全新的事件循环
- 运行传入的协程直到完成
- 关闭事件循环并清理资源

## 01.3 async def 

普通函数用 `def`，异步函数或协程函数用 `async def`。区别在哪？调用异步函数不会执行函数体，而是返回一个**协程对象**

```python
# 普通函数：调用就执行，立刻得到结果
def make_coffee():
    return "☕"

drink = make_coffee()  # drink 就是 "☕"

# 异步函数：调用不执行！只是拿到一个"承诺"（协程对象）
async def make_coffee_slow():
    return "☕"

promise = make_coffee_slow()  # promise 不是 "☕"！是一个协程对象
print(type(promise))          # <class 'coroutine'>
```

**理解这一步至关重要**：调用 `async def` 函数，就像下了一个外卖单，系统给出一个订单号。你拿着订单号本身并不能吃，你得等外卖送到（用 `await`）才能拿到食物。这是初学者最常犯的错误。Python 会好心提醒你：

```bash
RuntimeWarning: coroutine 'make_coffee_slow' was never awaited
```

翻译：你下了单但从来没去取餐

## 01.4 await：暂停与恢复

`await` 做两件事：

1. 告诉事件循环："我现在要等了，你去忙别的吧"
2. 等到结果回来，把结果拿到手

```python
import asyncio

async def cook(dish, seconds):
    print(f"🍳 开始做 {dish}...")
    await asyncio.sleep(seconds)  # 模拟做菜耗时，这里把控制权还给事件循环
    print(f"✅ {dish} 做好了！")
    return dish

async def main():
    # 串行等待（一道一道做）
    dish1 = await cook("番茄炒蛋", 2)   # 等 2 秒
    dish2 = await cook("红烧肉", 3)     # 再等 3 秒
    # 总共等了 5 秒

asyncio.run(main())
```

到这里可能会说：这跟普通函数有什么区别？一道一道做，还是串行的啊。**没错！单纯用 await 就是串行的。** 要并发，我们需要下一个概念

**可以被 await 的对象（Awaitable）有三种：**

| 类型             | 示例                                | 说明                           |
| ---------------- | ----------------------------------- | ------------------------------ |
| 协程 (Coroutine) | `await my_coroutine()`              | async def 返回的对象           |
| Task             | `await asyncio.create_task(coro())` | 包装后的协程，已注册到事件循环 |
| Future           | `await future`                      | 底层对象，一般不直接用         |

## 01.5 create_task：协程调度

回到餐厅的比喻：你不是一道菜做完再做下一道，而是同时告诉厨房"三道菜一起做"

```python
import asyncio
import time

async def cook(dish, seconds):
    print(f"🍳 {dish} 开始")
    await asyncio.sleep(seconds)
    print(f"✅ {dish} 完成")
    return dish

async def main():
    start = time.time()

    # 同时下三个单（三道菜一起做）
    task1 = asyncio.create_task(cook("番茄炒蛋", 2))
    task2 = asyncio.create_task(cook("红烧肉", 3))
    task3 = asyncio.create_task(cook("蒸鱼", 1))

    # 三道菜都等着上桌
    r1 = await task1
    r2 = await task2
    r3 = await task3

    print(f"总耗时: {time.time() - start:.1f}秒")
    # 3 秒！不是 2+3+1=6 秒

asyncio.run(main())
```

**关键的思维模型：**

```css
create_task = 把订单递给厨房（开始做了）
await task   = 等这道菜上桌（拿结果）

如果你先把三个订单都递进去，厨房同时做三道菜
最终等待时间 = 最慢的那道菜 = 3秒
```

**一个微妙但重要的细节**：`create_task()` 之后菜并没有立即开始做。准确地说，是"注册到了厨房的待做列表"。实际开始做，是在你下一次 `await` 的时候——因为 await 把你（当前协程）暂停了，事件循环才有机会去执行那些 task

```python
async def main():
    task = asyncio.create_task(cook("蛋炒饭", 1))
    # 此刻 cook 还没开始执行！你还霸占着执行权

    print("我还在说话...")   # 这行先执行
    print("我说完了")        # 这行也先执行

    await task  # 现在你让出执行权了，事件循环才开始做蛋炒饭
```

## 01.6 task取消

有时候你等了太久想取消，或者用户关闭了页面不再需要结果

```python
import asyncio

async def download_huge_file():
    try:
        print("开始下载 10GB 文件...")
        await asyncio.sleep(100)  # 模拟漫长下载
        print("下载完成")
    except asyncio.CancelledError:
        # 被取消时会抛这个异常，你可以在这里做清理
        print("下载被取消，清理临时文件...")
        raise   # ⚠️ 必须重新抛出！否则取消不生效

async def main():
    task = asyncio.create_task(download_huge_file())
    await asyncio.sleep(2)  # 等了 2 秒不想等了
    task.cancel()           # 发出取消请求

    try:
        await task
    except asyncio.CancelledError:
        print("确认：下载已取消")

asyncio.run(main())
```

## 01.7 易错点

**错误一：在 async 函数里用了阻塞操作**

```python
import time

async def bad_example():
    time.sleep(5)  # 💀 这会卡死整个事件循环！
    # 想象你这个服务员站在原地发呆 5 秒，所有桌的客人都在干等

async def good_example():
    await asyncio.sleep(5)  # ✅ 让出控制权，事件循环去忙别的
```

怎么区分？简单的判断标准：**如果一个操作是 I/O 相关的（网络、文件、sleep），==在 async 函数里必须用它的 async 版本。==** 如果没有 async 版本（比如某些第三方库），用 `run_in_executor` 扔到线程池：

```python
import asyncio

def cpu_heavy_work(n):
    """这是个 CPU 密集型的同步函数，没有 async 版本"""
    return sum(i * i for i in range(n))

async def main():
    loop = asyncio.get_running_loop()
    # 把阻塞操作扔到线程池，不会卡住事件循环
    result = await loop.run_in_executor(None, cpu_heavy_work, 10000000)
    print(result)
```

**错误二：创建了协程但忘记驱动它**

```python
async def fetch():
    return "数据"

async def main():
    fetch()   # ❌ 什么都不会发生！你只是创建了协程对象然后扔掉了
    await fetch()  # ✅ 这才会执行
```

**错误三：搞混了"并发"和"串行"**

```python
async def main():
    # ❌ 这是串行！一个等完再等下一个
    r1 = await cook("菜A", 2)
    r2 = await cook("菜B", 3)
    # 总耗时 5 秒

    # ✅ 这才是并发！先下单再等
    t1 = asyncio.create_task(cook("菜A", 2))
    t2 = asyncio.create_task(cook("菜B", 3))
    r1 = await t1
    r2 = await t2
    # 总耗时 3 秒
```

# 02 并发执行模式

上一章用 `create_task` + `await` 手动管理并发。但任务多了（比如同时发 100 个请求），手动管理太累了。asyncio 提供了三个高级工具

## 02.1 asyncio.gather

**性格**：把一堆任务全部并发执行，等所有人都做完，按你给的顺序返回结果。`gather` 接收多个 awaitable，**并发执行所有任务**，等全部完成后按**传入顺序**返回结果列表

```python
import asyncio

async def query_db(db_name, delay):
    await asyncio.sleep(delay)
    return f"{db_name}的数据"

async def main():
    # 同时查三个数据库
    results = await asyncio.gather(
        query_db("用户库", 1),
        query_db("订单库", 2),
        query_db("商品库", 1.5),
    )

    # results 的顺序和你传入的顺序一致，不是谁先完成谁在前
    print(results)
    # ['用户库的数据', '订单库的数据', '商品库的数据']

asyncio.run(main())
```

**那如果某个任务出错了怎么办？** 这是 gather 最需要注意的地方：

```python
async def might_fail(name, should_fail=False):
    await asyncio.sleep(0.5)
    if should_fail:
        raise ValueError(f"{name} 炸了💥")
    return f"{name} OK"

async def main():
    # 默认行为：一个炸了，gather 直接把异常抛给你
    # 但注意！其他任务并不会被取消，它们还在后台跑
    try:
        results = await asyncio.gather(
            might_fail("A"),
            might_fail("B", should_fail=True),  # 这个会炸
            might_fail("C"),
        )
    except ValueError as e:
        print(f"出错了: {e}")

    # 换一种策略：return_exceptions=True
    # 意思是"别急着炸，把异常也当成结果收集起来，最后一起给我"
    results = await asyncio.gather(
        might_fail("A"),
        might_fail("B", should_fail=True),
        might_fail("C"),
        return_exceptions=True,   # 关键参数
    )
    for r in results:
        if isinstance(r, Exception):
            print(f"  失败: {r}")
        else:
            print(f"  成功: {r}")

    # 输出：
    #   成功: A OK
    #   失败: B 炸了💥
    #   成功: C OK
```

**什么时候用 gather：** 你需要并发做一批事情，等全部完成后统一处理结果。比如一个页面需要同时请求用户信息、通知列表、推荐内容三个接口，全拿到后再渲染

## 02.2 asyncio.wait

gather 是"全部做完再说"，但有时候你想要更灵活的策略：

- "谁先做完我先处理谁"
- "给你 3 秒，做完多少算多少"
- "只要有一个失败就别等了"

`wait` 返回两个集合：`(done, pending)`，支持三种等待模式

```python
import asyncio

async def task(name, delay):
    await asyncio.sleep(delay)
    return f"{name} done"

async def main():
    tasks = [
        asyncio.create_task(task("A", 3), name="A"),
        asyncio.create_task(task("B", 1), name="B"),
        asyncio.create_task(task("C", 2), name="C"),
    ]

    # ① ALL_COMPLETED（默认）：等全部完成
    done, pending = await asyncio.wait(tasks, return_when=asyncio.ALL_COMPLETED)
    print(f"完成: {len(done)}, 未完成: {len(pending)}")

    # ② FIRST_COMPLETED：有一个完成就返回
    tasks2 = [
        asyncio.create_task(task("X", 3)),
        asyncio.create_task(task("Y", 1)),
    ]
    done, pending = await asyncio.wait(tasks2, return_when=asyncio.FIRST_COMPLETED)
    first = done.pop()
    print(f"最先完成: {first.result()}")  # Y done

    # 取消剩余任务（可选）
    for t in pending:
        t.cancel()

    # ③ FIRST_EXCEPTION：有一个异常就返回
    # 适合 "fail-fast" 场景

    # ④ 超时控制
    tasks3 = [asyncio.create_task(task("慢任务", 10))]
    done, pending = await asyncio.wait(tasks3, timeout=2)
    print(f"2秒内完成: {len(done)}, 超时: {len(pending)}")

asyncio.run(main())
```

**⚠️ 注意**：`wait` 的输入必须是 Task 或 Future 集合（需要自己 `create_task`），而 `gather` 可以直接传协程

**gather vs wait 对比：**

| 特性     | gather                | wait                     |
| -------- | --------------------- | ------------------------ |
| 输入     | awaitable（自动包装） | 必须是 Task/Future 集合  |
| 返回     | 有序结果列表          | (done, pending) 两个集合 |
| 错误处理 | 全部或抛异常          | 逐个检查 task.result()   |
| 超时     | 不支持                | 支持 timeout 参数        |
| 部分完成 | 不支持                | 支持 FIRST_COMPLETED     |

**gather vs wait 怎么选：**

用 gather 的场景——"我需要所有结果，一个都不能少"：合并多个 API 的响应

用 wait 的场景——"我需要更灵活的控制"：有超时限制、想先处理快的、想在第一个失败时就停下来

## 02.3 asyncio.as_completed

`as_completed` 返回一个迭代器，**按完成时间顺序**逐个产出结果，不需要等全部完成。gather 和 wait 都是"等一批完成后处理"，as_completed 是"谁好了我就马上处理谁"

```python
import asyncio

async def search_engine(name, seconds):
    await asyncio.sleep(seconds)
    return f"{name}: 找到 {seconds * 100} 条结果"

async def main():
    searches = [
        search_engine("百度", 0.5),
        search_engine("Google", 2),
        search_engine("Bing", 1),
    ]

    print("同时搜索三个引擎，谁先出结果就先展示：\n")

    for coro in asyncio.as_completed(searches):
        result = await coro    # 按完成时间顺序返回
        print(f"  🔍 {result}")
        # 第 0.5 秒：百度
        # 第 1.0 秒：Bing
        # 第 2.0 秒：Google

asyncio.run(main())
```

**什么时候用**：搜索聚合（先展示最快的结果）、爬虫进度展示、批量任务实时反馈

## 02.4 Semaphore

并发限流，不加控制地并发可能导致目标服务被打爆、文件描述符耗尽、内存溢出。Semaphore（信号量）限制同时执行的协程数量



如果你有 1000 个 URL 要爬，直接 gather 1000 个请求会怎样？

- 对方服务器可能把你封了
- 你的机器可能内存或文件描述符用完
- 网络带宽被打满，反而每个都很慢

Semaphore（信号量）就是一个**限流器**：门口最多同时进 N 个人，有人出来了，外面的才能进

```python
import asyncio

async def fetch(session_id, sem):
    async with sem:  # 获取信号量，超过限制时自动等待
        print(f"  任务{session_id} 开始（当前并发: {sem._value} 剩余名额）")
        await asyncio.sleep(1)
        print(f"  任务{session_id} 完成")
        return session_id

async def main():
    sem = asyncio.Semaphore(3)  # 最多同时 3 个并发

    # 虽然有 10 个任务，但同时最多跑 3 个
    tasks = [fetch(i, sem) for i in range(10)]
    results = await asyncio.gather(*tasks)
    print(f"全部完成: {results}")

asyncio.run(main())
```

**把限流做成装饰器，更优雅：**

```python
import asyncio
from functools import wraps

def limit_concurrency(n):
    """装饰器：限制被装饰函数的最大并发数"""
    sem = asyncio.Semaphore(n)
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            async with sem:
                return await func(*args, **kwargs)
        return wrapper
    return decorator

# 使用：无论外面怎么调用，同时最多 5 个在执行
@limit_concurrency(5)
async def call_api(url):
    await asyncio.sleep(1)
    return f"响应: {url}"
```
