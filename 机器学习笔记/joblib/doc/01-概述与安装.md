## 1.1 joblib 是什么？

**joblib** 是 Python 生态中的一个轻量级工具库，专门为科学计算和机器学习场景设计。它不是在"发明轮子"，而是在 Python 标准库提供的两个已有能力上，做了针对 **大数据量、数值计算、模型训练** 场景的专项优化。

一句话定位：

> joblib = 比 pickle 更高效的序列化 + 透明的函数缓存 + 零心智负担的本地并行

它有且仅有三个核心能力：

| 核心能力 | 一句话解释 | 你的工作里怎么用 |
|----------|-----------|-----------------|
| **高效序列化** | 存/取 numpy 数组和模型，比 pickle 快几倍、省几倍磁盘 | 训练完的模型保存、大特征矩阵落盘 |
| **透明缓存** | 给函数加一个装饰器，同样的输入直接返回缓存不用重算 | 数据预处理、特征工程、昂贵的中间计算结果复用 |
| **简易并行** | 把普通 for 循环改成 Parallel + delayed，一行不动逻辑 | 并行调参、交叉验证、批量特征提取、数据增强 |

---

## 1.2 为什么你要专门学它？

### 1.2.1 三个词覆盖你工作流里最痛的环节

作为一名 **数据分析 / 机器学习 / 深度学习** 从业者，你的日常工作流基本长这样：

```
原始数据 → [预处理] → [特征工程] → [模型训练+调参] → [模型部署+推理]
```

joblib 的三个核心能力恰好映射到其中的关键环节：

```
原始数据 → [预处理] ────────── Memory 缓存，避免每次重跑
         → [特征工程] ──────── Parallel 并行，多CPU加速
         → [模型训练+调参] ── Parallel 并行搜索 + Memory 缓存中间结果
         → [模型部署+推理] ── dump 模型保存，load + mmap 大模型快速加载
```

### 1.2.2 你大概率已经在用它了

如果你用 scikit-learn，那你**已经在用 joblib**，只是可能没有意识到：

- `sklearn.ensemble.RandomForestClassifier(n_jobs=-1)` 的并行引擎就是 joblib
- `GridSearchCV(n_jobs=-1)` 的并行搜索用的也是 joblib
- scikit-learn 官方推荐的模型保存方式就是 `joblib.dump(model, 'model.pkl')`
- 官方文档原话：*"It is recommended to use joblib instead of pickle for scikit-learn estimators."*

### 1.2.3 跟标准库比，到底快在哪省在哪？

**序列化对比（1000万元素 numpy 数组）**：

| 方法 | 写入速度 | 文件体积 | 加载速度 |
|------|---------|---------|---------|
| `pickle.dump()` | 1×（基准） | 1×（基准） | 1×（基准） |
| `joblib.dump()` | **2×–5× 更快** | **2×–4× 更小** | 支持 mmap 零拷贝 |
| `joblib.dump(compress=3)` | ≈ pickle 速度 | **5×–10× 更小** | 支持 mmap 零拷贝 |

**关键**：joblib 检测到 numpy 数组时，会走特殊的序列化路径——直接写原始内存缓冲区，而不是像 pickle 那样逐个元素遍历。

## 1.3 与其他序列化工具的对比

作为 ML 从业者，你一定会接触到这四个工具，它们的关系是：

| 特性 | pickle | cloudpickle | dill | **joblib** |
|------|--------|-------------|------|-----------|
| Python 内置 | ✅ | ❌ | ❌ | ❌ |
| 序列化 lambda / 闭包 | ❌ | ✅ | ✅ | ❌（委托 cloudpickle） |
| 序列化生成器 | ❌ | ❌ | ✅ | ❌ |
| **numpy 数组速度** | 慢 | 慢 | 慢 | ✅ **快（优化）** |
| **内置压缩** | ❌ | ❌ | ❌ | ✅ |
| **内存映射（mmap）** | ❌ | ❌ | ❌ | ✅ |
| **并行计算** | ❌ | ❌ | ❌ | ✅ |
| **缓存（Memoization）** | ❌ | ❌ | ❌ | ✅ |

**关键理解**：joblib 和 cloudpickle/dill **不是竞争关系，是协作关系**：

- joblib 的并行后端 `loky` 内部用 cloudpickle 来序列化发送给子进程的函数
- 如果你有连 cloudpickle 也处理不了的对象（比如某些科学计算模拟器），可以用 `set_loky_pickler("dill")` 切换到 dill
- joblib 专注于**数值密集场景**的序列化效率（numpy 大数组、scikit-learn 模型）
- cloudpickle / dill 专注于**复杂 Python 对象**的序列化覆盖范围

**决策指南**：

| 你的场景 | 用什么 |
|----------|--------|
| 保存 scikit-learn 模型到生产环境 | **joblib** |
| 保存大 numpy 数组（含内存映射） | **joblib** |
| 笔记本里定义的 lambda 传给子进程 | **cloudpickle**（joblib 已内置） |
| 要序列化生成器或极端复杂的动态对象 | **dill** |
| 简单对象，追求最大兼容性 | **pickle** |

## 1.4 核心 API 速览

在你深入各章节之前，先看每个核心 API 的「最小可用」示例，建立直觉。

### 1.4.1 序列化：dump & load

```python
import joblib
import numpy as np
from sklearn.ensemble import RandomForestClassifier

# 训练一个模型
model = RandomForestClassifier().fit(X_train, y_train)

# 保存——比 pickle 更快更小
joblib.dump(model, 'model.joblib', compress=3)

# 加载——支持内存映射，大文件不占满内存
model = joblib.load('model.joblib')

# 加载大数组时使用内存映射（零拷贝）
big_array = joblib.load('features.joblib', mmap_mode='r')
```

### 1.4.2 缓存：Memory

```python
from joblib import Memory

memory = Memory('./cache_dir', verbose=1)

@memory.cache
def preprocess_data(raw_path):
    """耗时一小时的数据预处理，同样的 raw_path 只跑一次"""
    # ... 复杂的加载、清洗、特征提取 ...
    return processed_data

# 第一次调用：真的跑
processed = preprocess_data('data/2024.csv')  # [Memory] Calling preprocess_data...
# 第二次调用：直接读缓存
processed = preprocess_data('data/2024.csv')  # [Memory] 0.000s = ~1h saved
```

### 1.4.3 并行：Parallel & delayed

```python
from joblib import Parallel, delayed

# 普通写法：
results = [train_and_evaluate(params) for params in param_list]

# joblib 并行写法——只改了这行：
results = Parallel(n_jobs=-1, verbose=10)(
    delayed(train_and_evaluate)(params) for params in param_list
)
# ↑ n_jobs=-1：用满全部CPU    ↑ delayed：延迟执行，封装成任务
```

一个更复杂的例子——三行代码实现 10 折交叉验证：

```python
from sklearn.model_selection import KFold

kf = KFold(n_splits=10)
scores = Parallel(n_jobs=-1)(
    delayed(train_fold)(train_idx, test_idx)
    for train_idx, test_idx in kf.split(X)
)
```

### 1.4.4 三者组合

## 1.5 安装

### 1.5.1 基础安装

如果从头创建项目：

```bash
# 创建新项目 + 添加依赖
uv init joblib-learning
cd joblib-learning
uv add joblib numpy scikit-learn scipy pandas matplotlib
```

### 1.5.2 可选依赖

joblib 本身极轻量——除了 `numpy` 没有任何硬依赖。但某些功能需要可选依赖：

| 功能 | 安装方式 |
|------|---------|
| LZ4 压缩（最快压缩算法） | `uv add lz4` 或 `uv sync --extra full` |
| LZMA 压缩（最高压缩率） | Python ≥ 3.3 内置 |
| Zstandard 压缩 | `uv add zstandard` 或 `uv sync --extra full` |
| 并行后端的 progress bar | Python ≥ 3.8 内置（`print`） |

> 本项目已通过 `uv sync --extra full` 预装了 lz4、zstandard 等可选依赖。

### 1.5.3 版本兼容性

| Python 版本 | joblib 1.5.x 支持 |
|-------------|-------------------|
| 3.8 | ❌ 1.5.0+ 已放弃 |
| 3.9 – 3.12 | ✅ 完全支持 |
| 3.13（含 free-threaded / 无 GIL） | ✅ 支持 |
| 3.14 | ✅ 支持（1.5.0+） |
| PyPy | ❌ 1.5.0+ 已放弃 |

**建议**：生产环境用 Python 3.10–3.12 + joblib 1.5.3。

## 1.6 版本历史要点

了解版本演进有助于读懂旧代码和博客：

| 版本 | 日期 | 跟你相关的变化 |
|------|------|---------------|
| 1.4.0 | 2024 | `return_as` 参数；`parallel_config` 上下文管理器 |
| 1.5.0 | 2025-05 | **大版本**：`Memory` 自动创建 `.gitignore`；Parallel 漂亮打印进度；`ensure_native_byte_order` 参数；放弃 Python 3.8 |
| 1.5.1 | 2025-05 | Bug 修复（loky 更新到 3.5.5） |
| 1.5.2 | 2025-08 | Bug 修复（多节点并发缓存冲突；Python 3.13.7+ 兼容） |
| **1.5.3** | **2025-12** | **安全加固**：`.gitignore` 不覆盖已有文件；`pre_dispatch` 安全检查；cloudpickle 3.1.2 |

### 1.5.0 最重要的三个变化

1. **`Memory` 自动创建 `.gitignore`**：缓存目录不会再被误提交到 git
2. **`check_call_in_cache` 扩展**：现在还能验证缓存是否有效，不只是存不存在
3. **`ensure_native_byte_order`**：加载不同字节序系统保存的数组时自动转换
