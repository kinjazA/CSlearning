## 11.1 joblib 在 2026 年 ML 栈中的角色

首先搞清楚一件事：**joblib 不是什么都能做，它只做三件事，但这三件事在单机 ML 开发里几乎没有替代品。**

```
❌ joblib 不管的（有更好的工具）:
  结构化超参搜索      → Optuna / Ray Tune（贝叶斯、剪枝、多目标）
  分布式计算          → Ray / Dask / Spark
  跨框架模型部署      → ONNX / TorchScript
  深度学习训练        → PyTorch / TensorFlow 自己的 DataLoader

✅ joblib 管的（目前在单机 sklearn 生态里没有替代品）:
  模型存取            → dump / load（存模型、存特征、存任何 Python 对象）
  本地函数缓存        → Memory（notebook 调试时，同样的输入不重算）
  单机并行            → Parallel（读数据、提取特征、交叉验证、批量推理）
```

**更准确的理解**：Optuna 管的是"搜哪组参数"这个决策问题；joblib 管的是"让 for 循环跑得更快"这个执行问题。两者是协作关系——你理解了 joblib 的嵌套并行和 n_jobs，Optuna 调参时才不会被"为什么开了并行反而更慢"卡住。

### 11.1.1 三大能力的真实权重（实际工作视角）

三个核心能力在真实 ML 开发中的**出现频率**和**不可替代性**不是均匀的。以下按实际工作中的权重排序：

#### 🥇 dump / load — 刚需，绕不开

- **频率**：每个项目都用，每天用
- **不可替代性**：sklearn 官方唯一推荐的模型持久化方案。pickle 能替代吗？能，但没理由——joblib 对 numpy 数组有专项优化，sklearn 模型本质就是一堆 numpy 数组
- **容易被忽略的高价值用法**：
  1. **存中间数据，不只是模型**。训练可能跑几小时，中间的特征矩阵、CV 结果都应该 dump 下来。调试时不从原始数据重跑，而是 load 中间产物接着往下调——这比 Memory 缓存更灵活、更可控
  2. **mmap 模式加载大数据**。10GB 特征矩阵用 `mmap_mode='r'` 加载，多进程共享一份磁盘数据，不占内存

#### 🥈 Memory — 极度好用，但场景比你想象的窄

- **频率**：开发/调试阶段高频，生产环境低频
- **不可替代性**：唯一一个零配置、磁盘持久化、重启不丢的 Python 函数缓存。`functools.lru_cache` 重启就没了，Redis 要运维
- **关键约束**：Memory 只适合**纯函数**——相同输入一定产生相同输出。很多真实预处理函数不是纯函数（读数据库、调 API、依赖外部文件内容变化），这些没法直接用 Memory
- **最真实的高价值场景**：

  | 场景 | 为什么值钱 |
  |------|-----------|
  | **Notebook 里反复调试** | 预处理 20 分钟，改个下游参数又重跑——加 `@memory.cache`，同样的数据只跑一次 |
  | **特征工程实验** | 试了 50 种特征组合，每一种都从原始数据开始算——缓存让只有第一次慢 |
  | **超参搜索的中间结果** | 跑过一轮 100 组参数的搜索，后面加 50 组新参数——前面的自动命中缓存 |

- **一条务实的判断**：在决定用 Memory 之前，先问自己——「这个函数的输出，能不能用 `dump` 直接存一个中间文件？」如果能，dump/load 往往比 Memory 更直观、更好调试。Memory 胜在**自动化和透明**，适合探索阶段；dump/load 胜在**显式和可控**，适合生产 pipeline

#### 🥉 Parallel — 理解原理比直接调用更重要

- **频率**：间接使用（通过 sklearn）> 直接使用
- **你真正需要「会写」Parallel 的场景只有一种**：要并行执行的任务**不是** sklearn 内置的

  ```python
  # 这些你必须自己写 Parallel：
  # - 并行读 1000 个文件
  # - 并行调 API
  # - 并行处理图片/文本
  # - 对 50 个城市分别训练模型

  # 这些 sklearn 已经帮你并行好了，你只需要设 n_jobs：
  # - RandomForestClassifier(n_jobs=-1)
  # - GridSearchCV(n_jobs=-1)
  # - cross_val_score(n_jobs=-1)
  ```

- **但你更需要「理解」Parallel 的原理，因为**：
  1. 嵌套并行是最常见的性能陷阱——外层 Parallel 开 8 进程 × 内层 sklearn 的 n_jobs=-1 = 64 线程抢 8 核
  2. `parallel_backend` 是调试利器——任务 I/O 密集时一行切换全局后端
  3. 理解数据如何传递给子进程——大数组默认 pickle 拷贝 N 份，mmap 让所有子进程共享

- **一句话总结 Parallel 的定位**：直接写 `Parallel(n_jobs=-1)` 的频率远低于 `dump/load`，但**理解它的行为**是 debug sklearn 性能问题的基础能力

## 11.2 用途一：模型存取（dump / load）

### 这是 joblib 最不可替代的功能

训练完的 sklearn Pipeline、特征矩阵、中间结果——开发阶段存取这些，joblib 是标准答案。比 pickle 快 2-5×，文件小 30-70%，支持零拷贝加载。

> **不只是存模型**：dump/load 最容易忽视的用法是存**中间产物**。训练跑了几小时的中间特征矩阵、交叉验证结果、预处理后的数据，都应该 dump 下来。调试时不从原始数据重跑，而是 load 中间产物接着往下调——这比 Memory 缓存更灵活、更可控、更容易审计。

### 速查

```python
# 保存单个模型
joblib.dump(model, f'model_v1_20240625.joblib', compress=3)

# 加载
model = joblib.load('model_v1_20240625.joblib')

# ⭐ 保存整个 Pipeline（生产环境的标准做法）
from sklearn.pipeline import Pipeline
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('pca', PCA(n_components=50)),
    ('clf', LogisticRegression())
])
pipe.fit(X_train, y_train)
joblib.dump(pipe, 'pipe_v1.joblib', compress=3)

# 部署时一行加载，直接 predict——预处理逻辑不会丢
pipe = joblib.load('pipe_v1.joblib')
pipe.predict(X_new)  # scaler → pca → clf，和训练时完全一致

# 大文件（> 可用内存的一半）→ mmap 加载
X = joblib.load('X.joblib', mmap_mode='r')
```

> **为什么保存整个 Pipeline 是铁律**：只保存模型不保存预处理，是 ML 部署里排名第一的 bug 来源。训练时做了标准化、编码、降维——推理时漏掉任何一步，结果就是错的。把整个 Pipeline 存成一个文件，加载回来直接 `predict()`，预处理逻辑永远不会和模型脱节。

### 压缩决策

```
日常开发                    → compress=3
需要 mmap 零拷贝            → compress=0（互斥！压缩后 mmap 失效）
生产服务冷启动（速度优先）    → compress=('lz4', 0)
最佳全能（装了 zstd）        → compress=('zstd', 3)
长期归档（体积优先）          → compress=('lzma', 9)
```

### 这个用途下的坑

| 坑 | 错误 | 正确 |
|------|------|------|
| 大文件 OOM | `load('big.joblib')` | `load('big.joblib', mmap_mode='r')` |
| 压缩 + mmap 互斥 | `dump(x,f,compress=3); load(f,mmap_mode='r')` | 二选一 |
| 覆盖历史模型 | `dump(m, 'model.joblib')` | `dump(m, f'model_v2_{date}.joblib')` |
| 数据打一个包 | `dump({'X':X,'y':y}, f)` | `dump(X,f); dump(y,f)` 分别存 |
| 加载外部文件 | `load(user_uploaded)` | 只加载信任来源（load 会执行代码） |
| **只存模型不存预处理** | `dump(clf, 'model.joblib')` 丢了 scaler/encoder | `dump(pipe, 'pipe.joblib')` 把整个 Pipeline 存下来 |

## 11.3 用途二：本地函数缓存（Memory）

### 这个功能同样没有替代品

`functools.lru_cache` 重启就丢；Redis 太重。Memory 卡在中间——**磁盘级透明缓存，重启还在，零配置**。

**核心场景**：你在 notebook 里反复调试，预处理和特征工程每次重跑 20 分钟。加一个 `@memory.cache`，同样的数据只跑一次。

### 速查

```python
from joblib import Memory
memory = Memory('./cache', verbose=1)

@memory.cache
def preprocess(data_path):
    """同样的路径只跑一次"""
    df = pd.read_csv(data_path)
    # ... 30 行清洗 ...
    return df
```

### 常用操作

| 操作 | 代码 | 说明 |
|------|------|------|
| 装饰函数 | `@memory.cache` | 自动缓存返回值 |
| 排除不影响结果的参数 | `@memory.cache(ignore=['verbose'])` | verbose/log 类参数不计入缓存 key |
| 检查是否已缓存 | `memory.check_call_in_cache(fn, *args)` | 返回 True/False，不执行函数 |
| 清空缓存 | `memory.clear()` | 数据源或逻辑大改后清一下 |
| 按时间清理 | `memory.reduce_size(age_limit=timedelta(days=30))` | 清理实验残留下来的旧缓存 |
| 不落盘 | `Memory(None)` | 只存内存，进程重启消失 |

### 缓存什么时候命中、什么时候不命中

```
命中   ← 函数源代码相同 + 参数值相同（ignore 的除外）
不命中 ← 函数源代码变了 / 参数值变了
```

### 这个用途下的坑

| 坑 | 说明 |
|------|------|
| 副作用不执行 | 缓存命中时函数体不跑 → 写文件、发请求、改全局变量都不会发生 |
| 修改已缓存的可变对象 | 缓存返回的 list/dict 别原地改——先 copy |
| ignore 后参数变了不重算 | `ignore=['random_state']` → 改了 seed 也拿旧结果 |
| 改了数据源但参数路径没变 | Memory 不知道文件内容变了 → 手动清缓存 |

### 什么时候不该用 Memory — 适用边界

不是所有「跑得慢的函数」都适合加缓存。以下情况**不要**用 Memory：

| 不适合的情况 | 原因 | 更好的方案 |
|-------------|------|-----------|
| 函数有副作用（写文件、发请求、更新数据库） | 缓存命中时副作用不执行 → 文件不更新、请求不发 | 去掉副作用，或不用缓存 |
| 函数依赖外部可变状态（读数据库、调 API） | Memory 不知道外部数据变了，返回过期结果 | 手动管理中间文件（dump/load） |
| 函数很简单、跑得很快 | 缓存 lookup 的开销 > 重算的开销 | 不缓存 |
| 需要精确控制缓存失效时机 | Memory 的失效逻辑是自动的（源码变了就失效），不够灵活 | 用 dump/load 显式管理 |
| 输出是几百 MB 的大对象 | 缓存落盘慢，读取也慢 | 用 dump/load 存为 .joblib 文件 |
| 生产环境需要跨机器共享 | Memory 是本地磁盘缓存 | Redis / 共享文件系统 |

### dump/load vs Memory — 选型指南

两者都能「避免重复计算」，但适用场景完全不同：

```
                     Memory                        dump/load
─────────────────────────────────────────────────────────────
触发方式              自动（装饰器）                  显式调用
缓存 key              函数源码 + 参数值              你自己定的文件名
失效控制              源码变了自动失效                你手动管理版本
适合阶段              探索 / 调试 / notebook         工程化 pipeline
可审计性              低（黑盒）                     高（文件即记录）
灵活性                低（绑定函数签名）              高（想存什么存什么）
重启后                保留                           保留
─────────────────────────────────────────────────────────────
结论：探索阶段用 Memory（自动省心），生产 pipeline 用 dump/load（显式可控）
```

## 11.4 用途三：单机并行（Parallel）

### 当前角色：管"执行"，不管"决策"

Optuna / Ray Tune 管的是**搜哪组参数**（搜索策略）。但你的工作流里到处都是和调参无关的 for 循环，这些东西 Optuna 不碰，Parallel 管：

```
Parallel 的典型用武之地（都不是调参）:
  - 1000 个文件，并行读 + 预处理
  - 200 张图，并行提取特征
  - 10 折交叉验证，每折独立训练
  - 对 50 个城市分别训练模型
  - 批量推理：100 万条数据分 8 份同时 predict
```

### 速查

```python
from joblib import Parallel, delayed

# 串行 → 并行，只改这一行
results = Parallel(n_jobs=-1)(
    delayed(fn)(args) for args in iterable
)
```

### 什么时候需要自己写 Parallel，什么时候不需要

这是 Parallel 最容易误解的地方——不是所有「并行」都需要你亲自写 `Parallel + delayed`：

```
你必须自己写 Parallel                     sklearn 已经帮你并行好了（只需设 n_jobs）
─────────────────────────────────────    ──────────────────────────────────────────
并行读 1000 个文件                        RandomForestClassifier(n_jobs=-1)
并行调 API / 发请求                       GridSearchCV(n_jobs=-1)
并行处理图片 / 文本 / 音频                cross_val_score(n_jobs=-1)
对 50 个城市 / 客户分别训练模型            cross_validate(n_jobs=-1)
批量推理：自定义分片并行 predict          任何 sklearn 估计器的 n_jobs 参数
```

**关键**：当你的任务可以用 sklearn 内置函数表达时，直接用它的 `n_jobs` 参数，不要自己套一层 Parallel。只有当任务**不是 sklearn 内置的**（读文件、调 API、自定义逻辑），才需要自己写。

**但无论如何你都需要理解 Parallel 的行为**——即使你从不自己写，sklearn 内部的 `n_jobs` 就是 Parallel。嵌套并行、数据传递、后端选型这些坑，你不理解就会在调 sklearn 性能时踩到。

### 常用参数

| 参数 | 常用值 | 什么时候用 |
|------|--------|-----------|
| `n_jobs` | `-1` | 日常——用满所有核心 |
| | `1` | 调试——单进程错误信息清晰 |
| `backend` | `'loky'`（默认） | CPU 密集：训练模型、numpy 运算 |
| | `'threading'` | I/O 密集：读文件、调 API、数据库查询 |
| `verbose` | `10` | 显示进度和剩余时间 |
| `return_as` | `'generator_unordered'` | 边算边拿结果，不按顺序 |
| `timeout` | `30` | 单个任务卡死超过 30 秒就抛异常 |

### parallel_backend — 不改代码切换后端

```python
from joblib import parallel_backend

# 范围内所有 Parallel（包括 sklearn 内部的）都用 threading
with parallel_backend('threading'):
    RandomForestClassifier(n_jobs=-1).fit(X, y)
```

### 后端选型

```
CPU 密集（训练模型、numpy、纯 Python） → loky（默认）
I/O 密集（读文件、调 API）            → threading
numpy C 扩展运算                     → 两个都行（C 扩展释放 GIL）
不确定                                → loky
```

### 这个用途下的坑

| # | 坑 | 错误 | 正确 |
|---|------|------|------|
| 1 | **嵌套并行** | 外层 8 进程 × 内层 n_jobs=-1 = 64 竞争 | 内层设 `n_jobs=1` |
| 2 | 全局变量 | 子进程改 `list.append(x)` | 用 Parallel 的返回值 |
| 3 | lambda 传不进子进程 | `delayed(lambda x: x**2)(i)` | 用命名函数 |
| 4 | 大数组每个子进程复制 | 每个子进程各收一份 5GB X | `load(X, mmap_mode='r')` 共享 |
| 5 | 一个任务失败全丢 | 抛异常 → Parallel 终止 | 任务内 `try/except`，返回 `{'status':'error'}` |
| 6 | Windows 上顶层代码 | 没加保护 | `if __name__ == '__main__':` |

### 安全任务模板

```python
def safe_task(params):
    try:
        result = actual_work(params)
        return {'status': 'ok', 'result': result}
    except Exception as e:
        return {'status': 'error', 'error': str(e)}

results = Parallel(n_jobs=-1)(delayed(safe_task)(p) for p in param_list)
ok = [r for r in results if r['status'] == 'ok']
```

## 11.5 三合一：一个完整工作流模板

```python

from joblib import Memory, Parallel, delayed
from datetime import date

TODAY = date.today().isoformat()
memory = Memory('./cache', verbose=1)

# ── 用途二：缓存昂贵预处理 ──
@memory.cache
def preprocess(data_path):
    df = pd.read_csv(data_path)
    # ... 清洗 ...
    return X, y

# ── 用途三：并行执行 ──
def train_one(params, X, y):
    model = SomeModel(**params, n_jobs=1)  # ← 内层 n_jobs=1
    model.fit(X, y)
    return {'score': model.score(X_val, y_val), 'model': model}

results = Parallel(n_jobs=-1, verbose=10)(
    delayed(train_one)(p, X, y) for p in param_list
)

# ── 用途一：持久化 ──
best = max(results, key=lambda r: r['score'])
joblib.dump(best['model'], f'model_v1_{TODAY}.joblib', compress=3)
```

## 11.6 十条铁律

| # | 铁律 | 违反的后果 |
|---|------|-----------|
| 1 | **内层 n_jobs=1** | 嵌套并行 → 线程爆炸，比串行慢 |
| 2 | **大数组用 mmap** | 每个子进程复制一份 → OOM |
| 3 | **压缩和 mmap 互斥** | `compress=3` + `mmap_mode='r'` → mmap 静默失效 |
| 4 | **文件名带版本号** | `dump(m,'m.joblib')` → 旧模型永久丢失 |
| 5 | **缓存的函数是纯函数** | 副作用在缓存命中时不执行 |
| 6 | **任务内 try/except** | 一个任务抛异常 → 全部结果丢失 |
| 7 | **CPU 用 loky，IO 用 threading** | 用错后端 → 性能反而变差 |
| 8 | **load 只加载信任来源** | 恶意文件 → 任意代码执行 |
| 9 | **sklearn 版本对齐** | 1.5 存 → 1.2 读 → crash |
| 10 | **先 n_jobs=1 调通，再 n_jobs=-1** | 多进程错误难定位 |

## 11.7 性能排查清单

工作流不够快时，按这个顺序排查：

```
□ 0. 真正的瓶颈在哪？（先别急着优化）
      → 用 time.perf_counter() 或 cProfile 定位最慢的环节
      → 不要把时间花在优化只占 5% 运行时间的代码上
      → 瓶颈可能在 I/O、数据加载、特征工程，不在模型训练本身

□ 1. 是否开了 n_jobs=-1？
      → 确认不是 n_jobs=1

□ 2. 后端选对了吗？
      → CPU 密集用 loky，I/O 密集换 threading

□ 3. 有嵌套并行吗？
      → 被 Parallel 调用的函数内部，模型设 n_jobs=1

□ 4. 大数组是否在重复复制？
      → dump(X, compress=0) → load(X, mmap_mode='r')

□ 5. 预处理是否在重复执行？
      → 加 @memory.cache

□ 6. 压缩算法对吗？
      → 热路径用 lz4，归档用 lzma，日常 zstd

□ 7. 任务耗时差异大吗？
      → 设 batch_size=1 改善负载均衡

□ 8. 结果收集阻塞吗？
      → 换 return_as='generator_unordered'
```

## 11.8 场景代码速查

```python
# ── 保存 sklearn 模型 ──
joblib.dump(model, f'model_v1_{date}.joblib', compress=3)

# ── 保存整个 Pipeline（⭐ 生产环境标准做法）──
pipe = Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())])
pipe.fit(X, y)
joblib.dump(pipe, 'pipe_v1.joblib', compress=3)
pipe = joblib.load('pipe_v1.joblib')
pipe.predict(X_new)  # 预处理 + 推理，一步到位

# ── 加载模型 ──
model = joblib.load('model_v1.joblib')

# ── 存 + 取大数组（mmap）──
joblib.dump(X, 'X.joblib', compress=0)

# ── 缓存预处理 ──
@Memory('./cache', verbose=1).cache

Parallel(n_jobs=-1)(

# ── 并行读文件（I/O 密集）──

# ── 并行批量提取特征 ──
Parallel(n_jobs=-1, verbose=10)(
    delayed(extract)(path) for path in image_paths
)

# ── Memory + Parallel 组合 ──
@memory.cache
def train_one(p, X, y):
    return SVC(n_jobs=1, **p).fit(X, y).score(X_val, y_val)
Parallel(n_jobs=-1)(delayed(train_one)(p, X, y) for p in grid)

# ── 安全并行 ──
def safe(p):
    try: return {'ok':True, 'result':work(p)}
    except Exception as e: return {'ok':False, 'error':str(e)}
results = Parallel(n_jobs=-1)(delayed(safe)(p) for p in grid)

# ── 增量训练大矩阵 ──

m = SGDClassifier(loss='log_loss')
for s in range(0, len(X), 50000):
    batch = np.array(X[s:s+50000])
    m.partial_fit(batch, y[s:s+50000], classes=[0,1])

# ── 控制 sklearn 内部并行后端 ──

    RandomForestClassifier(n_jobs=-1).fit(X, y)
```
