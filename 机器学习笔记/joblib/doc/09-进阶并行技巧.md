## 9.1 解决什么问题？

第 07 章的基础用法已经覆盖 80% 场景。但生产环境里你会遇到这些情况：

- "10000 个任务，每个结果都很大，我不想等全部算完才拿到第一个结果"
- "有些任务偶尔卡死，能不能设个超时自动跳过？"
- "任务有大有小，有些 worker 闲着有些忙死，怎么办？"
- "配合 tqdm 显示进度条，不是只看 verbose 的文字"

本章就是为这些"更进一步"的场景准备的。

## 9.2 return_as — 边算边拿结果

默认情况下，`Parallel` 等所有任务算完才一次性返回全部结果。但如果你有 1000 个任务，可能想先拿到先算完的：

```python
# 默认：全部算完才拿到结果
results = Parallel(n_jobs=4)(delayed(task)(i) for i in range(1000))
# 等了 10 分钟 → 一次性拿到 1000 个结果

# 边算边拿：算完一个就 yield 一个
for result in Parallel(n_jobs=4, return_as='generator_unordered')(
    delayed(task)(i) for i in range(1000)
):
    print(f"刚完成一个: {result}")
    # 你可以边收结果边处理，不需要等全部算完
```

### 三种 return_as 模式

| 值 | 行为 | 使用场景 |
|------|------|---------|
| `'list'`（默认） | 全部算完，按输入顺序返回 | 常规场景 |
| `'generator'` | 边算边返回，**保持输入顺序** | 需要流式处理但顺序重要 |
| `'generator_unordered'` | 边算边返回，**谁先算完先给谁** | 顺序不重要，追求最快拿到结果 |

```python
# generator：保证顺序，但快的任务要等前面的任务完成才能返回
for r in Parallel(n_jobs=4, return_as='generator')(tasks):
    process(r)

# generator_unordered：不保证顺序，最快的先返回
for r in Parallel(n_jobs=4, return_as='generator_unordered')(tasks):
    process(r)  # 可能先收到任务#5，再收到任务#0
```

## 9.3 timeout — 超时跳过

有些任务可能因为数据问题、死循环、资源竞争而卡住。设个超时，超过时间自动放弃：

```python
results = Parallel(n_jobs=4, timeout=30)(
    delayed(task)(i) for i in range(100)
)
# 任何单个任务超过 30 秒没有返回 → 抛出 TimeoutError
```

```python
# 配合 return_as='generator_unordered' 逐个捕获超时
from joblib import Parallel, delayed
import time

def maybe_slow(i):
    if i == 7:
        time.sleep(100)  # 模拟卡死
    return i ** 2

for result in Parallel(n_jobs=4, timeout=5, return_as='generator_unordered')(
    delayed(maybe_slow)(i) for i in range(20)
):
    print(f"完成: {result}")
# 任务 7 超时 → 整个 Parallel 抛出异常
```

**注意**：`timeout` 是单个任务的超时，不是总超时。而且超时后整个 Parallel 会抛异常，不是只跳过那个任务。

## 9.4 batch_size 和 pre_dispatch — 控制调度

### 9.4.1 问题

默认情况下，Parallel 把任务尽可能多地分发给 worker。当任务数量远大于 worker 数时（10000 个任务、8 个 worker），每个 worker 被"预分配"了大量任务。如果任务执行时间差异大，短的 worker 闲下来时，任务已经被分配给了慢的 worker——**负载不均**。

### 9.4.2 batch_size

```python
# 把任务分成批次，每个批次作为一个调度单元
# worker 完成一个 batch 后，才拿下一个 batch
results = Parallel(n_jobs=4, batch_size=10)(
    delayed(task)(i) for i in range(10000)
)
```

- `batch_size=1`：最细粒度调度——负载最均衡，但调度开销最大
- `batch_size='auto'`（默认）：让 joblib 根据任务数自动选择
- `batch_size=50`：每个批次 50 个任务——减少调度开销，适合任务都很短的情况

**经验**：任务耗时均匀 → 默认就好。任务耗时差异大 → `batch_size=1` 或较小值。

### 9.4.3 pre_dispatch

控制"提前分发给 worker 的任务数量"：

```python
# 任何时候最多只有 20 个任务被分发出去
# 剩下的留在主进程的生成器里，按需分发
results = Parallel(n_jobs=4, pre_dispatch=20)(
    delayed(task)(i) for i in range(10000)
)
```

**什么时候用**：任务生成器本身很耗内存（如从大文件逐行生成），控制 `pre_dispatch` 避免一次把所有任务都生成出来。

## 9.5 max_nbytes — 大数组自动 mmap

当函数返回大 numpy 数组时，joblib 可以自动把它们转成 mmap 来减少内存占用：

```python
# 任何返回值大于 1MB 的 numpy 数组，自动转为 memmap
results = Parallel(n_jobs=4, max_nbytes='1M')(
    delayed(return_big_array)(i) for i in range(100)
)
```

- `max_nbytes=None`（默认）：不转换，直接返回
- `max_nbytes='1M'`：大于 1MB 的数组自动转为 memmap
- `max_nbytes='auto'`：自动选择阈值

**场景**：每个任务返回一个大数组，100 个任务的返回结果加起来可能 OOM。用 `max_nbytes` 让结果留在磁盘上。

## 9.6 错误处理策略

```python
# 默认行为：任何一个子任务出错 → 整个 Parallel 立即抛异常
try:
    results = Parallel(n_jobs=4)(
        delayed(risky_task)(i) for i in range(100)
    )
except Exception as e:
    print(f"任务失败了: {e}")
    # 但之前成功的结果也丢了
```

实践中更好的做法：**在任务函数内部捕获异常，返回错误信息而非让它抛出**：

```python
def safe_task(i):
    try:
        return {"status": "ok", "i": i, "result": risky_operation(i)}
    except Exception as e:
        return {"status": "error", "i": i, "error": str(e)}

results = Parallel(n_jobs=4)(delayed(safe_task)(i) for i in range(100))

# 分类处理
ok_results = [r for r in results if r["status"] == "ok"]
errors = [r for r in results if r["status"] == "error"]
print(f"成功: {len(ok_results)}, 失败: {len(errors)}")
```

这样哪怕 1 个任务报错，其他 99 个的结果也不丢。

## 9.7 常见坑

### 坑 1：timeout + generator_unordered 的交互

```python
# ❌ timeout 对 generator 模式的行为可能不符合预期
# 超时后整个迭代会抛异常，而不是只跳过一个任务
for r in Parallel(n_jobs=4, timeout=5, return_as='generator_unordered')(tasks):
    print(r)  # 某个任务超时 → 整体抛异常
```

### 坑 2：batch_size 太大导致负载不均

```python
# 10 个任务，batch_size=5 → 只有 2 个 batch
# 如果前 5 个都慢，后 5 个都很快 → worker 2 闲等 worker 1
results = Parallel(n_jobs=4, batch_size=5)(...)
# 10 个任务不需要设 batch_size，默认就好
```

### 坑 3：generator 模式下忘记消费完

```python
# generator 模式需要全部消费完，子进程才会正常退出
gen = Parallel(n_jobs=4, return_as='generator')(tasks)
first = next(gen)       # 拿第一个
# ... 没有继续遍历 → 子进程可能僵死

# ✅ 确保遍历完，或手动关闭
```

## 9.8 本章要点

- [ ] `return_as='generator_unordered'` — 边算边拿，不用等全部完成
- [ ] `timeout` — 防止单个任务卡死拖垮整体
- [ ] `batch_size` — 任务耗时差异大时调小，改善负载均衡
- [ ] `max_nbytes='1M'` — 返回值是大数组时自动 mmap，省内存
- [ ] 错误处理：在任务函数内部 try/except，不要让 Parallel 直接接异常
