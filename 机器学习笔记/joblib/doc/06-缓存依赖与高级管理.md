## 6.1 解决什么问题？

第 05 章的 `@memory.cache` 默认行为已经解决 80% 的问题。但实际工作中你会遇到这些情况：

- "我有 100 个参数，其中一个 `verbose` 标志不影响结果——但它一变缓存就失效"
- "我想知道某个函数 + 参数的组合是否已经缓存了，而不是直接调用"
- "缓存文件太多了，我想只保留最近一个月的"
- "我在多台机器上跑同一个项目，能不能共享缓存？"

本章就是为这些场景准备的。

## 6.2 ignore — 让某些参数不影响缓存

### 6.2.1 问题

```python
@memory.cache
def train_model(data_path, model_type, verbose=False):
    # verbose=True 只是多打几行日志，不影响训练结果
    ...

# 这两个调用结果完全一样，但参数不同 → 缓存不命中，白白重算
train_model('data.csv', 'rf', verbose=False)  # 执行
train_model('data.csv', 'rf', verbose=True)   # 又执行一遍！
```

### 6.2.2 解决

```python
@memory.cache(ignore=['verbose'])
def train_model(data_path, model_type, verbose=False):
    ...

train_model('data.csv', 'rf', verbose=False)  # 执行
train_model('data.csv', 'rf', verbose=True)   # 缓存命中 ✓
# verbose 被忽略，不影响缓存 key
```

**适用参数**：任何不影响函数**业务结果**的参数——`verbose`、`debug`、`logger`、`random_state`（如果你不关心可复现性）。

### 6.2.3 重要：ignore 不会"用新参数重算"

`ignore` 的含义是：**计算缓存 key 时，假装这个参数不存在。** 但它**不会**让你用新参数值重算一遍。

```python
@memory.cache(ignore=['verbose'])
def compute(x, verbose=False):
    if verbose:
        print(f"计算 {x}...")      # ← 这行有副作用（打印）
    return x ** 2

# 第一次：verbose=False，函数真的执行了
compute(5, verbose=False)
# → 函数体执行，verbose=False，不打印
# → 返回 25，缓存 key = hash(compute源码, x=5)，不含 verbose

# 第二次：verbose=True，但缓存 key 和上次一样（因为 ignore）
compute(5, verbose=True)
# → 缓存命中，函数体根本没执行
# → 虽然你传了 verbose=True，但 print 没触发
# → 返回的是上次算好的 25
```

**关键理解**：第二次调用时，`verbose=True` 这个值被传入了——但它没有任何机会生效，因为函数体根本没跑。缓存命中 = 跳过函数体 = 忽略所有参数的实际值。

所以 `ignore` 的适用前提是：**你确信这个参数的变化不会影响函数返回值。** 如果你 ignore 了一个实际会影响结果的参数，你会拿到旧参数对应的结果。

## 6.3 检查缓存状态

### 6.3.1 缓存是否存在

```python
from joblib import Memory
memory = Memory('./cache')

@memory.cache
def expensive(x):
    return x ** 2

# 检查是否已有缓存（不执行函数）
if memory.check_call_in_cache(expensive, 5):
    print("已有缓存，直接读")
    result = expensive(5)  # 这次真的会命中
else:
    print("没缓存，需要计算")
    result = expensive(5)  # 这次执行
```

`check_call_in_cache(fn, *args)` 的返回值：
- `True` — 已有缓存，`fn(*args)` 会直接返回缓存值
- `False` — 没缓存，调用会真正执行

### 6.3.2 实际用途

在并行计算或批量任务中，先检查缓存再决定是否提交计算：

```python
# 1000 个任务，只提交那些还没缓存的
tasks_to_run = []
for params in param_grid:
    if not memory.check_call_in_cache(train_one, params):
        tasks_to_run.append(params)

print(f"1000 个任务中 {len(tasks_to_run)} 个需要计算，其余已缓存")
```

## 6.4 缓存清理策略

### 6.4.1 全量清空

```python
memory.clear()  # 删除该 Memory 实例的所有缓存文件
```

### 6.4.2 按时间清理

```python
from datetime import timedelta

# 删除 30 天前的缓存
memory.reduce_size(age_limit=timedelta(days=30))

# 删除 6 小时前的缓存
memory.reduce_size(age_limit=timedelta(hours=6))
```

场景：定期清理实验过程中产生的大量中间结果缓存，只保留近期有用的。

### 6.4.3 手动删除特定函数的缓存

```python
import shutil

# 缓存目录结构：./cache/joblib/模块名/函数名/
cache_path = './cache/joblib/__main__/train_model'
shutil.rmtree(cache_path)  # 只删 train_model 的缓存
```

## 6.5 不落盘的缓存

```python
memory = Memory(None, verbose=1)  # location=None → 只存内存，不写磁盘

@memory.cache
def compute(x):
    time.sleep(1)
    return x ** 2

compute(5)  # 执行 → 结果存在内存
compute(5)  # 缓存命中 ✓（但不会写入磁盘）
# 进程重启后缓存消失
```

场景：
- 同一进程内临时避免重复计算（比如一个 web 请求周期内）
- 不想留下磁盘痕迹
- 单元测试中替代 mock

## 6.6 跨项目共享缓存

### 6.6.1 多 notebook 共享

```python
# notebook_A.ipynb
memory = Memory('/shared/cache')
@memory.cache
def preprocess(path):
    ...

# notebook_B.ipynb
memory = Memory('/shared/cache')  # 同一个目录
@memory.cache
def preprocess(path):             # 同一个函数的代码必须一样
    ...

# notebook_A 跑完 preprocess('data.csv') 后
# notebook_B 调用 preprocess('data.csv') → 缓存命中！
```

**关键条件**：函数源代码必须完全相同。如果两个 notebook 里的 `preprocess` 函数体有哪怕一个注释不同，缓存 key 就不匹配。

### 6.6.2 实际建议

```python
# 把共享的缓存函数放在一个公共模块里
# shared_utils.py
from joblib import Memory

memory = Memory('/shared/cache')

@memory.cache
def preprocess(path):
    """统一的预处理逻辑"""
    ...

# 所有 notebook 都从同一个模块 import
# notebook_A.ipynb & notebook_B.ipynb
from shared_utils import preprocess

# 保证函数源码一致，缓存互相命中
```

## 6.7 缓存结果的 mmap 加载

如果缓存的返回值包含大 numpy 数组，可以用 `mmap_mode` 加载：

```python
memory = Memory('./cache', mmap_mode='r')

@memory.cache
def load_features(file_path):
    return np.load(file_path)  # 返回大数组

# 缓存的数组通过 mmap 访问，不占内存
features = load_features('big_features.npy')
```

**注意**：这和压缩不兼容——如果缓存时用了压缩，`mmap_mode` 会被忽略。

## 6.8 常见坑

### 坑 1：ignore 的参数变化不会触发重新计算

```python
@memory.cache(ignore=['random_state'])
def train(data, random_state=42):
    rng = np.random.RandomState(random_state)
    # ...

train(data, random_state=42)  # 执行 → 缓存
train(data, random_state=123) # 命中缓存 → 返回的是 random_state=42 的结果！
```

你声明了 `ignore`，就要接受"改了它也不会重算"。如果某个参数真的影响结果，就不要 ignore。

### 坑 2：多机器共享缓存目录的并发冲突

```python
# 机器 A 和 机器 B 同时向 /shared/cache 写缓存
# joblib 1.5.2+ 已修复大部分并发问题
# 但跨机器的文件系统竞态仍然可能发生，建议：
# - 每台机器用自己的缓存目录（除非网络文件系统确实可靠）
# - 或先在一台机器上预热缓存，其他机器只读
```

### 坑 3：缓存的函数返回了生成器

```python
@memory.cache
def generate_numbers(n):
    return (i for i in range(n))  # 返回生成器

gen = generate_numbers(100)  # 缓存的是生成器对象
list(gen)                    # [0, 1, 2, ...] 第一次消费
list(gen)                    # [] 第二次消费！生成器已耗尽
```

**规则**：缓存的函数应该返回**可反复使用的结果**（list、ndarray、DataFrame），不要返回生成器或迭代器。

## 6.9 本章要点

- [ ] `ignore=['verbose']` — 不影响结果的参数不计入缓存 key
- [ ] `check_call_in_cache(fn, *args)` — 先问再算，避免重复提交
- [ ] `memory.clear()` 全清，`memory.reduce_size(age_limit=...)` 按时间清
- [ ] `Memory(None)` — 不落盘，只在内存缓存
- [ ] 多 notebook 共享缓存 → 函数放在公共模块 import
- [ ] `Memory(..., mmap_mode='r')` — 缓存的大数组用 mmap 加载
- [ ] 不要 ignore 影响结果的参数；不要缓存生成器
