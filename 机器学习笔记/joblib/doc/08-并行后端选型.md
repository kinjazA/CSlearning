# 第 08 章 · 并行后端选型

> **难度**：⭐⭐⭐ | **核心**：`backend`, `parallel_backend()` | **一句话**：知道什么时候该用什么后端，别用错

---

## 8.1 解决什么问题？

第 07 章你学会了 `Parallel(n_jobs=-1)`，它默认用的是 `loky` 后端（多进程）。但并不是所有场景都适合多进程：

- "我的任务是读 1000 个文件，CPU 几乎不干活，多进程有帮助吗？"
- "我的数据在 GPU 上，多进程复制来复制去反而更慢？"
- "我在 Jupyter 里跑，多进程老是报错？"

**本章回答：什么场景选什么后端。**

---

## 8.2 三种后端一览

| 后端 | 类型 | 适用场景 | 一句话 |
|------|------|---------|--------|
| `loky`（**默认**） | 多进程 | CPU 密集型 | 最安全，默认用这个就行 |
| `threading` | 多线程 | I/O 密集型 | 读文件、发请求、等网络 |
| `multiprocessing` | 多进程 | 兼容旧代码 | 和 stdlib 行为一致，一般不选 |

---

## 8.3 loky — 默认多进程，90% 的情况用这个

```python
Parallel(n_jobs=-1, backend='loky')(
    delayed(task)(i) for i in range(100)
)
# 或省略 backend——默认就是 loky
Parallel(n_jobs=-1)(...)
```

**loky 做了什么**：
- 启动 N 个独立的 Python 子进程
- 每个子进程有自己独立的内存空间和 GIL
- 通过 pickle/cloudpickle 把函数和数据序列化传给子进程
- 子进程算完后把结果序列化传回主进程

**适用**：纯 Python 计算、numpy 运算、sklearn 模型训练——任何 CPU 密集型的任务。

**不适用**：任务需要共享大量 Python 对象（序列化开销大）、或子进程启动时间长但任务本身很短。

### loky 的一个关键细节

loky 在 **Linux/Mac** 上默认用 `fork` 创建子进程——快，且子进程能自动继承父进程的内存（包括 mmap 映射）。在 **Windows** 上用 `spawn`——慢一些，且子进程需要重新 import 所有模块。

这就是为什么 Windows 上必须有 `if __name__ == '__main__':` 保护——`spawn` 模式下子进程会重新执行整个脚本。

## 8.4 threading — 多线程，I/O 密集型首选

```python
Parallel(n_jobs=-1, backend='threading')(
    delayed(task)(i) for i in range(100)
)
```

**多线程的特点**：
- 所有线程共享同一个进程的内存空间
- **不需要序列化函数和数据**——直接访问，零开销传递
- 受 Python GIL 限制——同一时刻只有一个线程能执行 Python 代码
- 但 I/O 操作（读文件、网络请求、数据库查询）会释放 GIL

**所以多线程适合**：

```python
# ✅ 读大量文件
def read_one(path):
    return pd.read_csv(path)

dfs = Parallel(n_jobs=8, backend='threading')(
    delayed(read_one)(p) for p in file_paths
)

# ✅ 调 API
def fetch(url):
    return requests.get(url).json()

results = Parallel(n_jobs=8, backend='threading')(
    delayed(fetch)(url) for url in urls
)

# ✅ numpy 操作（numpy 的 C 扩展会释放 GIL）
def matrix_multiply(i):
    return np.dot(np.random.randn(1000,1000), np.random.randn(1000,1000))

results = Parallel(n_jobs=4, backend='threading')(
    delayed(matrix_multiply)(i) for i in range(20)
)
```

**多线程不适合**：

```python
# ❌ 纯 Python 计算——GIL 导致串行化
def pure_python_sum(n):
    total = 0
    for i in range(n):
        total += i
    return total

# 用 threading → 实际只有一个线程在跑，和串行一样慢
# 用 loky → 每个进程独立 GIL → 真正并行
```

### 多线程的核心优势：零序列化开销

```python
# threading：传的是引用，不复制数据
big_data = load_10GB_file()  # 主进程里的一份数据
Parallel(n_jobs=8, backend='threading')(
    delayed(process_chunk)(big_data, i) for i in range(8)
)
# 所有线程访问同一份 big_data，不需要序列化 10GB 数据 × 8 次
```

## 8.5 两种指定后端的方式

### 方式 1：每次调用指定

```python
Parallel(n_jobs=4, backend='threading')(...)
```

### 方式 2：上下文管理器（影响范围内所有 Parallel）

```python
from joblib import parallel_backend

with parallel_backend('threading', n_jobs=4):
    # 这个 block 里的所有 Parallel 都用 threading
    results1 = Parallel()(delayed(f1)(i) for i in range(10))
    results2 = Parallel()(delayed(f2)(i) for i in range(10))

# 退出 block 后恢复默认 loky
results3 = Parallel()(delayed(f3)(i) for i in range(10))  # loky
```

方式 2 在你不能直接控制 `Parallel` 调用时特别有用——比如 sklearn 内部用的 `Parallel`：

```python
with parallel_backend('threading'):
    # sklearn 内部用 Parallel(n_jobs=-1) 的地方会变成 threading
    model = RandomForestClassifier(n_jobs=-1).fit(X, y)
```

## 8.6 快速决策

```
你的任务：                                       → 用什么后端
─────────────────────────────────────────────────────────────
训练 sklearn 模型、numpy 运算、纯 Python 计算      → loky（默认）
批量读文件、调 API、数据库查询                     → threading
大部分 numpy 操作（dot、einsum、SVD）              → threading 也可以（C 扩展释放 GIL）
需要共享大对象，不想序列化                         → threading
任务在 Jupyter 里，fork 有问题                     → 试 threading
不确定                                             → loky，默认就是对的
```

## 8.7 常见坑

### 坑 1：CPU 密集任务用了 threading → 比串行还慢

```python
# ❌ 纯 Python 循环 + threading = GIL 串行化 + 线程切换开销
def pure_python_work(n):
    return sum(i**2 for i in range(n))

Parallel(n_jobs=4, backend='threading')(...)  # 比 n_jobs=1 还慢
```

### 坑 2：I/O 密集任务用了 loky → 浪费序列化开销

```python
# ❌ 读文件用 loky → 每个子进程都要启动 + 序列化参数
def read_file(path):
    return open(path).read()

Parallel(n_jobs=8, backend='loky')(...)  # 进程启动开销 > 读文件时间
# ✅ 用 threading
Parallel(n_jobs=8, backend='threading')(...)
```

### 坑 3：Jupyter 里 loky 可能报错

Jupyter 的交互式环境和 `fork` 有兼容性问题。如果 loky 报错，试试 `backend='threading'` 或者把函数定义放在单独的 `.py` 文件里 import。

### 坑 4：threading 下修改共享对象导致数据竞争

```python
# ❌ 多线程同时写一个 list → 数据竞争
shared = []
def append_to_shared(i):
    shared.append(i)     # 非线程安全

Parallel(n_jobs=4, backend='threading')(...)
# shared 可能丢失数据或顺序错乱
```

**规则**：多线程下只读共享数据是 OK 的。写入必须用锁或队列。

---

## 8.8 本章要点

- [ ] `loky`（默认）→ CPU 密集型：训练模型、numpy 运算、纯 Python
- [ ] `threading` → I/O 密集型：读文件、调 API、数据库查询
- [ ] threading 的优势：零序列化开销，共享大对象不需要复制
- [ ] threading 的劣势：GIL 限制，纯 Python 计算无法并行
- [ ] `parallel_backend()` 上下文管理器 → 影响范围内所有 Parallel 调用
- [ ] 不确定选什么 → 用 loky，默认就是对的
