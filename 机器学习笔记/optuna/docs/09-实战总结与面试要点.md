## 第一部分：实战开发手册

### 9.1 完整工作流（从数据到最佳模型）

```
1. 加载数据 → train/val/test 拆分
       │
2. 定义搜索空间 → 确定哪些参数要搜，范围多少
       │
3. 编写 objective 函数 → 闭包传入数据，自包含，无副作用
       │
4. 创建 Study → 选 direction、sampler、pruner
       │
5. study.optimize(...) → 设置 n_trials、timeout、catch、callback
       │
6. 分析结果 → best_params、trials_dataframe、超参重要性
       │
7. 用最佳参数 + 全量训练数据重新训练 → 最终模型
       │
8. 在独立测试集上评估 → 这才是真正的成绩
       │
9. 保存模型 + 搜索记录 → joblib / pickle / SQLite
```

### 9.2 搜索空间设计规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 范围宁宽勿窄 | 范围宽了 TPE 会自动收敛到好区域；窄了可能错过最优解 | lr: `[1e-5, 1e-1]` 不要 `[1e-4, 1e-3]` |
| 跨越数量级用 log | 学习率、正则化系数、C 值天生适合对数空间 | `suggest_float('lr', 1e-5, 1e-1, log=True)` |
| 有序值用 int 不用 categorical | batch_size `[16, 32, 64, 128]` 用 `suggest_int` + step，不用 categorical | TPE 能利用数值间的顺序关系 |
| 条件参数加前缀 | 不同分支的同类型参数用前缀区分 | `rf_n_estimators` vs `xgb_n_estimators` |
| 循环中的参数加索引 | 第 i 层的参数加 `_l{i}` 后缀 | `n_units_l0`, `dropout_l1` |
| 不要搜你能确定的参数 | 如果 community 共识 `Adam` + `lr=1e-3` 就是好的，不用搜 | 把 trial 预算留给不确定的参数 |

### 9.3 Sampler 与 Pruner 选型速查

**Sampler 选择**：

```
你的搜索空间是？
├── 主要连续参数（< 20 个）+ 一些离散参数
│   └── TPESampler ✅ 默认首选
│
├── 连续参数 > 20 个（高维）
│   └── CmaEsSampler ✅ 高维连续空间更高效
│
├── 只想验证 objective 函数能跑通
│   └── RandomSampler ✅ 快速基线
│
├── 搜索空间很小（< 100 种组合）
│   └── GridSampler ✅ 穷举即可
│
└── 需要严格可复现的结果
    └── RandomSampler(seed=42) 或 GridSampler
```

**Pruner 选择**（回顾第 3 章 3.3.5 节决策树）：

| 条件 | Pruner | 关键参数 |
|------|--------|---------|
| 通用，不想多想 | `MedianPruner` | `n_startup_trials=5, n_warmup_steps=10` |
| Trial 多（> 50）且耗时长 | `HyperbandPruner` | `min_resource=3, reduction_factor=3` |
| 有先验知识知道什么值不可接受 | `ThresholdPruner` | `lower=0.5`（准确率低于 50% 直接剪） |
| 指标波动大 | `PatientPruner(MedianPruner)` | `patience=3` |
| 不需要剪枝 | 不传 `pruner` 参数 | `should_prune()` 永远返回 `False` |

### 9.4 Trial 数量指南

| 场景 | 推荐 n_trials | 理由 |
|------|-------------|------|
| 快速原型 / 验证代码 | 10~20 | 确认 objective 能跑通，没有报错 |
| 单模型调参（sklearn） | 50~100 | TPE 收敛的甜区 |
| 多模型对比搜索 | 100~200 | 每个模型至少被采样 30+ 次才有意义 |
| 深度学习 | 20~50 | 单次训练耗时长，配合 Hyperband 剪枝 |
| 竞赛 / 论文 | 200~500 | 追求极致，时间和算力允许 |
| 生产环境 | 50~100 | 平衡效果和成本 |

> **经验公式**：`n_trials ≈ 搜索空间维度 × 10`。10 个参数 → 至少 100 个 trial。

### 9.5 可复现性检查清单

一个可复现的 Optuna 实验需要锁定以下所有随机源：

- [ ] `TPESampler(seed=42)` — 采样器种子
- [ ] `model.random_state = 42 + trial.number` — 模型种子（每个 trial 不同但可追溯）
- [ ] `StratifiedKFold(..., random_state=seed)` — CV 拆分种子
- [ ] `train_test_split(..., random_state=42)` — 数据拆分种子
- [ ] `np.random.seed(42)` — NumPy（如果需要）
- [ ] `torch.manual_seed(42)` — PyTorch（如果需要）

> **为什么 `42 + trial.number` 而不是 `42`？** 所有 trial 用同一个种子 → 同一个参数组合每次出相同的 CV 分数 → Optuna 区分不了哪个更好 → 采样器学到的是噪声而非真实的参数-性能关系。

### 9.6 常见错误与排查

| 症状 | 可能原因 | 排查方法 |
|------|---------|---------|
| 所有 trial 的 score 几乎一样 | 模型 random_state 没设 | 检查所有随机源是否都设了种子 |
| `best_value` 远差于手动调参 | 搜索范围太窄或 scoring 选错了 | 扩大范围，确认 scoring 和 direction 匹配 |
| 某个 trial 报错后全部停止 | 没设 `catch` | `study.optimize(..., catch=(Exception,))` |
| 多进程跑不起来 | `objective` 用了全局变量 | 改用闭包 `make_objective(X, y)` |
| 内存越跑越大 | trial 间对象没释放 | `study.optimize(..., gc_after_trial=True)` |
| `should_prune()` 永远返回 False | 没创建 Pruner | `create_study` 时加上 `pruner=...` |
| `best_params` 里有奇怪的键 | 条件参数+命名冲突 | 检查条件分支中的参数名是否有前缀区分 |

### 9.7 性能优化技巧

**速度优化**：

```python
# 1. 大数据集用子采样 + 少折数 CV
cv = 3 if len(X) > 100_000 else 5
idx = np.random.choice(len(X), min(50_000, len(X)), replace=False)

# 2. 热启动——把已知的好参数先跑掉
study.enqueue_trial({'lr': 0.001, 'n_layers': 3})
study.enqueue_trial({'lr': 0.01, 'n_layers': 2})
study.optimize(objective, n_trials=100)  # 前 2 个 trial 用预设参数

# 3. 多进程（仅单机，objective 必须用闭包传数据）
study.optimize(objective, n_trials=100, n_jobs=4)
```

**内存优化**：

```python
# GPU 训练场景
def objective(trial):
    model = create_model(trial).to(device)
    score = train_and_evaluate(model)
    del model
    torch.cuda.empty_cache()
    gc.collect()
    return score

study.optimize(objective, n_trials=100, gc_after_trial=True)
```

### 9.8 与 MLOps 工具的集成

```python
# 集成 MLflow
import mlflow

def objective(trial):
    with mlflow.start_run(nested=True):
        params = {k: trial.suggest_float(k, ...) for k in [...]}
        mlflow.log_params(params)
        score = evaluate(params)
        mlflow.log_metric('val_score', score)
        return score

# 集成 Weights & Biases
import wandb

def wandb_callback(study, trial):
    if trial.state == optuna.trial.TrialState.COMPLETE:
        wandb.log({
            'trial_number': trial.number,
            'value': trial.values[0],
            **trial.params,
        })

study.optimize(objective, n_trials=100, callbacks=[wandb_callback])
```

## 第二部分：面试要点 & 技术深问

### 9.9 面试高频题：Optuna 基础

**Q1：Optuna 和 GridSearchCV 的区别是什么？为什么选择 Optuna？**

| 维度 | GridSearchCV | Optuna |
|------|-------------|--------|
| 搜索策略 | 穷举所有组合 | 贝叶斯优化（TPE），用历史结果指导下一个采样点 |
| 效率 | O(组合数)，维度灾难 | 用更少 trial 找到更优解 |
| 搜索空间 | 必须预先指定所有候选值 | 定义-by-run，支持条件参数 |
| 剪枝 | 不支持 | 支持中途提前终止差 trial |
| 分布式 | 仅 `n_jobs` | 支持多机分布式 + SQLite/PostgreSQL |

**Q2：TPE 的 `l(x)/g(x)` 为什么能选出好参数？**

面试标准回答：TPE 把历史 trial 按分位数分成好组（l(x)）和差组（g(x)），分别用 KDE（Parzen 窗）估计参数分布。然后从 l(x) 中采样候选参数，用 l(x)/g(x) 给候选打分——在"好参数常见但差参数少见"的区域采样。这个比值本质上是 Expected Improvement 的等价形式。

**Q3：`suggest_int` 和 `suggest_categorical` 的区别？batch_size 用哪个？**

- `suggest_int`：对数值有顺序关系，TPE 能利用这种顺序（如 `[16, 32, 64, 128]`）。
- `suggest_categorical`：无序选择，TPE 用计数估计概率，不利用顺序关系。
- batch_size 应该用 `suggest_int('batch_size', 16, 256, step=16)`。

### 9.10 面试高频题：Sampler 与 Pruner

**Q4：TPE 和 CMA-ES 各适合什么场景？为什么？**

- **TPE**：混合类型空间（连续 + 离散 + 条件参数）。因为是树结构 + 单变量 KDE，天然支持离散参数和条件依赖。
- **CMA-ES**：纯连续高维空间（> 20 维）。它是一个进化策略算法，在连续空间中比 KDE 更有效地搜索。但不支持离散参数。

**Q5：MedianPruner 和 HyperbandPruner 的决策逻辑有什么区别？**

- **MedianPruner**：在每个 step，比较当前 trial 的指标 vs 所有已完成 trial 的**同期中位数**。低于中位数 → 剪。判断标准是**相对比较**。
- **HyperbandPruner**：让所有 trial 跑同样的步数，排序后只保留前 1/η，给它们更多步数。判断标准是**锦标赛排名**。Hyperband 更激进，因为不考虑"后面可能追上来"。

**Q6：被剪枝的 trial 会影响最终结果吗？**

- ❌ 不参与 `study.best_trial` 的竞争。
- ✅ 它的中间 report 值被保留，帮助后续 trial 的 Pruner 做更准确的判断。
- ✅ 它的参数组合被采样器记录为"不好的区域"，指导后续不往那里采。

### 9.11 面试高频题：工程实践

**Q7：Optuna 如何保证分布式环境下的正确性？**

核心是 **共享 Storage**（SQLite/PostgreSQL/MySQL）。所有 worker 进程读到同一个 Study 状态：
- 每个 worker 在开始新 trial 前，从 storage 读到所有已完成 trial 的数据
- Sampler 基于全部 history 采样下一组参数（通过数据库锁保证不冲突）
- Worker 完成后把结果写回 storage（原子写入）

**Q8：`study.optimize()` 中的 `n_trials` 和 `timeout` 同时设置了会怎样？**

**满足其一即停止**。先到的条件触发停止。常用于"跑 1000 个 trial 或 2 小时，哪个先到就用哪个"。

**Q9：optuna 的 `enqueue_trial` 和 `tell` (ask-tell) 有什么区别？什么场景用哪个？**

- **`enqueue_trial`**：在 `optimize()` 之前插入预设参数。这些 trial 先执行，然后 TPE 在收集到的数据基础上继续采样。适用场景：有先验知识（之前跑过的实验参数），想热启动。
- **`tell` (ask-tell 接口)**：完全绕过 `optimize()`，手动控制 trial 的创建和结果报告。`study.ask()` 获取新 trial，训练完后 `study.tell(trial, value)` 报告结果。适用场景：分布式集群中每个节点独立跑 trial，或需要自定义调度逻辑。

**Q10：如何在 Optuna 中处理训练不稳定的情况？**

1. **Pruner 侧**：用 `PatientPruner(MedianPruner(), patience=3)`，减少误剪
2. **Sampler 侧**：增大 `TPESampler(n_startup_trials=...)`，多收集数据再建模
3. **Objective 侧**：对同一组参数跑多次取平均（如 `np.mean([cross_val_score(...) for _ in range(3)])`）
4. **容错侧**：`study.optimize(..., catch=(RuntimeError,))` 标记失败 trial 但不中断

### 9.12 面试高频题：与其他框架对比

**Q11：Optuna vs Hyperopt vs Ray Tune 怎么选？**

| | Optuna | Hyperopt | Ray Tune |
|---|---|---|---|
| 采样算法 | TPE (默认), CMA-ES, 网格, 随机 | TPE (默认), 随机, 自适应 TPE | TPE, BayesOpt, HyperBand, BOHB, PBT 等 |
| 剪枝 | ✅ Median, Hyperband, Threshold, Patient | ❌ 无内置剪枝 | ✅ Hyperband, Median, ASHA, BOHB, PBT |
| 分布式 | SQLite/PostgreSQL/MySQL | MongoDB | Ray 原生分布式 |
| 可视化 | Dashboard, matplotlib, plotly | 基础 matplotlib | 集成 TensorBoard, MLflow |
| 定义方式 | define-by-run | 需要预先定义空间 | define-by-run |
| 上手难度 | ⭐ 低 | ⭐⭐ 中 | ⭐⭐⭐ 高（需了解 Ray） |
| 适用场景 | 单机~小集群 ML/DL | 单机 ML | 大规模分布式 DL |

**一句话选型**：单机 ML 调参 → Optuna；大规模分布式 DL → Ray Tune；只用最简单功能 → Hyperopt 够了。

**Q12：什么情况下不应该用 Optuna？**

1. **单次训练极慢**（> 数小时）：跑不了几个 trial，贝叶斯优化的优势发挥不出来，手动调参可能更高效。
2. **只需要调 1~2 个参数**：网格搜索 10×10=100 次就穷举完了，不需要 TPE。
3. **已有强先验知识**：你已经知道什么参数组合好，直接固定即可。
4. **大模型（LLM）全量训练**：一次训练几万美金，不可能跑 N 个 trial 去搜。用 scaling law + 启发式。

### 9.13 面试踩分细节

这些是面试官可能追问的"等一下，你确定吗？"类问题：

| 细节 | 正确回答 |
|------|---------|
| `direction='maximize'` 但 scoring 是 `'neg_MSE'`，对吗？ | 对。sklearn 的 scorer 都是越大越好，`neg_MSE` 越大 = MSE 越小。所以 `direction='maximize'` 是优化目标，实际是在最小化 MSE。 |
| `study.best_trial` 返回的一定是剪枝后的最优？ | 是。`best_trial` 只从 `COMPLETE` 状态的 trial 中选。被剪枝的（`PRUNED`）不参与。 |
| CV 分数可以直接当最终成绩吗？ | 不能。CV 分数是搜索过程中的"内部排名"，有乐观偏差。最终模型必须在独立测试集上评估。 |
| 多目标优化中 `best_params` 返回哪个解？ | 返回 Pareto 前沿上**第一个目标**最优的解。要用 `best_trials` 获取完整前沿。 |
| 条件参数的名字冲突会有什么后果？ | TPE 会错误地把两个不同含义的参数当成同一个来建模 l(x)/g(x)，导致采样严重偏差。 |

### 9.14 一句话速记卡

- **Study** = 一次完整的优化任务，Trial = 一次参数尝试，Epoch = 权重更新一次
- **TPE** = 好参数常见 + 差参数少见的区域 → 采样
- **剪枝** = `report()` → `should_prune()` → `raise TrialPruned()`
- **step 必须单调递增**，**先 report 再 should_prune**
- **闭包传数据**，不用全局变量
- **三处种子**：采样器 + 模型 + CV
- **best_value ≠ 最终成绩**，测试集评估才是
- **direction 看 scoring**，sklearn 的 scorer 都是越大越好
- **条件参数加前缀**，循环参数加索引
- **结果保存**：`joblib.dump(study, ...)` 或 SQLite storage
