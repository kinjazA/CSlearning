# 第3章：剪枝策略实战

## 本章目标
理解剪枝（Pruning）的原理和决策机制，能够根据场景选择合适的 Pruner，并在不同框架中正确集成。

---

## 3.1 为什么需要剪枝？

假设你在搜索一个神经网络的学习率，要跑 100 个 epoch：

| Trial | 学习率 | Epoch 5 的准确率 | 是否值得继续？ |
|-------|--------|-----------------|--------------|
| #0 | 0.001 | 0.72 | ✅ 继续 |
| #1 | 0.01 | 0.68 | ✅ 还可以 |
| #2 | 10.0 | 0.10 | ❌ 明显没救了 |
| #3 | 0.00001 | 0.15 | ❌ 收敛太慢 |

**没有剪枝**：每个 trial 都硬跑到 100 epoch。Trial #2 和 #3 白白浪费了 95% 的训练时间。

**有剪枝**：在第 5 个 epoch 发现 Trial #2 的表现远不如同期其他 trial，**提前终止**，省下后面 95 个 epoch 的时间。

> **核心思想**：用少量资源（前几个 epoch）快速筛选，把大量资源留给有希望的 trial。

---

## 3.2 剪枝的工作原理

### 3.2.1 核心 API：两个方法 + 一个异常

剪枝只需要三个元素，都在目标函数内部完成：

```python
def objective(trial):
    for epoch in range(100):
        train(...)
        val_score = evaluate(...)

        # ① 报告：告诉 Optuna "我现在跑到了第几步，效果如何"
        trial.report(val_score, epoch)

        # ② 判断：询问 Pruner "我应该继续还是放弃？"
        if trial.should_prune():
            # ③ 终止：抛出异常，Optuna 捕获后跳到下一个 trial
            raise optuna.TrialPruned()

    return val_score
```

| 元素 | 签名 | 作用 |
|------|------|------|
| `trial.report(value, step)` | `value: float, step: int` | 把当前指标和步数上报给 Study |
| `trial.should_prune()` | `→ bool` | 让 Pruner 基于所有 trial 的 report 数据做判断 |
| `raise TrialPruned()` | — | 通知 Optuna 这个 trial 提前终止（**不算异常，算正常控制流**） |

### 3.2.2 一次剪枝的完整流程

```
Trial #5 开始 (lr=10.0)
│
├── Epoch 0: train → val_acc=0.25 ──→ trial.report(0.25, step=0)
│   └── trial.should_prune()
│       Pruner 查看：同期其他 trial 在 step=0 的 report 值
│       其他 trial: 0.45, 0.52, 0.48, 0.50
│       中位数 ≈ 0.49，当前 0.25 < 0.49 → 建议剪枝！
│       → return True
│
├── raise optuna.TrialPruned()
│   Optuna 捕获该异常
│   标记 Trial #5 状态为 PRUNED
│   不会把 0.25 当作最终结果
│   立即开始 Trial #6
│
└── （Epoch 1~99 被跳过，节省约 95% 的时间）
```

### 3.2.3 被剪枝后发生了什么？

这是最容易混淆的地方：

```
被剪枝的 trial：
├── ❌ 不参与 study.best_trial 的竞争
├── ❌ 它的返回值被丢弃
├── ✅ 它已经 report 的中间值被保留（供后续 trial 剪枝参考）
├── ✅ 它的参数组合被记录（供采样器学习"哪些参数区域不好"）
└── ✅ 已消耗的训练时间确实是沉没成本（所以 Hyperband 会极早期筛选）
```

> **关键理解**：剪枝的 trial 仍然为 Optuna 提供了**负面信号**——"参数区域 X + 中间结果差 → 以后少来这里"。它的 report 数据帮助 Pruner 更准确地识别差 trial。

### 3.2.4 关键约束（踩坑点）

| 约束 | 说明 | 错误示例 |
|------|------|---------|
| `step` 必须**单调递增** | 不能倒退或重复 | `report(v, 5)` 之后不能再 `report(v, 3)` |
| **先 report 再 should_prune** | 否则 Pruner 没有数据做判断 | 直接调 `should_prune()` 会报错或永远返回 False |
| 没有配置 Pruner 时 `should_prune()` 永远返回 `False` | Study 创建时没传 `pruner=` 就不会触发剪枝 | `study = optuna.create_study(direction='maximize')` ← 无 Pruner |
| 返回值必须能到达 | 不要在剪枝检查前就 `return` | 如果 epoch 0 就 return，Pruner 根本没机会判断 |

## 3.3 四种 Pruner 的决策逻辑

Pruner 的本质是回答一个问题：**"给定所有 trial 在 step=k 的 report 值，当前 trial 是否应该被淘汰？"**

四种 Pruner 用不同的逻辑回答这个问题。

### 3.3.1 MedianPruner — 中位数淘汰

**决策逻辑**：当前 trial 的指标 < 同期所有已完成 trial 的**中位数** → 剪枝。

```
Step=5 时的局面：
已完成 trial 在 step=5 的 report 值：[0.45, 0.52, 0.55, 0.48, 0.61, 0.50]
排序：[0.45, 0.48, 0.50, 0.52, 0.55, 0.61]
中位数 = (0.50 + 0.52) / 2 = 0.51

当前 trial 在 step=5 的 report 值 = 0.38
0.38 < 0.51 → 剪枝！
```

```python
pruner = optuna.pruners.MedianPruner(
    n_startup_trials=5,      # 前 5 个 trial 不剪枝（没有足够的参考数据）
    n_warmup_steps=10,       # 每个 trial 的前 10 步不剪枝（训练早期波动大）
    interval_steps=1,        # 每隔 1 步检查一次（每步都检查）
)
```

| 参数 | 默认值 | 作用 |
|------|--------|------|
| `n_startup_trials` | 5 | 前 N 个 trial 不做判断——它们负责"建立基准" |
| `n_warmup_steps` | 0 | 每个 trial 的前 N 步不剪枝——避免训练早期的噪声误杀 |
| `interval_steps` | 1 | 检查频率。设 5 表示每 5 个 step 才检查一次 |

**适合**：通用场景，训练曲线比较平滑（没有剧烈波动）。

**不适合**：训练初期指标波动大的任务（如 GAN、强化学习）。

### 3.3.2 HyperbandPruner — 锦标赛淘汰

**决策逻辑**：不按中位数比，而是用**锦标赛制**——让所有 trial 跑少量步数，只保留表现最好的 1/η，给它们更多步数，重复直到只剩最优的几个。

```
Hyperband 的一轮（rung）：
初始：16 个 trial 各跑 3 个 epoch
  ↓ 排序，只保留前 1/3 ≈ 5 个 trial
剩余 5 个 trial 继续跑到 9 个 epoch
  ↓ 排序，只保留前 1/3 ≈ 2 个 trial
剩余 2 个 trial 继续跑到 27 个 epoch
  ↓ 排序，只保留前 1/3 ≈ 1 个 trial
剩余 1 个 trial 跑完全程
```

```python
pruner = optuna.pruners.HyperbandPruner(
    min_resource=3,          # 最少跑 3 个 step（太小会导致判断不准）
    reduction_factor=3,     # 每轮淘汰到 1/3
    max_resource='auto',    # 最大 step 数（'auto' = 从 report 自动推断）
    n_brackets=4,           # Hyperband bracket 的数量（越大探索越充分）
)
```

| 参数 | 默认值 | 作用 |
|------|--------|------|
| `min_resource` | 1 | 每个 trial 至少跑多少步才能进入首次筛选 |
| `reduction_factor` | 3 | 每轮保留比例 = 1/η。η 越大淘汰越激进 |
| `max_resource` | `'auto'` | 单个 trial 最多跑多少步（自动从最大的 report step 推断） |
| `n_brackets` | 4 | 并行跑几组不同策略的 bracket（越大越全面但越慢） |

**适合**：训练时间长、trial 数量多的场景（如深度学习的 100+ trial 搜索）。

**不适合**：trial 总数很少（< 20）的情况——还没筛选几轮就没 trial 了。

> **Hyperband vs Median 的核心区别**：Median 按"同期中位数"判断——你的 trial 在 step=5 比别人的 step=5 差就剪。Hyperband 按"锦标赛排名"判断——跑同样的步数后，排名靠后的直接淘汰。Hyperband 更激进，因为它不考虑"也许后面会追上来"。

### 3.3.3 ThresholdPruner — 硬阈值淘汰

**决策逻辑**：指标低于（或高于）某个**绝对阈值** → 剪枝。不做任何相对比较。

```python
pruner = optuna.pruners.ThresholdPruner(
    lower=0.5,   # 准确率低于 50% 直接剪掉
    upper=1.0,   # 准确率不可能超过 100%，只是形式上设置上限
)
```

**适合**：你已经从经验中知道什么值是绝对不可接受的。比如"验证准确率低于 60% 的模型没有部署价值"。

**不适合**：没有先验知识的一般场景——你并不知道"好"的绝对标准是什么。

### 3.3.4 PatientPruner — 带容忍的包装器

**PatientPruner 不是独立的 Pruner**，它**包裹**另一个 Pruner，给它增加"耐心"——即使被建议剪枝，也允许多跑几步确认。

```python
pruner = optuna.pruners.PatientPruner(
    wrapped_pruner=optuna.pruners.MedianPruner(),  # 被包装的 Pruner
    patience=3,         # 被建议剪枝后再容忍 3 个 step
    min_delta=0.01,     # 如果指标在 patience 步内提升了 min_delta，就再给一次机会
)
```

**工作流程**：
```
Step 5: MedianPruner 建议剪枝 → PatientPruner："再观察 3 步"
Step 6: 指标 0.52 → 没有显著提升（< 0.01 delta）
Step 7: 指标 0.53 → 没有显著提升
Step 8: 指标 0.52 → patience 用尽，真正剪枝
```

**适合**：训练指标波动大的任务（小 batch 训练、GAN、强化学习）。

**不适合**：训练很稳定的任务——避免"耐心"反而浪费了本可以省下的时间。

### 3.3.5 Pruner 选型决策

```
你需要剪枝吗？
├── 不需要 → 创建 Study 时不传 pruner 参数（默认行为）
│
└── 需要 → 有先验知识知道绝对阈值吗？
    ├── 有 → ThresholdPruner
    │
    └── 没有 → 训练指标波动大吗？
        ├── 波动大 → PatientPruner(MedianPruner, patience=3)
        │
        └── 平稳 → trial 总数多吗（> 50）？
            ├── 多 + 训练时间长 → HyperbandPruner
            │
            └── 少 或 训练快 → MedianPruner（最简单、最安全）
```

## 3.4 按框架集成

剪枝的核心逻辑（report → should_prune → TrialPruned）在所有框架中是一样的。区别只在于**"步"是什么**。

### 3.4.1 PyTorch / TensorFlow — 按 Epoch 剪枝

深度学习训练天然分 epoch，"步"就是 epoch 编号。剪枝直接嵌入训练循环即可，不需要额外适配。

```python
def objective(trial):
    model = create_model(trial)
    for epoch in range(EPOCHS):
        train(model, train_loader)
        val_acc = evaluate(model, val_loader)

        trial.report(val_acc, epoch)       # step = epoch

        if trial.should_prune():
            raise optuna.TrialPruned()

    return val_acc
```

**要点**：`step` 参数直接传 epoch 编号（0, 1, 2, ...）。可以加上 `n_warmup_steps` 避免前几个 epoch 的噪声误判。

### 3.4.2 XGBoost / LightGBM — 按 Boosting Round 剪枝

梯度提升树没有 epoch 概念，但逐轮构建树。Optuna 提供了专门的回调函数，不需要手动写 `report()` 和 `should_prune()`。

```python
import optuna.integration.xgboost as optuna_xgb
import xgboost as xgb

def objective(trial):
    param = {
        'max_depth': trial.suggest_int('max_depth', 2, 10),
        'eta': trial.suggest_float('eta', 0.01, 0.3, log=True),
        'eval_metric': 'auc',
    }

    dtrain = xgb.DMatrix(X_train, label=y_train)
    dvalid = xgb.DMatrix(X_val, label=y_val)

    # Optuna 提供的剪枝回调——在每轮 boosting 后自动 report + should_prune
    pruning_callback = optuna_xgb.XGBoostPruningCallback(
        trial,
        observation_key='validation-auc'   # 用哪个评估指标来判断
    )

    bst = xgb.train(
        param, dtrain,
        num_boost_round=500,
        evals=[(dvalid, 'validation')],
        callbacks=[pruning_callback],       # 传入回调即可
        verbose_eval=False
    )

    return bst.best_score
```

**要点**：回调自动完成 report + should_prune，"步"是 boosting round 编号。LightGBM 同理，用 `optuna.integration.lightgbm.LightGBMPruningCallback`。

### 3.4.3 sklearn — 按 CV Fold 剪枝

sklearn 的模型（随机森林、SVM 等）通常一次性 `fit()` 完成，没有天然的分段点。要集成剪枝，必须**手动拆分交叉验证**，每完成一个 fold 报告一次：

```python
from sklearn.model_selection import StratifiedKFold
import numpy as np

def objective(trial):
    params = {
        'n_estimators': trial.suggest_int('n_estimators', 10, 300),
        'max_depth': trial.suggest_int('max_depth', 2, 32),
    }

    scores = []
    kf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

    for step, (train_idx, val_idx) in enumerate(kf.split(X, y)):
        model = RandomForestClassifier(**params, random_state=42)
        model.fit(X[train_idx], y[train_idx])
        score = model.score(X[val_idx], y[val_idx])
        scores.append(score)

        # 每完成一个 fold，报告当前平均分
        trial.report(np.mean(scores), step)

        if trial.should_prune():
            raise optuna.TrialPruned()

    return np.mean(scores)
```

**要点**：
- "步" = fold 编号（只有 5 个 fold 的话，只有 5 个 report 点，不够精细）
- 对于 sklearn 模型，通常 **fold 数少、判断样本不足**，剪枝效果不如 DL 明显
- `OptunaSearchCV` **不支持剪枝**——需要剪枝就必须手动写上面的循环

### 三种框架对比

| | PyTorch | XGBoost/LightGBM | sklearn |
|---|---|---|---|
| "步"的粒度 | epoch（通常 20~200 步） | boosting round（通常 100~500 步） | CV fold（通常 5~10 步） |
| 集成方式 | 手动 `report()` + `should_prune()` | 回调 `XGBoostPruningCallback` | 手动拆分 CV + `report()` |
| 代码量 | 中等 | 低 | 中等 |
| 剪枝效果 | ⭐⭐⭐ 好（步数多，判断充分） | ⭐⭐⭐ 好（步数多，判断充分） | ⭐⭐ 一般（步数少，判断粗糙） |

## 3.5 剪枝效果预期

| 配置 | 预期耗时（相对无剪枝） | 预期最佳值 | 说明 |
|------|----------------------|-----------|------|
| 无剪枝 | 100% | 基线 | 每个 trial 跑完全程 |
| MedianPruner | 40~60% | 约等于基线 | 温和剪枝，误杀率低 |
| HyperbandPruner | 30~50% | 约等于基线 | 激进剪枝，早期快速筛选 |
| ThresholdPruner | 取决于阈值设置 | 约等于基线 | 适合有先验知识的场景 |

> **关键洞察**：剪枝的"最佳值"通常不会被拉低——因为被剪掉的 trial 本来就没有竞争力。剪枝节省的是本来会浪费在差参数上的时间。

## 3.6 常见问题

**Q：剪枝后 `study.best_trial` 会不会是被剪掉的 trial？**
不会。`best_trial` 只从 `COMPLETE` 状态的 trial 中选。被剪枝的 trial 状态是 `PRUNED`，不参与竞争。

**Q：我的指标波动很大，MedianPruner 误杀了怎么办？**
用 `PatientPruner(MedianPruner(), patience=3)`，或者增大 `n_warmup_steps` 跳过波动期。

**Q：Hyperband 和 Median 怎么选？**
一个简单规则：
- trial 数 < 30 或单次训练 < 1 分钟 → MedianPruner（安全）
- trial 数 > 50 且单次训练 > 5 分钟 → HyperbandPruner（高效）
- 不确定 → MedianPruner（最简单，不容易出错）

**Q：没有配置 Pruner 时 `should_prune()` 会怎样？**
永远返回 `False`。你的代码不会报错，但等于没做剪枝——所有 trial 都会跑完全程。

**Q：可以用多个 Pruner 吗？**
不能直接组合多个独立 Pruner。但你可以用 `PatientPruner` 包装任意一个 Pruner，给它增加容忍度。

**Q：剪枝在分布式环境中安全吗？**
安全。所有进程共享同一个 storage，`should_prune()` 查看的是**所有进程**已完成的 trial 的 report 数据。但注意多进程场景下，不同 trial 的 report 到达顺序可能不一致，Hyperband 对此有专门处理。

---

## 3.7 本章小结

- 剪枝通过 `report()` + `should_prune()` + `TrialPruned()` 三步实现
- `step` 必须单调递增，必须先 report 再判断
- **MedianPruner**：用中位数淘汰——你的 trial 比同期中位数差就剪，最通用
- **HyperbandPruner**：用锦标赛淘汰——跑同样的步数、排名靠后的淘汰，最高效
- **ThresholdPruner**：用绝对阈值淘汰——低于某值直接剪，需要先验知识
- **PatientPruner**：包装其他 Pruner，给它们增加容忍度
- PyTorch 在 epoch 循环中集成，XGBoost 用回调，sklearn 需手动拆 CV fold
- 被剪枝的 trial 不参与最佳值竞争，但提供了负面信号帮助后续采样
