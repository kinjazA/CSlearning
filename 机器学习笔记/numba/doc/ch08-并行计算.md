# 第八章：并行计算

> **核心问题**：现代 CPU 都是多核的，你的笔记本可能有 8 核 16 线程。`@njit` 默认只用 1 个核——怎么让 Numba 用上所有核心，把计算时间除以核心数？

---

## 8.1 并行化的基础概念

### 8.1.1 数据并行 vs 任务并行

**数据并行（Data Parallelism）**：同一个操作，应用到不同的数据上。

```
数组: [1, 2, 3, 4, 5, 6, 7, 8]
任务: 每个元素 × 2

单核:  核心 1: [1×2, 2×2, 3×2, 4×2, 5×2, 6×2, 7×2, 8×2]

并行:  核心 1: [1×2, 2×2, 3×2, 4×2]
      核心 2: [5×2, 6×2, 7×2, 8×2]
```

**任务并行（Task Parallelism）**：不同的任务，并行执行。

```
任务 A: 读取数据
任务 B: 清洗数据
任务 C: 训练模型

并行：线程 1 跑 A，线程 2 跑 B，线程 3 跑 C
```

**Numba 的并行主要是数据并行**——把一个大数组拆成 N 份，每个核心处理一份。任务并行需要用 Python 的 `multiprocessing` 或 `concurrent.futures`，Numba 不直接支持。

### 8.1.2 为什么 Python 默认不能并行

Python 有一个全局解释器锁（**GIL, Global Interpreter Lock**）——同一时刻只有一个线程能执行 Python 字节码。这意味着多个 Python 线程无法真正并行，只能**并发**（快速切换）。

```
Python 多线程（有 GIL）:
线程 1: ━━ ━━ ━━ ━━    ← 轮流占用 CPU
线程 2:    ━━ ━━ ━━ ━━

Numba 编译后（无 GIL）:
线程 1: ━━━━━━━━━━━━    ← 同时占用不同核心
线程 2: ━━━━━━━━━━━━
```

Numba 的 `parallel=True` 和 `prange` 能释放 GIL，让编译后的代码真正并行。

## 8.2 `parallel=True` —— 一行代码启用自动并行

### 8.2.1 最简单的并行化

```python
from numba import njit
import numpy as np

# 串行版本
@njit
def sum_serial(arr):
    total = 0.0
    for i in range(len(arr)):
        total += arr[i]
    return total

# 并行版本：只加一个参数
@njit(parallel=True)
def sum_parallel(arr):
    total = 0.0
    for i in range(len(arr)):
        total += arr[i]
    return total
```

加上 `parallel=True` 后，Numba 会**自动分析循环**，判断哪些能并行化，然后生成多线程代码。不是所有循环都能自动并行——需要满足条件（无依赖、归约操作等）。

### 8.2.2 `parallel=True` 的自动并行能力

Numba 能自动并行化这些模式：

```python
@njit(parallel=True)
def auto_parallel_examples(arr):
    # 1. 元素级映射
    result1 = arr * 2 + 1              # ✅ 自动并行
    
    # 2. 归约（reduction）
    total = np.sum(arr)                # ✅ 自动并行
    
    # 3. 简单的逐元素循环
    result2 = np.empty_like(arr)
    for i in range(len(arr)):
        result2[i] = arr[i] ** 2       # ✅ 自动并行（编译器识别独立性）
    
    return result1, total, result2
```

### 8.2.3 什么时候自动并行会失败

```python
@njit(parallel=True)
def cannot_auto_parallel(arr):
    # ❌ 有依赖：每次迭代依赖上一次的结果
    for i in range(1, len(arr)):
        arr[i] = arr[i] + arr[i-1]     # 依赖 arr[i-1]，不能并行
```

这种情况下，`parallel=True` 不会报错，但也不会并行化这个循环——它会回退到串行执行。

## 8.3 `prange` —— 显式并行循环

### 8.3.1 基本用法

```python
from numba import njit, prange

@njit(parallel=True)
def explicit_parallel(arr):
    n = len(arr)
    result = np.empty_like(arr)
    for i in prange(n):              # ← 用 prange 代替 range
        result[i] = arr[i] * arr[i]
    return result
```

**`prange` = parallel range**，告诉 Numba：这个循环的每次迭代是独立的，可以安全地并行。

### 8.3.2 `range` vs `prange`

| | `range` | `prange` |
|--|---------|----------|
| **执行方式** | 顺序执行，迭代 0→1→2→... | 多线程并行，迭代顺序不确定 |
| **需要** | 无特殊要求 | 必须配合 `parallel=True` |
| **依赖性** | 可以有依赖 | 每次迭代必须独立 |
| **归约支持** | 支持 | 支持（自动检测） |

**关键规则**：`prange` 循环内，每次迭代不能依赖其他迭代的结果。

### 8.3.3 典型的 `prange` 模式

```python
@njit(parallel=True)
def typical_patterns(X, Y):
    n = len(X)
    
    # 模式 1: 逐元素映射
    result = np.empty_like(X)
    for i in prange(n):
        result[i] = np.sin(X[i]) * np.cos(Y[i])  # ✅ 每个 i 独立
    
    # 模式 2: 归约（求和、求最大值等）
    total = 0.0
    for i in prange(n):
        total += X[i]                # ✅ Numba 自动识别归约操作
    
    # 模式 3: 矩阵的行操作
    m, n = X.shape
    row_sums = np.zeros(m)
    for i in prange(m):
        for j in range(n):           # 内层用 range（列顺序访问）
            row_sums[i] += X[i, j]   # 每行独立 → 可并行
    
    return result, total, row_sums
```

### 8.3.4 `prange` 的常见错误

```python
# ❌ 错误 1: 忘记加 parallel=True
@njit                               # 缺少 parallel=True
def bad_prange(arr):
    for i in prange(len(arr)):      # prange 不起作用！
        arr[i] = arr[i] * 2

# ❌ 错误 2: 迭代之间有依赖
@njit(parallel=True)
def bad_dependency(arr):
    for i in prange(1, len(arr)):
        arr[i] = arr[i] + arr[i-1]  # 依赖 arr[i-1] → 竞态条件！
```

## 8.4 归约（Reduction）操作

### 8.4.1 什么是归约

归约 = 把一个数组"归约"成一个标量值。常见归约操作：

- `sum` —— 求和
- `max` / `min` —— 求最大/最小值
- `product` —— 求积
- `any` / `all` —— 逻辑或/与

### 8.4.2 Numba 自动识别归约

```python
@njit(parallel=True)
def reductions(arr):
    # 求和
    total = 0.0
    for i in prange(len(arr)):
        total += arr[i]              # ✅ 自动归约
    
    # 求最大值
    max_val = arr[0]
    for i in prange(len(arr)):
        if arr[i] > max_val:
            max_val = arr[i]         # ✅ 自动归约
    
    # 计数
    count = 0
    for i in prange(len(arr)):
        if arr[i] > 0:
            count += 1               # ✅ 自动归约
    
    return total, max_val, count
```

Numba 会为每个线程创建局部累加器，最后合并——你不需要手动管理。

### 8.4.3 归约的并行原理

```
串行归约：
total = 0
total += arr[0]  → 1
total += arr[1]  → 3
total += arr[2]  → 6
total += arr[3]  → 10

并行归约：
线程 1: local1 = arr[0] + arr[1] = 3
线程 2: local2 = arr[2] + arr[3] = 7
合并:   total = local1 + local2 = 10
```

## 8.5 竞态条件（Race Condition）

### 8.5.1 什么是竞态条件

当多个线程**同时写同一个内存位置**，结果就不确定了——这叫竞态条件。

```python
# ❌ 竞态条件示例
@njit(parallel=True)
def race_condition(arr):
    counter = 0
    for i in prange(len(arr)):
        if arr[i] > 0:
            counter += 1           # 多个线程同时写 counter
    return counter
```

每个线程读到的 `counter` 可能是过时的值，导致最终结果错误。

**Numba 对简单归约能自动处理**（如上面的求和），但**复杂的写入模式**会有问题：

```python
# ❌ 竞态：多个线程写同一个数组位置
@njit(parallel=True)
def histogram_wrong(data, bins):
    hist = np.zeros(len(bins) - 1)
    for i in prange(len(data)):
        bin_idx = np.searchsorted(bins, data[i]) - 1
        hist[bin_idx] += 1         # 多个线程可能写同一个 hist[bin_idx]
    return hist
```

### 8.5.2 避免竞态条件的方法

**方法 1：确保每个迭代写不同位置**

```python
# ✅ 安全：每个 i 写 result[i]，互不冲突
@njit(parallel=True)
def safe_parallel(arr):
    result = np.empty_like(arr)
    for i in prange(len(arr)):
        result[i] = arr[i] * 2     # 每个线程写自己的位置
    return result
```

**方法 2：用归约模式**（Numba 自动处理竞态）

```python
# ✅ 归约操作，Numba 自动做线程局部累加
@njit(parallel=True)
def safe_reduction(arr):
    total = 0.0
    for i in prange(len(arr)):
        total += arr[i]
    return total
```

**方法 3：回退到串行**（当无法并行时）

```python
# 对于有依赖或竞态的，就用 range
@njit(parallel=True)
def mixed_approach(arr):
    # 这部分可以并行
    squared = np.empty_like(arr)
    for i in prange(len(arr)):
        squared[i] = arr[i] ** 2
    
    # 这部分有依赖，必须串行
    cumsum = np.empty_like(arr)
    cumsum[0] = squared[0]
    for i in range(1, len(squared)):  # 用 range，不是 prange
        cumsum[i] = cumsum[i-1] + squared[i]
    
    return cumsum
```

## 8.6 线程控制

### 8.6.1 设置线程数

```python
import numba
import os

# 方法 1: 环境变量（在 import numba 之前）
os.environ['NUMBA_NUM_THREADS'] = '4'

# 方法 2: 代码中设置
numba.set_num_threads(4)

# 方法 3: 查看当前线程数
print(f"当前线程数: {numba.get_num_threads()}")
```

**默认线程数 = CPU 核心数**。大部分情况下不需要改。

### 8.6.2 线程 ID

```python
from numba import get_thread_id

@njit(parallel=True)
def show_thread_id(n):
    for i in prange(n):
        tid = get_thread_id()
        print(f"迭代 {i} 在线程 {tid}")
```

用途：调试并行代码，或者做线程局部存储。

## 8.7 `parallel=True` vs `nogil=True` —— 两种并行的区别

### 8.7.1 核心区别

这是两种**完全不同的并行方式**，很多人会混淆：

| | `parallel=True` | `nogil=True` |
|--|----------------|--------------|
| **谁创建线程** | Numba 自动创建内部线程 | 你在 Python 代码里用 `threading` / `ThreadPoolExecutor` 创建 |
| **典型用法** | `@njit(parallel=True)` + `prange` | `@njit(nogil=True)` + Python `ThreadPoolExecutor` |
| **释放 GIL** | 自动释放（Numba 内部线程不受 GIL 限制） | 显式释放，让多个 Python 线程能并行调用 |
| **适用场景** | 单个函数内部的数据并行（数组操作） | 多个任务各自独立，用 Python 调度 |
| **是否互斥** | 否——可以同时使用 | 否——可以同时使用 |

### 8.7.2 `parallel=True` —— Numba 内部并行

```python
# Numba 自己管理线程，你不需要写任何 threading 代码
@njit(parallel=True)
def process_large_array(arr):
    result = np.empty_like(arr)
    for i in prange(len(arr)):      # Numba 自动把 i 分给多个线程
        result[i] = heavy_computation(arr[i])
    return result

# 调用：就像普通函数，Numba 内部自动并行
arr = np.random.randn(10_000_000)
result = process_large_array(arr)   # Numba 自动用 8 个核心处理
```

**适用场景**：
- 单个大数组，需要对每个元素做相同操作
- 矩阵运算、图像处理、特征工程
- 你不想自己管理线程

### 8.7.3 `nogil=True` —— 配合 Python 多线程

```python
# Python 端创建多个线程，每个线程调用 Numba 函数
from concurrent.futures import ThreadPoolExecutor

@njit(nogil=True)               # ← 关键：释放 GIL
def process_single_task(task_id, data):
    # 每个任务独立处理自己的数据
    result = 0.0
    for x in data:
        result += heavy_computation(x)
    return task_id, result

# 10 个独立任务，每个任务处理不同的数据
tasks = [(i, np.random.randn(100000)) for i in range(10)]

# 用 Python ThreadPoolExecutor 并行执行
with ThreadPoolExecutor(max_workers=8) as executor:
    # 如果没有 nogil=True，这些线程会因为 GIL 而串行执行
    # 有了 nogil=True，Numba 函数释放 GIL，8 个任务真正并行
    results = list(executor.map(lambda t: process_single_task(t[0], t[1]), tasks))
```

**适用场景**：
- 多个独立的任务（不同的数据集、不同的参数配置）
- 需要用 Python 做任务调度、结果收集
- 配合 `concurrent.futures` / `threading` 等 Python 多线程框架

### 8.7.4 为什么需要 `nogil=True`

默认情况下，即使用 `@njit` 编译，**函数执行时仍然持有 GIL**（为了安全调用 Python 对象）。这意味着：

```python
# ❌ 没有 nogil=True
@njit
def worker(data):
    return np.sum(data)

# Python 多线程调用
with ThreadPoolExecutor(max_workers=8) as executor:
    tasks = [np.random.randn(1000000) for _ in range(8)]
    results = list(executor.map(worker, tasks))
    # 8 个线程因为 GIL 而排队执行，实际还是串行！
```

```python
# ✅ 加上 nogil=True
@njit(nogil=True)               # ← 释放 GIL
def worker(data):
    return np.sum(data)

# 现在 8 个线程可以真正并行
with ThreadPoolExecutor(max_workers=8) as executor:
    tasks = [np.random.randn(1000000) for _ in range(8)]
    results = list(executor.map(worker, tasks))
    # 8 个线程真正并行，加速 ~8×
```

### 8.7.5 两者能否组合

**可以！**`parallel=True` 和 `nogil=True` 可以同时使用：

```python
@njit(parallel=True, nogil=True)
def best_of_both(arr):
    result = np.empty_like(arr)
    for i in prange(len(arr)):      # Numba 内部并行
        result[i] = arr[i] ** 2
    return result

# 外层：Python 多线程调用不同任务
# 内层：每个任务内部用 Numba 并行处理大数组
with ThreadPoolExecutor(max_workers=4) as executor:
    tasks = [np.random.randn(10_000_000) for _ in range(4)]
    results = list(executor.map(best_of_both, tasks))
    # 总并行度 ≈ 4 (Python 线程) × CPU核心数 (Numba 内部)
```

**注意**：这种嵌套并行可能造成**线程过载**（oversubscription）——总线程数超过 CPU 核心数，反而变慢。通常需要限制：

```python
import numba

# 限制 Numba 内部线程数
numba.set_num_threads(2)

# Python 线程数 = 4，Numba 内部线程数 = 2
# 总并行度 ≈ 4 × 2 = 8，适合 8 核 CPU
with ThreadPoolExecutor(max_workers=4) as executor:
    ...
```

### 8.7.6 选择指南

```
┌──────────────────────────────────────────────┐
│ 你的场景是什么？                              │
├──────────────────────────────────────────────┤
│                                              │
│ 单个大数组，逐元素操作                         │
│   → @njit(parallel=True) + prange           │
│                                              │
│ 多个独立任务，用 Python 调度                   │
│   → @njit(nogil=True) + ThreadPoolExecutor  │
│                                              │
│ 多个任务，每个任务内部有大数组                  │
│   → @njit(parallel=True, nogil=True)        │
│      但要限制 Numba 线程数避免过载            │
│                                              │
└──────────────────────────────────────────────┘
```

### 8.7.7 `nogil=True` 的限制

```python
@njit(nogil=True)
def cannot_use_python_objects(arr):
    # ✅ 可以：纯 NumPy/Numba 操作
    return np.sum(arr) + 1

@njit(nogil=True)
def cannot_do_this(arr):
    # ❌ 不行：调用 Python 对象需要 GIL
    result = []                    # Python list
    for x in arr:
        result.append(x)           # 需要 GIL → 编译失败
    return result
```

`nogil=True` 要求函数内部**完全不接触 Python 对象**——只能用 NumPy 数组、Numba typed 容器、标量等。

## 8.8 并行性能基准

### 8.8.1 并行加速的理想与现实

```
理想加速比 = 核心数

8 核 CPU:
  单核耗时 8s
  理想并行耗时 = 8s / 8 = 1s
  理想加速比 = 8×

实际情况:
  实际并行耗时 ≈ 1.5s
  实际加速比 ≈ 5.3×  ← 不到理想的 8×
```

**为什么达不到理想加速比？**

| 原因 | 影响 |
|------|------|
| 线程创建/同步开销 | 固定成本，小任务时占比大 |
| 内存带宽瓶颈 | 8 个核心争抢同一块内存 |
| 缓存失效（cache miss） | 多线程导致缓存行竞争 |
| 不能完全并行的代码部分 | Amdahl 定律限制 |

**经验值**：对于计算密集型任务，实际加速比 ≈ 0.6 ~ 0.8 × 核心数。

### 8.8.2 什么时候不该用并行

```python
# 太小的数组 → 线程开销大于收益
arr_small = np.random.randn(100)       # 并行反而慢
arr_large = np.random.randn(10_000_000) # 并行有收益

# 经验值：数组大小 > 10,000 时才考虑并行
```

## 8.9 本章关键概念

| 概念 | 一句话 |
|------|--------|
| **数据并行** | 同一操作，应用到不同数据，Numba 的主打方向 |
| **`parallel=True`** | Numba 内部自动并行，配合 `prange` 使用 |
| **`nogil=True`** | 释放 GIL，让 Python 多线程能真正并行调用 Numba 函数 |
| **`prange`** | 显式并行循环，每次迭代必须独立 |
| **归约** | 数组→标量（sum/max/min），Numba 自动处理线程合并 |
| **竞态条件** | 多线程同时写同一位置 → 结果不确定，确保每次迭代写不同位置 |
| **GIL** | Python 全局锁，`@njit` 默认持有，`nogil=True` 释放 |
| **理想 vs 实际加速比** | 理想 = 核心数，实际 ≈ 0.6~0.8 × 核心数 |
| **何时并行** | 数组 > 10,000 + 计算密集（不是 I/O 密集） |
