# 第十一章：调试与诊断

> **核心问题**：Numba 编译失败了怎么办？性能不如预期怎么排查？类型推断错误怎么定位？本章提供系统化的调试方法。

---

## 11.1 常见编译错误及解决方案

### 11.1.1 TypingError —— 类型推断失败

**错误示例**：

```python
@njit
def bad_typing(arr):
    result = []                    # Python list
    for x in arr:
        result.append(x)           # ❌ TypingError
    return result
```

**错误信息**：

```
numba.core.errors.TypingError: Failed in nopython mode pipeline
...
Use of unsupported NumPy function 'numpy.append' or unsupported use of the function.
```

**解决方案**：

```python
# 方案 1: 用 NumPy 数组
@njit
def good_typing_1(arr):
    result = np.empty_like(arr)
    for i in range(len(arr)):
        result[i] = arr[i] * 2
    return result

# 方案 2: 用 numba.typed.List
from numba.typed import List
@njit
def good_typing_2(arr):
    result = List.empty_list(types.float64)
    for x in arr:
        result.append(x)
    return result
```

### 11.1.2 LoweringError —— 无法生成机器码

**错误示例**：

```python
@njit
def bad_lowering():
    import json                    # ❌ 不能在 @njit 内 import
    return json.dumps({'a': 1})
```

**错误信息**：

```
numba.core.errors.LoweringError: Failed in nopython mode pipeline
...
'import' is not supported
```

**解决方案**：把 import 移到函数外，或者放弃用不支持的模块。

### 11.1.3 NumbaPerformanceWarning —— 回退到 object 模式

**警告示例**：

```python
@njit
def slow_code(arr):
    result = []                    # ⚠️ reflected list
    for x in arr:
        result.append(x)
    return result
```

**警告信息**：

```
NumbaPerformanceWarning: 
@njit is using object mode, performance may be degraded.
```

**解决方案**：
1. 用 `nopython=True` 或 `@njit` 强制报错（而不是警告）
2. 检查 `inspect_types()` 找出回退的地方
3. 替换为 Numba 支持的类型

## 11.2 `inspect_types()` —— 查看类型推断结果

### 11.2.1 基本用法

```python
@njit
def compute(a, b):
    x = a + b
    y = x * 2
    return y

# 触发编译
compute(1, 2)

# 查看类型推断
print(compute.inspect_types())
```

**输出示例**：

```
compute (int64, int64)
--------------------------------------------------------------------------------
# File: example.py
# --- LINE 2 ---

@njit

# --- LINE 3 ---

def compute(a, b):

    # --- LINE 4 ---
    # label 0
    #   a = arg(0, name=a)  :: int64
    #   b = arg(1, name=b)  :: int64
    #   $4load_global.0 = global(+: <built-in function add>)  :: Function(<built-in function add>)
    #   x = call $4load_global.0(a, b, func=$4load_global.0, args=[Var(a, example.py:4), Var(b, example.py:4)], kws=(), vararg=None)  :: (int64, int64) -> int64

    x = a + b

    # --- LINE 5 ---
    #   y = x * 2  :: int64

    y = x * 2

    # --- LINE 6 ---
    #   $12return_value.2 = cast(value=y)  :: int64
    #   return $12return_value.2

    return y
```

**解读**：
- `x = a + b` → `x :: int64`（x 的类型是 int64）
- `y = x * 2` → `y :: int64`
- 所有类型都正确推断，没有 `pyobject` 出现 → ✅ 编译成功

### 11.2.2 定位 object 模式回退

```python
@njit
def mixed_mode(arr):
    result = []                    # ← 这里会回退
    for x in arr:
        result.append(x)
    return result

mixed_mode(np.array([1, 2, 3]))
print(mixed_mode.inspect_types())
```

在输出中搜索 `pyobject`：

```
result = []  :: pyobject
```

看到 `pyobject` 就说明这里没有被编译，性能会很差。

## 11.3 环境变量调试开关

### 11.3.1 `NUMBA_DISABLE_JIT` —— 禁用编译

```bash
# Linux/Mac
export NUMBA_DISABLE_JIT=1
python your_script.py

# Windows
set NUMBA_DISABLE_JIT=1
python your_script.py
```

效果：所有 `@njit` 装饰的函数**不编译**，直接用 Python 解释器执行。用于：
- 调试逻辑错误（可以用 `pdb` / IDE 断点）
- 排除是否是 Numba 编译导致的 bug

### 11.3.2 `NUMBA_WARNINGS` —— 显示所有警告

```bash
export NUMBA_WARNINGS=1
```

显示所有性能警告、类型推断警告等，帮助排查隐藏的问题。

### 11.3.3 `NUMBA_DUMP_ANNOTATION` —— 导出类型注解

```bash
export NUMBA_DUMP_ANNOTATION=1
```

把每个函数的类型推断结果输出到 HTML 文件，方便可视化查看。

## 11.4 性能诊断工具

### 11.4.1 `@njit(debug=True)` —— 生成调试信息

```python
@njit(debug=True)
def slow_function(arr):
    ...
```

效果：
- 保留源码行号信息
- 生成更详细的 LLVM IR
- 可以用 `gdb`（Linux/Mac）调试

**代价**：生成的代码稍慢（~5-10%）

### 11.4.2 查看编译后的 LLVM IR

```python
@njit
def func(x):
    return x * x + 1

func(1)  # 触发编译
print(func.inspect_llvm())
```

输出 LLVM 中间表示（IR），用于高级性能分析。

### 11.4.3 查看生成的汇编代码

```python
print(func.inspect_asm())
```

输出 x86-64 汇编代码。用于：
- 检查是否向量化（SSE/AVX 指令）
- 确认循环展开
- 高级优化验证

## 11.5 常见性能陷阱及排查

### 11.5.1 循环内分配内存

```python
# ❌ 性能陷阱
@njit
def bad_alloc(n):
    for i in range(n):
        temp = np.zeros(100)       # 每次分配 → 慢
        ...

# ✅ 修复
@njit
def good_alloc(n):
    temp = np.zeros(100)           # 只分配一次
    for i in range(n):
        ...
```

**排查方法**：用 `line_profiler` 逐行计时，看哪行占比最高。

### 11.5.2 类型不稳定

```python
# ❌ 类型变化 → Numba 生成复杂的分发代码
@njit
def unstable_type(n):
    x = 0
    for i in range(n):
        if i % 2 == 0:
            x = i                  # int
        else:
            x = float(i)           # float ← 类型不稳定
    return x

# ✅ 统一类型
@njit
def stable_type(n):
    x = 0.0                        # 统一用 float
    for i in range(n):
        if i % 2 == 0:
            x = float(i)
        else:
            x = float(i)
    return x
```

**排查方法**：`inspect_types()` 看变量类型是否一致。

### 11.5.3 小数组用并行反而慢

```python
# ❌ 数组太小，并行开销 > 收益
@njit(parallel=True)
def bad_parallel():
    arr = np.arange(100)           # 只有 100 个元素
    for i in prange(len(arr)):
        arr[i] = arr[i] ** 2
```

**排查方法**：对比串行和并行版本的实际运行时间。

## 11.6 错误信息速查

| 错误类型 | 常见原因 | 解决方案 |
|---------|---------|---------|
| `TypingError` | 使用了不支持的类型 | 改用 Numba 支持的类型 |
| `LoweringError` | 语法不支持（import/yield） | 移除不支持的语法 |
| `UntypedAttributeError` | 访问未定义的属性 | 检查拼写、`@jitclass` 字段 |
| `NumbaPerformanceWarning` | 回退到 object 模式 | 用 `inspect_types()` 排查 |
| `CompilationError` | 递归深度、复杂度超限 | 简化函数、拆分逻辑 |
| `ValueError: `<br>`not supported` | 函数签名不匹配 | 检查类型声明 |

## 11.7 调试流程图

```
Numba 函数不工作？
    │
    ├─→ 能编译吗？
    │   ├─ 能 → 跳到"性能问题"
    │   └─ 不能 → 看错误类型
    │       ├─ TypingError → 检查类型（inspect_types）
    │       ├─ LoweringError → 检查语法（不支持的特性）
    │       └─ 其他 → 查看完整错误栈
    │
    └─→ 性能问题？
        ├─ 比预期慢 → 排查
        │   ├─ inspect_types() 有 pyobject？ → 回退到 object 模式
        │   ├─ 循环内分配数组？ → 移到外面
        │   ├─ 类型不稳定？ → 统一类型
        │   ├─ 数组太小用 parallel？ → 改为串行
        │   └─ 用 line_profiler 找热点行
        │
        └─ 结果错误 → 逻辑 bug
            ├─ NUMBA_DISABLE_JIT=1 运行 → 用 Python 调试器
            └─ 对比 Numba 版本和纯 Python 版本结果
```

## 11.8 实战案例：一个完整的调试流程

### 11.8.1 问题：函数很慢

```python
@njit
def slow_function(arr):
    result = []
    for x in arr:
        result.append(x * 2)
    return result

arr = np.random.randn(1000000)
%timeit slow_function(arr)  # 很慢！
```

### 11.8.2 步骤 1：检查类型

```python
slow_function(arr[:10])
print(slow_function.inspect_types())
```

发现：`result :: pyobject` → 回退到 object 模式

### 11.8.3 步骤 2：修复类型

```python
@njit
def fast_function(arr):
    result = np.empty_like(arr)
    for i in range(len(arr)):
        result[i] = arr[i] * 2
    return result
```

### 11.8.4 步骤 3：验证性能

```python
%timeit fast_function(arr)  # 快很多！
```

### 11.8.5 步骤 4：检查是否还有优化空间

```python
# 尝试向量化
@njit
def vectorized(arr):
    return arr * 2

%timeit vectorized(arr)  # 更快！
```

## 11.9 本章关键概念

| 概念 | 一句话 |
|------|--------|
| **`inspect_types()`** | 查看类型推断结果，找出 `pyobject` 回退 |
| **`TypingError`** | 类型推断失败，使用了不支持的类型 |
| **`LoweringError`** | 语法不支持，如 `import`/`yield` |
| **`NUMBA_DISABLE_JIT`** | 禁用编译，用 Python 调试器排查逻辑错误 |
| **`debug=True`** | 生成调试信息，保留行号 |
| **性能陷阱** | 循环内分配、类型不稳定、小数组并行 |
| **调试流程** | 先看能否编译 → 检查类型 → 排查性能陷阱 |
