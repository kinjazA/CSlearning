# 第六章：性能优化策略

> **核心问题**：`@njit` 不是万能加速器。写出"对的 Numba 代码"和写出"最快的 Numba 代码"之间，有 5–10 倍的差距。本章讲怎么填平这个差距。

---

## 6.1 先搞清楚：Numba 到底在哪一步省了时间

回顾一下执行一个 Numba 函数时的完整时间线：

```
┌────────── 热身 ──────────┐   ┌──── 执行（每次）────┐
│ 类型推断 → LLVM → 机器码  │   │  直接跑机器码       │
│ ~0.05 ∼ 2s（一次性）     │   │  ~0.001s            │
└──────────────────────────┘   └─────────────────────┘
```

Numba 加速的来源：
1. **消除解释器开销** — 没有循环中每条字节码的 C 层分派
2. **类型确定后的优化** — LLVM 知道所有变量类型，可以做 C 编译器级别的优化
3. **CPU 向量化 (SIMD)** — LLVM 自动把循环转为 SSE/AVX 指令，一条指令处理多个数据

**Numba 省不掉的**：
- 内存分配和拷贝的时间
- 算法本身的计算量（O(n²) 编译后还是 O(n²)）
- I/O、网络等和计算无关的操作

## 6.2 循环优化 —— 收益最大的方向

### 6.2.1 不要在循环内分配数组

这是 Numba 新手最常见的性能错误：

```python
# ❌ 每次循环都创建一个新数组 —— 灾难
@njit
def bad_loop(arr):
    for i in range(len(arr)):
        # 每次迭代都在堆上分配内存，+ 垃圾回收压力
        chunk = np.zeros(100)
        arr[i] = np.sum(chunk * i)

# ✅ 在循环外分配，循环内只读写
@njit
def good_loop(arr):
    chunk = np.zeros(100)            # 只分配一次
    for i in range(len(arr)):
        arr[i] = np.sum(chunk * i)
```

**为什么这很重要**：内存分配（`malloc`）比加法运算慢 100–1000 倍。循环内分配数组会让你的编译加速全部作废。

### 6.2.2 用局部标量代替数组元素反复读取

```python
# ❌ 每次循环都从内存取 arr[i] —— 多次内存访问
@njit
def slow_sum(arr):
    s = 0.0
    for i in range(len(arr)):
        s += arr[i] * arr[i] + arr[i] * 2.0 + arr[i]
    return s

# ✅ 把 arr[i] 读到局部变量，重复使用
@njit
def fast_sum(arr):
    s = 0.0
    for i in range(len(arr)):
        x = arr[i]              # 一次内存读
        s += x * x + x * 2.0 + x  # 三次运算都用 x
    return s
```

**原理**：CPU 寄存器（局部变量）的访问延迟 ~0，内存访问 ~100 个时钟周期。把频繁用到的值装进局部变量，LLVM 才有机会把它放进寄存器。

### 6.2.3 减少分支（分支预测失败代价高）

```python
# ❌ 循环内 if-else —— 每次迭代都要分支
@njit
def branch_heavy(arr):
    result = np.empty_like(arr)
    for i in range(len(arr)):
        if arr[i] > 0:
            result[i] = np.sqrt(arr[i])
        else:
            result[i] = -np.sqrt(-arr[i])
    return result

# ✅ 用 np.where / np.sign 等无分支写法 —— 不是总能做到
@njit
def branch_light(arr):
    result = np.sqrt(np.abs(arr))
    result = np.where(arr > 0, result, -result)
    return result
```

### 6.2.4 把多次数据遍历合并为一次（循环融合）

```python
# ❌ 三次独立的数据遍历 —— 数据走了三遍，产生了两个中间数组
@njit
def separate_passes(arr):
    a = arr + 1         # 第1遍遍历 → 分配临时数组 a
    b = a * 2           # 第2遍遍历 → 分配临时数组 b
    c = np.sqrt(b)      # 第3遍遍历 → 最终结果 c
    return c

# ✅ 合并为一次遍历 —— 数据只走一遍，零中间数组
@njit
def fused_pass(arr):
    n = len(arr)
    c = np.empty_like(arr)
    for i in range(n):
        tmp = arr[i] + 1
        tmp = tmp * 2
        c[i] = np.sqrt(tmp)
    return c
```

**优势**：
- 减少数据遍历次数（3 pass → 1 pass）
- 消灭中间数组的内存分配（2 个临时数组 → 0，对于百万级数组节省 ~16MB）
- 数据留在 CPU cache 中，不需要反复从主存加载

## 6.3 内存布局 —— 第三章的回顾 + 实战

关于 C-contiguous vs F-contiguous 的原理已在第三章 §3.3.3 详讲。这里补充两个实战技巧：

### 6.3.1 用 `np.ascontiguousarray` 确保布局

```python
@njit
def process(arr):
    # 如果 arr 是 F-contiguous，Numba 会自动做一次拷贝
    return arr * 2.0 + 1.0

# 更好的做法：调用前主动转换
arr = some_external_function()            # 不确定返回什么布局
arr = np.ascontiguousarray(arr)           # 确保 C-contiguous
result = process(arr)                     # 无隐式拷贝
```

### 6.3.2 遍历方向的选择

这是最容易犯的错，直接决定 3–5 倍的性能差异：

```python
# ✅ C 数组：外层沿行，内层沿列
for i in range(n):          # 外层行
    for j in range(m):      # 内层列 ← 连续访问内存
        out[i, j] = arr[i, j] ** 2

# ❌ C 数组：外层沿列，内层沿行
for j in range(m):          # 外层列
    for i in range(n):      # 内层行 ← 内存跳跃
        out[i, j] = arr[i, j] ** 2
```

## 6.4 预分配输出数组

```python
# ❌ 函数内部每次都创建新数组
@njit
def bad_prealloc(a, b):
    result = np.empty(len(a))         # 每次调用都分配
    for i in range(len(a)):
        result[i] = a[i] + b[i]
    return result

# ✅ 调用方预分配，传入输出数组（out parameter pattern）
@njit
def good_prealloc(a, b, out):
    """结果写入 out，不分配新内存"""
    for i in range(len(a)):
        out[i] = a[i] + b[i]

# 使用：
a = np.random.randn(1000000)
b = np.random.randn(1000000)
out = np.empty_like(a)
for epoch in range(1000):
    good_prealloc(a, b, out)          # 1000 次调用，0 次内存分配
```

## 6.5 `fastmath=True` —— 什么时候值得冒险

> 详见第二章 §2.5.3

快速判断是否适合 `fastmath=True`：

| 场景 | 建议 |
|------|------|
| 图形渲染、游戏物理、动画 | ✅ 放心开启 |
| 数值模拟（天气、流体）、信号处理 | ⚠️ 测试后决定 |
| 金融计算、加密、科学论文级别计算 | ❌ 不要开 |
| 机器学习训练 | ⚠️ 推理可以，训练慎用（梯度可能有微小差异） |

## 6.6 编译缓存 —— 避免重复热身

> 详见第二章 §2.5.1

```python
@njit(cache=True)      # 开发时就加上
def my_func(arr):
    ...
```

- 第一次运行 → 编译 + 写入 `__pycache__/*.nbc`
- 以后每次运行 → 直接加载，跳过编译
- 源码改了 → 自动失效，重新编译

## 6.7 基准测试的正确姿势

### 6.7.1 必须包含热身（编译成本不能被计为运行时间）

```python
import time

@njit
def target_func(arr):
    ...

# ❌ 错误基准测试：包含了编译时间
arr = np.random.randn(1000000)
t0 = time.perf_counter()
result = target_func(arr)          # 第一次调用 → 编译 + 执行
t = time.perf_counter() - t0      # 这个时间不真实！

# ✅ 正确基准测试：先热身，再计时
arr = np.random.randn(1000000)
target_func(arr[:10])              # 热身：用小数据触发编译
t0 = time.perf_counter()
for _ in range(100):
    result = target_func(arr)      # 热身后的执行
t = (time.perf_counter() - t0) / 100
```

### 6.7.2 用 `%timeit`（Jupyter）/ `timeit` 自动处理热身

```python
import timeit

# timeit 默认会跑多轮，自动排除热身
t = timeit.timeit(
    lambda: target_func(arr),
    number=100
) / 100
```

## 6.8 加速比和热点——别给所有代码都加 @njit

### 6.8.1 先定义两个概念

**加速比 (Speedup)**：优化后比优化前快了多少倍。

```
加速比 = 优化前的运行时间 / 优化后的运行时间

例：原来跑 10 秒，优化后跑 2 秒 → 加速比 = 10/2 = 5×
```

加速比 > 1 说明变快了，加速比 < 1 说明变慢了（做负优化了）。

**热点 (Hotspot)**：程序运行时间里，**占比最大的部分**。

```
程序总运行时间 100 秒：
  ├── load_data()     2 秒  (2%)    ← 不热
  ├── preprocess()    8 秒  (8%)    ← 不热
  ├── compute()      80 秒 (80%)    ← 🔥 热点！
  └── save_result()  10 秒 (10%)    ← 不热
```

`compute()` 占了 80% 的时间，它就是热点。优化它 → 收益最大；优化 `load_data()` → 就算把它优化到 0 秒，整体也只快了 2%。

### 6.8.2 一个"挑水"的故事

想象你每天要去河边挑水，一天的任务是：

```
                    耗时      占比
去河边（走路）       20 分钟    10%
在河边舀水（热点）   3 小时     90%   ← 🔥
挑水回家（走路）     0 分钟     0%（算在去程里）
```

你嫌太慢，决定买一辆跑车——**加速去河边的路程**。走路 20 分钟变成开车 2 分钟，快了 10 倍！

```
加速前：20 + 180 = 200 分钟
加速后： 2 + 180 = 182 分钟
加速比 = 200/182 ≈ 1.1×  ← 几乎没快！
```

你优化错了地方。真正应该优化的是**舀水**——比如换个更大的桶，让 3 小时变成 30 分钟：

```
加速前：20 + 180 = 200 分钟
加速后：20 +  30 =  50 分钟
加速比 = 200/50 = 4×  ← 快了4倍！
```

**这个故事的教训**：谁的占比大，就优化谁。给占比 10% 的部分加速 10 倍，整体只快 1.1 倍。给占比 90% 的部分加速 10 倍，整体快将近 10 倍。

### 6.8.3 Amdahl 定律——数学版

Amdahl 定律把这个直觉用公式表达出来：

```
加速比 = ──────────────────────
         (1 - f)  +  (f / s)

其中：
  f  = 热点占程序总时间的比例（0~1 之间）
  s  = 你对热点加速的倍数
```

套入"挑水"故事验证：

```
舀水的占比 f = 0.9（90%）
你给舀水加速 s = 6 倍（3小时 → 30分钟）

加速比 = 1 / ((1 - 0.9) + 0.9/6)
       = 1 / (0.1 + 0.15)
       = 1 / 0.25
       = 4×  ✅ 和上面手算一致
```

再套入 Numba 的典型场景：

```
你的机器学习代码：
  - 数据预处理: 10%
  - 特征工程（Numba 可加速 50×）: 80%  ← 热点
  - 模型训练: 10%

f = 0.8, s = 50

加速比 = 1 / ((1 - 0.8) + 0.8/50)
       = 1 / (0.2 + 0.016)
       = 1 / 0.216
       = 4.6×

即使 Numba 把特征工程加速了 50 倍，
整个程序"只"快了 4.6 倍。
```

**为什么会这样？** 因为那 20%（数据预处理 + 模型训练）没有被加速，它们成了新的瓶颈。Amdahl 定律揭示了一个冰冷的现实：**整体加速受限于不可加速部分**。

### 6.8.4 怎么找到热点

不靠猜，靠工具。这里讲两种方法。

#### 方法 1：cProfile（Python 自带，无需安装）

**命令拆解**：

```bash
python -m cProfile -s cumtime your_script.py
│      │  │         │  │           │
│      │  │         │  │           └─ 你要分析的脚本文件
│      │  │         │  └─ 排序依据：cumtime = 按累计时间排序
│      │  │         │                （可用 tottime、ncalls、calls 等）
│      │  │         └─ -s = sort（排序）
│      │  └─ cProfile 模块名
│      └─ -m = module（把模块当脚本运行）
└─ Python 解释器
```

**完整示例**。假设你有一个脚本 `my_script.py`：

```python
# my_script.py
import numpy as np
import time

def load_data():
    time.sleep(0.1)                        # 模拟 I/O
    return np.random.randn(10000)

def compute_features(data):
    result = np.zeros_like(data)            # 🔥 热点在这里
    for i in range(len(data)):
        for j in range(100):
            result[i] += np.sin(data[i]) * j
    return result

def save_result(result):
    time.sleep(0.05)                       # 模拟写文件

# 主程序
data = load_data()
features = compute_features(data)
save_result(features)
```

**运行分析**：

```bash
$ python -m cProfile -s cumtime my_script.py

         52345 function calls in 1.847 seconds

   Ordered by: cumulative time

   ncalls  tottime  percall  cumtime  percall filename:lineno(function)
        1    0.000    0.000    1.847    1.847 my_script.py:1(<module>)
        1    0.000    0.000    1.747    1.747 my_script.py:10(compute_features)  ← 🔥
        1    0.100    0.100    0.100    0.100 my_script.py:5(load_data)
        1    0.050    0.050    0.050    0.050 my_script.py:16(save_result)
    10000    1.747    0.000    1.747    0.000 {built-in method numpy.sin}
        ...
```

**怎么读这个输出**：

| 列名 | 含义 | 这个例子中 |
|------|------|----------|
| `ncalls` | 函数被调用了多少次 | `compute_features` 调用了 1 次 |
| `tottime` | 函数**自身**花了多少时间（不含它调用的子函数） | `compute_features` 自身花了 0 秒——时间都在它调用的子函数里 |
| `cumtime` | 函数**总共**花了多少时间（含子函数） | `compute_features` 总共花了 **1.747 秒**——这就是热点！ |
| `percall` | 平均每次调用的时间 | cumtime / ncalls |

> **看 `tottime` 找自身慢的函数，看 `cumtime` 找整条调用链慢的函数。两个都要看。**

在这个例子里，`compute_features` 的 `cumtime = 1.747` 秒，占总时间 95%——它就是热点。`load_data` 和 `save_result` 加起来才 0.15 秒，优化它们没意义。

#### 方法 2：在代码里嵌入 cProfile（不需要命令行）

如果你不想每次敲命令行，也可以在代码里写：

```python
import cProfile
import pstats

# 方法 2a：直接跑并打印
cProfile.run('my_function()', sort='cumtime')

# 方法 2b：保存到文件然后分析
cProfile.run('my_function()', 'profile_output.stats')
p = pstats.Stats('profile_output.stats')
p.sort_stats('cumtime').print_stats(20)   # 打印前 20 名
```

#### 方法 3：line_profiler（逐行分析，更精细）

cProfile 告诉你**哪个函数**是热点。如果想更进一步——知道函数里**哪一行**最慢，用 `line_profiler`：

```bash
uv pip install line_profiler
```

```python

@profile                       # ← 不需要 import，kernprof 会自动注入
def compute_features(data):
    result = np.zeros_like(data)
    for i in range(len(data)):          # 哪一行最慢？
        for j in range(100):
            result[i] += np.sin(data[i]) * j
    return result
```

```bash
kernprof -l -v my_script.py

# 输出：
# Line #  Hits    Time     % Time  Line Contents
# ==============================================
#     11                                    def compute_features(data):
#     12     1    15237      0.0%              result = np.zeros_like(data)
```

### 6.8.5 什么不该用 Numba

```
✅ 应该用 Numba：
  - 热点中有大量的数值循环
  - 热点代码被调用了很多次
  - 循环内包含不能向量化的分支逻辑

❌ 不该用 Numba：
  - 占比 < 10% 的代码 → 即使加速 100×，整体也只快 ~1.1×
  - 已经是纯 NumPy 向量化操作 → np.sum(a+b) 编译后不会更快
  - 只调用一次的短函数 → 热身时间 > 运行时间
  - I/O、网络、文件读写 → Numba 根本不支持
```

## 6.9 性能优化决策树

```
你的 @njit 函数慢？
    │
    ├─→ 跑一遍，用 inspect_types() 确认没有回退到 object 模式
    │
    ├─→ 检查循环内是否在分配内存？
    │       是 → 移到循环外
    │
    ├─→ 检查遍历方向是否匹配数组布局？
    │       不匹配 → 交换内外层循环顺序
    │
    ├─→ 检查是否反复读同一个数组元素？
    │       是 → 用局部变量缓存
    │
    ├─→ 检查是否有多个独立循环可以合并？
    │       是 → 循环融合
    │
    ├─→ 检查调用方是否可以预分配输出数组？
    │       可以 → out parameter pattern
    │
    ├─→ 可以接受微量浮点误差吗？
    │       可以 → fastmath=True
    │
    └─→ 函数已被多次调用？
            是 → cache=True
```

## 6.10 本章关键概念

| 概念 | 一句话 |
|------|--------|
| **循环内不分配** | 内存分配比计算贵 100–1000 倍，把 `np.zeros` 等移出循环 |
| **局部变量缓存** | 反复读的数组元素存到局部变量，让它进 CPU 寄存器 |
| **循环融合** | 多个顺序循环合并为一个，减少遍历次数和中间数组 |
| **out parameter** | 调用方预分配输出数组，函数只管写入 |
| **正确的基准测试** | 必须热身后再计时，否则量的是编译时间+运行时间 |
| **Amdahl 定律** | 只加速热点，不要给所有函数都加 `@njit` |
| **决策树** | 按优先级排查：object 回退 → 内存分配 → 布局 → 缓存 → 融合 → 预分配 → fastmath |
