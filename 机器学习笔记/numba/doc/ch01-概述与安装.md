# 第一章：Numba 概述与安装

> **核心问题**：Python 写起来快，跑起来慢。Numba 如何在不改变代码风格的前提下，把 Python 函数编译成接近 C 速度的机器码？

---

## 1.1 为什么需要 Numba

### 1.1.1 Python 的性能瓶颈

Python 慢，根源在于它的**解释执行**模型和**动态类型**系统。

```python
def add(a, b):
    return a + b
```

当 Python 执行这行代码时，发生了什么？

```
1. 字节码解释器取指：LOAD_FAST a, LOAD_FAST b, BINARY_ADD, RETURN_VALUE
2. 对于 BINARY_ADD，解释器必须：
   a. 检查 a 和 b 的类型（PyObject 的 type 指针）
   b. 如果是 int，调用 PyLong_Add；如果是 str，调用 PyUnicode_Concat...
   c. 分配新的 PyObject 来存结果
3. 巨大的解释循环 overhead：每条字节码都是一次 C 函数调用
```

**三个核心瓶颈**：

| 瓶颈 | 原因 | 影响 |
|------|------|------|
| **解释器开销** | 每条字节码需要 C 层分发处理 | 循环 → 灾难 |
| **动态类型检查** | 每次操作都查类型、分派具体实现 | 数值计算 → 大量浪费 |
| **GIL（全局解释器锁）** | CPython 同一时刻只有一个线程执行 Python 字节码 | 多核 → 用不上 |

### 1.1.2 常见加速方案对比

| 方案 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **NumPy 向量化** | 把循环推到 C 层执行 | 简单，一行代码 | 复杂逻辑写不出向量化；中间数组多，内存爆炸 |
| **Cython** | 给 Python 加类型声明 → 编译为 C 扩展 | 成熟，生态好 | 需要写 .pyx 文件、学类型语法、构建系统 |
| **PyPy** | 用 JIT 编译器替代 CPython 解释器 | 无需改代码 | C 扩展兼容性差，生态不完整 |
| **C/C++ 扩展** | 手写 C 用 ctypes/cffi/pybind11 调用 | 极致性能 | 开发慢、维护贵、脱离 Python 生态 |
| **Numba** | 给 Python 函数加 `@jit` 装饰器 → LLVM 即时编译为机器码 | **零侵入**、纯 Python 开发 | 只支持 Python/NumPy 子集 |

### 1.1.3 Numba 的定位

> **Numba = 用装饰器标记 Python 函数 → 函数调用时即时编译为原生机器码 → 后续调用直接跑机器码**

听起来像魔法？看一个例子：

```python
import numpy as np
from numba import njit

# 原始 Python 函数
def sum_python(arr):
    total = 0
    for x in arr:
        total += x
    return total

# 加一行装饰器
@njit
def sum_numba(arr):
    total = 0
    for x in arr:
        total += x
    return total

# 性能对比
arr = np.random.randn(10_000_000)
# sum_python(arr)  → ~2.5 秒
# sum_numba(arr)   → ~0.02 秒（首次 0.05 秒含编译）
# np.sum(arr)      → ~0.01 秒（和 Numba 同数量级）
```

> **关键洞察**：对于无法向量化的复杂循环逻辑（多层嵌套、依赖前一迭代结果），Numba 往往比 NumPy 快，因为省掉了大量中间数组的分配。

## 1.2 Numba 工作原理

### 1.2.1 整体架构

```
 Python 函数（.py）
        │
        ▼
┌──────────────────┐
│   @njit 装饰器    │  ← 标记需要编译的函数
└──────────────────┘
        │
        ▼  (首次调用时触发)
┌──────────────────────────┐
│  Numba 前端：             │
│  1. 获取 Python 字节码    │  ← 用 byteplay/dis 反汇编
│  2. 类型推断引擎          │  ← 根据输入参数推断变量类型
│  3. 生成 Numba IR         │  ← 类型化的中间表示
└──────────────────────────┘
        │
        ▼
┌──────────────────────────┐
│  Numba 后端：             │
│  1. IR 优化 (passes)      │
│  2. 生成 LLVM IR          │  ← 工业级编译器中间语言
└──────────────────────────┘
        │
        ▼
┌──────────────────────────┐
│  LLVM 编译器：            │
│  1. 更多优化 (O2/O3)      │
│  2. 生成目标架构机器码     │  ← x86-64 / ARM64 / ...
│  3. JIT 链接             │
└──────────────────────────┘
        │
        ▼
   原生机器码函数（内存中）
```

### 1.2.2 关键环节详解

**（1）类型推断 —— Numba 的核心魔法**

```python
@njit
def foo(a, b):
    c = a + b        # c 的类型取决于 a 和 b 的实际类型
    return c * 2.0
```

当你第一次调用 `foo(1, 2)` 时：
- Numba 看到 `a: int64, b: int64`
- 推断 `c = a + b` → `c: int64`
- 编译出 `foo(int64, int64) → float64` 版本

当你后来调用 `foo(1.0, 2.0)` 时：
- 类型变了！Numba 会重新编译一个 `foo(float64, float64) → float64` 版本
- 每个不同的类型组合都会触发一次编译（并缓存）

**（2）nopython 模式 vs object 模式**

```
nopython 模式 (推荐)：
  全部代码编译为机器码，不经过 Python C API
  → 快，但有 Python 子集限制
  
object 模式 (兼容备选)：
  能编译的编译，不能的回退到 Python C API
  → 兼容性好，但通常比纯 Python 还慢
```

`@njit` 就是 `@jit(nopython=True)` —— 强制 nopython 模式，编译失败直接报错。

**（3）编译缓存**

每编译一个新签名组合，Numba 都会把结果写入磁盘缓存（`__pycache__` 目录下的 `.nbc` 文件），这样程序重启后就不用重新编译了。

## 1.3 安装与环境配置

### 1.3.1 基础安装

```bash
# 使用 uv 安装（推荐）
uv add numba

# 或者用 pip
uv pip install numba
```

依赖关系：
```
numba
├── llvmlite   ← 轻量级 LLVM Python 绑定
├── numpy      ← 必需（Numba 的数组支持基于 NumPy）
└── importlib-metadata (Python < 3.8)
```

### 1.3.2 验证安装

```bash
# 方法1：查看版本
python -c "import numba; print(numba.__version__)"

# 方法2：查看详细的系统/编译信息
numba -s
```

`numba -s` 会输出：
- Python 版本、平台
- NumPy 版本
- LLVM 版本
- CUDA 信息（如果安装了）
- 支持的 CPU 特性（SSE、AVX 等）

## 1.4 热身（Warm-up）是什么——理解 Numba 的双阶段执行

### 1.4.1 一个直观类比

想象你去一家**现点现做的寿司店**：

- **普通 Python** = 点一份、大厨做一份、端给你、再点下一份……每份都重新做一遍相同的事
- **Numba** = 你第一次点「三文鱼寿司」时，大厨说"等一下":
  1. 🕐 他先花 **3 分钟**把三文鱼的切法、饭团的标准流程写下来，试做一遍确认好吃
  2. 然后把这个"标准化菜谱"（机器码）贴在墙上
  3. 做好第一份端给你
  4. 🚀 以后再点「三文鱼寿司」—— 瞄一眼菜谱，**10 秒**出锅
- 如果你第一次点「三文鱼」，大厨写了菜谱。第二次点「金枪鱼」，**又要花 3 分钟**写金枪鱼的菜谱（不同输入类型 = 不同菜谱）

**"热身"就是大厨写菜谱的那几分钟。做完之后，后面的菜就都是秒出了。**

### 1.4.2 实际运行时间线

```python
import time
import numpy as np
from numba import njit

@njit
def my_sum(arr):
    total = 0.0
    for x in arr:
        total += x
    return total

arr = np.random.randn(1_000_000)
```

```
时间线 (假设):

第 1 次调用 my_sum(arr):
  ╔═══编译阶段══════╗   ╔═执行═╗
  ║ 类型推断  0.02s ║   ║0.001s║
  ║ 生成 LLVM IR    ║   ╚══════╝
  ║ LLVM 优化 0.08s ║
  ║ 生成机器码      ║   合计 ≈ 0.1s
  ╚═════════════════╝
  ↑                                     ↑
  这是"热身"时间                         实际计算

第 2 次调用 my_sum(arr):
  ╔═执行═╗
  ║0.001s║    ← 直接用编译好的机器码，没有编译开销！
  ╚══════╝

第 3 次调用 my_sum(arr):
  ╔═执行═╗
  ║0.001s║    继续秒出……
  ╚══════╝
```

### 1.4.3 热身开销的实际大小

热身时间取决于函数**有多长、逻辑有多复杂**：

```
简短函数 (10 行循环)          ≈ 0.01 ∼ 0.05 秒   ← 基本无感
中等函数 (50 行，多个循环)     ≈ 0.05 ∼ 0.3 秒    ← 能感觉到
长函数 (200 行，分支嵌套)      ≈ 0.3 ∼ 2 秒        ← 明显停顿
极复杂函数 (含内联展开等)      ≈ 2 ∼ 10 秒          ← 很卡
```

### 1.4.4 如何应对热身开销

```python
# 策略 1: cache=True —— 写入磁盘缓存，下次运行程序时跳过编译
@njit(cache=True)
def heavy_func(arr):
    ...

# 策略 2: 预热身 —— 程序启动时用 dummy 数据跑一遍
@njit
def expensive_op(x, y):
    ...

expensive_op(0.0, 0.0)  # ← 提前触发编译，用户调用时就是秒出的

# 策略 3: 显式签名 —— 省去类型推断时间，直接进入 LLVM 编译
@njit(float64(float64, float64))
def typed_func(a, b):
    return a * b + 1.0
```

### 1.4.5 热身 vs 加速比的权衡

```
假设你的函数：
  - 热身编译时间：0.1 秒
  - 编译后单次执行：纯 Python 要 1 秒，Numba 要 0.01 秒 (100× 加速)

调用 1 次:    Numba 总时间 = 0.1 + 0.01 = 0.11s    vs Python 1s     ✅ 还是快
调用 10 次:   Numba 总时间 = 0.1 + 0.10 = 0.20s    vs Python 10s    ✅ 快很多
调用 100 次:  Numba 总时间 = 0.1 + 1.00 = 1.10s    vs Python 100s   ✅ 极快
调用 1 次:    但函数只跑 0.001 秒 → Numba 反而更慢  ❌ 不值得
```

> **规则**：函数被调用次数越多，热身成本摊得越薄。如果函数只跑一次且本身很快，Numba 可能得不偿失。反之——循环里被反复调用的函数、处理大数据量的函数，热身成本可以忽略不计。

### 1.4.6 各方法性能对比

配套代码见 `code/ch01/first_numba.py`，核心对比：

| 方法 | 特点 | 典型加速比 |
|------|------|-----------|
| 纯 Python 循环 | 逐元素 Python 解释器处理 | 基线 (1×) |
| NumPy 向量化 | C 层循环，但中间数组多 | 10× – 100× |
| Numba `@njit` | 编译为机器码，无解释开销（首次有热身） | 100× – 500×（循环密集场景） |
| Numba + parallel | 多核并行 | 额外 2× – 8×（取决于核心数） |

## 1.5 本章关键概念

| 概念 | 一句话理解 |
|------|-----------|
| **JIT** (Just-In-Time) | 不是提前编译，而是在函数**第一次被调用时**编译 |
| **热身 (Warm-up)** | 首次调用的编译过程：输入参数 → 类型推断 → LLVM 优化 → 机器码。一次编译，永久复用 |
| **nopython 模式** | 完全脱离 Python C API，编译为纯机器码 |
| **object 模式** | 回退到 Python 解释器，慢，不推荐 |
| **类型推断** | Numba 自动分析代码推断每个变量的类型，无需手写 |
| **编译缓存** | 同一类型组合只编译一次；`cache=True` 可跨重启保留 |
| **LLVM** | 业界标准的编译器基础设施，Numba 的"后端引擎" |
| **热身成本** | 短函数 ~0.01s，复杂函数可达数秒；多次调用时可摊薄 |
